# 数据链路层的基本概念

> **数据链路层**：OSI/RM 参考模型中的第二层，介于**物理层**和**网络层**之间，负责通过一条链路从一个结点向另一个物理链路直接相连的相邻结点传送数据报（帧）。

数据链路层使用的信道：
- 点对点信道；一对一通信，通常通过一条专用物理链路连接两个节点
- 广播信道：一对多通信，单个节点发送的数据可被连接在同信道上的所有节点接收

| 概念                  | 说明                                          |
| ------------------- | ------------------------------------------- |
| **链路（Link）**        | 从一个结点到相邻结点的一段**物理线路**（有线或无线），中间没有任何其他交换结点   |
| **数据链路（Data Link）** | 链路 + 实现通信协议的硬件和软件（如网卡、驱动程序），是**逻辑概念**       |
| **帧（Frame）**        | 数据链路层传输的协议数据单元（PDU），将网络层交下来的 IP 数据报**封装成帧** |

> [!warning] 链路 vs 数据链路
> - **链路**：纯物理概念，指物理线路本身
> - **数据链路**：逻辑概念，指在物理线路上实现控制数据传输的硬件和软件
> - 类比：链路 = 公路，数据链路 = 公路 + 交通规则

# 数据链路层为网络层提供的服务

> 数据链路层为网络层提供服务，其基本任务是将源机器中来自网络层的数据传输到目标机器的网络层。

### 三种服务类型

| 服务类型 | 是否需要确认 | 是否需要连接 | 特点 | 适用场景 |
|----------|:----------:|:----------:|------|----------|
| **无确认的无连接服务** | ❌ | ❌ | 源机器直接发送帧，不建立连接也不确认，**尽最大努力交付** | 误码率很低的信道（如光纤），或实时通信（如语音、视频） |
| **有确认的无连接服务** | ✅ | ❌ | 不建立连接，但每帧单独确认；超时未确认则重传 | 无线信道等误码率较高的信道 |
| **有确认的面向连接服务** | ✅ | ✅ | 分三阶段：建立连接 → 传输帧 → 释放连接；每帧编号，保证**有序、可靠交付** | 可靠性要求高的场景 |

> [!tip] 记忆要点
> - **无连接**：发送前无需建立连接，每个帧独立处理
> - **有确认**：接收方需要发回确认帧，发送方超时重传
> - **面向连接**：类似电话通信，先"拨号"建立连接，再传输，最后"挂断"

> [!warning] 注意
> - 数据链路层是否提供可靠传输取决于**具体的协议**和**信道条件**
> - 对于有线局域网（误码率低），数据链路层通常采用无确认的无连接服务
> - 对于无线网络（误码率高），数据链路层通常使用有确认的服务

# 链路管理

> **链路管理**：对数据链路层连接的建立、维持和释放的管理。

| 阶段 | 说明 |
|------|------|
| **建立连接** | 收发双方交换必要信息，确认已处于就绪状态 |
| **维持连接** | 在数据传输过程中维护链路状态，防止链路意外中断 |
| **释放连接** | 数据传输完毕后，双方释放链路资源 |

> [!info] 适用场景
> 链路管理主要用于**面向连接的服务**。对于无连接服务，不需要链路管理。

# 链路聚合

> **链路聚合（Link Aggregation）**：将多条物理链路捆绑成一条逻辑链路，实现**带宽叠加、负载分担和链路冗余**。

| 要点       | 说明                                       |
| -------- | ---------------------------------------- |
| **基本思想** | N 条物理链路 → 1 条逻辑链路，对上层透明                  |
| **增加带宽** | 逻辑带宽 = 各物理链路之和（如 4×1Gbps = 4Gbps）        |
| **负载分担** | 通过哈希算法将不同数据流分发到不同物理链路，同一流走同一链路避免乱序       |
| **链路冗余** | 某条物理链路故障时，流量自动切换至其余正常链路                  |
| **LACP** | IEEE 802.1AX 标准协议，自动协商聚合；至少一端为 Active 模式 |
| **前提**   | 成员端口速率、双工模式、VLAN 配置必须一致                  |

# 封装成帧和透明传输

## 封装成帧

> **封装成帧**：数据链路层将网络层交下来的 IP 数据报作为数据部分，加上**帧首部**和**帧尾部**，封装成一个完整的**帧（Frame）**。

### 帧定界（Frame Delimiting）

> 物理层传输的是**比特流**，数据链路层需要在帧的首部和尾部添加**定界符**（特殊的控制字符或比特序列），用以标识每一帧的起始和结束。这就是**帧定界**。

在 HDLC 协议中，使用**标识字段 F （01111110）** 标识帧的开始和结束；接收方将第一个 F 视为帧的开始，下一个 F 标志该帧的结束

### 帧同步

> **帧同步**：接收方能够从收到的比特流中，借助帧定界符正确识别每个帧的边界（每个帧的开始和结束），从而将比特流还原为一个个独立的帧。

> [!tip] 封装成帧、帧定界与帧同步的关系
> - **封装成帧**：整体过程——将网络层数据加上首部和尾部构成帧
> - **帧定界**是**手段**（在帧首部和尾部添加定界标记）
> - **帧同步**是**目的**（接收方借助定界符正确识别帧边界）

## 透明传输

> **透明传输**：无论什么样的比特组合（包括与控制字符相同的比特组合），都应当能够在数据链路上传送而不被误解释为控制信息；使网络层感知不到底层的帧封装过程——即**数据链路对上层是透明的**

> [!info] 透明传输的必要性
> 如果帧的定界符恰好出现在数据部分，接收方会错误地认为帧在此结束。因此需要某种机制来确保**数据部分中的特殊字符不会被误解释为控制字符**。常见的解决方法包括**字节填充**和**比特填充**。


# 流量控制

> **流量控制**：限制发送方的发送速率，使其不超过接收方的接收能力，防止因发送过快导致接收方**缓存溢出（帧丢失）**。

| 特点 | 说明 |
|------|------|
| **目的** | 使发送方的发送速率与接收方的接收能力匹配 |
| **常见机制** | 停止-等待协议、滑动窗口协议（后退 N 帧、选择重传） |
| **与网络层的区别** | 数据链路层控制的是**相邻节点间链路上的流量**；传输层控制的是**端到端的流量** |

> [!tip] 流量控制的本质
> 流量控制本质上是**由接收方控制发送方**，接收方通过反馈信息告知发送方自己的接收能力。

> [!warning] 流量控制 ≠ 拥塞控制
> - **流量控制**：点到点（或端到端），控制发送方与接收方之间的速率匹配
> - **拥塞控制**：全局性概念，防止过多的数据注入网络，保证网络能够承受现有负荷

>[!note] 实现流量控制的层
>在 OSI 模型中，流量控制由数据链路层实现，用于调节相邻节点间的流量
>在 TCP/IP 模型中，该功能主要由传输层承担，负责端到端的流量控制

# 差错控制

> **差错控制**：在数据通信过程中能发现（检测）差错并采取措施纠正，将差错控制在尽可能小的允许范围内。

### 位错与帧错

数据链路层出现的差错可分为**位错**和**帧错**两类：

| 分类     | 说明                               | 典型表现                           |
| ------ | -------------------------------- | ------------------------------ |
| **位错** | 帧中个别比特在传输过程中发生翻转（0→1 或 1→0）      | 某比特因噪声干扰由 0 变成 1               |
| **帧错** | 整帧层面的错误，包括帧丢失、帧重复、帧失序；属于更高层的传输错误 | 帧未到达接收方；同一帧被收到多次；帧到达顺序与发送顺序不一致 |

> [!info] 位错 vs 帧错
> - **位错**：局部性错误——帧还在，只是内部个别比特出错 → 通过**检错码 / 纠错码**（如 CRC、海明码）处理
> - **帧错**：整体性错误——整帧丢失、重复或乱序 → 通过**编号 + 确认 + 重传**（ARQ）处理
> - 位错是帧错的可能原因之一：严重的位错导致帧校验失败，接收方丢弃该帧，形成**帧丢失**

### 差错的物理成因

| 类型 | 说明 |
|------|------|
| **随机差错（热噪声引起）** | 信道本身固有的、持续存在的随机热噪声（白噪声），比特随机出错 |
| **突发差错（冲击噪声引起）** | 外界特定短暂原因造成的冲击噪声（如雷电），通常**成片出错**，是传输中产生差错的主要原因 |

> [!info] 误码率（BER, Bit Error Rate）
> 传输错误的比特数与传输总比特数的比率。如 $10^{-6}$ 表示平均每传输 $10^6$ 比特有 1 比特出错。

### 差错控制方式

| 方式 | 原理 | 特点 |
|------|------|------|
| **检错重传（ARQ）** | 发送方发送带有检错码的数据，接收方检测出错则要求重传 | 广泛使用，需要**反馈信道** |
| **前向纠错（FEC）** | 发送方发送带有纠错码的数据，接收方不仅能检错还能纠错 | 不需要反馈信道，但编码效率较低 |
| **混合纠错（HEC）** | 结合 ARQ 和 FEC，先尝试纠错，无法纠正时再要求重传 | 折中方案 |

### 常见的差错控制技术

| 技术 | 分类 | 说明 |
|------|------|------|
| **奇偶校验码** | 检错码 | 增加一个校验位使 1 的个数为奇数（奇校验）或偶数（偶校验） |
| **循环冗余码（CRC）** | 检错码 | 数据链路层最广泛使用的检错技术，检错能力强 |
| **海明码** | 纠错码 | 可以纠正单比特错误，通过多重奇偶校验实现 |

# 可靠传输

> **可靠传输**：确保数据链路层发送的帧能够**无差错、不丢失、不重复且按序**地交付给接收方的网络层。

## 可靠传输的实现机制

| 机制 | 说明 |
|------|------|
| **确认（ACK）** | 接收方收到正确的帧后，向发送方返回确认帧 |
| **超时重传** | 发送方在发送帧时启动计时器，若超时未收到确认则重传该帧 |
| **帧编号** | 为每个帧分配序号，接收方据此判断是否有重复或丢失 |
| **自动重传请求（ARQ）** | 综合以上机制的可靠传输协议 |

## 可靠传输在数据链路层的定位

> [!warning] 数据链路层不一定提供可靠传输
> - **有线局域网**（误码率低，如以太网）：通常**不提供**可靠传输，将可靠性问题交给上层（传输层）处理
> - **无线网络**（误码率高，如 Wi-Fi、4G/5G 链路层）：通常**提供**可靠传输（如通过确认 + 重传）
> - 是否实现可靠传输取决于**信道质量**和**协议设计**

> [!tip] 可靠传输 ≠ 流量控制
> - **可靠传输**：解决帧**出错、丢失、重复、乱序**问题 → 保证**正确性**
> - **流量控制**：解决发送方**过快**导致接收方溢出的问题 → 保证**速率匹配**
> - 二者通常配合使用（如滑动窗口协议同时提供流量控制和可靠传输）

> 📌 详细机制（停止-等待、后退 N 帧、选择重传）见 [[4.流量控制与可靠传输机制|流量控制与可靠传输机制]]

# ❗ 易错点总结

1. **链路 ≠ 数据链路**：链路是纯物理的，数据链路 = 物理线路 + 通信协议
2. **帧是数据链路层的 PDU**，封装的对象是网络层的 IP 数据报
3. **透明传输**中"透明"的含义：无论数据部分是什么比特组合，都能正确传输而不被误解释为控制信息
4. **帧定界 = 识别帧边界**，**帧同步 = 接收方正确识别帧**，前者是方法，后者是目的
5. **流量控制**在数据链路层和传输层都存在，区别在于：数据链路层控制**相邻结点间**，传输层控制**端到端**
6. **流量控制**解决发送速率过快问题，**差错控制**解决传输错误问题，二者是不同概念
7. **无确认的无连接服务**适用于误码率低的信道（如有线局域网），不是为了"简化"而省略确认
8. **ARQ = 检错 + 重传**，需反馈信道；**FEC = 检错 + 纠错**，无需反馈信道
9. **链路聚合**：多条物理链路 → 一条逻辑链路，实现带宽叠加 + 负载分担 + 链路冗余，逐流分担保证不乱序
