# 流量控制与滑动窗口机制

> **流量控制（Flow Control）**：限制发送方的发送速率，使其不超过接收方的接收能力，防止因发送过快导致接收方**缓存溢出（帧丢失）**。流量控制本质是**由接收方控制发送方**，通过反馈机制告知发送方自己的接收能力。

> [!important] 数据链路层和传输层的区别
> - 数据链路是控制相邻节点间的流量，传输层是控制端到端的流量
> - 数据链路的手段是当接收方无法接收时，**不返回确认帧**；而传输层的手段是接收方在确认报文中携带**窗口字段值**，动态调整发送方的发送窗口

## 停止-等待流量控制基本原理

> **原理**：发送方每发送一帧就**停下来等待确认**，收到接收方的确认帧（ACK）后才发送下一帧。若超时未收到确认则**重传**。

> [!warning] 关键问题：重复帧
> ACK 丢失导致发送方重传，接收方会收到**重复帧**。需要**帧序号**来区分新帧和重复帧。停止-等待协议**只需 1 比特序号**（0 和 1 交替）。

## 滑动窗口流量控制基本原理

>[!important] 核心思想
> 发送方维持一个**发送窗口**（Send Window），允许在未收到确认的情况下**连续发送多个帧**，同时在收到确认后**滑动窗口**以发送新帧。滑动窗口机制是停止-等待协议的改进，通过**流水线（Pipelining）** 提高信道利用率。

流量控制的本质是**接收方控制发送方的发送速率**，滑动窗口通过以下机制实现：
1. **窗口大小限制发送速率**：$W_T$ 决定了发送方"提前发送"的最大帧数，接收方可通过**拒绝返回 ACK** 或**延迟 ACK** 来间接限制发送速率
2. **窗口满则停**：当发送窗口内的所有帧都已发出但未确认时，发送方停止发送，等待 ACK → 自然形成**背压（Backpressure）**
3. **接收窗口过滤**：接收方仅接收窗口内的帧，窗口外的帧直接丢弃，防止缓冲区溢出


### 滑动窗口的重要特性

1. **发送窗口的滑动依赖于确认信息**
   - 发送方每收到一个 ACK，便将窗口下限前移，上限也随之同步前移
   - 若 ACK 丢失（但后续 ACK 到达，如 GBN 的累积确认），窗口仍可**跳跃式**前移
   - 若无任何 ACK 返回 → 窗口停滞 → 超时重传

2. **发送窗口和接收窗口的大小**（设 n 比特对帧编号，序号空间 $0 \sim 2^n-1$）：
	- **停止-等待协议**：$W_T = 1,\ W_R = 1$（只需 1 bit 编号）
	- **后退 N 帧（GBN）**：$1 \leq W_T \leq 2^n-1,\ W_R = 1$
	  GBN 的 $W_T$ 不能取 $2^n$，否则重传帧序号与新帧序号重叠，接收方无法区分
	- **选择重传（SR）**：$W_T \leq 2^{n-1},\ W_R \leq 2^{n-1}$，通常取 $W_T = W_R = 2^{n-1}$
	  约束条件 $W_T + W_R \leq 2^n$，保证新旧窗口序号不重叠

> [!important] n 比特编号下窗口大小的约束详解
> **基础事实**：n 比特能表示 $2^n$ 个不同序号（$0 \sim 2^n-1$），序号**模 $2^n$ 循环使用**——帧 $2^n-1$ 之后下一帧序号又回到 0。
> 
> **核心矛盾**：若窗口太大，发送方重传的旧帧序号可能与接收方期望的新帧序号重叠 → 接收方**无法区分新帧和重传帧** → 协议失效。
> 
> **约束推导**（以 n=3，序号空间 0~7 为例）：
> 
> **① GBN：$W_T \leq 2^n - 1$**
> 
> 若 $W_T = 8 = 2^n$：发送方连续发帧 0~7，所有 ACK 丢失 → 超时重传 0~7。此时接收方已正确收到第一批 0~7，期望下一批也是 0~7 → 收到重传帧 0 时**误以为是新帧** → 出错！
> 
> 若 $W_T = 7$：发送方发帧 0~6，ACK 丢失 → 超时重传 0~6。接收方期望序号 7 → 收到重传帧 0 在窗口外 → 丢弃 ✓
> 
> **② SR：$W_T \leq 2^{n-1}$，$W_R \leq 2^{n-1}$，且 $W_T + W_R \leq 2^n$**
> 
> 若 $W_T = 5, W_R = 5$（均 $\geq 2^{n-1} = 4$）：发送窗口 [0,1,2,3,4]，接收窗口 [0,1,2,3,4]。全部发送并收到后 ACK 丢失 → 超时重传 0~4。此时接收窗口已移至 [5,6,7,0,1] → 收到重传帧 0,1 **落入新窗口内** → 误判为新帧！
> 
> 若 $W_T = 4, W_R = 4$（$W_T + W_R = 8 = 2^n$，恰好安全）：发送 0~3，接收窗口移至 [4,5,6,7]。重传 0~3 全部在窗口外 → 丢弃 ✓。但若 $W_T + W_R > 2^n$（如 5+4=9），则存在重叠风险。
> 
> | 约束条件 | GBN | SR |
> |:-------:|:---:|:--:|
> | 发送窗口 $W_T$ | $\leq 2^n-1$ | $\leq 2^{n-1}$ |
> | 接收窗口 $W_R$ | $= 1$ | $\leq 2^{n-1}$ |
> | 通用条件 | — | $W_T + W_R \leq 2^n$ |
> | 典型取值 | $W_T = 2^n-1$ | $W_T = W_R = 2^{n-1}$ |
> 
> **记忆技巧**：GBN 因为接收窗口 = 1，"吃掉"一个序号作缓冲，所以发送窗口可以到 $2^n-1$；SR 是收发窗口对半分，各取 $2^{n-1}$。

3. **当接收窗口大小为 1 时**（$W_R = 1$）：
   - 接收方**只能按序接收** → 失序帧直接丢弃，不缓存
   - 这就是 GBN 的行为特征：简单但高误码率下效率低
   - 若 $W_R > 1$（如 SR），接收方可以**缓存失序帧**，待前面帧到达后连续上交

4. **在数据链路层的滑动窗口协议中，窗口大小在传输过程中是固定的**（与传输层 TCP 的动态调整窗口不同）

# 可靠传输机制详解

> 可靠传输的核心机制包括：**确认（ACK）**、**超时重传**、**帧序号**、**捎带确认**。

结合确认和超时重传这两种机制的可靠传输协议称为**自动重传请求（Automatic Repeat reQuest，ARQ）**；重传由发送方自主触发，无需接收方显式请求重传；在 ARQ 协议中，**数据帧和确认帧都必须编号**，以便发送方和接收方能够准确识别确认帧对应哪个数据帧；ARQ 协议主要包括三个类型：停止-等待（Stop-and-Wait）协议，退后 N 帧（Go-Back-N）协议和选择重传（Selective Repeat）协议

## 确认机制

| 确认类型 | 说明 | 使用协议 |
|----------|------|:-------:|
| **ACK（肯定确认）** | 告知发送方帧已正确收到 | 所有 ARQ |
| **NAK（否定确认）** | 告知发送方帧出错，请重传 | SR 可选 |
| **累积确认** | ACK N 表示 N-1 及之前全部收到 | GBN |
| **单独确认** | 每帧单独发送 ACK | SR、停止-等待 |

## 超时重传

> **超时计时器**是可靠传输的核心。发送方每发送一帧（或一批帧），启动定时器；若超时未收到确认则重传。

| 协议 | 计时器策略 |
|------|----------|
| **停止-等待** | 1 个计时器，对应当前帧 |
| **GBN** | 1 个计时器，对应最旧的未确认帧 |
| **SR** | 每个未确认帧各有独立计时器 |

## 帧序号

> 帧序号用于**区分新帧和重传帧**，防止重复帧被错误上交。

| 协议 | 所需序号位数 | 说明 |
|------|:---------:|------|
| **停止-等待** | 1 bit（0/1 交替） | 只需区分前后两帧 |
| **GBN** | n bit | 序号范围 $0 \sim 2^{n}-1$，窗口 $\leq 2^n-1$ |
| **SR** | n bit | 序号范围 $0 \sim 2^{n}-1$，窗口 $\leq 2^{n-1}$ |

## 捎带确认（Piggybacking）

> **原理**：将 ACK 信息**附加在反向数据帧**中，而不单独发送 ACK 帧，节省带宽。

> [!warning] 捎带确认的问题
> 如果反向没有数据帧要发送，ACK 就会被"捎带延迟"。
> 实际协议通常会设置一个**捎带定时器**：超时后若还没有反向数据帧，就单独发送 ACK。

## 单帧滑动窗口与停止-等待协议（S-W）

> **本质**：停止-等待协议是滑动窗口的**最简特例**——发送窗口 $W_T = 1$，接收窗口 $W_R = 1$。窗口内始终只有 1 帧，发完即停，等 ACK 后窗口滑动至下一帧。

> [!tip] 为什么叫"单帧滑动窗口"
> 窗口大小 = 1 意味着流水线长度为 1，链路上最多同时只有 **1 个未确认帧**。窗口虽然也在"滑动"，但没有流水线效果——这正是停止-等待与 GBN/SR 的本质区别。

> [!warning] 关键机制：帧序号区分新旧帧
> ACK 丢失导致发送方重传，接收方可能收到**两个帧 0**。借助 **1 bit 序号**（0/1 交替），接收方能识别哪些是重复帧并丢弃，同时**重复发送 ACK** 确保发送方能收到确认。

实际传输中出现的两类差错：
- **数据帧出错或丢失**：接收方未收到（或收到有差错的帧）→ 不发送 ACK → 发送方**超时重传**该帧。接收方只需等待，不感知丢失
- **确认帧出错或丢失**：发送方收不到 ACK → **超时重传**同一帧 → 接收方收到**重复帧**。借助帧序号（0/1 交替）**识别并丢弃重复帧**，但仍**重发 ACK**

发送方和接收方均需设置*帧缓冲区*：
- **发送方**：缓存已发送但未确认的帧，以便超时后重传，在收到确认后清除该副本
- **接收方**：缓存已正确接收的**帧序号**，以判断新到的帧是否重复

### 停止-等待协议要点总结

| 维度        | 说明                  |
| --------- | ------------------- |
| **窗口大小**  | $W_T = 1,\ W_R = 1$ |
| **序号位数**  | 1 bit（0 和 1 交替）     |
| **计时器**   | 1 个，对应当前帧           |
| **确认方式**  | 逐帧确认                |
| **重传策略**  | 超时重传当前帧             |
| **信道利用率** | $U = 1/(1+2a)$      |
| **优点**    | 实现极简，无需大量缓冲区        |
| **缺点**    | 信道利用率低，尤其长距离链路      |
| **适用场景**  | 短距离、低速率、误码率低的简单链路   |

## 多帧滑动窗口与后退 N 帧协议（GBN, Go-Back-N）⭐

> **原理**：发送方维持一个**发送窗口**，可以连续发送多个帧而不必等待确认。接收方采用**累积确认**，若某个帧出错/丢失，则**丢弃该帧及之后所有帧**，发送方需**回退到出错帧重传**。

## 发送方行为

GBN 发送方需要处理三件事：

1. **上层调用**：检查发送窗口是否已满。窗口未满时封装成帧并发送，若计时器未启动则启动；窗口已满时拒绝上层数据或将数据缓存等待。
2. **收到 ACK**：采用累积确认。收到 ACK N 表示序号 N 之前的所有帧均已确认收到，发送窗口随之滑动到 N。
3. **超时事件**：GBN 只有一个计时器，对应最旧的已发送未确认帧。超时后重传所有已发送但未确认的帧，然后重新从最旧的未确认帧开始计时。

## 接收方行为

GBN 接收方只维持一个接收窗口（$W_R = 1$），即按序接收：

1. **收到期望序号的帧且无差错**：上交网络层，发送 ACK 进行累积确认，期望序号加 1。
2. **收到非期望序号的帧（失序帧）**：直接丢弃，重新发送最近一次的 ACK。
3. **收到有差错的帧**：直接丢弃，重新发送最近一次的 ACK。

> [!warning] GBN 接收方的关键特征
> - 接收窗口 $W_R = 1$（只接收期望的下一帧）
> - **不缓存失序帧**，直接丢弃
> - 累积确认：ACK N 意味着 "N 之前所有帧都正确收到了"

## 窗口大小

> [!important] GBN 的窗口大小约束（n 比特序号）
> $$1 \leq W_T \leq 2^n - 1$$
> $$W_R = 1$$

**为什么发送窗口不能等于 $2^n$？**

若 $W_T = 2^n$（如 n=3，共有 0~7 共 8 个序号）：

场景：发送方连续发送帧 0~7，所有 ACK 丢失；接收方收到帧 0~7 并上交，但 ACK 全部丢失；发送方超时后重传帧 0~7。此时接收方期望下一批帧的序号也是 0~7 → **无法区分"新帧 0"和"重传的旧帧 0"** → 造成混乱。

若 $W_T = 2^n - 1$（如 n=3，最多发 7 帧，范围 0~6）：

发送方发帧 0~6，ACK 全部丢失 → 超时重传 0~6。此时接收方期望序号 7，收到重传的帧 0 落在窗口外 → 知道是旧帧 → 丢弃。✓ 可以区分！


> [!tip] 408 常见考点
> 给定 n 比特序号，问 GBN 的最大窗口大小 → **答案是 $2^n-1$，不是 $2^n$！**

## 累积确认的含义

> **累积确认（Cumulative ACK）**：ACK N 表示序号 N-1 及之前的所有帧都已正确收到。

发送方收到 ACK 4 表示帧 0, 1, 2, 3 全部正确收到，帧 4 的状态未知（可能未收到或正在处理）。即使 ACK 2 中途丢失，只要后续收到 ACK 4，帧 0~3 就全部确认。

| 特点 | 说明 |
|------|------|
| **优点** | 个别 ACK 丢失不影响确认（后面的 ACK 覆盖前面的） |
| **缺点** | 一个帧出错 → 所有之后的帧都要重传（即使它们正确到达） |

## 计时器管理

> GBN 发送方**只维持一个计时器**，对应**最旧的已发送但未确认的帧**。

| 特点       | 说明                                                               |
| -------- | ---------------------------------------------------------------- |
| **优点**   | ① 提高了信道利用率（流水线发送）<br>② 只需一个计时器，实现相对简单<br>③ 接收方简单（$W_R=1$，不缓存失序帧） |
| **缺点**   | ① 一个帧出错 → 大量重传（回退 N 帧）<br>② 误码率高时性能极差<br>③ 带宽浪费（正确到达的帧也可能被重传）    |
| **适用场景** | 误码率较低的信道                                                         |

## 多帧滑动窗口与选择重传协议（SR, Selective Repeat）⭐

> **原理**：只重传**出错或丢失**的帧，正确到达的帧**被缓存**而不丢弃。接收方对每个正确帧单独发送确认。

## 窗口机制

| 参数 | GBN | SR |
|------|:---:|:--:|
| **发送窗口 $W_T$** | $1 \leq W_T \leq 2^n-1$ | $1 \leq W_T \leq 2^{n-1}$ |
| **接收窗口 $W_R$** | $= 1$ | $1 \leq W_R \leq 2^{n-1}$ |
| **通常取值** | — | $W_T = W_R = 2^{n-1}$ |

> [!important] SR 窗口大小约束（n 比特序号）
> $$W_T \leq 2^{n-1}, \quad W_R \leq 2^{n-1}, \quad W_R \leq W_T$$
> 通常取 $W_T = W_R = 2^{n-1}$；因为需要保证新旧窗口的序号不重叠，避免接收方无法区分新帧和重传帧；若接收窗口大于发送窗口则永远不会填满，造成资源浪费

**窗口不重叠条件：$W_T + W_R \leq 2^n$**

## 发送方行为

SR 发送方需要处理三件事：

1. **上层调用**：检查发送窗口是否已满。窗口未满时封装成帧并发送，启动该帧专用的独立计时器；窗口已满时拒绝或缓存。
2. **收到 ACK**：对该帧标记为"已确认"。若该帧恰好是窗口下限，则窗口前移（可能连续前移多个已确认的帧）。
3. **超时事件**：SR 每个未确认帧有**独立的计时器**。某个帧超时后只重传该帧，然后重新启动该帧的计时器。

## 接收方行为

接收方需要设置足大的帧缓冲区（其数量等于接收窗口大小），用于暂存正确的失序帧
SR 接收方需要处理四种情况：

1. **收到窗口范围内的帧（无差错）**：缓存该帧，发送该帧的 ACK。若该帧是接收窗口下限，则将连续的已缓存帧上交网络层，窗口前移。
2. **收到窗口范围前的帧（重复帧）**：丢弃该帧，但仍发送 ACK（以防发送方的 ACK 丢失）。
3. **收到窗口范围外的帧**：直接丢弃。
4. **收到有差错的帧**：丢弃该帧，可选发送 NAK（否定确认）请求重传。

> [!tip] SR 接收方的关键特征
> - 接收窗口 $W_R > 1$，**缓存失序帧**
> - 对每个正确帧**单独发送 ACK**（非累积确认）
> - 可以连续上交已缓存的连续帧


| 特点 | 说明 |
|------|------|
| **优点** | ① **只重传出错帧**，避免无效重传<br>② 信道利用率更高（尤其高误码率时）<br>③ 接收方可连续上交已缓存帧 |
| **缺点** | ① 实现复杂（每个帧独立计时器、接收方需缓存失序帧）<br>② 接收方需要足够的缓冲区<br>③ 需要更复杂的序号管理 |
| **适用场景** | 高误码率、对带宽利用率要求高的信道 |

# 三种 ARQ 协议对比总结

| 对比维度 | 停止-等待 | 后退 N 帧（GBN） | 选择重传（SR） |
|----------|:-------:|:-------------:|:-----------:|
| **发送窗口 $W_T$** | 1 | $1 \sim 2^n-1$ | $1 \sim 2^{n-1}$ |
| **接收窗口 $W_R$** | 1 | 1 | $1 \sim 2^{n-1}$ |
| **确认方式** | 逐帧确认 | **累积确认** | **逐个确认** |
| **计时器数量** | 1 | **1 个** | **每帧 1 个** |
| **重传范围** | 仅超时帧 | 出错帧及之后所有 | **仅出错帧** |
| **接收方缓存** | 不缓存 | 不缓存 | **缓存失序帧** |
| **信道利用率** | 低 | 较高 | **最高** |
| **实现复杂度** | 简单 | 中等 | 复杂 |
| **适用误码率** | 低 | 低~中 | 中~高 |


# 信道利用率与吞吐量分析

> [!info] 符号说明
> | 符号 | 含义 | 单位 |
> |------|------|:--:|
> | $T_D$ | 数据帧发送时延 = 数据帧长 / 信道带宽 | s |
> | $T_A$ | 确认帧（ACK）发送时延 = ACK 帧长 / 信道带宽 | s |
> | $T_P$ | 单向传播时延 = 信道长度 / 信号传播速率 | s |
> | $a$ | 传播时延与数据帧发送时延之比：$a = T_P / T_D$ | — |
> | **RTT** | 往返时延（忽略 $T_A$ 时 = $2T_P$，考虑 $T_A$ 时 = $T_D + 2T_P + T_A$） | s |
> | $U$ | 信道利用率 = **有效发送时间**（数据帧发送时间） / 总时间 | — |
> | $W_T$ | 发送窗口大小（滑动窗口协议中可连续发送的最大帧数） | 帧 |
> | $p$ | 帧出错率 / 丢失率 | — |

> [!tip] 关键参数 $a$ 的物理意义
> - $a \ll 1$：帧发送时延远大于传播时延 → 短链路 / 低速率 → 停等也还行
> - $a \gg 1$：传播时延远大于帧发送时延 → 长链路 / 高速率（如卫星） → 停等效率极低，必须用滑动窗口
> - $a$ 本质上是"链路中能容纳的帧数"：一条链路从发送端到接收端能同时"装下"多少帧（需用 $T_D$ 衡量）

---

## 停止-等待协议（S-W）

> 每发一帧等一个 ACK，周期 = $T_D + 2T_P + T_A$。
![[停止-等待协议中数据帧和确认帧的发送时间关系.png]]
**无差错信道利用率：**

$$U = \frac{T_D}{T_D + 2T_P + T_A} = \frac{T_D}{T_D + 2T_P + T_A}$$

实际中 $T_A \ll T_D$（ACK 帧极短），常简化为 $U = \dfrac{T_D}{T_D + 2T_P} = \dfrac{1}{1 + 2a}$。

- 当 $a$ 很大时 $U \approx 0$，信道几乎空转——这是停止-等待的根本缺陷

**有差错（丢包率 $p$）：**

$$U = \frac{T_D \cdot (1-p)}{T_D + 2T_P + T_A} \approx \frac{1-p}{1+2a}$$

---

## 连续 ARQ 协议

> 窗口内连续发送，接收方累积确认。一个周期内最多发 $W_T$ 帧。

![[连续 ARQ 协议的流水线传输可提高信道利用率.png]]

**无差错（$p = 0$）信道利用率：**

$$U = \begin{cases} \dfrac{W_T \cdot T_D}{T_D + 2T_P + T_A} = \dfrac{W_T \cdot T_D}{T_D + 2T_P + T_A}, & W_T \cdot T_D < T_D + 2T_P + T_A \\[10pt] 1, & W_T \cdot T_D \geq T_D + 2T_P + T_A \end{cases}$$

近似为 $U = \min\!\left(\dfrac{W_T}{1+2a},\ 1\right)$（忽略 $T_A$ 时）。

- **铺满条件**：$W_T \geq 1 + 2a$ 时利用率可达 100%，发送方可连续不断发送
- **窗口不够时**：利用率与 $W_T$ 成正比

**有差错（丢包率 $p$）：**

$$U \approx \frac{1-p}{1 + (2a + T_A/T_D) \cdot p} \approx \frac{1-p}{1 + 2a \cdot p}$$

- $p$ 较大时性能骤降：一个帧出错 → 回退重传最多 $W_T$ 帧 → 大量无效重传
- 分母中的 $1 + 2a \cdot p$ 反映"重传代价随 $a$ 和 $p$ 放大"

---

| 协议 | 无差错利用率 | 有差错利用率（$p$） | 特点 |
|:---:|:----------:|:----------------:|------|
| **S-W** | $\dfrac{1}{1+2a}$ | $\dfrac{1-p}{1+2a}$ | $a$ 大时趋近 0 |
| **GBN** | $\min\!\left(\dfrac{W_T}{1+2a}, 1\right)$ | $\dfrac{1-p}{1+2a \cdot p}$ | $p$ 大时性能骤降 |
| **SR** | $\min\!\left(\dfrac{W_T}{1+2a}, 1\right)$ | $1-p$ | 高误码率最优 |

> [!tip] 理解公式背后的直觉
> - 停止-等待：每次成功发送一帧的代价是完整的一个 RTT 周期，$a$ 越大代价越高
> - GBN：$W_T$ 帧铺满 RTT 前利用率与窗口成正比；但 $p$ 大时一个错全回退，严重浪费
> - SR：出错只影响单帧，利用率近似 $1-p$，几乎不受 $a$ 影响

# ❗ 易错点总结

**关于窗口大小（408 高频考点）：**

1. **停止-等待**的发送窗口和接收窗口都是 **1**，只需 **1 bit 序号**

2. **GBN 的发送窗口最大为 $2^n-1$**，接收窗口 = 1。考试中容易错写成 $2^n$

3. **SR 的发送窗口最大为 $2^{n-1}$**，接收窗口也为 $2^{n-1}$（满足 $W_T + W_R \leq 2^n$）

4. **GBN 和 SR 中序号个数都是 $2^n$**（n bit 序号），但窗口大小限制不同

**关于确认机制：**

5. **GBN 使用累积确认**：ACK N 表示 N-1 及之前的所有帧均已正确收到

6. **SR 使用单独确认**：每个帧独立发送 ACK，不累积

7. GBN 中即使收到失序帧**也不会发送 ACK**（严格来说会重发上一个 ACK）

8. SR 收到窗口范围内的失序帧**会缓存并单独 ACK**

**关于计时器：**

9. **GBN 只有一个计时器**，对应最旧的已发送未确认帧

10. **SR 每个未确认帧各有一个计时器**

11. GBN 超时时**重传所有已发送未确认的帧**，不只是超时的那个

12. SR 超时时**只重传超时的那个帧**

**关于捎带确认：**

13. 捎带确认是**将 ACK 放在反向数据帧中**，而非单独发送 ACK 帧，目的是节省带宽

14. 若长时间没有反向数据帧，需**单独发送 ACK**，否则发送方会超时重传

**关于流量控制与可靠传输的关系：**

15. **流量控制**解决发送速率匹配问题，**可靠传输**解决出错后重传问题——两者独立但常结合

16. 数据链路层的流量控制是**相邻结点**间的；传输层（TCP）的流量控制是**端到端**的

**关于信道利用率：**

17. 停止-等待的信道利用率 $U = T_F / (T_F + 2T_P)$，随 $T_P$ 增大而**降低**

18. GBN 和 SR 通过**流水线（窗口）** 提高利用率，理想情况下窗口足够大可达到 **100%**

19. 在高误码率环境下，**SR 的利用率远高于 GBN**（因为 SR 只重传出错帧）

**关于重传：**

20. GBN 的"回退 N"意味着**回退到出错的帧**重新开始发送其后所有帧

21. SR 接收方**需要足够大的缓冲区**来缓存失序帧，缓冲区大小至少等于接收窗口大小
