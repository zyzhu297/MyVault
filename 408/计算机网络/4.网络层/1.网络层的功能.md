# 网络层的基本概念

> **网络层**：OSI 参考模型中的第三层，介于**数据链路层**和**传输层**之间，负责将分组从源主机经过多个网络节点传送到目的主机，提供**主机到主机的通信服务**。

网络层是**通信子网的最高层**，其主要任务是：

| 核心任务 | 说明 |
|----------|------|
| **路由选择** | 为分组选择从源端到目的端的合适路径 |
| **分组转发** | 路由器将收到的分组从合适的端口转发出去 |
| **网际互连** | 将异构的网络（不同协议、不同拓扑）连接成一个统一的互联网 |
| **拥塞控制** | 防止过多的分组注入网络，避免网络性能急剧恶化 |

## OSI 与 TCP/IP 模型中网络层的对比

| 比较维度     | OSI                          | TCP/IP                    |
| -------- | ---------------------------- | ------------------------- |
| **名称**   | 网络层                          | **网际层**（Internet Layer）   |
| **核心协议** | 通用标准（CLNP 等）                 | **IP 协议**                 |
| **服务类型** | **既支持面向连接（虚电路），也支持无连接（数据报）** | **仅支持无连接服务**（IP 数据报）      |
| **可靠性**  | 可以提供可靠/不可靠服务                 | **不可靠、尽力交付（Best-Effort）** |

在互联网采用 TCP/IP 体系结构中，网络层向上只提供**简单灵活的、无连接的、尽最大努力交付的数据报服务**；而可靠性由**更高层的传输层来负责**

# 异构网络互联

> **异构网络互联**：将两个或两个以上使用**不同协议、不同拓扑、不同技术**的计算机网络，通过**路由器**连接在一起，形成一个逻辑上统一的**互联网（Internet）**，使不同网络中的主机可以互相通信。

## 互联需要解决的主要问题

| 问题 | 说明 |
|------|------|
| **物理层差异** | 不同网络可能使用不同的物理介质和信号编码方式 |
| **数据链路层差异** | 不同网络可能使用不同的 MAC 协议（以太网、Wi-Fi、令牌环等） |
| **网络层差异** | 不同网络可能使用不同的网络层协议 |
| **寻址差异** | 不同网络使用不同的寻址方式 |
| **MTU 差异** | 不同网络的最大传输单元（MTU）可能不同，需要分片 |

## 网络互联的核心设备——路由器

> **路由器**是一台专用计算机，用于在互联网中进行路由选择；工作在**网络层**，是异构网络互联的核心设备。

| 要点 | 说明 |
|------|------|
| **工作层次** | 网络层（参考 OSI 第三层） |
| **互联方式** | 将多个网络通过路由器的不同端口连接起来 |
| **核心功能** | 根据 IP 地址进行**路由选择**和**分组转发** |
| **协议转换** | 路由器两端可以连接不同协议的网络，负责协议的适配和转换 |

> [!tip] 路由器 vs 交换机
> - **二层交换机**：工作在数据链路层，基于 MAC 地址转发，不能连接异构网络
> - **路由器**：工作在网络层，基于 IP 地址转发，可以连接不同类型的网络
> - 路由器是**构成互联网的基础**——"用路由器把不同的网络连接起来"


整个互联网采用统一的 **IP 协议**（Internet Protocol），屏蔽底层网络的差异，通过 IP 协议就可使这些性能各异的网络在网络层上看起来像一个统一的网络，所以可以把互连后的网络视为一个*虚拟互连网络*，简称 IP 网络

1. **统一寻址**：所有主机使用统一的 IP 地址标识
2. **统一分组格式**：所有网络传输统一格式的 IP 数据报
3. **逐跳转发**：路由器在每个网络中逐跳转发 IP 分组
4. **底层透明**：IP 协议屏蔽了下层网络（以太网、Wi-Fi、PPP 等）的差异

# 路由与转发

> **路由选择**与**分组转发**是网络层的两个核心功能，它们紧密相关但又彼此独立

## 路由选择（Routing）

> **路由选择**：根据**路由算法**，为分组确定从源端到目的端的**传输路径**（整条路径或下一跳节点）。

| 要点       | 说明                                       |
| -------- | ---------------------------------------- |
| **本质**   | 确定端到端的**路径**，是**控制平面（Control Plane）**的功能 |
| **核心**   | 动态构建和维护**路由表**                           |
| **工作方式** | 路由器之间通过**路由协议**交换路由信息，计算最佳路径             |
| **结果**   | 生成路由表，包含目的网络和对应的下一跳地址                    |

## 分组转发（Forwarding）

> **分组转发**：路由器从输入端口收到 IP 分组后，根据**转发表**（由路由表导出），将分组从合适的输出端口转发出去。

| 要点 | 说明 |
|------|------|
| **本质** | 将分组从输入接口转移到输出接口，是**数据平面（Data Plane）**的功能 |
| **核心** | 查找**转发表**（Forwarding Table） |
| **工作方式** | 提取分组目的 IP → 查找转发表 → 从对应端口发出 |
| **速度要求** | 必须**高速执行**（纳秒级），通常由硬件实现 |

## 路由表与转发表

| 比较维度     |  路由表（Routing Table）  | 转发表（Forwarding Table） |
| -------- | :------------------: | :-------------------: |
| **所属平面** |         控制平面         |         数据平面          |
| **生成方式** |       路由协议计算得出       |        由路由表导出         |
| **侧重点**  |         拓扑计算         |        使查找最优化         |
| **数据结构** | 较复杂，包含路由信息（下一跳、度量值等） |    优化为查找速度（最长前缀匹配）    |
| **更新频率** |     相对较低（几秒~几分钟）     |       随路由表更新而更新       |
| **实现方式** |         软件实现         |     通常由硬件（TCAM）实现     |

> 路由器通过路由协议（如 RIP、OSPF、BGP）交换信息 → 计算最佳路径 → 生成路由表 → 导出转发表 → 分组到达时，高速查转发表转发。

> [!warning] 路由选择 vs 分组转发
> - **路由选择**：选择"走哪条路"，关注的是**全局路径**，运行频率较低
> - **分组转发**：执行"把分组送出"，关注的是**单跳转发**，每分组到达即执行
> - 二者是**不同的功能**，路由选择为分组转发生成"地图"，转发是"按地图走"

> 📌 路由算法的详细内容见 [[4.路由算法与路由协议|路由算法与路由协议]]

# 网络层提供的两种服务

> 网络层可以向传输层提供两种不同类型的服务：**面向连接的虚电路服务**和**无连接的数据报服务**。

## 虚电路服务（Virtual Circuit）

> **虚电路**：通信前先建立一条**逻辑上的连接**（虚电路），该连接一旦建立，其对应的**端到端传输路径即被固定**，后续分组沿同一路径传输，通信结束后释放连接。类似于**电话系统**。

每次建立虚电路时，系统为其分配一个未使用过的虚电路号（VCID），以区别于同一节点上的其他虚电路，此后双方就沿着已建立的虚电路传送分组；**仅在连接建立阶段使用完整的源和目的地址**，后续分组的首部**仅需携带虚电路号**；每个中间节点都维护一个*虚电路表*，其中记录了每条虚电路的 VCID 及下一跳信息，这些内容在连接建立过程中确定

### 工作过程

1. **建立虚电路**：源端发送呼叫请求，沿途路由器记录虚电路映射，分配虚电路号；若目标同意建立连接，则返回”呼叫应答”分组予以确认
2. **数据传输**：每个分组携带虚电路号（而非目的地址），路由器根据虚电路号转发；其中**数据传输是有确认的**，接收方收到分组后需返回确认
3. **释放虚电路**：源端发送”释放请求”分组，**逐段拆除虚电路**，释放沿途资源

### 特点

1. **需要建立连接**：通信前必须先建立虚电路，引入额外时延
2. **分组按序到达**：所有分组沿同一路径传输，无失序问题
3. **分组首部开销小**：除建立连接阶段外，分组仅携带虚电路号，开销较小
4. **可靠性较高**：可在网络层提供可靠传输，支持端到端的流量控制
5. **对故障敏感**：某节点或链路故障时，经过该处的所有虚电路均中断
6. **路由器维护状态**：路由器需为每条虚电路维护虚电路表，占用内存资源
7. **适合长时间、大量数据传输**：不适合突发性通信场景

> [!info] 虚电路 ≠ 电路交换
> - **虚电路是"逻辑"的连接**：**不需要独占**物理通路，可被多条虚电路共享，分组仍经过存储转发
> - **电路交换是"物理"的连接**：独占物理通路，比特流直接传输
> - 虚电路只是确保所有分组**走同一条路径**，本身仍是分组交换

## 数据报服务（Datagram）

> **数据报**：每个分组独立路由，携带完整的源地址和目的地址，不需要预先建立连接。各分组可能走不同路径。类似于**邮政系统**。

### 工作过程

1. 发送方将数据分组，每个分组携带完整的目的 IP 地址，逐个发送至与其直连的节点，该节点对分组进行短暂缓存
2. 根据**目的地址**独立为每个分组选择转发路径；节点查询其转发表；网络状态动态变化，导致转发表随之更新，各分组独立进行路由选择，可能经由不同路径转发
3. 后续节点以相同的方式转发分组，直至其最终抵达目的
4. 接收方收到的分组可能**乱序、重复或丢失**

在传输过程中，**分组仅占用当前链路资源**；得益于存储转发机制和资源共享，多个主机可以同时通信

### 特点

1. **无需建立连接**：发送方随时可以发送分组，通信灵活
2. **尽最大努力交付**：不保证分组一定到达，不保证按序，不保证不重复，可靠性由上层负责
3. **分组携带完整地址**：每个分组都携带完整的源和目的地址，开销较大
4. **存在排队时延和丢包风险**：分组经过路由器时需排队等待，网络拥塞时可能被丢弃
5. **故障适应能力强**：某路由器或链路故障时，分组可自动绕行其他路径
6. **资源利用率高**：不需要预留资源，多个用户的分组可交错传输，充分共享网络资源

## 两种服务的对比

| 比较维度 | 虚电路服务 | 数据报服务 |
|----------|:--------:|:--------:|
| **是否需要建立连接** | ✅ 需要 | ❌ 不需要 |
| **寻址方式** | 分组携带**虚电路号**（较小） | 分组携带**完整目的地址** |
| **路由选择** | 仅在建立时选择一次路径 | 每个分组独立选路 |
| **分组顺序** | 全部按序到达 | 可能失序到达 |
| **路由器状态维护** | 需维护每条虚电路状态 | 无状态，不必维护 |
| **路由器故障影响** | 所有经过的虚电路中断 | 可自动绕行，仅丢失故障途中的分组 |
| **可靠性** | 可以在网络层提供可靠服务 | 由上层（传输层）负责可靠性 |
| **适用场景** | 持续大量数据传输 | 突发数据（现代互联网） |
| **典型代表** | X.25、帧中继、ATM | **IP 协议（Internet 核心）** |

> [!important] 为什么 TCP/IP 选择数据报？
> 1. **适应性强**：互联网是异构网络，路由器无状态有利于适应网络变化
> 2. **健壮性**：某节点故障不影响全局（与虚电路形成鲜明对比）
> 3. **灵活性**：上层协议（TCP/UDP）根据需要选择可靠性还是实时性
> 4. **端到端原则**：可靠性由端系统负责，网络核心保持"尽最大努力交付"的简洁设计

# SDN 的基本概念

> **SDN（Software Defined Networking，软件定义网络）**：一种新型网络架构，核心思想是将网络设备的**控制平面与数据平面分离**，控制平面集中化，数据平面分布式部署，通过**逻辑集中式控制器**对整个网络进行统一管理和编程；该控制器掌握全网拓扑及主机状态，能**为每个分组计算最优转发路径**，并通过 OpenFlow 等协议将相应的**流表（SDN 中的转发表）** 下发至各路由器

其中**数据平面**（也称转发平面）负责**分组转发**，而**控制平面**负责**路由选择**，生成并维护路由信息；

## 传统网络 vs SDN

| 比较维度 | 传统网络 | SDN |
|----------|----------|-----|
| **控制方式** | **分布式控制**，每台设备独立决策 | **集中式控制**，由控制器统一决策 |
| **控制平面位置** | 与数据平面**耦合在同一设备内** | 与数据平面**分离**，集中在控制器上 |
| **转发表生成** | 设备通过分布式路由协议自行计算 | 由**控制器统一下发**转发表 |
| **网络可编程性** | 难以编程，通过 CLI 逐台配置 | **高度可编程**，通过统一 API 自动配置 |
| **全局视图** | 每台设备只有**局部视图** | 控制器拥有**全局网络视图** |
| **创新灵活性** | 新功能需要新硬件或固件升级 | 通过软件即可快速部署新功能 |
| **典型架构** | 路由器/交换机运行 OSPF、BGP 等路由协议 | **OpenFlow 交换机 + SDN 控制器** |
如此一来，网络架构又变为集中控制模式，而互联网本质上是分布式的；显然不具有可行性；然而在特定条件下，例如像一些大型数据中心互联的广域网，采用 SDN 架构能够显著提升网络的运行效率

## SDN 架构的三层模型

SDN 将网络划分为三层：应用层、控制层、基础设施层，并通过标准化的**编程接口**，赋予网络高度的**可编程性**，使开发者能够以**软件方式**灵活定义、动态调整并高效控制网络行为

- **北向接口（Northbound API）**
	- 位于**应用层与控制层之间**
	- 向上层网络管理应用（如负载均衡、防火墙、流量工程等）提供**抽象的网络视图**，屏蔽底层细节
	- **使开发者能以软件方式灵活定义和调整网络行为**
	- 目前北向接口标准化程度较低，多采用 RESTful API

- **南向接口（Southbound API）**
	- 位于**控制层与基础设施层之间**
	- 控制器通过南向接口协议向交换机下发**流表（Flow Table）**，交换机按流表进行分组转发，实现控制器与转发设备之间的双向通信
	- 最核心的协议是 **OpenFlow**，控制器可统一**管理异构硬件**，并将上层应用逻辑**动态下发**至数据平面
	- 控制器负责流表的生成、维护和下发，交换机仅根据下发至自身的流表执行分组转发

- **东西向接口（East-West API）**
	- 位于**多个 SDN 控制器之间**
	- 用于控制器集群间的**内部状态同步、信息交换和协同决策**，尤其在**大规模 SDN 部署**（如大型数据中心）中不可或缺
	- 解决单一控制器性能有限和单点故障的问题，提升系统的可用性与扩展性

## SDN 与传统网络的关键区别

| 传统路由器/交换机 | SDN 交换机 |
|:---------------:|:----------:|
| 自行运行路由协议，计算转发表 | 不运行路由协议，转发表（流表）由控制器下发 |
| 控制+转发紧密耦合 | 控制分离，只负责转发（可编程交换机） |
| 分组转发基于目的 IP 地址 | 分组转发基于**多字段流表匹配**（可匹配 MAC、IP、端口等） |
| 难以全局优化 | 控制器拥有全局视图，可全局优化 |

## SDN 的优缺点

| 优点                                   | 缺点                                                            |
| ------------------------------------ | ------------------------------------------------------------- |
| **全局视图**：集中式控制器掌握全网拓扑，可做最优路由决策       | **单点故障风险**：集中式控制器若失效，全网受影响（可通过多控制器集群缓解）                       |
| **网络可编程**：通过软件快速部署新服务、新协议，无需更换硬件     | **可扩展性瓶颈**：单一控制器性能有限，大规模网络中控制平面可能成为瓶颈                         |
| **简化设备**：交换机只需支持流表转发，无需运行复杂路由协议，成本降低 | **标准化仍在演进**：北向接口尚未统一，不同厂商控制器存在兼容性问题                           |
| **灵活组网**：控制器可动态调整网络拓扑和流量策略，适应业务变化    | **适用场景受限**：互联网本质是分布式的，集中式 SDN 不适用于全网，更适合**数据中心、园区网、广域网**等可控环境 |

# 拥塞控制

> **拥塞（Congestion）**：网络中出现**资源需求超过可用资源**的现象——当过多的分组注入网络后，某些节点因资源不足而拥塞，分组被大量丢弃，导致网络性能急剧恶化。

判断网络是否进入拥塞状态，通常依据的是吞吐量和网络负载之间的关系：
- 若负载增大但吞吐量增长甚微，则表明网络可能已处于轻度拥塞状态
- 若负载增大但吞吐量反而下降，则表明网络很可能已进入拥塞状态
拥塞控制的核心在于**及时感知拥塞并采取措施**，避免因缓冲区溢出导致分组丢失；拥塞控制的目标是确保网络能有效承载当前流量，这是一个**全局**过程，涉及所有主机、路由器以及影响网络传输能力的各种因素；**仅靠增加资源无法根本解决拥塞**

## 区分拥塞控制和流量控制

| 比较维度     | 拥塞控制                              | 流量控制                            |
| -------- | --------------------------------- | ------------------------------- |
| **作用范围** | **全局性**，涉及整个通信子网中的所有主机、路由器以及链路    | **端到端 / 点到点**，仅涉及发送方与接收方之间      |
| **目的**   | 防止过多数据注入网络，保证网络能够承受现有负荷           | 使发送方速率与接收方接收能力匹配，防止接收方被"淹没"     |
| **关注点**  | 网络整体资源的合理分配与网络整体性能                | 个体通信双方之间的速率匹配                   |
| **实现层**  | 网络层（分组丢弃、流量整形等）和传输层（TCP 拥塞控制）     | 数据链路层（相邻节点间）和传输层（端到端）           |
| **典型机制** | 慢开始、拥塞避免、快重传、快恢复（TCP）；漏桶/令牌桶（网络层） | 停止-等待、滑动窗口（数据链路层）；TCP 滑动窗口（传输层） |

> [!important] 拥塞控制与流量控制的密切关系
> 二者目的不同但方法相似。某些拥塞控制算法会向发送端发送控制报文，告诉发送端网络已出现问题、必须减慢发送速率，这又类似于流量控制——**拥塞的缓解最终往往需要通过限制源端的发送速率来实现**。
## 拥塞控制的两大类方法

### 开环控制（Open-Loop Control）

> **开环控制**：在设计网络时就考虑拥塞因素，通过**预防**手段力求网络正常运行时不发生拥塞；通过**静态**策略防止拥塞发生；系统一旦投入运行，控制机制就不再依赖实时反馈

| 方法 | 说明 |
|------|------|
| **准入控制** | 判断是否有足够资源接纳新的流量，否则拒绝 |
| **资源预留** | 提前为某些流预留带宽和缓存资源 |
| **流量整形** | 规整发送方的流量使其平滑，避免突发性拥塞（如漏桶、令牌桶算法） |

### 闭环控制（Closed-Loop Control）

> **闭环控制**：通过**反馈机制动态**调整发送速率，当检测到拥塞时采取措施缓解。

| 步骤 | 说明 |
|------|------|
| **监测** | 持续监测网络状态（延迟、丢包率、队列长度等） |
| **反馈** | 将拥塞信息反馈给流量源（如显式拥塞通知 ECN、丢包检测） |
| **调整** | 源节点根据反馈调整发送速率（如降低发送窗口） |

> [!note] 漏桶算法（Leaky Bucket）
> 将流量整形为一个恒定的输出速率，消除突发。无论进水（到达分组）的速率如何波动，出水（发送分组）的速率始终恒定。
> - 优点：输出平滑，确保流量不会突发
> - 缺点：当输出速率过低时，严格按照恒定速率输出不够灵活

> [!note] 令牌桶算法（Token Bucket）
> 以恒定速率向桶中添加令牌，分组发送前必须获取令牌。允许一定程度的突发——只要桶里有足够的令牌，就可以一次发送多个分组。
> - 优点：允许一定程度的"有控制的突发"
> - 缺点：实现比漏桶复杂

# ❗ 易错点总结

1. **网络层是通信子网的最高层**，传输层及以上属于资源子网
2. **路由器**是网络层的核心设备，用于异构网络互联；二层交换机不能连接异构网络
3. **路由选择**（控制平面）= 选哪条路；**分组转发**（数据平面）= 把分组发出去
4. **路由表**由路由协议生成，**转发表**由路由表导出；转发表用于高速硬件查表
5. **最长前缀匹配**：CIDR 下转发时，选掩码更长、更精确的表项
6. **虚电路服务** = 面向连接（先建连接，再传数据）；**数据报服务** = 无连接（每个分组独立路由）
7. **OSI 网络层支持两种服务**；**TCP/IP 网际层仅支持无连接的数据报服务**（IP 协议）
8. **SDN**：控制平面与数据平面**分离**，**集中式控制器**统一下发流表
9. **拥塞控制**是**全局性**概念（维持全网负载），**流量控制**是**端到端**概念（发送方-接收方速率匹配）
10. **漏桶算法**严格控制输出速率，**令牌桶算法**允许有控制的突发
11. TCP/IP 选择数据报而非虚电路的原因：互联网对健壮性、灵活性的需求大于对可靠性保障的需求
