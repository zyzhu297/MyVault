# 计算机网络的分层结构


> [!note] 为什么要分层
> 计算机网络是一个**复杂的系统**，分层可以将庞大而复杂的问题转化为若干较小的**局部问题**，各层之间**相互独立**，每一层只需要考虑本层的功能实现，便于研究、设计和标准化。

**分层的原则**：
- 每一层只实现一种相对独立的功能，降低整个系统的复杂度
- 每层之间的交互通过**接口**进行，低层向高层提供服务
- 各层功能的定义独立于具体实现方法，可采用最适合的计数实现
- 保持下层对上层的独立性，上层**单向**使用下层提供的服务
- 整个分层结构应有利于标准化工作

| 概念      | 全称                           | 说明                          |
| ------- | ---------------------------- | --------------------------- |
| **SDU** | Service Data Unit            | **层与层**之间交换的数据单位，从上层传到下层    |
| **PDU** | Protocol Data Unit           | **对等**实体之间交换的数据单位，包含本层的控制信息 |
| **PCI** | Protocol Control Information | 协议控制信息，即 PDU 中本层添加的头部/尾部    |

$$n\text{-}PDU = n\text{-}SDU + n\text{-}PCI=(n-1)\text{-}SDU$$

> [!important] 三者关系
> 协议是"规则"，接口是"通道"，服务是"功能"。
> - 协议：**水平的**（对等层之间）
> - 接口和服务：**垂直的**（相邻层之间）

**分层结构的含义**：

- 第 $n$ 层实体在实现自身功能时，**只能使用第 $n-1$ 层提供的服务**，不能跨层调用更低层的服务
- 第 $n$ 层向第 $n+1$ 层提供的服务，**不仅包含第 $n$ 层本身的功能，还包含其下层服务的功能总和**
- **接口仅存在于相邻两层之间**，不同主机的对等层之间没有直接接口，对等层通过协议进行逻辑通信
- **对等层之间的通信是"虚拟"的**，实际数据流是：发送方自上而下封装 → 物理介质传输 → 接收方自下而上解封装

# 分层结构中的基本概念

## 实体（Entity）

> [!note] 实体
> 每一层中的**活动元素**称为实体。同一层的实体称为**对等实体**（Peer Entity）。

实体可以是软件实体（如进程）或硬件实体（如网卡）。

## 协议（Protocol）

> [!note] 协议
> **协议**：为进行网络中的数据交换而建立的**规则、标准或约定**。协议是**水平的**，控制对等实体之间的通信，不对等实体之间不存在协议

**协议的三要素**：

| 要素         | 含义                           | 举例                 |
| ---------- | ---------------------------- | ------------------ |
| **语法**     | 数据与控制信息的**结构或格式**            | 每个字段多长、什么类型        |
| **语义**     | 需要发出**何种**控制信息、完成何种动作、做出何种响应 | 请求报文、确认报文          |
| **同步**（时序） | 事件实现**顺序**的详细说明              | 先建立连接，再传输数据，最后释放连接 |

## 接口（Interface）

> [!note] 接口
> **接口**：同一节点内**相邻**两层之间交换信息的**连接点**，也称为**服务访问点、服务接入点（SAP, Service Access Point）**；不允许跨层定义接口

- 接口是**垂直的**，是上下层之间交互的通道
- 通过接口，下层向上层提供服务

## 服务（Service）

> [!note] 服务
> **服务**：下层为紧邻的上层提供的**功能调用**。服务是**垂直的**，即下层通过接口向上层提供服务。



**服务原语**：上层使用下层服务时的操作命令：

| 原语 | 含义 |
|------|------|
| **请求（Request）** | 服务用户请求服务提供者做某项工作 |
| **指示（Indication）** | 服务提供者告知服务用户某事件发生 |
| **响应（Response）** | 服务用户对某事件的响应 |
| **证实（Confirm）** | 服务提供者告知服务用户请求已完成 |

### 服务与协议的关系与区别

只有本层协议正确实现，才能保证向上一层提供服务

| 比较维度 | 服务 | 协议 |
|----------|------|------|
| **方向** | **垂直的**，下层通过接口向上层提供 | **水平的**，控制对等实体间的通信 |
| **可见性** | 对上层**可见**，通过服务原语调用 | 对上层**不可见**，协议细节被封装 |
| **本质** | 功能调用，描述"做什么" | 规则约定，描述"怎么做" |
| **交互对象** | 相邻层之间 | 对等实体之间 |
| **载体** | 服务原语（Request / Indication / Response / Confirm） | PDU（协议数据单元） |

**二者关系**：
- **协议是服务的实现手段**，只有本层协议正确实现，才能保证向上一层提供服务
- **服务是协议的目的**，协议的存在是为了向上层提供某种承诺的服务质量
- 同一服务可以由不同的协议实现（如 TCP 和 UDP 都提供传输层服务，但协议机制完全不同）

### 面向连接服务与无连接服务

| 类型 | 特点 | 典型例子 |
|------|------|----------|
| **面向连接服务** | 通信前必须**建立连接**，通信后**释放连接**；按序传输，可靠性高 | TCP、电话系统 |
| **无连接服务** | 无需建立连接，每个分组**独立传输**，可能乱序、丢失 | IP、UDP、邮政系统 |

### 可靠服务与不可靠服务

| 类型 | 特点 | 典型例子 |
|------|------|----------|
| **可靠服务** | 保证数据正确、完整地到达目的端，有确认和重传机制 | TCP |
| **不可靠服务** | 尽量交付，不保证一定正确到达，无误恢复机制 | IP、UDP |

### 有应答服务与无应答服务

| 类型 | 特点 | 典型例子 |
|------|------|----------|
| **有应答服务** | 接收方收到数据后给发送方一个**确认/应答** | TCP 的 ACK |
| **无应答服务** | 不需要对方确认 | 大多数无连接服务 |

> [!warning] 易错提醒
> 面向连接 ≠ 可靠服务。面向连接也可以在连接建立后不提供可靠传输（如在 ATM 中使用无确认的 CBR 服务）。

# ISO/OSI 参考模型（七层模型）

## OSI 模型概述

> [!note] OSI/RM 定义
> **OSI/RM**（Open System Interconnection / Reference Model）：国际标准化组织（ISO）提出的网络体系结构标准，目的是使不同厂家的网络能够**互相通信**。

OSI 模型采用**七层**结构，从下到上依次为：

| 层次编号 | 层次名称      | 功能摘要                                     |
| ---- | --------- | ---------------------------------------- |
| 7    | **应用层**   | 为用户应用程序提供网络服务接口                          |
| 6    | **表示层**   | 数据格式转换、加密解密、压缩解压缩                        |
| 5    | **会话层**   | 建立、管理、终止会话（同步）                           |
| 4    | **传输层**   | 差错控制、流量控制，复用和分用、连接管理、可靠传输管理              |
| 3    | **网络层**   | 差错控制、流量控制、**拥塞控制**，网际互连、路由选择，连接管理、可靠传输管理 |
| 2    | **数据链路层** | 差错控制、流量控制、成帧、介质访问控制                      |
| 1    | **物理层**   | 透明传输比特流，定义物理接口特性                         |

## 各层详细功能

### 物理层（Physical Layer）

> [!note] 物理层核心任务
> 在**物理媒体**上**透明地传输比特流**；因为没有下一层所以不需要数据封装

- **传输单位**：**比特（bit）**
- 定义物理设备的**机械特性、电气特性、功能特性、规程特性**
- 解决：用多大的电压表示 1 和 0，一个比特持续多长时间，初始连接/断开连接的方式等

**主要功能**：
- 物理连接的建立、维持和释放
- 比特流的透明传输
- 物理层接口的标准定义

> [!warning] 易错提醒
> 物理层**不是指传输介质本身**（双绞线、光纤等属于第 0 层，不在 OSI 模型之内）。

### 数据链路层（Data Link Layer）

> [!note] 数据链路层核心任务
> 在相邻节点间的**无差错**数据帧传输。

- **传输单位**：**帧（Frame）**
- 负责将网络层传下来的 IP 数据报组装成帧
- 主要设备：网桥、二层交换机

**主要功能**：
- **成帧**（Framing）：定义帧的开始和结束
- **差错控制**：通过 CRC 等检错码发现并纠正错误
- **流量控制**：协调相邻节点间的数据流量（如停止-等待、滑动窗口）
- **介质访问控制（MAC）**：解决广播信道中多台设备争用信道的问题
- 将不可靠的物理链路变为**相对可靠**的逻辑链路

### 网络层（Network Layer）

> [!note] 网络层核心任务
> 将分组从**源端**经过多个网络节点传送到**目的端**，提供**通信子网**的最高层功能；**既有提供连接的虚电路服务，由提供无连接的数据报服务**

- **传输单位**：**分组/包（Packet）**
- 主要设备：路由器、三层交换机

**主要功能**：
- **路由选择**：为分组选择从源到目的的合适路径
- ⭐**拥塞控制**：网路节点因过载而大量丢弃分组时，网络处于拥塞状态，需采取措施缓解
- **流量控制**：协调源主机与目的主机的收发速率
- **差错控制**：通过检错机制确保向上层提交的都是无差错的分组，**无法纠正则丢弃**
- **网际互连**：将异构网络连接起来
- **逻辑寻址**：使用 IP 地址标识网络中的主机

### 传输层（Transport Layer）

> [!note] 传输层核心任务
> 负责主机中两个**进程之间**的通信，实现**端到端**的**可靠**数据传输；OSI 模型中只提供面向连接的可靠服务

- **传输单位**：**报文段（Segment）** / 用户数据报（User Datagram）
- **端到端**：从源主机的进程到目的主机的进程，是整个协议栈中**最关键的一层**

**主要功能**：
- **复用和分用**：将多个应用层进程的数据复用到一条网络连接，或从网络层接收数据后分发到不同进程
- **可靠传输**：通过确认、重传等机制保证数据无差错、不丢失、不重复、按序到达
- **流量控制**：端到端的流量控制（区别于数据链路层的点对点流量控制）
- **拥塞控制**：TCP 的拥塞控制算法

> [!important] 数据链路层 vs 传输层的流量控制
> - 数据链路层：**相邻节点之间**（点对点），单条链路
> - 传输层：**源端到目的端**（端到端），跨越整个网络

### 会话层（Session Layer）

> [!note] 会话层核心任务
> 允许不同主机上各**进程之间**进行对话（建立、管理、终止会话）。

- 传输单位：报文
- **同步功能**：在数据流中插入**同步点**（检查点），当传输中断后可以从最近的同步点恢复，避免从头重传
- 管理会话：单工、半双工、全双工

### 表示层（Presentation Layer）

> [!note] 表示层核心任务
> 处理在两个通信系统中交换信息的**表示方式**。

**主要功能**：
- **数据格式转换**：不同系统的数据表示方式不同（如 ASCII 与 EBCDIC 编码转换）
- **数据加密与解密**：保证传输安全
- **数据压缩与解压缩**：减少传输的数据量

### 应用层（Application Layer）

> [!note] 应用层核心任务
> 为用户应用程序提供**网络服务**的接口，是 OSI 模型的**最高层**。

- 传输单位：报文
- **直接面向用户**，提供最终用户使用的网络应用

**主要应用层协议**：
- 文件传输（FTP）
- 电子邮件（SMTP）
- 万维网（HTTP）
- 域名解析（DNS）

> [!warning] 易错提醒
> 应用层**不是**指应用程序本身（如浏览器、Word），而是指应用层协议。

# TCP/IP 参考模型（四层模型）

## TCP/IP 模型概述

> [!note] TCP/IP 模型定义
> **TCP/IP 模型**：因特网（Internet）事实上的标准，由 ARPANET 发展而来，采用**四层**结构。

| 层次 | 名称 | 对应 OSI 层次 | 主要协议 |
|------|------|:-----------:|----------|
| 4 | **应用层** | 应用层 + 表示层 + 会话层 | HTTP、FTP、SMTP、DNS、Telnet |
| 3 | **传输层** | 传输层 | TCP、UDP |
| 2 | **网际层**（网络互联层） | 网络层 | IP、ICMP、IGMP、ARP、RARP |
| 1 | **网络接口层** | 数据链路层 + 物理层 | Ethernet、PPP、HDLC |

## 各层详细功能

### 网络接口层

> [!note] 网络接口层
> TCP/IP 模型**没有具体定义**该层内容，仅指出主机必须使用某种协议与网络连接，以便能够在网络上传输 IP 分组。

- 对应 OSI 的数据链路层 + 物理层
- 可以是各种物理网络，如以太网、令牌环、PPP 等

### 网际层（Internet Layer）

> [!note] 网际层核心任务
> 将分组发往任何网络，并使分组**独立地**到达目的端（可能不按顺序）。

- **核心协议**：**IP（Internet Protocol）**
- 提供**无连接、不可靠**的分组传输服务
- 主要功能：路由选择、分组转发、网际互连

**网际层的主要协议**：

| 协议 | 全称 | 功能 |
|------|------|------|
| **IP** | Internet Protocol | 核心协议，负责分组传输和路由 |
| **ICMP** | Internet Control Message Protocol | 传递差错报告和控制信息（如 ping 使用） |
| **IGMP** | Internet Group Management Protocol | 管理多播组成员 |
| **ARP** | Address Resolution Protocol | IP 地址 → MAC 地址 |
| **RARP** | Reverse ARP | MAC 地址 → IP 地址 |

> [!info] IP 协议三大特点
> **无连接、不可靠、尽力交付（Best-Effort）**

### 传输层（Transport Layer）

> [!note] 传输层核心任务
> 提供**应用进程之间**的端到端通信。

| 协议 | 特点 | 应用场景 |
|------|------|----------|
| **TCP**（传输控制协议） | **面向连接、可靠**，按序传输，流量控制，拥塞控制 | 文件传输、邮件、Web 浏览 |
| **UDP**（用户数据报协议） | **无连接、不可靠**，开销小，实时性好 | DNS、视频流、VoIP、在线游戏 |

### 应用层（Application Layer）

> [!note] 应用层核心任务
> 包含所有**高层协议**，直接为用户提供服务。

合并了 OSI 模型的**应用层、表示层、会话层**三层功能。

**常见应用层协议**：

| 协议 | 全称 | 端口 | 传输层 |
|------|------|:----:|:------:|
| **HTTP** | 超文本传输协议 | 80 | TCP |
| **HTTPS** | 安全超文本传输协议 | 443 | TCP |
| **FTP** | 文件传输协议 | 20, 21 | TCP |
| **SMTP** | 简单邮件传输协议 | 25 | TCP |
| **POP3** | 邮局协议 v3 | 110 | TCP |
| **DNS** | 域名系统 | 53 | UDP/TCP |
| **Telnet** | 远程登录协议 | 23 | TCP |
| **DHCP** | 动态主机配置协议 | 67, 68 | UDP |

# OSI 模型与 TCP/IP 模型的比较

## 相同点

| 共同点 | 说明 |
|--------|------|
| **分层结构** | 都采用分层的体系结构，各层相互独立 |
| **功能对应** | 都有传输层和网络层，功能相似 |
| **协议独立** | 都是基于独立的协议栈概念 |
| **面向互连** | 目标都是实现异构网络的互连 |

## 不同点

| 比较维度 | OSI/RM | TCP/IP |
|----------|--------|--------|
| **层数** | 7 层 | 4 层 |
| **设计理念** | **先有模型后有协议**，偏向理论，通用性强 | **先有协议后有模型**，贴近实际，实用性强 |
| **服务/协议/接口** | 三个概念**严格区分**，定义清晰 | 三个概念**没有明确区分** |
| **网络层** | **支持无连接和面向连接**两种服务 | **仅支持无连接**服务（IP） |
| **会话层 / 表示层** | 独立的分层 | 合并到应用层中 |
| **核心协议** | 通用标准 | TCP 和 IP（故得名 TCP/IP） |
| **应用范围** | 学术研究和教学（理论上更完整） | 实际因特网标准 |

> [!important] 关键理解
> - OSI 是**法律上的国际标准**，但未被广泛采用（协议复杂、实现困难、制定周期长）
> - OSI 精确定义了“服务”“协议”“接口”这三个核心概念，其分层思想与现代面向对象程序设计高度契合；而 TCP/IP 模型对这三者没有明确区分
> - TCP/IP 是**事实上的国际标准**，因特网的实际标准
> - 教学中常用的是**五层协议体系结构**（结合两者优点）

# 五层协议体系结构（综合模型）

> [!note] 五层体系结构
> 教学和考试中常用的**折中方案**：综合 OSI 和 TCP/IP 的优点，采用**五层结构**。

| 层次 | 名称 | 功能 | 传输单位 |
|:----:|------|------|:--------:|
| 5 | **应用层** | 为用户提供网络应用程序和接口 | 报文 |
| 4 | **传输层** | 进程到进程的端到端通信 | 报文段（TCP）/ 用户数据报（UDP） |
| 3 | **网络层** | 源到目的的分组路由和转发 | 分组/数据报 |
| 2 | **数据链路层** | 相邻节点间的帧传输和差错控制 | 帧 |
| 1 | **物理层** | 比特流的透明传输 | 比特 |

## 各层数据传输过程

> [!note] 数据封装与解封装
> 数据在各层之间的封装与解封装过程如下：

发送方自上而下，每一层对上层数据添加**头部**（或尾部），封装成本层的 PDU：

$$\text{应用层报文} \xrightarrow{\text{+TCP头}} \text{传输层报文段} \xrightarrow{\text{+IP头}} \text{网络层分组} \xrightarrow{\text{+帧头/帧尾}} \text{数据链路层帧} \xrightarrow{\text{物理层}} \text{比特流}$$

接收方自下而上，每一层剥去头部（或尾部），恢复上层数据：

$$\text{比特流} \xrightarrow{\text{物理层}} \text{帧} \xrightarrow{\text{去帧头/帧尾}} \text{分组} \xrightarrow{\text{去IP头}} \text{报文段} \xrightarrow{\text{去TCP头}} \text{应用层报文}$$


## 各层常见设备

| 层次 | 常见设备 |
|:----:|----------|
| **应用层** | 网关（应用层网关） |
| **传输层** | 网关（传输层网关） |
| **网络层** | **路由器**、三层交换机 |
| **数据链路层** | **网桥**、**二层交换机** |
| **物理层** | **集线器（Hub）**、**中继器** |


# ❗ 易错点总结

1. **协议是"水平的"**（对等实体之间），**服务是"垂直的"**（相邻层之间）
2. **面向连接 ≠ 可靠服务**：面向连接重在"建立连接"，不一定保证可靠性
3. **物理层不是传输介质**：双绞线、光纤等属于第 0 层
4. **OSI 是"法律上的标准"**，**TCP/IP 是"事实上的标准"**
5. **TCP/IP 的网络层仅支持无连接**（IP），OSI 的网络层支持有/无连接两种
6. **数据链路层**和**传输层**都有"流量控制"：前者是**点对点**（相邻节点），后者是**端到端**（源到目的）
7. **应用层不是指应用程序本身**，而是指应用层协议
8. **封装**：自上而下加头加尾；**解封装**：自下而上去头去尾
9. **TCP/IP 模型中无会话层和表示层**，这两层的功能合并到了应用层
10. **协议三要素**：语法、语义、同步（时序）— 常考选择题
