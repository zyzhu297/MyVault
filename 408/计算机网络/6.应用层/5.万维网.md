# 万维网的概念与组成结构

## 万维网 (World Wide Web, WWW) 概念

- **定义**：万维网是一个大规模的、**联机式**的信息存储空间，是**分布式超媒体系统**，是**超文本系统**的扩充。
- **本质**：万维网并非某种特殊的计算机网络，而是运行在因特网上的一个**分布式应用**。
- **信息单位**：任何有用的事物都被称为*资源*，并由一个**统一资源定位符（URL）** 唯一标识；万维网以**页面**为单位组织信息，每个页面称为一个"网页"或"Web 页面"。
- **核心思想**：通过**超链接** (Hyperlink) 将分布在世界各地的文档链接在一起，用户只需点击即可从一个页面跳转到另一个页面。
- **客户/服务器模式**：万维网采用 **C/S（Client/Server，客户/服务器）模式**，由**万维网客户程序（浏览器）** 和**万维网服务器程序（Web 服务器）** 组成。

## 万维网的组成结构

万维网的核心由以下三部分组成：

| 组成部分             | 说明                                       | 核心协议/标准                 |
| ---------------- | ---------------------------------------- | ----------------------- |
| **统一资源定位符 URL**  | 标识万维网上的各种资源，使每一个资源都有唯一的标识符               | RFC 3986                |
| **超文本传输协议 HTTP** | 定义了浏览器与服务器之间的交互规则，基于 **TCP** 连接提供可靠的数据传输 | RFC 2616, RFC 7230-7235 |
| **超文本标记语言 HTML** | 定义页面的结构、内容和格式，使不同作者创作的页面都能在浏览器上正确显示      | W3C 标准                  |

上述三部分共同构成了万维网的基础架构。

## URL（统一资源定位符）

### URL 的一般格式

```
<协议>://<主机>:<端口>/<路径>
```

- **协议**：访问资源所使用的协议，如 `http`、`https`、`ftp` 等。
- **主机**：资源所在主机的域名或 IP 地址。
- **端口**：协议对应的端口号（HTTP 默认 80，可省略；HTTPS 默认 443）。
- **路径**：资源在主机上的具体位置。

### 示例

```
http://www.example.com:80/path/to/file.html
```

> [!warning]
> URL 中的字母**不区分大小写**，<端口>和<路径>在某些情况可以省略

# 超文本传输协议

## HTTP 概述

- **定义**：HTTP（HyperText Transfer Protocol，超文本传输协议）是万维网的应用层协议，定义了浏览器如何向服务器请求文档以及服务器如何将文档传送给浏览器。
- **层次**：HTTP 位于应用层，使用 **TCP** 作为传输层协议（HTTP/1.0 和 HTTP/1.1），HTTP/3 使用 **QUIC（基于 UDP）**。
- **端口**：HTTP 服务器默认监听 **TCP 端口 80**。
- **无状态协议**：HTTP 本身是**无状态**的，服务器不保存客户的状态信息。Cookie 机制是为了弥补这一不足。

> [!warning] HTTP 是**面向事务**的
> 每一次请求—响应对应一个事务。

## HTTP 的操作过程

### 万维网的工作流程

1. **DHCP 获取网络配置**：主机启动时通过 DHCP 获取 IP 地址、子网掩码、默认网关和 DNS 服务器地址。
2. **浏览器分析 URL**：解析协议、主机名、端口和路径。
3. **DNS 解析域名**：向 DNS 服务器查询域名对应的 IP 地址（发送 DNS 报文前需 **ARP** 获取 DNS 服务器或默认网关的 MAC 地址）。
4. **建立 TCP 连接**：三次握手（发送 SYN 前需 **ARP** 获取 Web 服务器或默认网关的 MAC 地址）。
5. **发送 HTTP 请求**：浏览器发送 HTTP 请求报文。
6. **服务器处理并响应**：返回 HTTP 响应报文。
7. **释放 TCP 连接**：四次挥手。
8. **浏览器解析并显示**：渲染 HTML 页面。

### 用户访问页面的具体过程

以一个用户点击超链接为例：

> 用户点击 `http://www.tsinghua.edu.cn/chn/yxsz/index.htm`，该链接指向清华大学院系设置页面。

1. 浏览器分析 URL。
2. DNS 解析 `www.tsinghua.edu.cn`（过程中需 ARP）。
3. 与服务器建立 TCP 连接（过程中需 ARP）。
4. 发送 `GET /chn/yxsz/index.htm`。
5. 服务器返回 `index.htm`。
6. TCP 连接释放。
7. 浏览器渲染页面。

### 页面请求时间分析

请求一个万维网文档所需的时间 = **RTT（往返时延）相关开销** + **文档传输时延**。

下面以一个典型场景分析不同连接方式下的 RTT 开销。

> [!example] 场景设定
> 用户访问一个网页，页面包含 **1 个 HTML 主文件**和 **N 个内嵌对象**（如图片、CSS、JS 等），忽略传输时延，只计算 RTT 开销。

#### 1. 非持续连接（HTTP/1.0 默认）

每个对象都需要单独建立 TCP 连接、请求、接收后释放：

- TCP 三次握手：1 RTT
- 发送请求 + 接收响应：1 RTT
- **每个对象 2 RTT**，共 (1 + N) 个对象

$$ \text{总时间} = 2 \times (N + 1) \ \text{RTT} $$

> [!warning] 缺点
> 每请求一个对象都需要 2 个 RTT，当 N 很大时开销巨大；且每次新建 TCP 连接都要经历慢启动。

> [!note] 补充：并行 TCP 连接（实际浏览器的优化）
> 上述 2(N+1) RTT 是**纯串行**的理论值。实际上浏览器会**同时建立多个并行的 TCP 连接**来加速加载。
>
> 设浏览器同时打开 **M 条**并行 TCP 连接（早期浏览器通常 M = 4~6，现代浏览器 M ≈ 6~8）：
>
> - 第 1 条连接获取 HTML：2 RTT
> - 浏览器解析 HTML 后，将 N 个内嵌对象分配到 M 条连接上并行获取
> - 每条连接仍为**非持续模式**（一个对象完成后释放，再建新连接取下一个）
> - 每"轮"最多并行处理 M 个对象，每轮 2 RTT，共需 ⌈N/M⌉ 轮
>
> $$\text{总时间} = 2 + 2 \times \left\lceil \frac{N}{M} \right\rceil \ \text{RTT}$$
>
> **举例**：N = 10 张图片，M = 4 条并行连接：
>
> - HTML：2 RTT
> - 第 1 轮：4 张图 × 2 RTT → 完成后还剩下 6 张
> - 第 2 轮：4 张图 × 2 RTT → 完成后还剩下 2 张
> - 第 3 轮：2 张图 × 2 RTT → 全部完成
> - **总计 = 2 + 2 × 3 = 8 RTT**（远小于纯串行的 22 RTT）
>
> > 并行 TCP 是非持续连接在**实际应用中的折中方案**：通过多条并发连接部分缓解了串行等待的问题，但每个连接仍要经过 TCP 握手和慢启动，效率不如后续的持续连接方案。

#### 2. 持续连接 — 非流水线方式（HTTP/1.1）

只建立一次 TCP 连接，但必须收到前一个响应后才能发送下一个请求：

- TCP 三次握手：**1 RTT**
- 第一个对象（HTML）：**1 RTT**（请求 + 响应）
- 客户端解析 HTML，依次请求 N 个对象，每个需 **1 RTT**（串行）

$$ \text{总时间} = 1 + 1 + N = N + 2 \ \text{RTT} $$

> 相比非持续连接节省了 N+1 个 RTT（N+1 次 TCP 握手），但每个内嵌对象仍需排队等待。

#### 3. 持续连接 — 流水线方式（HTTP/1.1 默认）

只建立一次 TCP 连接，且客户端可以**连续发出所有请求**而不必等待响应：

- TCP 三次握手：**1 RTT**
- 第一个对象（HTML）：**1 RTT**（请求 + 响应）
- 客户端解析 HTML 后，**一次性发出全部 N 个请求**
- 服务器按序返回全部响应：**1 RTT**（所有 N 个对象并行传输）

$$ \text{总时间} = 1 + 1 + 1 = 3 \ \text{RTT} $$

> [!important] 关键结论
> 流水线方式下，无论页面有多少内嵌对象，**总 RTT 恒为 3**，与 N 无关。这是 HTTP/1.1 相比 HTTP/1.0 的最大性能提升之一。

#### 四种方式对比总结

| 方式 | TCP 握手 | HTML | N 个内嵌对象 | 总 RTT |
|-----|:---:|:---:|:---:|:---:|
| **非持续—串行** | (N+1) 次，每次 1 RTT | 1 RTT | N RTT | **2(N+1)** |
| **非持续—并行 M 条** | ⌈N/M⌉ + 1 次，每次 1 RTT | 1 RTT | 2⌈N/M⌉ RTT | **2 + 2⌈N/M⌉** |
| **持续—非流水线** | 1 RTT（共享） | 1 RTT | N RTT（串行） | **N + 2** |
| **持续—流水线** | 1 RTT（共享） | 1 RTT | 1 RTT（并行） | **3** |

> [!important] 传输 RTT 与 MSS 和拥塞窗口的关系
> 前述分析假设每个对象在 **1 个 RTT** 内传完（即对象足够小）。但当对象较大时，TCP 的拥塞控制和流量控制会显著影响实际需要的 RTT 数。
>
> **1. MSS 决定分段数**
> - 设对象大小为 $S$，每个 TCP 报文段的最大载荷为 **MSS**（Maximum Segment Size）
> - 该对象需要拆分为 $L = \lceil S / \mathrm{MSS} \rceil$ 个报文段
>
> **2. 拥塞窗口决定每 RTT 能发多少段**
> - TCP 的**拥塞窗口 cwnd**（Congestion Window）限制了每 RTT 最多能发送的报文段数
> - 若当前 cwnd = $W$（以 MSS 为单位），则每 RTT 最多发送 $W$ 个报文段
> - 传输该对象需要的 RTT 数：$\lceil L / W \rceil$
>
> **3. 慢启动阶段的特殊考量**
> - TCP 刚建立连接时处于**慢启动**阶段，cwnd 从 1 MSS 开始，每经过一个 RTT **翻倍**
> - 第 1 个 RTT：cwnd = 1，累计发出 1 个 MSS
> - 第 2 个 RTT：cwnd = 2，累计发出 1 + 2 = 3 个 MSS
> - 第 3 个 RTT：cwnd = 4，累计发出 1 + 2 + 4 = 7 个 MSS
> - 第 $k$ 个 RTT：cwnd = $2^{k-1}$，累计发出 $(2^k - 1)$ 个 MSS
>
> **4. 结合 HTTP 的完整 RTT 分析**
>
> 以"持续连接—流水线"获取一个大小为 $S$ 的对象为例（$L = \lceil S / \mathrm{MSS} \rceil$）：
>
> - 若 $L \le 1$：对象本身只需 1 RTT 传完 → 总 3 RTT（如前所述）
> - 若 $L > 1$：对象传输本身也需要多个 RTT，取决于当前 cwnd
>   - 慢启动阶段：找到最小 $k$ 使 $2^k - 1 \ge L$，则传输需 $k$ 个 RTT
>   - 拥塞避免阶段：若 cwnd = $W$ 稳定，则传输需 $\lceil L / W \rceil$ 个 RTT
>
> > 因此，**实际总 RTT = HTTP 交互 RTT + 数据传输 RTT**。对于小对象（一个网页通常是几十 KB），传输 RTT 可忽略；但对于大文件下载（图片、视频等），必须考虑分段和窗口的影响。

## HTTP 的特点

### 1. 无连接与无状态

| 特点 | 说明 |
|-----|------|
| **无连接** | HTTP/1.0 在交换 HTTP 报文之前不需要先建立 HTTP 连接，但底层依赖 TCP 连接。每次请求—响应后释放 TCP 连接。 |
| **无状态** | 服务器不记忆客户的任何状态信息，每次请求都是独立的。简化了服务器设计，使其能支持大量并发请求。 |

### 2. 持续连接与非持续连接

| 连接方式 | 说明 | 优缺点 |
|---------|------|--------|
| **非持续连接** | 每次请求/响应对应一个单独的 TCP 连接，传完后立即释放（HTTP/1.0 默认）。 | **优点**：简单；**缺点**：每请求一个对象需要 2 个 RTT 开销，效率低。 |
| **持续连接** | 建立一次 TCP 连接后，可以连续发送多个请求和接收多个响应（HTTP/1.1 默认）。 | **优点**：减少 RTT 开销和 TCP 慢启动的影响。 |

### 3. 持续连接的两种工作方式

| 工作方式 | 说明 |
|---------|------|
| **非流水线方式** | 客户收到前一个响应后才能发送下一个请求。服务器在发送完一个对象后 TCP 连接空闲，浪费资源。每个引用对象需要 1 个 RTT。 |
| **流水线方式** | 客户可以连续发送多个请求而不必等待响应，服务器按序返回响应。所有引用对象共需 1 个 RTT，效率较高。HTTP/1.1 默认使用流水线方式。 |

### 4. 无状态 → Cookie

- HTTP 是无状态的，但实际应用中常需要识别用户身份（如购物车、登录状态）。
- **解决方案**：使用 **Cookie** 机制。
- **Cookie 工作流程**：
  1. 服务器在响应报文中添加 `Set-Cookie` 首部行。
  2. 浏览器将 Cookie 保存在本地。
  3. 浏览器后续请求同一服务器时，自动在请求报文中添加 `Cookie` 首部行。

> **Cookie 不是 HTTP 协议的一部分**，而是对 HTTP 无状态特性的一种补充机制。

### 5. HTTP 与 HTTPS

| 对比项 | HTTP | HTTPS |
|-------|------|-------|
| **端口** | 80 | 443 |
| **安全性** | 明文传输，不安全 | SSL/TLS 加密，安全 |
| **协议栈** | HTTP → TCP → IP | HTTP → SSL/TLS → TCP → IP |
| **证书** | 不需要 | 需要 CA 证书 |

## HTTP 的报文结构

HTTP 报文分为**请求报文**和**响应报文**两种，结构相似。

### 报文的一般结构

![[HTTP 的报文结构.png]]```

- **开始行**：区分请求报文和响应报文。
- **首部行**：说明浏览器、服务器或报文主体的一些信息，由 **关键字: 值** 组成。
- **空行**：CRLF，分隔首部行和实体主体。
- **实体主体**：通常不用，仅在 POST 等方法中使用。

### 请求报文

#### 常用请求方法

| 方法          | 说明                                  | 幂等性 |
| ----------- | ----------------------------------- | --- |
| **GET**     | 请求读取由 URL 标识的信息，可附带查询参数（?key=value） | 是   |
| **POST**    | 向服务器提交数据（如表单数据），数据放在报文体中            | 否   |
| **HEAD**    | 与 GET 类似，但只返回响应头部，不返回主体（用于测试链接有效性）  | 是   |
| **PUT**     | 向指定 URL 上传资源（更新），存储一个文档             | 是   |
| **DELETE**  | 删除指定 URL 的资源                        | 是   |
| **CONNECT** | 用于代理服务器，建立隧道                        | 否   |
| **OPTIONS** | 查询服务器支持的方法                          | 是   |
| **TRACE**   | 回显服务器收到的请求，用于诊断                     | 是   |

> [!note] 幂等性
> **幂等性（Idempotence）** 是指同一个请求执行**一次**和执行**多次**，对服务器资源产生的**副作用是相同的**，即多次重复请求不会产生额外的影响。
>
> - **幂等的方法**（GET、HEAD、PUT、DELETE 等）：无论发送多少次相同的请求，结果都一样。例如 GET 请求只是读取数据，不会修改服务器状态；DELETE 删除资源，第一次删除成功后，后续请求返回 404，但不会改变服务器上资源已不存在的事实。
> - **非幂等的方法**（POST、CONNECT 等）：每次请求都可能产生新的副作用。例如 POST 提交表单，每次都会创建一条新的记录。
>
> 幂等性是设计 RESTful API 和实现**可靠请求重传**的重要基础——幂等的请求在网络超时时可以安全地重试。

### 响应报文

#### 常见状态码

| 状态码 | 类别 | 示例 |
|-------|------|------|
| **1xx** | 信息性 | 100 Continue（继续发送请求） |
| **2xx** | 成功 | **200 OK**（请求成功）、204 No Content |
| **3xx** | 重定向 | **301 Moved Permanently**（永久重定向）、**302 Found**（临时重定向）、304 Not Modified |
| **4xx** | 客户端错误 | **400 Bad Request**（请求语法错误）、**403 Forbidden**（拒绝访问）、**404 Not Found**（资源不存在） |
| **5xx** | 服务器错误 | **500 Internal Server Error**（服务器内部错误）、**502 Bad Gateway**、**503 Service Unavailable** |

#### 常见首部行字段

| 首部字段 | 说明 | 出现位置 |
|---------|------|---------|
| Host | 请求的主机名和端口号 | 请求 |
| User-Agent | 客户端浏览器的类型和版本 | 请求 |
| Accept | 客户端能处理的媒体类型 | 请求 |
| Accept-Language | 客户端偏好的语言 | 请求 |
| Connection | 连接管理（keep-alive / close） | 通用 |
| Content-Type | 实体主体的媒体类型 | 实体 |
| Content-Length | 实体主体的字节数 | 实体 |
| Set-Cookie | 服务器向客户端设置 Cookie | 响应 |
| Cookie | 客户端向服务器发送的 Cookie | 请求 |
| Server | 服务器软件信息 | 响应 |
| Location | 重定向的目标 URL | 响应 |
| Cache-Control | 缓存控制指令 | 通用 |
| ETag | 资源的实体标签（用于缓存验证） | 响应 |

## 代理服务器

### 概念

**代理服务器（Proxy Server）** 是一种网络实体，又称为**万维网高速缓存（Web Cache）**。

### 工作原理

1. 代理服务器把最近的一些**请求和响应**暂存在本地磁盘中。
2. 当新请求到达时，若代理服务器发现该请求的响应**已在缓存中**，就直接返回缓存中的响应，而不需要按 URL 的地址再次去因特网访问该资源。

### 作用

- **减少延迟**：就近获取缓存内容，减少访问因特网的时间。
- **减少网络流量**：避免重复从远程服务器获取相同内容，减轻骨干网负担。
- **节省带宽成本**：对于机构内部网络，代理服务器可以大大节省访问因特网的带宽费用。

## HTTP 请求报文举例

### 完整的 GET 请求示例

当用户在浏览器地址栏输入 `http://www.example.com/index.html` 并回车时，浏览器发送的请求报文大致如下：

```
GET /index.html HTTP/1.1\r\n
Host: www.example.com\r\n
Connection: keep-alive\r\n
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)\r\n
Accept: text/html,application/xhtml+xml\r\n
Accept-Language: zh-CN,zh;q=0.9\r\n
Accept-Encoding: gzip, deflate\r\n
\r\n
```

### POST 请求示例（提交表单）

假设用户在登录页面提交用户名和密码：

```
POST /login HTTP/1.1\r\n
Host: www.example.com\r\n
Content-Type: application/x-www-form-urlencoded\r\n
Content-Length: 32\r\n
\r\n
username=admin&password=123456
```

### 对应的响应报文示例

```
HTTP/1.1 200 OK\r\n
Date: Mon, 08 Jul 2024 10:30:00 GMT\r\n
Server: Apache/2.4.41\r\n
Content-Type: text/html; charset=utf-8\r\n
Content-Length: 125\r\n
Connection: keep-alive\r\n
\r\n
<html>
<head><title>Example</title></head>
<body>
<h1>Hello, World!</h1>
</body>
</html>
```

### 304 Not Modified 响应示例

当浏览器已缓存了页面且页面未更新时，服务器返回：

```
HTTP/1.1 304 Not Modified\r\n
Date: Mon, 08 Jul 2024 10:35:00 GMT\r\n
Server: Apache/2.4.41\r\n
\r\n
(空实体主体)
```

## HTTP 版本演进

| 版本 | 特点 |
|-----|------|
| **HTTP/0.9** | 仅支持 GET，无首部，无版本号，已废弃 |
| **HTTP/1.0** | 引入首部、POST/HEAD 方法、状态码；默认非持续连接 |
| **HTTP/1.1** | 默认持续连接、流水线、更多缓存控制、Host 首部（支持虚拟主机）、范围请求 |
| **HTTP/2** | 二进制分帧、多路复用、头部压缩（HPACK）、服务器推送 |
| **HTTP/3** | 基于 QUIC（UDP），解决队头阻塞，0-RTT 连接恢复 |

> **总结**：目前主流仍是 HTTP/1.1 和 HTTP/2，HTTP/3 正在逐步推广中。
