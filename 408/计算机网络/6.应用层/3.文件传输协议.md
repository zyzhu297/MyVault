# FTP 概述

> **文件传输协议（File Transfer Protocol，FTP）**：互联网中使用最广泛的文件传送协议，采用**客户/服务器（C/S）模型**。⭐FTP 提供**交互式**访问，**允许客户指明文件的类型与格式**，并允许文件具有**存取权限**

FTP 使用 **TCP** 作为传输层协议，使用**两个并行的 TCP 连接**——一个**控制连接**和一个**数据连接**，这一"控制与数据分离"的设计是 FTP 与大多数其他应用层协议（如 HTTP、SMTP）最显著的区别。

服务器进程由两部分组成：
- **主进程**：负责接受新的客户端连接请求（监听 21 号端口），为每个客户创建一个从属进程
- **若干从属进程**：每个从属进程负责处理对应的客户请求——包括命令解析、数据传输、权限验证等。控制连接和数据连接均由从属进程管理；

FTP 主要提供的功能：
1. **文件的上传与下载**——客户端可以将本地文件上传到服务器（STOR）或从服务器下载文件到本地（RETR）
2. **目录浏览与管理**——列出远程目录内容（LIST）、切换工作目录（CWD）、删除文件（DELE）等
3. **用户认证与权限控制**——通过用户名/密码登录，不同用户可设置不同的文件访问权限
4. **异构系统间的文件传输**——通过 ASCII / Binary 模式自动处理不同操作系统之间的文件格式差异（如换行符转换）
5. **通过匿名 FTP 方式**，实现公用文件的共享

| 属性 | 说明 |
|------|------|
| **应用模型** | 客户/服务器（C/S） |
| **传输层协议** | TCP |
| **连接数** | **2 个**（控制连接 + 数据连接） |
| **控制连接端口** | 服务器端 **21** |
| **数据连接端口** | 服务器端 **20** |
| **工作模式** | 主动模式（PORT）、被动模式（PASV） |

# FTP 的工作原理

> FTP 的"双连接"架构是其最核心的设计特征——控制信息和数据传输走不同的通道，各司其职。

FTP 服务器在整个会话期间维持两个 TCP 连接：

| 连接 | 建立时机 | 生存周期 | 功能 | 数据流向 |
|------|----------|:--------:|------|:--------:|
| **控制连接** | 客户发起 FTP 会话时建立 | **整个会话期间**一直保持 | 传送**FTP 命令**（如 USER、PASS、LIST、RETR、STOR）和服务器的**应答** | 命令/应答 |
| **数据连接** | 每次文件传输时**临时建立** | **仅传输期间**存在，传输完毕立即关闭 | 传送**文件数据**和**目录列表** | 文件内容 |

> [!important] 控制连接与数据连接的核心区别
>
> | 对比维度 | 控制连接 | 数据连接 |
> |----------|:--------:|:--------:|
> | **端口号** | 服务器端口 **21** | 服务器端口 **20** |
> | **持续时间** | **整个会话** | **仅数据传输期间** |
> | **传送内容** | FTP 命令和应答 | 文件数据、目录列表 |
> | **连接方式** | 始终由客户发起 | 主动模式：服务器发起；被动模式：客户发起 |
> | **数量** | **1 条** | 每次传输**1 条**（可先后建立多条） |

> [!tip] 为什么 FTP 要使用两条连接？
> 将**控制**和**数据**分离的好处是：
> 1. **简化协议设计**——控制信息与数据各自独立，无需在数据流中插入控制标记
> 2. **传输期间仍能控制**——文件传输过程中，客户依然可以通过控制连接发送**中止命令**（ABOR）
> 3. **支持第三方传输**——数据连接可以在两个 FTP 服务器之间建立（代理传输）

## FTP 是有状态协议

> **FTP 是一种有状态协议（Stateful Protocol）**——FTP 服务器在整个会话期间会**持续维护**客户端的状态信息，同一个会话内的后续命令依赖之前命令建立的上下文。

FTP 的"状态"来源于**控制连接在整个会话期间一直保持**这一设计——因为控制连接不关闭，服务器可以在内存中为每个客户维护一份**会话状态**：

| 状态信息       | 建立方式           | 说明                                            |
| ---------- | -------------- | --------------------------------------------- |
| **用户身份**   | USER + PASS 登录 | 登录后服务器记住当前用户，后续命令无需重复认证                       |
| **当前工作目录** | CWD 切换         | 服务器持续跟踪用户的当前目录，LIST / RETR / STOR 等命令相对于该目录执行 |
| **传输模式**   | TYPE 设置        | 服务器记住当前是 ASCII 还是 Binary 模式，后续传输沿用该设置         |
| **工作模式**   | PORT / PASV    | 服务器记住当前是主动还是被动模式，数据连接据此建立                     |
| **传输进度**   | 数据连接进行中        | 服务器知道当前正在传输哪个文件、传输了多少字节                       |

> [!important] FTP 有状态 vs HTTP 无状态（408 常考对比）
>
> | 对比维度 | FTP | HTTP |
> |----------|:---:|:---:|
> | **状态特性** | **有状态** | **无状态**（每个请求独立，服务器不保留上下文） |
> | **根本原因** | 控制连接**一直保持**（TCP 持久连接） | 早期 HTTP/1.0 每次请求一个 TCP 连接，请求完成即关闭 |
> | **用户信息** | 登录一次后服务器一直记住用户身份 | 每次请求需携带 Cookie / Session / Token 来模拟状态 |
> | **工作目录** | 服务器维护当前目录，命令相对于它执行 | 无"目录状态"概念，每个 URL 是绝对路径 |
> | **状态存储** | 服务器内存中直接维护 | 依靠 **Cookie + Session**（额外机制）来模拟状态 |

> [!tip] 有状态带来的影响
> **优点**：
> - 同一会话中无需重复认证——USER/PASS 后所有操作自动关联到该用户
> - 上下文连续——CWD 切换到某目录后，后续 LIST / RETR 自动针对该目录，无需每次指定全路径
> - 状态一致——TYPE 设置后所有传输统一使用该模式
>
> **缺点**：
> - **服务器资源消耗大**——每个活动会话都要占用服务器内存维护状态，大量并发用户时服务器压力显著
> - **故障恢复困难**——控制连接意外断开后，所有状态丢失，客户必须重新登录、重新切换目录、重新设置模式
> - **负载均衡复杂**——不支持将同一会话的不同请求分发到不同服务器（状态在特定服务器内存中，而非共享存储）

> [!warning] 注意
> FTP 的有状态性仅体现在**控制连接层面**——数据连接本身是**无状态**的：每次数据传输独立建立连接，传输完毕即关闭，下次传输再重新建立。

# 控制连接与数据连接

## 控制连接（Control Connection）

> **控制连接**：客户和服务器之间传送 FTP 命令和应答信息的 TCP 连接，在整个 FTP 会话期间**一直保持**。

控制连接的建立过程与一般的 TCP 连接相同——服务器在 **21 号端口**被动等待客户的连接请求，客户使用临时端口主动发起连接。

**控制连接的特点：**
- 始终由**客户**主动发起建立
- 在整个 FTP 会话期间**一直保持**，直到客户发送 QUIT 命令或连接异常断开
- 传送的是 **NVT ASCII** 格式的命令和应答（文本形式，人类可读）
- 每次命令对应一次应答（一发一收，同步交互）

## 数据连接（Data Connection）

> **数据连接**：为传输文件数据或目录列表而临时建立的 TCP 连接，传输完毕即**立即关闭**。每次需要传输数据时都重新建立一条新的数据连接。

**数据连接的特点：**
- **临时性**——只存在于数据传输期间，传输完成立即关闭
- **可多次建立**——一次 FTP 会话中可先后建立多条数据连接（传多个文件）
- 传送的是**文件的原始字节**（二进制或 ASCII 模式）
- 数据连接的建立方向取决于工作模式（主动/被动）

> [!warning] FTP 使用两个独立端口号（21 和 20），必须明确区分：
> - **21 号端口**：控制连接——传送命令和应答
> - **20 号端口**：数据连接——传送文件数据

## 控制信息的"带外"传送

> **带外（Out-of-Band）传送**：控制信息与数据信息通过**不同的物理/逻辑通道**分别传输，两者互不干扰。与之相对的是**带内（In-Band）传送**——控制信息与数据在同一通道中混合传输。

FTP 是**带外传送**的典型代表——控制命令通过独立的控制连接（端口 21）传输，文件数据通过数据连接（端口 20）传输。这个设计是 FTP 区别于大多数应用层协议的核心特征。

| 传送方式                |      通道划分       |     典型协议      | 说明                                                                                     |
| ------------------- | :-------------: | :-----------: | -------------------------------------------------------------------------------------- |
| **带外（Out-of-Band）** | 控制与数据**分离**不同连接 |  **FTP**、SSH  | FTP 控制命令走端口 21，数据走端口 20；SSH 控制信令和终端数据通过不同的 SSH 通道传输                                    |
| **带内（In-Band）**     | 控制与数据**混合**同一连接 | **HTTP**、SMTP | HTTP 的请求行/头部（控制信息）和实体主体（数据）在同一 TCP 连接中传输；SMTP 的 MAIL FROM / RCPT TO（控制）和邮件正文（数据）也在同一连接 |

> [!important] 408 考点：带外 vs 带内
> - **FTP = 带外**：控制连接和数据连接分离，控制信息"走外面"
> - **HTTP = 带内**：请求头/请求体在同一 TCP 连接中传送，控制信息与数据"走同一条路"
> - FTP 设计成带外的好处：**控制与数据互不干扰**——即使数据连接正传输大文件，控制连接仍能随时发送 ABOR 中止传输或发送其他命令

# FTP 的工作模式：主动模式与被动模式

FTP 数据连接的建立方式有两种——PORT（主动模式）和 PASV（被动模式），区别在于**谁发起数据连接**以及**数据连接的端口由谁指定**。

## 主动模式（PORT / Active Mode）

> **主动模式**：客户通过控制连接告知服务器自己监听的端口号，由**服务器主动**向客户发起数据连接。

**建立过程：**

| 步骤  | 发起方 | 操作                                                                                        |
| :-: | :-: | ----------------------------------------------------------------------------------------- |
|  ①  | 客户  | 通过控制连接发送 **PORT** 命令，携带自己的 IP 地址和一个临时端口号（如 `PORT 192,168,1,100,12,34` → 端口 = `12×256+34`） |
|  ②  | 服务器 | 收到 PORT 命令后，**从自己的 20 号端口**主动向客户指定的 IP:Port 发起 TCP 连接                                     |
|  ③  | 客户  | 在指定端口上被动等待并接受连接                                                                           |
|  ④  |  —  | 数据连接建立，开始传输数据                                                                             |
|  ⑤  |  —  | 传输完毕，服务器关闭数据连接                                                                            |

> [!warning] 主动模式的问题——防火墙
> 在主动模式下，数据连接由**外部服务器主动发起**连接到内部客户端，这在大多数客户端位于 **NAT/防火墙** 之后的现代网络环境中会**被防火墙拦截**。因此主动模式在现代网络中较少使用。

## 被动模式（PASV / Passive Mode）

> **被动模式**：客户通过控制连接发送 PASV 命令，服务器创建一个临时监听端口并将其告知客户，由**客户主动**向该端口发起数据连接。

**建立过程：**

| 步骤 | 发起方 | 操作 |
|:----:|:------:|------|
| ① | 客户 | 通过控制连接发送 **PASV** 命令 |
| ② | 服务器 | 在服务器端随机打开一个临时端口（如 30001），通过控制连接返回该端口号给客户（格式同 PORT） |
| ③ | 客户 | **自己主动**向服务器返回的端口发起 TCP 连接 |
| ④ | 服务器 | 在该临时端口上接受连接 |
| ⑤ | — | 数据连接建立，开始传输数据 |
| ⑥ | — | 传输完毕，服务器关闭数据连接 |

> [!tip] 被动模式的优势
> 被动模式中，数据连接由**客户主动发起**连接到服务器，符合 NAT/防火墙的"内部到外部"出站规则，通常不会被拦截。因此**现代 FTP 客户端默认使用被动模式**。

## 主动模式 vs 被动模式对比

| 对比维度 | 主动模式（PORT） | 被动模式（PASV） |
|----------|:--------------:|:--------------:|
| **数据连接发起方** | **服务器**主动连接客户 | **客户**主动连接服务器 |
| **服务器数据端口** | **固定 20** 号端口 | **随机**临时端口（>1023） |
| **客户要求** | 客户需开放端口等待服务器连接 | 客户只需出站连接（通常允许） |
| **防火墙友好性** | **差**——服务器→客户的连接常被防火墙拦截 | **好**——客户→服务器的连接符合常规出站规则 |
| **服务器端防火墙** | 需要开放 20 端口 | 需要开放大范围的临时端口（对服务器不友好） |
| **使用场景** | 早期网络，服务器和客户均无防火墙 | 现代网络环境，客户在 NAT/防火墙之后 |

> [!important] 408 考点：主动模式和被动模式的关键区别
> 1. PORT 模式：服务器用 **20 号端口**主动连接客户
> 2. PASV 模式：服务器开辟随机端口，**客户主动**连接该端口
> 3. **控制连接始终是客户发起的（21 端口）**——无论主动还是被动模式，这一点不变

# FTP 的传输模式

FTP 支持两种文件传输模式，由 TYPE 命令指定：

| 传输模式 | TYPE 命令 | 处理方式 | 适用文件类型 |
|----------|:--------:|----------|-------------|
| **ASCII 模式**（文本模式） | `TYPE A` | 传输时将文件按文本处理，自动转换**换行符**（如 Unix 的 `\n` ↔ Windows 的 `\r\n`） | 纯文本文件（.txt、.html、.c） |
| **二进制模式**（Binary / Image） | `TYPE I` | **逐位传输**，不做任何转换，原样复制每一个字节 | 图片、音频、视频、压缩包、可执行文件等所有非文本文件 |

> [!warning] 模式选错会导致文件损坏
> 用 ASCII 模式传输二进制文件会导致数据损坏——ASCII 模式会对字节进行换行符转换，破坏二进制文件的数据完整性。因此**传输非文本文件必须使用 Binary 模式**。

# 常见 FTP 命令

| 命令 | 全称 | 说明 |
|:----:|------|------|
| **USER** | User | 向服务器发送用户名 |
| **PASS** | Password | 向服务器发送密码 |
| **QUIT** | Quit | 注销并关闭控制连接 |
| **LIST** | List | 请求服务器返回当前目录的文件列表（通过数据连接） |
| **RETR** | Retrieve | 下载文件——从服务器复制一个文件到客户 |
| **STOR** | Store | 上传文件——从客户复制一个文件到服务器 |
| **DELE** | Delete | 删除服务器上的指定文件 |
| **PWD** | Print Working Directory | 显示当前工作目录 |
| **CWD** | Change Working Directory | 改变当前工作目录 |
| **PORT** | Port | 指定主动模式下的客户 IP 和端口 |
| **PASV** | Passive | 请求进入被动模式 |
| **TYPE** | Type | 指定传输模式（A = ASCII，I = Binary） |
| **ABOR** | Abort | 中止正在进行的文件传输 |
| **NOOP** | No Operation | 空操作，用于保持连接（防止超时断开） |

> [!info] FTP 使用 NVT ASCII
> FTP 控制连接上的命令和应答都是使用 **NVT ASCII**（Network Virtual Terminal ASCII）编码的文本，每条命令以 **CRLF（`\r\n`）** 结尾。这使得 FTP 的命令交互对人类可读，便于调试。

# FTP 应答码

FTP 服务器对每个命令返回一个三位数字的应答码，后面可附带可选的文字说明：

| 应答码范围 | 含义 | 示例 |
|:--------:|------|------|
| **1xx** | 初步应答——命令已接收，正在处理 | `150 File status okay; opening data connection.` |
| **2xx** | 完成应答——命令成功执行 | `200 Command okay.`<br>`227 Entering Passive Mode.` |
| **3xx** | 中间应答——命令已接受，还需后续操作 | `331 User name okay, need password.` |
| **4xx** | 暂时拒绝——命令未执行，可稍后重试 | `425 Can't open data connection.` |
| **5xx** | 永久拒绝——命令无法执行 | `500 Syntax error, command unrecognized.`<br>`530 Not logged in.` |

> [!tip] 常见 FTP 应答码速记
> | 应答码 | 含义 |
> |:------:|------|
> | `220` | 服务就绪 |
> | `230` | 登录成功 |
> | `331` | 用户名 OK，需要密码 |
> | `227` | 进入被动模式（返回 IP 和端口） |
> | `150` | 准备打开数据连接 |
> | `226` | 数据传输完成 |
> | `221` | 服务关闭（退出） |

# FTP vs TFTP

408 计网考试经常将 FTP 与 TFTP 对比考查：

| 对比维度 | FTP | TFTP（简单文件传送协议） |
|----------|:---:|:---:|
| **全称** | File Transfer Protocol | Trivial File Transfer Protocol |
| **传输层协议** | **TCP** | **UDP** |
| **端口号** | 21（控制）/ 20（数据） | **69** |
| **连接数** | **2 个**（控制 + 数据） | **无连接概念**（每个报文独立处理） |
| **可靠性** | TCP 提供可靠传输 | 由 **应用层自己**实现确认和重传（类似停止-等待协议） |
| **用户认证** | 支持（用户名 + 密码） | **不支持**（无认证机制） |
| **目录操作** | 支持（LIST、CWD、PWD 等） | **不支持** |
| **文件操作** | 丰富（上传/下载/删除/重命名等） | **仅支持文件传送**（上传和下载） |
| **交互性** | **强交互**（命令行交互式会话） | **非交互**（简单的请求-响应） |
| **传输模式** | ASCII / Binary | **仅二进制**（octet） |
| **数据连接** | 动态建立临时数据连接 | 无单独的数据连接，数据直接封装在 UDP 报文中 |
| **复杂度** | 高 | 低（代码量小，适合嵌入式设备） |
| **适用场景** | 通用文件传输、网站维护、文件管理 | 无盘工作站启动（PXE 引导）、配置文件备份、嵌入式系统固件升级 |

> [!important] TFTP 的关键特征（408 常考）
> 1. TFTP 使用 **UDP**，端口号 **69**
> 2. TFTP 的可靠性由**应用层自己实现**——使用确认与超时重传机制
> 3. TFTP 仅支持**文件上传和下载**，不支持目录列出、用户认证等交互功能
> 4. TFTP 报文块大小为 **512 字节**（除最后一块外，最后一块 < 512 表示传输结束）
> 5. TFTP 采用**停止-等待**（Stop-and-Wait）流量控制方式

# NFS（网络文件系统）

> **NFS（Network File System，网络文件系统）** ：由 Sun Microsystems 开发的**分布式文件系统协议**，允许客户端通过**挂载（Mount）** 的方式将远程服务器上的目录透明地接入本地文件系统，像访问本地文件一样访问远程文件。

NFS 的核心思路与 FTP 截然不同——FTP 是"复制"（先下载再操作），NFS 是"远程透明访问"（直接在远程目录上操作，无需下载整个文件）。

## NFS 的工作原理

NFS 采用**客户/服务器（C/S）模型**，允许同一目录被多个客户端同时挂载和共享：

| 步骤 | 操作 | 说明 |
|:----:|------|------|
| ① | 服务器**导出**目录 | 服务器将需要共享的目录配置为"导出"（Export），并设置访问权限 |
| ② | 客户端**挂载**目录 | 客户端使用 `mount` 命令将远程目录挂载到本地文件系统的一个挂载点 |
| ③ | **透明访问** | 挂载后，客户端中的应用程序可以像访问本地文件一样对远程文件进行读写（通过系统调用如 `read()`、`write()`） |
| ④ | **NFS 协议转发** | 客户端的文件操作请求被 Linux 内核中的 NFS 客户端模块拦截，转换为 NFS 协议请求通过网络发往服务器 |
| ⑤ | **服务器执行** | NFS 服务器收到请求后在本地文件系统执行操作，并将结果返回客户端 |

> [!tip] NFS 的"透明访问"是关键
> 对用户和应用程序来说，`/mnt/nfs_share/file.txt` 和 `/home/user/local_file.txt` 在使用上没有区别——都是通过 `open()`、`read()`、`write()` 等系统调用操作，底层是本地磁盘还是远程 NFS 服务器由操作系统内核透明处理。


FTP 和 NFS 都用于访问远程文件，但设计理念和应用场景完全不同：

| 对比维度       |             FTP              |                 NFS                 |
| ---------- | :--------------------------: | :---------------------------------: |
| **核心思路**   |  **复制**——文件从服务器完整传输到本地后再操作   | **透明访问**——像访问本地文件一样直接操作远程文件，不复制整个文件 |
| **文件位置**   | 传输后文件在**本地**（下载）或**服务器**（上传） |     文件始终存储在**服务器**上，客户端通过挂载点访问      |
| **访问粒度**   |   **整个文件**——传文件的最小单位是整个文件    |  **数据块/字节**——可以只读写文件的任意部分，无需传输整个文件  |
| **工作层次**   |     **应用层**——用户主动执行上传/下载     |    **操作系统层**——由内核文件系统驱动，对应用程序透明     |
| **使用方式**   |      FTP 客户端软件 + FTP 命令      |        `mount` 挂载 + 标准文件系统操作        |
| **断开网络影响** |    无法访问远程文件（但已下载的文件不受影响）     |       挂载点立即失效，正在读写的应用报 I/O 错误       |
| **并发访问**   |         通常单用户下载后本地操作         |     **多客户端可同时读写**同一共享目录（支持文件锁）      |
| **传输协议**   |      应用层 FTP 协议（基于 TCP）      |  RPC（远程过程调用），NFSv4 基于 TCP 端口 2049   |
| **认证机制**   |         用户名 + 密码（明文）         | 基于 IP / UID，NFSv4 支持 Kerberos 强身份认证 |
| **适用场景**   |      文件上传下载、网站部署、一次性传输       |      共享目录、无盘工作站、虚拟化镜像存储、集群共享存储      |

> [!important] 408 考点：FTP vs NFS 的核心区别
> 1. **FTP = 文件复制**——整个文件从服务器传到本地再操作，传输完毕后文件独立于服务器
> 2. **NFS = 透明访问**——文件始终在服务器上，客户端只读写所需的数据块，无需下载整个文件
> 3. NFS 修改文件时**直接作用于远程服务器上的原文件**，而 FTP 修改的是下载后在本地保存的**副本**

# 匿名 FTP

> **匿名 FTP（Anonymous FTP）**：一种不需账号和密码就能访问 FTP 服务器的机制。用户以 **"anonymous"** 为用户名、以**电子邮件地址**为密码登录，通常仅具备**下载权限**。

## 匿名 FTP 实现公用文件共享

匿名 FTP 是互联网早期实现**公用文件共享**（Public File Sharing）的基础机制。其核心思想是：服务器将部分文件目录设为**公开区域**，互联网上的任何用户都可以在不注册账号的情况下自由访问和下载这些文件。

**工作流程：**

| 步骤 | 操作 | 说明 |
|:----:|------|------|
| ① | 客户发起连接 | FTP 客户端连接服务器的 21 号端口（控制连接） |
| ② | 输入匿名账号 | 用户名输入 **"anonymous"**，密码输入**任意电子邮件地址**（如 `guest@example.com`） |
| ③ | 服务器验证 | 服务器接受 anonymous 登录，但将该用户限制在**特定的公开目录**中（通常为 `pub/` 目录） |
| ④ | 浏览下载 | 用户使用 LIST 查看目录、CWD 切换目录、RETR 下载文件（仅**只读权限**） |
| ⑤ | 结束会话 | 用户发送 QUIT 命令断开连接 |

> [!tip] 匿名 FTP 的历史地位
> 在 Web（HTTP）普及之前，匿名 FTP 曾是互联网上**最主要的公用文件分发方式**：
> - 各大高校和科研机构通过匿名 FTP 发布学术论文、研究成果
> - 开源软件社区（如 GNU 项目、Linux 内核早期版本）通过匿名 FTP 镜像站发布源代码
> - 硬件厂商通过匿名 FTP 发布驱动程序、固件更新和技术文档

## 公用文件共享的特点

| 特点 | 说明 |
|------|------|
| **无需注册** | 任何人只需知道 FTP 服务器地址即可访问公开文件，极大降低了使用门槛 |
| **只读共享** | 匿名用户默认只有**下载**和**浏览**权限，不能上传、删除或修改文件——保护服务器不被恶意破坏 |
| **目录隔离** | 服务器将匿名用户限制在指定目录（如 `pub/`），无法访问系统私密区域 |
| **全球可达** | 只要有互联网连接就能访问——是早期互联网"全球信息共享"理想的体现 |
| **镜像站网络** | 热门软件的匿名 FTP 服务器在全球设立**镜像站**（Mirror），分散负载、提升下载速度 |

## 匿名 FTP 的优缺点

|  | 优点 | 缺点 |
|------|------|------|
| **便利性** | 无需注册，即连即用，极大方便了临时和一次性用户 | 无法追踪用户行为——如果发生滥用，难以追溯责任 |
| **管理成本** | 无需为海量临时用户维护账号数据库 | 匿名用户的所有操作不受身份约束 |
| **传播效率** | 文件分发门槛低，有利于软件、文档的快速传播 | 可能被用于传播恶意软件或盗版内容 |
| **共享文化** | 促进了开源运动和互联网早期的知识共享精神 | 随着 HTTP/HTTPS 的普及，匿名 FTP 的流量逐渐被 Web 下载取代 |

> [!warning] 安全提示
> 1. 匿名 FTP 应仅授予**最小权限**（只读），防止恶意用户上传恶意文件或篡改现有文件
> 2. 匿名用户的目录应**严格隔离**于系统关键目录，防止越权访问
> 3. 管理员应定期审查匿名 FTP 目录中的内容，防止被滥用为"地下文件仓库"
> 4. 现代环境中，FTP 协议本身是**明文传输**（用户名、密码、文件内容均不加密），匿名 FTP 虽然无密码泄露风险，但传输的文件内容仍可被窃听。如需安全传输，应使用 **FTPS**（FTP over SSL/TLS）或 **SFTP**（SSH File Transfer Protocol）替代

## 匿名 FTP 与现代文件共享

随着互联网发展，匿名 FTP 的公用文件共享角色逐渐被替代，但其思想延续至今：

| 现代替代方案 | 说明 | 与匿名 FTP 的关系 |
|-------------|------|-------------------|
| **HTTP/HTTPS 文件下载** | Web 服务器提供文件直链下载，无需安装 FTP 客户端 | 功能上完全替代了匿名 FTP 的大多数场景 |
| **云存储共享链接** | 百度网盘、Google Drive、OneDrive 等生成分享链接 | 同样是"无需注册即可下载"的理念延续 |
| **BT / 磁力链接** | P2P 文件分发，用户间直接共享 | 进一步去中心化，无需中心服务器 |
| **CDN** | 内容分发网络在全球边缘节点缓存文件 | 继承匿名 FTP "镜像站"的分散负载思想

# ❗ 易错点总结

1. **FTP 使用两个 TCP 连接**——控制连接（端口 21）和数据连接（端口 20），这是 FTP 最核心的设计特征
2. **控制连接始终由客户发起**，在整个会话期间**一直保持**；数据连接是**临时**的，传输完毕立即关闭
3. **控制连接端口 21 保持不变**，无论主动模式还是被动模式
4. **主动模式（PORT）**：服务器用 **20 号端口**主动连接客户——防火墙不友好
5. **被动模式（PASV）**：客户主动连接服务器开放的**随机端口**——防火墙友好，现代默认模式
6. **数据连接在每次传输时重新建立**——一次 FTP 会话可以有多次数据连接（传多个文件）
7. **传输途中可以通过控制连接发送 ABOR 命令中止传输**——这正是"控制和数据分离"的优势
8. **FTP 使用 NVT ASCII** 格式传送控制信息（命令和应答），文件数据则按指定的传输模式处理
9. **ASCII 模式会转换换行符**，传输二进制文件必须使用 **Binary 模式**，否则文件损坏
10. **TFTP 使用 UDP（端口 69）**——与 FTP 完全不同！TFTP 的可靠性由应用层自己实现（停止-等待协议 + 超时重传）
11. **TFTP 的最后一个数据块 < 512 字节**表示传输结束；若文件大小恰好是 512 的整数倍，最后一个块为 0 字节
12. **TFTP 不支持认证和目录操作**——极简设计，仅支持上传和下载
13. **FTP 使用 C/S 模型**，服务器端口固定（21），客户端口随机
14. **控制连接和数据连接都使用 TCP**——FTP 不需要 UDP
15. **FTP 是带外（Out-of-Band）传送**——控制命令和数据走不同的连接；HTTP 是带内（In-Band）——控制信息和数据在同一连接中
16. **FTP 是有状态协议**——控制连接一直保持，服务器维护用户身份、当前目录、传输模式等会话状态
17. **NFS 的核心思路是"透明访问"**——挂载远程目录后像操作本地文件一样操作远程文件，无需下载整个文件；FTP 的核心思路是"复制"——整个文件传输到本地再操作
18. **NFSv2/v3 是无状态协议**（每个 RPC 请求独立），NFSv4 改为有状态；FTP 是有状态协议（控制连接一直保持）
19. **NFS 修改直接作用于远程原文件**——FTP 修改的是下载到本地的副本，如需更新服务器需重新上传
