# 域名系统概述

> **域名系统（Domain Name System，DNS）**：一种**分布式**的、**层次化**的命名系统，将人类可读的**域名**（如 `www.example.com`）映射为机器可路由的 **IP 地址**（如 `192.0.2.1`）

DNS 采用**客户/服务器模型**，协议运行在 UDP 上，使用 53 号端口

DNS 是互联网的核心基础设施之一——几乎所有的应用层协议（HTTP、FTP、SMTP 等）都依赖 DNS 将域名解析为 IP 地址后，才能进行通信。

---

# 层次域名空间

> **层次域名空间**：DNS 采用**层次树状结构**来组织域名，类似于操作系统的文件目录——各级域名之间用 `.` 分隔，从上到下依次为根、顶级域、二级域、三级域……

任何连接到互联网的主机或路由器都拥有唯一的层次结构名称，即**域名**（Domian Name）；**域**（Domain）是名字空间中一个可被管理的划分；域可进一步划分为子域

## 域名的层次结构

- 级别从左到右**升高**：`www`（三级）→ `example`（二级）→ `com`（顶级）→ `.`（根）

## 顶级域名分类

| 分类 | 说明 | 示例 |
|------|------|------|
| **国家/地区顶级域名（ccTLD）** | 以 ISO 3166 国家代码命名，由各国管理 | `.cn`（中国）、`.us`（美国）、`.jp`（日本）、`.uk`（英国） |
| **通用顶级域名（gTLD）** | 按组织类型划分，最初 7 个 | `.com`（商业）、`.org`（非营利）、`.net`（网络）、`.edu`（教育）、`.gov`（政府）、`.mil`（军事）、`.int`（国际组织） |
| **新通用顶级域名（new gTLD）** | 2012 年后 ICANN 批准新增 | `.xyz`、`.top`、`.app`、`.blog`、`.shop` |
| **基础结构域名（arpa）** | 用于反向域名解析 | `.arpa`（Address and Routing Parameter Area） |

## 域名的命名规则

| 规则 | 说明 |
|------|------|
| **字符集** | 英文字母（a-z，A-Z）、数字（0-9）、连字符（`-`） |
| **大小写** | **不区分**大小写（`Example.com` ≡ `example.com`） |
| **长度限制** | 每个标签（两个 `.` 之间的部分）不超过 **63 字符**，完整域名不超过 **255 字符** |
| **连字符限制** | `-` 不能出现在标签的开头或结尾 |
| **分级数量** | 总层级数不超过 **127 级** |

## 完全限定域名（FQDN）

> **完全限定域名（Fully Qualified Domain Name，FQDN）**：从主机名到根域的完整路径，以 `.` 结尾。

| 概念 | 示例 | 说明 |
|------|------|------|
| **FQDN** | `www.example.com.` | 末尾有 `.`（根），是域名的完整形式 |
| **相对域名** | `www.example.com` | 末尾无 `.`，相对当前域来解析 |
| **主机名** | `www` | 同一域内的主机标识 |

> [!info] 实战中的 FQDN
> 普通用户几乎从不输入末尾的 `.`，操作系统/浏览器会自动补全。但在 DNS 配置文件中（如 zone file），FQDN 末尾的 `.` 是**必须**的——漏写可能导致解析错误。

# 域名服务器

> **域名服务器（Name Server）**：存储域名空间中的域名信息、响应来自 DNS 客户端的查询请求的服务器。DNS 采用**分布式**架构，全球域名信息分散在各级域名服务器中，而非集中在一台服务器。

 一个服务器负责管辖（或具有管理权限）的范围称为**区**，其范围小于或等与**域**；一个区内的所有节点必须是连通的，每个区都设有相应的**权限域名服务器**（也称为*授权域名服务器*）用于保存该区中所有主机的域名到 IP 地址的映射信息；每个域名服务器不仅要能解析部分域名到 IP 地址的映射，还要能维护指向其他域名服务器的信息；当自身无法完成解析时，要能知道向哪个服务器进一步查询

## 域名服务器的层次

| 层次        | 服务器类型                                  | 存储内容                          |       数量        | 说明                                                 |
| --------- | -------------------------------------- | ----------------------------- | :-------------: | -------------------------------------------------- |
| **第 1 层** | 根域名服务器<br>（Root Name Server）           | 所有顶级域（TLD）的域名服务器的 IP 地址       | **13 组**（a ~ m） | DNS 查询的**最顶端入口**；13 组实为数百台镜像服务器（通过任播技术 Anycast 部署） |
| **第 2 层** | 顶级域名服务器<br>（TLD Name Server）           | 该顶级域下所有二级域的**权限域名服务器**的 IP 地址 |   按 TLD 数量分布    | 如 `.com` 服务器知道所有 `.com` 域的权限服务器地址                  |
| **第 3 层** | 权限域名服务器<br>（Authoritative Name Server） | 该域下所有主机的 IP 地址和 DNS 记录        |     每个域至少一台     | 能**直接返回**域名↔IP 的最终对应关系，是查询的"终点"                    |
| **辅助层**   | 本地域名服务器<br>（Local Name Server）         | **缓存**最近查询的结果                 | 每个 ISP / 机构至少一台 | 不属 DNS 层次结构，但承担绝大多数的实际查询                           |

> [!important] 根域名服务器
> 全球共有 **13 组**根域名服务器（编号 a ~ m，对应 `a.root-servers.net` ~ `m.root-servers.net`）。13 这个数字来源于 **UDP 报文长度限制**——DNS 早期使用 UDP 报文（512 字节），一个 DNS 响应报文最多容纳 13 个根服务器地址。现在通过 **任播（Anycast）** 技术，13 组对应全球**数百台**物理服务器。

> [!warning] 注意
> - **根域名服务器**不直接存储域名 ↔ IP 的映射，它只告诉你去**哪里**找顶级域名服务器
> - **顶级域名服务器**也不直接给出最终答案，它告诉你去**哪里**找权限域名服务器
> - **只有权限域名服务器**拥有某个域名的完整 DNS 记录，是解析的终点；**每台主机必须在所属区的权限服务器处登记，该服务器总能将其管辖的主机名解析为对应 IP 地址**

## 各服务器的具体功能

| 服务器 | 功能描述 |
|--------|----------|
| **根域名服务器** | 收到查询后，根据顶级域（如 `.com`）返回对应的顶级域名服务器的地址 |
| **顶级域名服务器** | 收到查询后，根据二级域（如 `example.com`）返回对应的权限域名服务器的地址 |
| **权限域名服务器** | 返回该域名对应的 DNS 记录（A 记录返 IP，MX 记录返回邮件服务器等） |
| **本地域名服务器** | 接收用户主机的 DNS 查询，代用户完成整个递归查询过程；缓存查询结果 |

> [!tip] 本地域名服务器的特殊地位
> 本地域名服务器不在 DNS 的正式层次结构中，但它在 DNS 解析过程中举足轻重：
> - 当用户主机发出 DNS 请求时，首先发给**本地域名服务器**
> - 本地域名服务器代理用户完成后续所有查询
> - 本地域名服务器通过**缓存**大幅减少网络中的 DNS 流量，提高响应速度

# 域名解析过程

> **域名解析**：将域名转换为对应 IP 地址的过程。DNS 采用**客户端-服务器**工作模式，由 DNS 客户端（通常为操作系统内置的 DNS 解析器）向 DNS 服务器发送查询请求。

## 两种查询方式

| 查询方式 | 谁发起 | 被查询服务器的行为 | 特点 |
|----------|--------|-------------------|------|
| **递归查询（Recursive Query）** | 客户 → 本地域名服务器 | 服务器必须返回**最终结果**（IP 地址或报错），不能只给中间参考；如果服务器不知道答案，它**替客户去查** | 对客户简单，但对服务器要求高——服务器承担全部查询工作 |
| **迭代查询（Iterative Query）** | 本地域名服务器 → 各级服务器 | 服务器返回**最佳已知信息**（如"我不知道，但你可以去问 XXX"），由查询方自己继续下一步查询 | 对服务器友好，但对查询方要求高——查询方需持续跟踪查询进度 |
![[两种域名解析工作方式.png]]
> [!important] 递归 vs 迭代——关键区别
> |  | 递归查询 | 迭代查询 |
> |------|:---:|:---:|
> | **返回内容** | 最终结果（IP 地址或报错） | 下一步要找的服务器地址（近似答案） |
> | **查询负担** | **被查询方**承担 | **查询方**承担 |
> | **典型场景** | 用户主机 → 本地域名服务器 | 本地域名服务器 → 根 / TLD / 权限服务器 |

## 完整的域名解析过程

本地域名服务器向根域名服务器的查询通常采用**迭代查询**方式，以用户在浏览器中输入 `www.example.com` 为例：

| 步骤 | 查询方 | 被查询方 | 查询方式 | 查询内容 | 返回内容 |
|:----:|--------|----------|:--------:|----------|----------|
| ① | 用户主机 | 本地域名服务器 | **递归** | `www.example.com` 的 IP | （最终会返回 IP） |
| ② | 本地域名服务器 | 根域名服务器 | **迭代** | 谁负责 `.com` 域？ | `.com` TLD 服务器的地址 |
| ③ | 本地域名服务器 | `.com` TLD 服务器 | **迭代** | 谁负责 `example.com` 域？ | `example.com` 权限服务器的地址 |
| ④ | 本地域名服务器 | `example.com` 权限服务器 | **迭代** | `www.example.com` 的 IP？ | `www.example.com` 的 IP 地址 |
| ⑤ | 本地域名服务器 | 用户主机 | 返回结果 | — | 最终的 IP 地址 |

> [!info] 涉及几个 UDP 报文？
> 上述完整过程共涉及 **4 次 DNS 请求/响应交换**，每次交换包含 1 个请求报文 + 1 个响应报文，因此：
> 
> $$\text{总 UDP 报文数} = 4 \text{ 次交换} \times 2 \text{ 报/次} = 8 \text{ 个 UDP 报文}$$
> 
> | 交换 | 请求方 → 响应方 | 报文 |
> |:----:|-----------------|:----:|
> | 1 | 主机 ↔ 本地域名服务器 | 2 个（① 的请求 + ⑤ 的响应） |
> | 2 | 本地域名服务器 ↔ 根域名服务器 | 2 个（② 请求 + ② 响应） |
> | 3 | 本地域名服务器 ↔ TLD 服务器 | 2 个（③ 请求 + ③ 响应） |
> | 4 | 本地域名服务器 ↔ 权限域名服务器 | 2 个（④ 请求 + ④ 响应） |
> | **合计** | | **8 个 UDP 报文** |
> 
> 即整个解析过程**至少**产生 8 个 UDP 报文。若考虑**缓存命中**的情况，部分步骤可省略——如本地域名服务器缓存了 `example.com` 的权限服务器地址，则跳过与根服务器的交换，减少 2 个 UDP 报文。

> [!warning] 注意！
> 这是一个理论上的"最完整"解析过程。实际情况中：
> 1. **几乎不会发生**全流程：本地域名服务器绝大多数查询都命中**缓存**，无需走完整个链路
> 2. 步骤②~④ 的迭代查询只在**缓存未命中**时才触发
> 3. 补充：获取 IP 地址的顺序并不是上面梳理的顺序，而是按照**主机 → 本地域名服务器 → 根域名服务器 → 顶级域名服务器 → 权限域名服务器**的层次顺序向下递归查找。

## DNS 缓存

> **DNS 缓存**：域名服务器在完成一次查询后，将解析结果临时存储一段时间（TTL），以便后续相同查询可以直接返回缓存结果，无需重新发起查询。

| 缓存类型 | 存储位置 | 说明 |
|----------|----------|------|
| **浏览器缓存** | 浏览器进程 | 浏览器自带 DNS 缓存，命中率较高（Chrome 默认缓存 1 分钟） |
| **操作系统缓存** | 操作系统 DNS 解析器 | 系统级 DNS 缓存（Windows: `ipconfig /flushdns` 清除） |
| **本地域名服务器缓存** | ISP 的本地域名服务器 | **最重要的缓存层**——绝大多数查询在此命中 |
| **递归过程中的缓存** | 各级 DNS 服务器 | 根服务器、TLD 服务器也能缓存信息 |

> [!tip] TTL（Time To Live）
> TTL 是 DNS 记录中的一个字段，指定缓存的有效时间（单位：秒）。TTL 到期后缓存失效，下次查询需重新解析。
> - TTL 较长：减少 DNS 流量，提高响应速度；但 DNS 记录更新后客户端需等 TTL 过期才能获取新结果
> - TTL 较短：记录更新能快速生效；但增加 DNS 查询量，降低性能

## 常见 DNS 记录类型

| 记录类型 | 全称 | 作用 | 示例 |
|:-------:|------|------|------|
| **A** | Address | 域名 → **IPv4** 地址 | `www.example.com.  A  192.0.2.1` |
| **AAAA** | IPv6 Address | 域名 → **IPv6** 地址 | `www.example.com.  AAAA  2001:db8::1` |
| **CNAME** | Canonical Name | 域名的**别名**（将域名指向另一个域名） | `blog.example.com.  CNAME  www.example.com.` |
| **NS** | Name Server | 指定该域的**权限域名服务器** | `example.com.  NS  ns1.example.com.` |
| **MX** | Mail Exchange | 指定该域的**邮件服务器**及其优先级 | `example.com.  MX  10  mail.example.com.` |
| **PTR** | Pointer | **反向解析**——IP 地址 → 域名（用于 `.arpa` 域） | `1.2.0.192.in-addr.arpa.  PTR  www.example.com.` |
| **SOA** | Start of Authority | 域的管理信息（主服务器、管理员邮箱、序列号等） | `example.com.  SOA  ns1.example.com.  admin.example.com.  ...` |
| **TXT** | Text | 存放任意文本信息（常用于验证域名所有权、SPF 反垃圾邮件等） | `example.com.  TXT  "v=spf1 include:_spf.google.com ~all"` |

> [!info] CNAME vs A
> - **A 记录**：域名 → IP 地址（终点是 IP）
> - **CNAME 记录**：域名 → 另一个域名（终点是另一个域名，需要继续解析）
> - CNAME 的优点是：当多个子域名指向同一台服务器时，只需修改目标域名的一条 A 记录即可全部生效

> [!warning] CNAME 的重要限制
> - 一个已经存在 CNAME 记录的名称，**不能再有其他任何类型的记录**（不能同时有 CNAME 和 A 记录）
> - **根域名**（如 `example.com`）不能使用 CNAME——因为根域的 SOA 和 NS 记录与 CNAME 冲突

## DNS 协议细节

| 协议细节 | 说明 |
|----------|------|
| **传输层协议** | 主要使用 **UDP**，端口号 **53** |
| **TCP 的使用场景** | ① 响应报文超过 **512 字节**时（UDP 限制）改用 TCP；② **区域传送（Zone Transfer）** 必须使用 TCP |
| **报文格式** | 固定的 12 字节首部 + 可变长的查询/回答/授权/附加信息四个部分 |
| **DNS 报文 ID** | 请求与响应使用相同的 **16 位标识符**，用于匹配查询和响应 |

> [!tip] DNS 为什么用 UDP？
> 1. **低延迟**：DNS 查询通常很小（域名 + 一个 IP 地址），TCP 三次握手开销太大
> 2. **简单性**：DNS 是无状态查询，UDP 的一发一收足够
> 3. **自修复**：如果 UDP 丢包，DNS 客户端只需**超时重传**即可，无需复杂的连接管理

> [!warning] 区域传送
> **区域传送（Zone Transfer）** 是主 DNS 服务器将完整的区域数据（zone data）复制到从 DNS 服务器的过程。由于数据量大、对可靠性要求高，区域传送**必须使用 TCP**。攻击者可能利用未限制的区域传送功能获取域名的完整信息——这被称为 **DNS 区域传送漏洞**。

# DNS 的安全问题

| 安全威胁 | 说明 | 防御措施 |
|----------|------|----------|
| **DNS 缓存投毒（DNS Cache Poisoning）** | 攻击者伪造 DNS 响应，将错误的 IP 地址写入 DNS 服务器的缓存，使用户被**重定向到恶意网站** | DNSSEC |
| **DNS 劫持（DNS Hijacking）** | 攻击者篡改用户 DNS 配置或 DNS 服务器的记录，将流量引导至非法服务器 | 使用可信的 DNS 服务器、加密 DNS |
| **DNS 放大攻击（DNS Amplification）** | 攻击者伪造源 IP 向开放的 DNS 服务器发送小查询，使服务器向受害者回复大响应，放大攻击流量 | 关闭开放 DNS、限制递归查询 |
| **中间人攻击** | 攻击者拦截 DNS 查询报文并篡改 | DNS over HTTPS（DoH）、DNS over TLS（DoT） |

> [!info] DNSSEC
> **DNSSEC（Domain Name System Security Extensions）** 通过**数字签名**来验证 DNS 数据的完整性和真实性，防止缓存投毒和伪造攻击。DNSSEC 使用公钥加密体系——各级域的 DNS 记录由上级域逐级签名，形成从根到目标域的**信任链**。

# ❗ 易错点总结

1. **DNS 是分布式系统**——全球域名信息分散在各级服务器，**不是一台服务器存储所有域名**
2. **域名层次**：从右向左级别升高——`www.example.com.` 中 `.`（根）→ `com`（顶级）→ `example`（二级）→ `www`（三级）
3. **根域名服务器 13 组**（而非 13 台）——13 来源于 UDP 报文 512 字节限制，实际上通过 Anycast 技术有**数百台**物理服务器
4. **根域名服务器不存储域名↔IP 映射**，只存储**顶级域名服务器的 IP 地址**
5. **权限域名服务器**才是 DNS 查询的**终点**——它拥有域名的完整 DNS 记录
6. **本地域名服务器不属于 DNS 层次结构**，但对实际查询至关重要——用户主机的第一跳
7. **递归查询 vs 迭代查询**：① 用户主机→本地域名服务器 是**递归**（返回最终结果）；② 本地域名服务器→各级服务器 是**迭代**（逐步缩小范围）
8. **DNS 缓存**是 DNS 高效运行的基石——绝大多数查询在本地域名服务器的缓存中命中，根本不到达根/TLD 服务器
9. **DNS 主要用 UDP（53 端口）**——但区域传送和超大响应**必须用 TCP**
10. **CNAME 不能与其他记录共存**（不能同时有 CNAME 和 A 记录），且**根域名不能使用 CNAME**
11. **TTL 决定缓存时间**——TTL 长则减少查询量但更新慢，TTL 短则更新快但查询量增大
12. **FQDN** 末尾必须有 `.`（根域标记），日常使用的域名省略了此点
13. **DNS 使用 C/S 模型**——DNS 客户端（解析器）主动向 DNS 服务器发送查询请求
