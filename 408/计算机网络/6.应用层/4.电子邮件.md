# 电子邮件系统的组成结构

> **电子邮件（E-mail）**：互联网上最广泛使用的应用之一，允许用户通过计算机网络交换消息。电子邮件系统采用**客户/服务器（C/S）模型**，由多个组件协同工作完成邮件的撰写、发送、传输、接收和读取。

一个完整的电子邮件系统由以下三部分组成：

| 组件 | 英文 | 角色 | 说明 |
|------|------|------|------|
| **用户代理** | User Agent，UA | 客户 | 用户与邮件系统的**接口**——编写、发送、读取、管理邮件 |
| **邮件服务器** | Mail Server | 服务器 | 邮件的**中转站**——接收、存储、转发邮件；同时充当客户和服务器角色 |
| **邮件协议** | Mail Protocols | 规则 | 定义邮件在 UA、邮件服务器之间如何传输——包括**发送协议**（SMTP）和**读取协议**（POP3、IMAP） |

## 电子邮件系统的信息流

一封邮件的完整旅程：

| 步骤  | 发送方      |          使用协议          | 接收方      | 操作                                         |
| :-: | -------- | :--------------------: | -------- | ------------------------------------------ |
|  ①  | 发件人 UA   |           —            | （用户操作）   | 撰写邮件                                       |
|  ②  | 发件人 UA   |        **SMTP**        | 发件人邮件服务器 | 将邮件**推送**到发件人的邮件服务器                        |
|  ③  | 发件人邮件服务器 |        **SMTP**        | 收件人邮件服务器 | 邮件服务器之间建立 **TCP** 连接，**中继转发**（可能经过多个中间服务器） |
|  ④  | 收件人邮件服务器 |           —            | （存储）     | 邮件存入收件人的**邮箱**中                            |
|  ⑤  | 收件人 UA   | **POP3 / IMAP / HTTP** | 收件人邮件服务器 | 收件人从邮件服务器**拉取**邮件到本地                       |

> [!important] 发送与接收使用不同协议
> - **发送邮件**：使用 **SMTP**——从 UA 到服务器、服务器到服务器，都是 SMTP
> - **接收/读取邮件**：使用 **POP3** 或 **IMAP**——从邮件服务器拉取到本地 UA
> - 这个"发用 SMTP，收用 POP3/IMAP"的架构是 408 常考的区分点

## 各组件详解

### 1. 用户代理（UA）

> **用户代理（User Agent）**：用户与电子邮件系统的**接口**——通常是运行在用户设备上的客户端软件（如 Outlook、Foxmail、Thunderbird）或 Web 浏览器中的邮件界面（如 Gmail、QQ 邮箱网页版）。

UA 应具备的核心功能：

| 功能 | 说明 |
|------|------|
| **撰写邮件** | 提供编辑器或模板，方便用户编写邮件内容 |
| **显示/读取邮件** | 以可读方式呈现邮件内容（文本、图片、附件等） |
| **发送邮件** | 通过 SMTP 协议将邮件提交到发件人邮件服务器 |
| **接收邮件** | 通过 POP3、IMAP 或 HTTP 协议从邮件服务器获取邮件 |
| **管理邮件** | 分类、归档、标记、删除、搜索、联系人管理等 |

> [!info] 用户代理的两种形式
> | 类型 | 代表 | 特点 |
> |------|------|------|
> | **客户端软件** | Outlook、Foxmail、Thunderbird | 需安装，功能较强，邮件可离线查看 |
> | **Web 界面** | Gmail、QQ 邮箱网页版 | 无需安装，跨平台，依赖网络，邮件存储在服务器端 |

### 2. 邮件服务器

> **邮件服务器（Mail Server）**：电子邮件系统的**核心**——承担邮件的接收、存储、转发功能。每个邮件服务器同时扮演**客户**和**服务器**双重角色：
> - 作为**服务器**：接收来自 UA 或其他邮件服务器发来的邮件
> - 作为**客户**：将邮件发送到下一个邮件服务器（中继转发）

邮件服务器的组成：

| 组件 | 英文 | 功能 |
|------|------|------|
| **邮件传输代理** | MTA（Mail Transfer Agent） | 负责邮件的**路由和转发**——使用 SMTP 在服务器之间传输邮件（如 Sendmail、Postfix、Exchange） |
| **邮件投递代理** | MDA（Mail Delivery Agent） | 负责将邮件**投递**到收件人的邮箱中（如 `procmail`、`maildrop`） |

邮件服务器的核心流程：

| 步骤 | 邮件服务器角色 | 操作 |
|:----:|:------------:|------|
| ① | **服务器** | 使用 SMTP 接收来自发件人 UA 的邮件 |
| ② | **客户** | 使用 SMTP 将邮件发送到收件人邮件服务器（可能经过多个中间服务器接力） |
| ③ | **服务器** | 使用 SMTP 接收来自发件人邮件服务器的邮件 |
| ④ | **内部** | MDA 将邮件投递到收件人的邮箱中（磁盘上的文件夹） |
| ⑤ | **服务器** | 等待收件人 UA 通过 POP3 或 IMAP 拉取邮件 |

> [!important] 邮件服务器的两个端口
> - **25 号端口**：SMTP 服务器之间传输邮件（服务器到服务器）
> - **110 号端口**（POP3）/ **143 号端口**（IMAP）：用户 UA 接收邮件

### 3. 电子邮件使用的协议

| 协议       | 全称                               | 作用                      |          方向           | 传输层 |  默认端口   |
| -------- | -------------------------------- | ----------------------- | :-------------------: | :-: | :-----: |
| **SMTP** | Simple Mail Transfer Protocol    | 发送邮件 + 服务器间中继           | UA → 服务器<br>服务器 → 服务器 | TCP | **25**  |
| **POP3** | Post Office Protocol v3          | 从服务器**下载**邮件到本地         |       UA ← 服务器        | TCP | **110** |
| **IMAP** | Internet Message Access Protocol | 在服务器上**管理**邮件（在线阅读、分类等） |       UA ↔ 服务器        | TCP | **143** |

> [!tip] 发送和接收使用不同协议的原因
> SMTP 是一种**推送（Push）协议**——发件方主动将邮件推向下一跳。POP3/IMAP 是**拉取（Pull）协议**——收件方主动从服务器拉取邮件。这两者的设计目标不同：SMTP 要求高效、可靠的邮件中继能力，而 POP3/IMAP 侧重于用户对邮件的管理和灵活的访问方式。

# 电子邮件的格式

> 电子邮件的格式由 **RFC 5322**（原 RFC 822）和 **RFC 2045 ~ 2049（MIME）** 共同定义。RFC 5322 定义了基本的文本邮件格式，MIME 将格式扩展到支持多媒体内容。

## 电子邮件地址的格式

> **电子邮件地址**格式固定为 `local-part@domain`——`@` 左边是**用户名**，右边是**邮件服务器域名**。

- **local-part**：该服务器上的用户名，同一服务器内必须唯一；大多数服务器不区分大小写
- **domain**：邮件服务器的 DNS 域名，大小写不敏感；SMTP 通过查询 DNS 的 **MX 记录**获取该域的邮件服务器地址（无 MX 记录时回退到 A 记录）

## 传统邮件格式（RFC 5322）

一封传统的电子邮件由两大部分组成：

| 组成部分 | 内容 | 说明 |
|:--------:|------|------|
| **信封** | 发件人地址、收件人地址 | 由 MTA 使用，用于邮件的**路由和投递**——用户看不见 |
| **内容** | 头部 + 主体 | 用户可见的邮件内容 |

信封和内容的关系类似于传统纸质邮件——信封告诉邮递员往哪儿送，内容（信纸）是收件人真正阅读的东西。

## 邮件内容的结构

邮件内容 = **邮件头部**（Header） + **空白行（CRLF）** + **邮件主体**（Body）

### 邮件头部（Header）

邮件头部由若干 **关键字段** 组成，每个字段一行，格式为 `字段名: 值`：

| 头部字段            | 全称                | 必须？ | 说明                                |
| --------------- | ----------------- | :-: | --------------------------------- |
| **To:**         |                   |  ✅  | 邮件接收者的地址（可多个，逗号分隔）                |
| **From:**       |                   |  ✅  | 邮件发送者的地址，通常由邮件系统自动填充              |
| **Subject:**    |                   |  否  | 邮件的主题/标题                          |
| **Date:**       |                   |  否  | 邮件的发送日期和时间                        |
| **Cc:**         | Carbon Copy       |  否  | **抄送**——收件人以外的其他接收者，所有收件人都能看到抄送对象 |
| **Bcc:**        | Blind Carbon Copy |  否  | **密送**——收件人看不到密送对象（密送对象之间也互相看不到）  |
| **Reply-To:**   |                   |  否  | 回复邮件的目标地址（若与 From 不同）             |
| **Message-ID:** |                   |  否  | 邮件的唯一标识符                          |
| **Received:**   |                   |  否  | 每经过一个 MTA 时添加——记录邮件的传输路径，用于追踪和排错  |

> [!info] Cc 与 Bcc 的关键区别
> |  | Cc（抄送） | Bcc（密送） |
> |------|:---:|:---:|
> | **To 收件人能看到？** | ✅ 能看到 | ❌ 看不到（隐藏） |
> | **Cc 中其他人能看到？** | ✅ 能看到 | ❌ 看不到 |
> | **用途** | 公开告知相关人 | 私下通知特定人 |

### 邮件主体（Body）

RFC 5322 最初规定邮件主体**只能包含 ASCII 文本字符**（7 位编码）。这意味着原始邮件协议**不支持非 ASCII 字符**（如中文、日文等）和**多媒体内容**（图片、音频、视频、附件等）。

> [!warning] RFC 5322 邮件的核心限制
> 1. **仅支持 7 位 ASCII 字符**——无法发送中文、日文、韩文、德文变音字母等
> 2. **不支持多媒体**——无法在邮件中嵌入图片、音频、视频或任何二进制数据
> 3. **附件功能缺失**——无法随邮件发送非文本文件（如 PDF、ZIP、图片）

# MIME（多用途互联网邮件扩展）

## MIME 的功能

RFC 5322 邮件只能传 **7 位 ASCII**，无法发送中文、图片、音频、附件等。**MIME** 就是为了解决这些问题——使邮件能够支持**非 ASCII 文本**、**多媒体内容**和**附件**。

MIME **没有修改** SMTP 或 RFC 5322，而是在邮件头部**新增字段**来描述内容，并通过**编码转换**把非 ASCII 数据转为 7 位 ASCII，SMTP 照常传输即可；接收端接收邮件后，再通过 MIME 对 ASCII 数据进行**解码还原**，进而恢复原始的非 ASCII 内容

> 核心思路：把"邮件内容是什么、怎么编码的"写进头部，SMTP 不用改，只负责传输。

## 主要内容

### 1. 新增头部字段

| 头部字段 | 作用 |
|----------|------|
| **MIME-Version:** | MIME 版本号，固定为 `1.0` |
| **Content-Description:** | 对邮件内容的简要文字描述（可选） |
| **Content-ID:** | 内容的唯一标识符（用于 HTML 邮件中引用内嵌图片等） |
| **Content-Transfer-Encoding:** | 说明内容**编码方式**（如何转为 7 位 ASCII） |
| **Content-Type:** | 说明内容**类型**（如 `text/plain`、`image/png`、`multipart/mixed`） |

### 2. 标准化了多种邮件内容的格式（Content-Type）

| 顶层类型 | 说明 | 子类型示例 |
|:--------:|------|------------|
| **text** | 文本 | `text/plain`、`text/html` |
| **image** | 图像 | `image/png`、`image/jpeg` |
| **audio** | 音频 | `audio/mpeg` |
| **video** | 视频 | `video/mp4` |
| **application** | 应用数据/二进制 | `application/pdf`、`application/octet-stream` |
| **multipart** | 多个独立部分 | `multipart/mixed`（正文+附件）、`multipart/alternative`（纯文本+HTML） |

### 3. 定义了传送编码机制（Content-Transfer-Encoding）

| 编码方式 | 原理 | 适用场景 |
|----------|------|----------|
| **7bit** | 本身就是 ASCII，不编码 | 纯英文 |
| **Quoted-Printable** | ASCII 原样保留，非 ASCII 字节转为 `=XX` | 少量非 ASCII 的文本 |
| **Base64** | 每 3 字节 → 4 个 ASCII 字符（64 个可打印字符映射） | **二进制附件** |

> [!important] Base64 膨胀率
> 3 字节 → 4 字符，大小增长约 **33%**（$\frac{4}{3}$ 倍）。

# SMTP

> **SMTP（Simple Mail Transfer Protocol，简单邮件传输协议）**：互联网电子邮件系统中**发送**和中继邮件的标准协议。SMTP 是电子邮件系统的核心传输协议——所有邮件从发件人到收件人走过的每一跳，都由 SMTP 承载。

| 属性 | 说明 |
|------|------|
| **应用模型** | 客户/服务器（C/S） |
| **传输层协议** | **TCP** |
| **端口号** | **25**（默认） |
| **邮件方向** | 发送 + 中继转发 |
| **通信模式** | "一发一收"——客户发一条命令，服务器回一个应答 |
| **命令与应答格式** | NVT ASCII 文本 |
| **状态特性** | **有状态**——一次 SMTP 会话中，服务器按顺序跟踪对话状态 |

## SMTP 的通信阶段

一次完整的 SMTP 会话分为三个紧密连接的阶段：

### 阶段一：连接建立

| 步骤 | 操作 | 说明 |
|:----:|------|------|
| ① | TCP 三次握手 | 发件方 SMTP 客户与收件方 SMTP 服务器的 **25 号端口**建立 TCP 连接 |
| ② | 服务器就绪 | 服务器发送 `220` 应答码，表示 SMTP 服务就绪，可以开始会话 |
| ③ | 客户 HELO / EHLO | 客户发送 `HELO`（或扩展的 `EHLO`）标识自己，服务器确认 |

>[!note] SMTP 的端到端直接交付
>与传统的"存储转发"不同，SMTP 采用**端到端直接交付**——发件人 SMTP 客户直接与收件人 SMTP 服务器建立 TCP 连接，将邮件一次性送达，**中间不需要任何邮件服务器暂存或中转**。即便跨越不同网络或地域，也是由发件人服务器直接向收件人服务器发起连接。

### 阶段二：邮件传送


| 步骤  |         命令          |    方向    | 说明                       |      服务器应答       |
| :-: | :-----------------: | :------: | ------------------------ | :--------------: |
|  ①  | `MAIL FROM:<发件人地址>` | 客户 → 服务器 | 告知服务器发件人地址               | 准备好则返回 `250 OK`  |
|  ②  |  `RCPT TO:<收件人地址>`  | 客户 → 服务器 | 告知服务器收件人地址（可重复多次指定多个收件人） | 准备好则返回 `250 0K`  |
|  ③  |       `DATA`        | 客户 → 服务器 | 请求开始传送邮件内容；收到 `354` 应答后，**逐行发送**邮件内容（头部 + 空白行 + 主体） | `354`——请开始输入邮件内容 |
|  ④  | `<CRLF>.<CRLF>`（单独一行中的句点） | 客户 → 服务器 | **邮件内容结束标志**——发送完毕，服务器确认接收 | `250 OK`——邮件接收成功 |

>`<CRLF>` 表示回车换行符

>[!note] RCPT 命令的作用
>是预先验证收件人地址的有效性，确保接收端系统已做好接收准备，进而避免在发送长篇邮件后才发现地址错误，造成通信资源浪费

### 阶段三：连接释放

| 步骤 | 命令 | 说明 |
|:----:|:----:|------|
| ① | `QUIT` | 客户发送 QUIT 命令，请求关闭会话 |
| ② | `221` | 服务器响应 221 应答码，关闭 TCP 连接 |

> [!tip] 阶段标题是笔者用来自洽解释模型的沟通工具——它们在逻辑上清楚地区分了 SMTP 会话的三个阶段,但本身**并不是 SMTP 协议的正式术语**。

## SMTP 命令详解

| 命令 | 全称/含义 | 说明 |
|:----:|-----------|------|
| **HELO** | Hello | 发起会话并标识客户（传统版本） |
| **EHLO** | Extended Hello | 发起会话并标识客户（**扩展版**——检查服务器是否支持 ESMTP 扩展功能） |
| **MAIL FROM** | Mail From | 指定发件人地址——对应邮件信封上的**反向路径**（Return-Path），退信时使用 |
| **RCPT TO** | Recipient To | 指定收件人地址——对应邮件信封上的**正向路径**，服务器据此路由邮件；可执行多次，每次指定一个收件人 |
| **DATA** | Data | 请求开始传输邮件内容（头部 + 主体）；服务器会回 `354` 应答，告知客户如何结束邮件 |
| **RSET** | Reset | 重置会话——取消当前的 MAIL FROM / RCPT TO / DATA 操作，回到初始状态；保留 TCP 连接不会被断开 |
| **VRFY** | Verify | 验证收件人邮箱是否存在（出于安全原因，现代服务器几乎禁用此命令） |
| **NOOP** | No Operation | 空操作——用于保持连接活跃（防超时断开），或测试服务器是否可响应 |
| **QUIT** | Quit | 结束会话，关闭 TCP 连接 |

> [!warning] `MAIL FROM` 信封 vs `From:` 头部（408 常考）
> SMTP 的 `MAIL FROM` 命令和邮件内容中的 `From:` 头部是**两个不同的东西**：
> 
> | | `MAIL FROM`（信封） | `From:` 头部 |
> |------|:---:|:---:|
> | **位置** | SMTP 命令（协议层） | 邮件头部（内容层） |
> | **用途** | 退信路径（bounce）——投递失败时原路返回 | 显示给收件人看的"发件人" |
> | **可达性要求** | 必须能接收退信 | 可以不指向真实邮箱 |
> | **用户可见？** | ❌ 不可见——在 DATA 命令之前发生 | ✅ 可见——邮件内容的 `From:` 行 |
> | **可被伪造？** | 难——MTA 可验证信封地址 | 易——垃圾邮件常伪造 `From:` 头部 |
> 
> 这种分离是**垃圾邮件伪造**的技术根源——垃圾邮件发送者可以在 `MAIL FROM` 中填入真实地址（自己收退信），同时在 `From:` 头部伪造为银行或知名公司地址以欺骗收件人。

## SMTP 应答码

SMTP 服务器用三位数字应答码（与 FTP 类似）：

| 应答码范围 | 含义 | 示例 |
|:--------:|------|------|
| **2xx** | 成功 | `250 OK`——命令成功执行 |
| **3xx** | 中间步骤——继续 | `354 Start mail input; end with <CRLF>.<CRLF>` |
| **4xx** | 暂时错误——可重试 | `450 Mailbox unavailable` |
| **5xx** | 永久错误——放弃 | `550 No such user` |

| 常见应答码 | 含义 |
|:--------:|------|
| `220` | SMTP 服务就绪 |
| `250` | 命令成功执行 |
| `354` | 请开始输入邮件内容，以 `<CRLF>.<CRLF>` 结束 |
| `221` | 服务关闭（退出） |
| `550` | 收件人不存在或邮箱不可达 |

## SMTP 的关键特征

| 特征 | 说明 |
|------|------|
| **基于 TCP** | 提供可靠的、面向连接的传输 |
| **NVT ASCII** | 命令、应答和古早邮件内容都使用 NVT ASCII（7 位编码） |
| **有状态协议** | 一次会话依次经过连接建立 → 邮件传送 → 连接释放三个阶段，服务器维护会话的状态信息 |
| **命令/应答模式** | 客户发一条命令，服务器回一个应答——交互过程同步且有序 |
| **直接交付** | 发件人邮件服务器与收件人邮件服务器**直接建立 TCP 连接**传送邮件，**不使用**中间邮件服务器进行中转 |

## SMTP 的优缺点

|  | 内容 |
|------|------|
| **优点** | ① 简单、成熟、可靠；② 基于 TCP 提供端到端的可靠传输；③ 使用 NVT ASCII 命令和应答——人类可读，易于调试；④ 支持 ESMTP 扩展（如 AUTH 认证、STARTTLS 加密、8BITMIME 等），功能持续完善 |
| **缺点** | ① **仅支持 7 位 ASCII**——不能直接传输非 ASCII 字符和二进制文件（MIME 的出现正是为了弥补这个不足）；② **明文传输**——默认不加密，用户名/密码/邮件内容都可被窃听；③ **发件人地址可伪造**——SMTP 本身没有发送方身份验证机制（需 SPF/DKIM/DMARC 来弥补）；④ **垃圾邮件泛滥**——SMTP 协议设计之初未考虑反垃圾邮件，导致垃圾邮件治理成本居高不下 |

# POP3 和 IMAP

邮件到达收件人邮件服务器后，收件人需要取出邮件。主要有两种方式：**POP3**（拉取下载）和 **IMAP**（在线阅读管理）。

## POP3

> 基于 **TCP**，端口 **110**。将邮件**下载到本地**，默认下载后从服务器删除。仅支持收件箱（INBOX），不支持文件夹管理。

**工作阶段：**

|     阶段     | 操作                        | 核心命令                                   |
| :--------: | ------------------------- | -------------------------------------- |
|  **特许阶段**  | 用户认证（用户名和口令）              | `USER` → `PASS`                        |
| **事务处理阶段** | 获取邮件列表、下载、标记删除            | `STAT`、`LIST`、`RETR`、`DELE`（仅标记，非真正删除） |
|  **更新阶段**  | 执行 DELE 标记的删除操作，释放 TCP 连接 | `QUIT`                                 |

> DELE 在事务处理阶段只是标记"已删除"，真正删除发生在发出 `QUIT` 后的更新阶段。

**两种工作模式**：下载并删除（节省服务器空间）、下载并保留（多设备可重复下载）。

## IMAP

> 基于 **TCP**，端口 **143**。邮件**始终保留在服务器上**，客户端只是"窗口"——所有操作（阅读、删除、移动、标记）直接作用于服务器，多设备自动同步。支持文件夹管理、**选择性获取邮件内容**（先拉取头部，再按需下载正文和附件）、服务器端搜索。

## 对比

| 对比维度 | POP3 | IMAP |
|----------|:----:|:----:|
| **端口** | **110** | **143** |
| **邮件存储** | 下载到**本地** | **服务器**上 |
| **多设备同步** | ❌ 不支持 | ✅ 自动同步 |
| **文件夹管理** | ❌ 仅 INBOX | ✅ 完整支持 |
| **部分下载** | ❌ | ✅ 先头部后正文 |
| **离线阅读** | ✅ 邮件在本地 | ⚠️ 需预先缓存 |
| **服务器压力** | 低 | 高 |

## 基于万维网的电子邮件

> 用户通过**浏览器**访问邮件服务，浏览器 ↔ Web 服务器之间使用 **HTTP/HTTPS**，但 Web 服务器 ↔ 邮件服务器之间**仍使用 SMTP / IMAP / POP3**。

| 属性                      | 说明                        |
| ----------------------- | ------------------------- |
| **协议（浏览器 ↔ Web 服务器）**   | **HTTP / HTTPS**          |
| **协议（Web 服务器 ↔ 邮件服务器）** | SMTP（发送）+ IMAP / POP3（接收） |
| **代表**                  | Gmail、QQ 邮箱、Outlook.com   |

> [!important] Web 邮件 ≠ 不用 SMTP
> 用户通过 HTTPS 提交到 Web 服务器，Web 服务器再通过 SMTP 发送给收件人邮件服务器——这是"HTTP(S) 前端 + SMTP/IMAP 后端"的组合

# ❗ 易错点总结

1. **发送用 SMTP，接收用 POP3 / IMAP**——不要混用！（出题点：给你一段描述，问使用了哪些协议）
2. **SMTP 端口 25、POP3 端口 110、IMAP 端口 143**——这是默认明文端口，它们的加密版本另有端口号（465/995/993）
3. **SMTP 只能传输 7 位 ASCII**——这是传统 SMTP 的硬限制。当下中文、图片、附件之所以能在邮件中轻松发送，靠的是 MIME 扩展，而不是 SMTP 本身的"进化"（出题点：SMTP 能否直接传输二进制文件？→ 不能，必须由 MIME 先编码）
4. **MIME 没有取代或修改 SMTP**——只是在邮件内容层面扩展了邮件格式和编码机制，SMTP 仍然是负责传输的那个协议（二者是"分工合作"的关系）
5. **Base64 编码导致 33% 的大小增长**——$\frac{4}{3}$ 倍。3 字节 → 4 个 ASCII 字符
6. **POP3 将邮件下载到本地**——默认删除服务器副本；**IMAP 保留在服务器上**——客户端是"窗口"
7. **POP3 不支持文件夹**——只有 INBOX；IMAP 支持完整文件夹管理
8. **SMTP 的 `MAIL FROM`（信封）≠ 邮件 `From:` 头部**——信封用于退信路径，头部是显示给用户看的"发件人"。信封地址和头部地址可以不同，这是垃圾邮件"伪造发件人显示地址"的技术基础
9. **SMTP 是有状态协议**——通过"连接建立 → 邮件传送 → 连接释放"三个阶段的紧密衔接维持会话状态，一个阶段出错意味着整个邮件传输失败
10. **SMTP 是"推送"协议**（Push）——主动将邮件推向下一站；**POP3/IMAP 是"拉取"协议**（Pull）——用户主动去服务器"取"邮件
11. **Web 邮件**——浏览器 ↔ Web 服务器之间只用 HTTP/HTTPS，但 Web 服务器 ↔ 邮件服务器之间**仍然使用 SMTP/POP3/IMAP**
12. **SMTP 使用 NVT ASCII 格式**的命令和应答——和 FTP（控制连接）、Telnet 一样使用 NVT ASCII
13. **MIME 没有改变 RFC 5322 或 SMTP 协议**——它是"附加"在原有系统之上的扩展层，SMTP 服务器无需任何改动即可传输 MIME 邮件（因为经过 Base64 编码后的数据完全是 ASCII）
14. **SMTP 的连接建立、邮件传送、连接释放三个阶段依次进行**——如果某个阶段失败，整个邮件传输过程将重新开始从头开始（不会断点续传；出题点：SMTP 如果发送中断，下次发送是"断点续传"吗？→ 不是。故障恢复后只能从头重新发送）
15. **`EHLO` vs `HELO`**：`HELO` 是原始 SMTP 的命令，`EHLO` 是 **ESMTP（Extended SMTP）** 的命令——`EHLO` 用于探测服务器是否支持邮件扩展功能（如认证、TLS 加密、8BITMIME 等）。现代 SMTP 客户应首先尝试 `EHLO`，若失败则回退到 `HELO`
