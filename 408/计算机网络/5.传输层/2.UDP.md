# UDP 数据报

> **UDP（User Datagram Protocol，用户数据报协议）**：TCP/IP 协议族中**无连接**的传输层协议，提供**尽最大努力交付**的不可靠传输服务。

UDP 只在 IP 数据报服务的基础上增加了两个最基本的传输层功能：**端口号**（实现进程到进程的通信）和**检验和**（检测数据在传输中是否出错）。

> [!tip] UDP 的设计哲学
> UDP 的设计遵循"**保持简单**"原则——它几乎不添加任何 IP 没有提供的东西。如果需要更复杂的服务（如可靠传输、拥塞控制），则由**应用层**自己实现（如 QUIC 协议在应用层基于 UDP 实现了 TCP 级别的可靠性）。

## UDP 概述

### UDP 的主要特点

| 特点          | 说明                                                              |
| ----------- | --------------------------------------------------------------- |
| **无连接**     | 发送数据前不需要建立连接，直接封装并发送                                            |
| **不可靠**     | 不保证数据一定到达、不保证顺序、不保证不重复——尽最大努力交付                                 |
| **面向报文**    | 保留应用层报文的边界——一次发送一个完整报文，接收方一次接收一个完整报文，**既不合并也不拆分**，由应用层选择合适的报文大小 |
| **无拥塞控制**   | 不会因网络拥塞而降低发送速率，发送速率不受控制                                         |
| **支持多播/广播** | UDP 支持一对一、一对多、多对多通信（TCP 仅支持一对一单播）                               |
| **首部开销小**   | 仅 **8 字节**固定首部（TCP 为 20~60 字节）                                  |
| **实时性好**    | 无需建立连接、无重传等待、无拥塞控制，传输效率高                                        |

### UDP 的适用场景

- **DNS 域名解析**：查询报文简短，UDP 低时延更高效
- **DHCP 动态主机配置**：客户端获取 IP 前无法建立 TCP 连接（尚无 IP），需要广播/多播支持
- **RIP 路由信息协议**：周期性广播路由信息，实时性优先于可靠性
- **SNMP 网络管理**：网络故障时 TCP 可能无法建立连接，UDP 能快速获取管理信息
- **实时音视频通信**（VoIP、RTP）：实时性要求极高，少量丢包人耳/人眼几乎察觉不到，而 TCP 重传造成的延迟和抖动反而严重影响体验
- **在线游戏**：实时性远重于数据完整性，少量的位置更新丢失可由下一次更新弥补

> [!important] UDP 面向报文 vs TCP 面向字节流
> 这是 UDP 与 TCP 的核心语义差异之一：
>
> |   | UDP（面向报文） | TCP（面向字节流） |
> |---|---|---|
> | **发送** | 应用进程给多少，UDP 就封装成一个数据报发多少 | 应用进程交付的字节流被 TCP 放入发送缓冲区，由 TCP 根据需要分片发送 |
> | **接收** | 接收方一次接收一个完整的数据报 | 接收方从接收缓冲区按字节流读取，没有"消息"概念 |
> | **消息边界** | **保留**——每个数据报是独立的，接收次数 = 发送次数 | **不保留**——多次发送的数据可能在一次 recv 中全部收到 |
> | **应用责任** | 应用层自己处理分包、排序等问题 | TCP 替应用层处理了分包、排序问题 |

## UDP 的首部格式

> UDP 的首部非常简洁，仅包含 **4 个字段**，固定 **8 字节**。

![[UDP 数据报格式.png]]

### 各字段详解

| 字段 | 长度 | 说明 |
|------|:----:|------|
| **源端口号** | 16 位 | 发送方进程的端口号。**可选项**——如果不需要对方回复，可填 0（如 DHCP 客户端首次请求时） |
| **目的端口号** | 16 位 | 接收方进程的端口号。**必填项**——分用的唯一依据之一 |
| **UDP 长度** | 16 位 | UDP 数据报的总长度（首部 8 字节 + 数据部分）。最小值为 **8**（仅有首部，无数据），单位：字节 |
| **UDP 检验和** | 16 位 | 用于差错检测，检验整个 UDP 数据报（首部 + 数据 + 伪首部）。**可选项**——IPv4 中可选填 0（表示不计算），IPv6 中**强制计算** |
| **数据部分** | 可变 | 来自应用层的有效载荷 |

> [!warning] UDP 长度字段
> - UDP 长度字段是**冗余**的——从 IP 首部的总长度减去 IP 首部长度也能推出 UDP 长度。之所以保留，是为了让 UDP 具备独立处理能力
> - 最小 UDP 长度为 **8 字节**（仅有首部），最大为 $2^{16} - 1 = 65535$ 字节，但实际受限于 IP 数据报最大长度 $65535 - 20 = 65515$ 字节

> [!info] 端口号的可选性
> - **目的端口号**：必须有，否则接收方不知道该交给哪个应用进程
> - **源端口号**：可选（填 0）。当不需要接收方回复时可以不填，如 DHCP 客户端在获得 IP 之前发送的 DHCPDISCOVER 报文
> - **检验和**：IPv4 中可选，IPv6 中强制——因为 IPv6 首部不再有检验和字段，端到端差错检测的责任完全落在传输层

# UDP 检验

> **UDP 检验和（Checksum）** 用于检测 UDP 数据报在传输过程中是否发生了比特差错。

## 检验和的计算范围

UDP 检验和的**独特之处**在于它会额外引入一个 **12B 伪首部（Pseudo Header）**，使得检测范围超越了 UDP 本身

## 伪首部（Pseudo Header）

> **伪首部**：计算检验和时，从 IP 首部中临时取出的部分字段组成的虚拟首部，**仅参与检验和计算，不向下传送也不向上递交**。

| 偏移  | 字段       |  长度  | 说明                                   |
| :-: | -------- | :--: | ------------------------------------ |
|  0  | 源 IP 地址  | 32 位 | 发送方 IP，验证数据来源是否正确                    |
|  4  | 目的 IP 地址 | 32 位 | 接收方 IP，验证数据是否到达正确的主机                 |
|  8  | 全零       | 8 位  | 保留字段，固定为 0                           |
|  9  | 协议号      | 8 位  | IP 首部中的协议字段，UDP 固定为 **17**，TCP 固定为 6 |
| 10  | UDP 长度   | 16 位 | UDP 数据报总长度（与 UDP 首部中的长度字段一致）         |

> [!important] 伪首部的作用
> 伪首部让 UDP 能够**双重验证**：不仅能检测数据报本身是否出错（首部 + 数据的检验和），还能验证数据报是否到达了**正确的目的 IP 和正确的传输层协议**——防止 IP 首部中的目的 IP 地址或协议号被篡改，导致数据报被错误交付。

## 检验和的计算方法

UDP 检验和采用 **16 位二进制反码求和**的算法：

1. 将**伪首部**、**UDP 首部**和**数据部分**拼接成一个整体
2. 在末尾**添零**使总长度为偶数字节（若数据部分为奇数字节，则补一个全 0 的字节）
3. 初始化检验和字段为 **0**
4. 每 **16 位**（2 字节）为一组，当作一个二进制数
5. 将所有 16 位二进制数进行**反码求和**（即常规二进制加法，**若有溢出则将溢出位回卷加到最低位**）
6. 将求和结果**取反码**，填入检验和字段

**接收方验证**：将收到的 UDP 数据报（包括检验和字段）与伪首部一起，按同样方式计算。若结果为 **全 1**（即反码和为 0 的反码），则说明无差错；否则数据报在传输中出错，直接被丢弃。

> [!tip] 反码求和的特点
> - **字节序无关**：大端和小端机器计算出的检验和相同，便于跨平台实现
> - **易于增量更新**：只改了少量数据时，无需重新计算整个检验和
> - **全零 vs 全一**：全零（用 0 表示未计算）和全一（反码中的 0）需要区分——⭐当计算结果恰好为全零时，填入**全 1**（即 0xFFFF），因为全 0 留作"未计算检验和"的标志

# UDP 与 TCP 核心对比

| 比较维度 | **UDP** | **TCP** |
|----------|:-------:|:-------:|
| **服务类型** | 无连接、不可靠 | 面向连接、可靠 |
| **连接建立** | 不需要 | 需要（三次握手） |
| **首部长度** | **8 字节**（固定） | 20~60 字节 |
| **数据单位** | **数据报（Datagram）** | **报文段（Segment）** |
| **数据边界** | 面向**报文**，保留消息边界 | 面向**字节流**，无消息边界 |
| **可靠性** | 尽最大努力交付 | 通过确认、重传、序号等保证可靠 |
| **顺序** | 不保证 | 保证按序到达 |
| **流量控制** | 无 | 有（滑动窗口） |
| **拥塞控制** | 无 | 有（慢开始、拥塞避免、快重传、快恢复） |
| **分用依据** | 目的 IP + 目的端口 | 四元组（源 IP、源端口、目的 IP、目的端口） |
| **通信方式** | 一对一、一对多（多播）、一对全部（广播） | 仅一对一（单播） |
| **检验和** | IPv4 可选，IPv6 强制 | 强制 |
| **适用场景** | DNS、DHCP、VoIP、实时视频、在线游戏 | 文件传输、电子邮件、Web 浏览 |

> [!important] UDP 存在的必要性
> 有 TCP 为什么还需要 UDP？
> 1. **无连接、开销小**：无需建立/释放连接，适合一闪而过的简单查询（如 DNS）
> 2. **无拥塞控制**：不会主动降速，适合实时应用（VoIP、实时视频——宁可丢包也不要延迟）
> 3. **支持广播/多播**：TCP 不支持，DHCP、RIP、mDNS 等协议必须有此能力
> 4. **首部开销极小**（8 vs 20+）：适合短报文（如 SNMP Trap）
> 5. **应用层可定制**：不需要 TCP 完整功能的应用可以基于 UDP 实现"够用就好"的传输协议（如 QUIC、KCP）

# ❗ 易错点总结

1. **UDP 提供无连接的不可靠传输**——不建立连接、无确认重传、无流量控制、无拥塞控制
2. **UDP 面向报文，TCP 面向字节流**：UDP 保留应用层报文的边界，发送几次就要接收几次（不合并、不拆分）；TCP 的发送/接收次数没有对应关系
3. **UDP 首部固定 8 字节**：源端口 + 目的端口 + 长度 + 检验和，各占 16 位
4. **UDP 长度字段是最小值 8**（仅有首部），最大值 $65535$ 字节（但实际受 IP 限制，约 $65515$ 字节）
5. **源端口号可选**（填 0，如 DHCP 客户端的初次请求）；**目的端口号必填**
6. **检验和在 IPv4 中可选**（填 0 = 不计算），在 **IPv6 中强制计算**——因为 IPv6 首部无检验和字段
7. **伪首部**参与检验和计算，但**不向下传送、不向上递交**——仅用于差错检测
8. **伪首部的作用**：不仅检测数据报本身是否出错，还验证是否到达正确的主机和正确的协议
9. **UDP 分用仅依据目的 IP + 目的端口**（二元组）——与 TCP 的四元组分用机制不同
10. **UDP 支持多播和广播**，TCP 仅支持单播——这是许多协议（DHCP、RIP）必须选用 UDP 的根本原因
11. **UDP 无拥塞控制**：发送方以恒定速率发送，不会因为网络拥塞而降速——对实时应用是优点，但可能加剧网络拥塞
12. **检验和计算方法**：16 位反码求和，以偶数字节为单位，结果为全 1 则无错
13. **协议号**：UDP 在 IP 首部中的协议号固定为 **17**（TCP 为 6）
14. **UDP 无真正的"连接"概念**：即使利用 connect() 函数绑定对端地址，也只是在内核中记录对端信息以便使用 send() 代替 sendto()，并没有三次握手等真正的连接建立过程
