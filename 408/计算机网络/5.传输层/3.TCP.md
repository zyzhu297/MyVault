# TCP 的特点

> **TCP（Transmission Control Protocol，传输控制协议）**：TCP/IP 协议族中**面向连接**的传输层协议，通过序号、确认、重传、流量控制、拥塞控制等机制，提供**可靠的、有序的、全双工的字节流**传输服务。

| 特点 | 说明 |
|------|------|
| **面向连接** | 通信前必须通过三次握手建立连接，通信后通过四次挥手释放连接 |
| **可靠传输** | 通过序号、确认应答、超时重传、检验和等机制，保证数据**不丢、不错、不乱、不重** |
| **面向字节流** | TCP 将应用层数据看作无结构的字节流，不保留消息边界——发送方写入的字节数与接收方读出的字节数无关 |
| **全双工通信** | 收发双方可以同时发送和接收数据，两端都设有发送缓冲区和接收缓冲区 |
| **仅支持单播** | TCP 只支持一对一的通信，不支持多播和广播 |
| **首部开销大** | TCP 首部为 20~60 字节（固定 20 字节 + 可变选项最多 40 字节） |
| **有流量控制** | 通过滑动窗口机制控制发送速率，防止发送方发送过快导致接收方缓冲区溢出 |
| **有拥塞控制** | 通过慢开始、拥塞避免、快重传、快恢复等算法，根据网络拥塞情况动态调整发送速率 |

> [!important] TCP 面向字节流 vs UDP 面向报文
> 这是 TCP 与 UDP 最核心的语义差异：
>
> |   | TCP（面向字节流） | UDP（面向报文） |
> |---|---|---|
> | **发送端** | 应用进程写到发送缓冲区的字节流，TCP 根据需要**分段**发送 | 应用进程给多少，UDP 就封装成一个完整数据报发送 |
> | **接收端** | TCP 将收到的字节流放入接收缓冲区，应用进程**不区分边界**地读取 | 接收方一次接收一个完整的数据报，**保留消息边界** |
> | **消息边界** | **不保留**——多次 write 的数据可能在某次 read 中一起读出 | **保留**——每次 send 对应一次 recv |
> | **粘连/拆分** | 存在粘包问题，需应用层自行界定消息边界 | 不存在粘包，每个数据报天然独立 |

> [!tip] TCP 的设计目标
> TCP 的核心目标是在不可靠的 IP 网络层之上构建一个**可靠的端到端数据传输管道**，使得：
> - 即使底层网络出现丢包、重复、失序、错误，上层应用也感觉不到
> - 适应不同网络条件——从局域网到拥塞的广域网
> - 能够公平地共享网络带宽

# TCP 报文段

> **TCP 报文段（Segment）**：TCP 传输的基本单位，由 TCP 首部和数据部分组成。首部长度 20~60 字节。

![[TCP 报文段.png]]

## TCP 首部各字段详解

| 字段                             |      长度      | 说明                                                                                                   |
| ------------------------------ | :----------: | ---------------------------------------------------------------------------------------------------- |
| **源端口号**                       |     16 位     | 发送方进程的端口号（必须填写）                                                                                      |
| **目的端口号**                      |     16 位     | 接收方进程的端口号（必须填写），分用的核心依据                                                                              |
| **序号（Sequence Number）**        |     32 位     | 本报文段所发送数据的**第一个字节的序号**（字节流中的偏移）。TCP 面向字节流，每个字节都有序号，采用模 $2^{32}$ 运算                                   |
| **确认号（Acknowledgment Number）** |     32 位     | 期望收到对方**下一个报文段的第一个字节的序号**，同时也是对之前数据的**累积确认**；若确认号为 N，则表明到序号 N-1 为止的所有数据已正确接收                         |
| **数据偏移（Data Offset）**          |     4 位      | **TCP 首部的长度**，以 **4 字节**为单位。最小值 5（20 字节），最大值 15（60 字节）。也即首部长度字段                                      |
| **保留（Reserved）**               |     6 位      | 保留位，目前置为 0                                                                                           |
| **URG（紧急指针有效）**                |     1 位      | =1 时表示紧急指针字段有效，报文段中含有紧急数据，应优先传送；且紧急数据被放置在报文段数据最前面                                                    |
| **ACK（确认号有效）**                 |     1 位      | =1 时确认号字段才有效。连接建立后**所有报文段的 ACK 都必须置 1**（SYN 报文除外）                                                    |
| **PSH（推送）**                    |     1 位      | =1 时提示接收方尽快将数据交付给应用进程，而不是等缓冲区填满再交付（如 Telnet 交互）                                                      |
| **RST（复位）**                    |     1 位      | =1 时表示 TCP 连接出现严重差错，必须立即释放并重新建立连接。也用于拒绝非法报文段或拒绝连接                                                    |
| **SYN（同步序号）**                  |     1 位      | =1 时表示这是一个连接请求或连接接受报文（仅在三次握手阶段使用）                                                                    |
| **FIN（终止）**                    |     1 位      | =1 时表示发送方数据已发送完毕，请求释放连接（仅在四次挥手阶段使用）                                                                  |
| **窗口（Window）**                 |     16 位     | 由本报文段的发送方填写，表示其作为接收方时当前还能接收的最大数据量；告诉对方：**本报文段确认号对应字节起，我方还能接收的字节数**，用于流量控制。最大值 65535 字节（$2^{16} - 1$） |
| **检验和（Checksum）**              |     16 位     | TCP **强制计算**检验和（首部 + 数据 + 伪首部），接收方检测出差错则丢弃报文段并等待重传                                                   |
| **紧急指针（Urgent Pointer）**       |     16 位     | 当 URG=1 时有效，指示本报文段中紧急数据的**字节数**（从序号字段值开始偏移）。紧急数据即便窗口为零也必须接收                                          |
| **选项（Options）**                | 可变（最长 40 字节） | 可选字段，常用选项有：MSS、窗口扩大因子、时间戳、SACK 支持等                                                                   |
| **填充（Padding）**                |      可变      | 使整个 TCP 首部长度成为 4 字节的整数倍，填充内容为全零                                                                      |

> [!important] 序号和确认号
> - **序号（seq）** 和 **确认号（ack）** 是 TCP 实现可靠传输的核心字段
> - 序号是**本端发送的数据**的编号，确认号是**期望收到的对方数据**的编号
> - SYN 和 FIN 报文段即使不携带数据，也**消耗一个序号**
> - 纯 ACK 报文段不消耗序号

## TCP 的常用选项

| 选项 | 长度 | 说明 |
|------|:----:|------|
| **MSS（最大报文段长度）** | 4 字节 | TCP 报文段中数据部分的最大长度（不含首部）。**仅在 SYN 报文中协商**，双方各自声明自己的 MSS。默认值为 **536 字节** |
| **窗口扩大因子（Window Scale）** | 3 字节 | 解决窗口字段仅 16 位、最大 65535 字节的瓶颈。扩大因子 n 表示窗口实际值为窗口字段值 × $2^n$，n 取值范围 0~14。**仅在 SYN 报文中协商** |
| **时间戳（Timestamp）** | 10 字节 | 包含发送方时间戳和回显时间戳两子字段。用于**计算 RTT** 和**防止序号回绕（PAWS）** |
| **SACK 允许（SACK-Permitted）** | 2 字节 | 仅在 SYN 报文中协商是否支持 SACK（选择性确认） |
| **SACK 信息** | 可变 | 当收到乱序数据时，告知对方已收到的不连续数据块——支持**选择性重传** |

> [!tip] MSS 与 MTU 的关系
> - **MTU**（链路层最大传输单元）：以太网默认 1500 字节
> - **MSS** = MTU − IP 首部（20 B）− TCP 首部（20 B）= 1460 字节（以太网典型值）
> - 若 TCP 报文段在网络中被分片，效率将大幅降低——协商合理的 MSS 是为了避免 IP 分片

# TCP 连接管理

> TCP 是面向连接的协议——通信前必须先**建立连接**，然后**数据传输**，通信结束后必须**释放连接**。连接建立采用**三次握手**，连接释放采用**四次挥手**

在 TCP 连接建立的过程中，需解决以下三个问题：
- 使通信双方确知对方的存在
- 允许双方协商相关参数（如最大窗口值、是否使用窗口扩大选项、时间戳选项等）
- 为传输实体分配必要的资源（如缓存空间、连接表项等）
TCP 将连接作为最基本的抽象；每条 TCP 连接有**两个端点**，端点是套接字；TCP 连接的建立采用**客户/服务器**模式；主动发起连接建立的应用程序称为**客户**（Client），被动等待连接请求的应用程序称为**服务器**（Server）

## 三次握手（Three-Way Handshake）

TCP 通过三次握手建立连接，核心目的是**确认双方的收发能力**、**同步初始序号**、**协商参数（MSS、窗口扩大因子等）**

![[三次握手.png]]

### 三次握手的过程

最初两端 TCP 进程均处于 CLOSED（关闭）状态；在连接建立前，服务器完成一些准备工作后进入 LISTEN（收听）状态，等待客户的连接请求

|   步骤    | 发送方 | 报文方向  |        标志位         | 序号/确认号                        | 作用                                                                    |
| :-----: | --- | :---: | :----------------: | ----------------------------- | --------------------------------------------------------------------- |
| **第一次** | 客户端 | → 服务器 | **SYN=1**<br>ACK=0 | seq = x（客户端初始序号 ISN）          | 客户端告知服务器：我要建立连接，我的初始序号是 x；客户进入 SYN-SENT状态                             |
| **第二次** | 服务器 | ← 客户端 | **SYN=1<br>ACK=1** | seq = y（服务器 ISN）<br>ack = x+1 | 服务器回应：我收到了，同意建立连接；我的初始序号是 y；服务器进入SYN-RCVD 状态                          |
| **第三次** | 客户端 | → 服务器 | SYN=0<br>**ACK=1** | seq = x+1<br>ack = y+1        | **此报文可携带数据；若不携带数据，则不消耗信号**，客户端最终确认：我收到了你的确认，连接正式建立，客户进入ESTABLISHED 状态 |

三次握手后，当服务器收到最终确认后，服务器也进入 **ESTABLISHED** 状态，连接正式建立，可以开始传输数据

>[!note] 最早发送数据时间
>从客户发出 SYN 报文段（第一次握手的连接请求报文段）时刻算起，客户最早可在 **1RTT** 后开始发送数据，而服务器最早在 **1.5RTT** 后才能发送数据

> [!important] SYN 消耗序号
> - SYN=1 的报文段**即使不携带数据，也消耗一个序号**
> - "消耗一个序号"意味着：若 SYN 报文段序号为 x，则下一个报文段的序号为 x+1
> - 同理，FIN 报文段也消耗一个序号

> [!tip] 三次握手时 TCP 连接的 socket 状态
> 客户端调用 connect() 时：操作系统为本连接分配一个**短暂端口**作为源端口——client socket =（client IP:短暂端口, server IP:熟知端口）
> 服务器调用 accept() 时：为**每个**已建立的连接返回一个**新的 socket**，同时继续使用原来的 socket 监听新连接（TCP 通过四元组区分不同连接，同一个 server 端口 80 可以服务数十万个并发连接）

### 为什么需要三次握手？

> **核心问题**：如何防止**已失效的连接请求报文段**突然又传到服务器，从而产生错误。

| 握手次数 | 能否工作 | 问题分析 |
|:--------:|:--------:|----------|
| **一次握手** | ✗ | 客户端无法确认服务器能收到数据，更无法确认自己能否收到服务器数据 |
| **两次握手** | ✗（有隐患） | 场景：客户端发出第一个 SYN 后因网络延迟很久才到达，客户端认为已超时并发出第二个 SYN；服务器收到第一个（已失效的）SYN 后回复 SYN-ACK，单向建立连接——此时客户端没有发起新请求却收到了 SYN-ACK，不会建立连接，而服务器**单方面认为连接已建立**，会一直等待客户端数据，白白浪费资源 |
| **三次握手** | ✅ | 服务器必须收到客户端的**第三次 ACK 确认**后才会分配资源进入 ESTABLISHED 状态——已失效的连接请求即使到达服务器，服务器回复 SYN-ACK 后，客户端不会回 ACK，服务器就不会误开连接，从而避免了资源浪费 |

> [!warning] "两次握手可不可行"是经典面试题
> 核心答案：**不可行**。主要原因是**防止已失效的连接请求造成服务器资源浪费**。此外，两次握手也无法让双方充分交换初始序号，无法协商 MSS 等参数。

## 四次挥手（Four-Way Teardown / Handshake）

> 通信双方的**任意一方**均可主动终止连接

TCP 是全双工通信，每个方向的数据发送都需要独立关闭——因此连接释放需要**四次交互**。

![[四次挥手.png]]

### 四次挥手的过程

|   步骤    | 发起方 | 报文方向  |        标志位         | 序号/确认号                             | 含义                                                                                                                       |
| :-----: | --- | :---: | :----------------: | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **第一次** | 客户端 | → 服务器 | **FIN=1<br>ACK=1** | seq = u<br>u 等于此前已发送数据的最后一个字节序号加 1 | 客户端：我数据发完了，请求关闭客户端→服务器方向；**并停止发送数据**，客户进入 FIN-WAIT-1 状态                                                                  |
| **第二次** | 服务器 | ← 客户端 |     **ACK=1**      | seq = v<br>**ack = u+1**           | 服务器：收到你的 FIN，确认（但我的数据可能还没发完）；服务进入 CLOSE-WAIT 状态；此时，从客户到服务器方向的连接已释放，但服务器到客户方向的连接未释放，TCP 处于**半关闭**状态，客户收到确认后，进入 FIN-WAIT-2 |
| **第三次** | 服务器 | ← 客户端 | **FIN=1<br>ACK=1** | seq = w, **ack = u+1**             | 服务器：我的数据也发完了，请求关闭服务器→客户端方向；服务器进入 LAST-ACK 状态                                                                             |
| **第四次** | 客户端 | → 服务器 |     **ACK=1**      | seq = u+1<br>ack = w+1             | 客户端：收到你的 FIN，发送**最终确认**，进入 TIME-WAIT 状态；等待 **2MSL** 后彻底关闭，才进入 CLOSED 状态                                                  |

在服务器收到最终确认后，也进入 CLOSED 状态，此时 TCP 连接断开

>[!note] 释放连接最短时间
>若服务器在收到客户的连接释放请求后不再发送数据，则从客户发出 FIN 报文段时刻起算，客户释放连接的最短时间为 **1RTT+2MSL** ，服务器释放连接的最短时间为 **1.5RTT**

### TIME_WAIT 状态

> 客户端在发送第四次挥手的 ACK 后，进入 **TIME_WAIT** 状态，等待 **2MSL**（Maximum Segment Lifetime，报文最大生存时间，通常取 2 分钟）后进入 CLOSED 状态。

**为什么需要 TIME_WAIT（等待 2MSL）？**

| 原因                            | 说明                                                                                                       |
| ----------------------------- | -------------------------------------------------------------------------------------------------------- |
| **① 确保最后的 ACK 能被服务器收到**       | 如果最后的 ACK 丢失，服务器会超时重传 FIN——客户端在 TIME_WAIT 期间可以重传 ACK。**2MSL = ACK 最大存活时间（1MSL）+ 对方重传的 FIN 最长存活时间（1MSL）** |
| **② 使本连接持续时间内产生的所有报文段从网络中消失** | 经过 2MSL 时间，可以保证本连接产生的所有报文段（包括迟到的、重传的）都已从网络中消失，不会影响下一次使用相同四元组的新连接                                         |

> [!warning] TIME_WAIT 只在主动关闭方出现
> - 主动发送第一个 FIN 的一方（主动关闭方）进入 TIME_WAIT
> - 被动关闭方收到 FIN 并回复 FIN 后进入 **LAST_ACK** 状态，收到最后 ACK 后直接进入 **CLOSED**，无需等待

### 为什么需要四次挥手？

> 核心原因：**TCP 是全双工通信**——关闭连接时，两个方向的"数据流"需要各自独立关闭。

- 客户端发送 FIN，表示**客户端→服务器方向**关闭（客户端不再发送数据，但仍可接收）
- 服务器收到 FIN 后先回复 ACK，此时**客户端→服务器方向**已关闭，但**服务器→客户端方向**可能还有数据要发
- 服务器数据发完后，再发送 FIN，表示**服务器→客户端方向**也关闭
- 客户端回复 ACK，等待 2MSL 后关闭

> [!important] 为什么不合并第二次和第三次？
> FIN 和 ACK 在方服务器端不能合并为一个报文段——因为服务器收到客户端的 FIN 后，可能**还有数据未发送完毕**，所以只能先回 ACK，等数据发完后再发 FIN。这就是**ACK 和 FIN 分别发送**的原因，也是比三次握手多一次交互的根本原因。

### TCP 连接状态总结

| 状态 | 说明 | 出现时机 |
|------|------|----------|
| **CLOSED** | 初始状态，无连接 | 初始状态 / 连接彻底释放后 |
| **LISTEN** | 服务器监听，等待连接请求 | 服务器调用 listen() 后 |
| **SYN-SENT** | 客户端已发送 SYN，等待服务器回复 | 客户端第一次握手后 |
| **SYN-RCVD** | 服务器已收到 SYN，已回复 SYN+ACK，等待客户端 ACK | 服务器第二次握手后 |
| **ESTABLISHED** | 连接已建立，正常数据传输 | 三次握手完成后 |
| **FIN-WAIT-1** | 主动关闭方已发送 FIN | 客户端第一次挥手后 |
| **FIN-WAIT-2** | 主动关闭方已收到对方对 FIN 的 ACK | 客户端收到服务器第二次挥手 ACK 后 |
| **TIME-WAIT** | 主动关闭方等待 2MSL | 客户端发送第四次挥手 ACK 后 |
| **CLOSING** | 双方同时关闭（罕见） | 双方几乎同时发送 FIN |
| **CLOSE-WAIT** | 被动关闭方收到 FIN，已回复 ACK，等待自身应用进程关闭 | 服务器第二次挥手后 |
| **LAST-ACK** | 被动关闭方已发送 FIN，等待最后的 ACK | 服务器第三次挥手后 |

# TCP 可靠传输

> **TCP 可靠传输的核心机制**：通过**检验、序号、确认、重传**等机制来保证数据在不可靠的 IP 网络层之上实现可靠交付。

其中检验机制与 UDP 类似，详见[[2.UDP#UDP 检验]]，TCP 协议号为 6

## 序号机制

TCP 将数据视为一个**无结构的字节流**，为流中的每个字节进行编号：

- **初始序号（ISN）**：连接建立时随机生成（避免与前一次连接的报文段混淆）
- **序号字段**：TCP 首部中的序号字段值为本报文段**数据部分第一个字节的序号**
- **确认号字段**：期望收到对方**下一个报文段第一个字节的序号**，同时也是对**该序号之前所有字节的累积确认**

> [!example] 序号示例
> 假设 ISN = 1000，客户端发送 500 字节数据。则：
> - 该报文段序号 = 1000
> - 数据占字节号：1000~1499
> - 服务器回复 ack = 1500（表示 1499 及之前全部收到，期待下一个字节 1500）

## 确认机制（ACK）

| 要点                        | 说明                                                                |
| ------------------------- | ----------------------------------------------------------------- |
| **累积确认（Cumulative ACK）**  | TCP 默认使用累积确认——确认号为 n，意味着**n-1 及之前的所有字节**均已收到                      |
| **捎带确认（Piggyback ACK）**   | 接收方可以在**自己发送数据时顺便携带 ACK**，减少网络中的纯 ACK 报文；即使没有数据要发，也必须在超时后发送单独 ACK |
| **延迟确认（Delayed ACK）**     | 接收方收到数据后不立即确认，而是**等待一段时间（通常 ≤ 500ms）**，希望有数据可以捎带确认，或者多个报文段合并一次确认  |
| **重复 ACK（Duplicate ACK）** | 收到**失序**报文段时，立即发送一个确认号等于期望序号的 ACK（该 ACK 与前一个 ACK 相同，故称重复 ACK）     |

> [!important] 累积确认的优缺点
> - **优点**：即使某个 ACK 丢失，后续 ACK 隐含了对之前数据的确认，不一定要重传
> - **缺点**：无法精细反映每个报文段的接收情况，发送方只知道"确认号之前都收到了"，不知道之后有哪些已到达（因此需要 SACK 补充）

## 重传机制

TCP 重传分为两种：**超时重传**（基于计时器）和**冗余确认重传**（基于重复 ACK，即快速重传）。

### 超时重传（Retransmission Timeout, RTO）

> 发送方每发一个报文段，就启动一个**重传计时器**。若计时器超时仍未收到确认，则重传该报文段；RTO 根据网络状况**动态计算**（基于往返时间 RTT 的加权平均值 RTTS）， RTO 应**略大于** RTTS

### 冗余确认

> **TCP 规定**：接收方收到**失序**报文段时，立即发送**重复 ACK**（确认号 = 期望收到的序号）。发送方若连续收到 **3 个相同的重复 ACK**（即总共 4 个相同确认号的 ACK），不等超时，**立即重传**该确认号对应的报文段。

| 重传方式 | 触发条件 | 响应速度 | 说明 |
|----------|:--------:|:--------:|------|
| **超时重传** | 重传计时器超时 | 慢（可能数百 ms 以上） | RTO 动态计算，重传后还会指数退避 |
| **冗余确认重传（快速重传）** | 收到 3 个重复 ACK | 快 | 不需要等超时，明显提高重传效率 |

## 选择性确认（Selective ACK, SACK）

> **问题**：累积确认 + 快速重传时，发送方只知道缺口处报文段丢了，但不知道缺口**之后**哪些报文段已到达——可能重传已收到的数据，浪费带宽。
>
> **SACK**：在 TCP 选项字段中，接收方告知发送方已收到的**不连续数据块**的边界，发送方可以精准地只重传**确实丢失的数据块**。

>SACK 需收发双方在 SYN 报文中协商（SACK-Permitted 选项）。
>SACK 不是替代累积确认，而是对累积确认的补充。即使在 SACK 下，确认号仍然表示连续收到的最高序号。

# TCP 流量控制

> **流量控制（Flow Control）**：防止**发送方发送速率过快**，导致**接收方缓冲区溢出**而丢弃数据。TCP 通过**滑动窗口**机制实现流量控制。

## 滑动窗口与接收窗口

| 概念             | 说明                                                                |
| -------------- | ----------------------------------------------------------------- |
| **接收窗口（rwnd）** | 接收方根据自己接收缓冲区的剩余空间动态设置，通过 TCP 首部的**窗口字段**告知发送方                     |
| **发送窗口**       | 发送方根据 rwnd 确定，表示**发送方在没有收到确认前最多可以连续发送的字节数**                       |
| **核心约束**       | 发送窗口 = min（拥塞窗口 cwnd, 接收窗口 rwnd）——既不过量（受 rwnd 限制），也不拥塞（受 cwnd 限制） |

## 零窗口与持续计时器

### 零窗口（Zero Window）

> 接收方缓冲区满，窗口字段 rwnd = 0，发送方**暂停发送**。

**问题**：接收方腾出空间后，发送一个 rwnd ≠ 0 的窗口更新报文段——但该报文段**可能丢失**；若发送方没收到，会一直等待造成**死锁**。

### 持续计时器（Persist Timer）

> 当发送方收到 rwnd = 0 的窗口通告后，启动**持续计时器**。计时器超时则发送一个**零窗口探测报文段**（仅 1 字节数据），接收方必须回复当前窗口值。如此循环，直到窗口打开。

> [!tip] 持续计时器与重传计时器的区别
> - **重传计时器**：已发送数据未确认 → 重传
> - **持续计时器**：窗口=0 无法发送新数据 → 探测窗口是否已打开

## 传输层流量控制 vs 数据链路层流量控制

| 比较维度      |             传输层（TCP）流量控制              |              数据链路层流量控制               |
| --------- | :-----------------------------------: | :----------------------------------: |
| **作用范围**  |      **端到端**（源主机 → 目的主机，跨越整个网络）       |        **点到点**（相邻两个节点之间的一条链路）        |
| **窗口可变性** |  接收窗口 rwnd **动态变化**（随接收缓冲区剩余空间实时调整）   | 窗口大小通常**固定**或在连接初始化时协定（如 HDLC 的窗口大小） |
| **复杂因素**  | 还需与**拥塞控制**配合（发送窗口 = min(cwnd, rwnd)） |       无需拥塞控制——链路只有两个节点，不存在网络拥塞       |

> 核心区别：数据链路层只关心**相邻两节点之间不要发太快**，传输层要应付**跨越整个网络**的不确定性——延迟波动、乱序到达、中间路由器的拥塞，复杂度远高于链路层。

# TCP 拥塞控制

> **拥塞控制（Congestion Control）**：防止过多的数据注入网络，导致网络中的**路由器或链路过载**（拥塞）。与流量控制不同——拥塞控制关注**网络全局**能力（多对发送方与路由器的互动），流量控制关注**端到端**的接收能力（点对点）。

TCP 还要求发送方维护一个**拥塞窗口（cwnd）**，其大小取决于当前网络的拥塞状况，并动态调整，其控制原理是：

- 若网络未出现拥塞，则适当增大 cwnd，以便把更多分组发送出去，提高网络利用率
- 若检测到网络拥塞，则适当减少 cwnd，以减少注入网络的分组数量，缓解网络拥塞

|   | 流量控制 | 拥塞控制 |
|---|---|---|
| **关注对象** | **接收方**的接收能力 | **网络**的承载能力 |
| **控制变量** | 接收窗口 rwnd | 拥塞窗口 cwnd |
| **作用范围** | 端到端（发送方 ↔ 接收方） | 全局（所有 TCP 连接公平共享链路） |
| **实现方式** | 接收方通告窗口 | 发送方根据网络反馈自主调节 |

> [!important] 发送窗口的最终确定
> $$\text{发送窗口} = \min(\text{cwnd}, \text{rwnd})$$
> - cwnd 是发送方根据拥塞情况自主维护的内部变量（不在首部传输）
> - rwnd 是接收方通过 TCP 首部窗口字段通告的值
> - 发送方取两者最小值，**既保护接收方也保护网络**

## 拥塞控制的核心算法

TCP 拥塞控制采用**四个核心算法**协同工作：

| 算法 | 阶段 | 核心思路 |
|------|:----:|----------|
| **慢开始（Slow Start）** | 连接初始 / 超时重传后 | cwnd 从 1 起步，每收到一个 ACK 翻倍（**指数增长**），快速探测带宽 |
| **拥塞避免（Congestion Avoidance）** | cwnd ≥ ssthresh 时 | cwnd 每 RTT 线性 +1（**加法增大，Additive Increase**），谨慎逼近网络容量 |
| **快重传（Fast Retransmit）** | 收到 3 个重复 ACK | 不等超时，立即重传缺失报文段 |
| **快恢复（Fast Recovery）** | 快重传后 | 不降到 1（不做慢开始），而是将 cwnd 和 ssthresh 减半后进入拥塞避免 |

> [!tip] 两个关键阈值
> - **ssthresh（慢开始门限）**：cwnd 增长到 ssthresh 时，从慢开始切换到拥塞避免
> - **ssthresh 的更新**：发生丢包时，ssthresh = max(cwnd / 2, 2 × MSS)

## 算法详解

### 慢开始（Slow Start）

> cwnd 从 **1 个 MSS**（一个最大报文段）起步，每收到一个对新数据的 ACK，cwnd 增加 1 个 MSS。由于一个 RTT 内发出 cwnd 个报文段、收到 cwnd 个 ACK → 每经过一个 RTT，cwnd **翻倍**（指数增长）。


> [!warning] "慢开始"并不慢
> 虽然 cwnd 从 1 起步看起来慢，但实际上指数增长的速度非常快——经过 log₂(ssthresh) 个 RTT 即可达到门限值。"慢"是相对于旧版 TCP 无需慢开始、直接以 rwnd 大小发送而言的。

### 拥塞避免（Congestion Avoidance）

> 当 cwnd ≥ ssthresh 时进入拥塞避免阶段，每经过一个 RTT，cwnd 增加 **1 个 MSS**（线性增长 / 加法增大）。
>
> 具体实现上通常采用：**每收到一个对新数据的 ACK，cwnd 增加 MSS²/cwnd**，使得一个 RTT 内 cwnd 约增加 1 个 MSS。

### 网络拥塞的处理

>无论处于慢开始阶段还是拥塞避免阶段，只要发送方判断网络出现拥塞，就执行以下操作：
> - 将满开始门限 **ssthresh 调整为当前 cwnd 的一半**（但不得小于 2）；
> - 将 **cwnd 重置为 1**
> - **重新执行慢开始**算法
>这样做的目的是*迅速减少*注入网络的分组数量，使得发生拥塞的路由器有足够时间把队列中积压的分组处理完

>[!warning]
>在慢开始阶段，若 2cwnd > ssthresh，则下一个传输轮次后的 cwnd 应等于 ssthresh，而不是 2cwnd；

### 两种丢包检测与处理

| 丢包检测方式 | 原因 | 严重性 | 处理方式 |
|:----------:|------|:------:|----------|
| **超时** | 多个报文段丢失，网络拥塞严重 | 严重 | ssthresh = cwnd/2，**cwnd = 1**，重新进入慢开始 |
| **3 个重复 ACK** | 个别报文段丢失，但后续数据到达了——网络仍在传输 | 较轻 | 快重传 + 快恢复 |

### 快重传（Fast Retransmit）

> 见上文 [[3.TCP#快速重传（Fast Retransmit）|快速重传]]。收到 3 个重复 ACK 后，不等超时立即重传。

### 快恢复（Fast Recovery）

> 在快重传后**跳过慢开始**，直接进入拥塞避免阶段。

步骤：
1. ssthresh = cwnd / 2
2. cwnd = ssthresh（不是 1！）
3. 直接进入**拥塞避免**阶段（加法增大）

> [!important] 超时 vs 3 重 ACK 的处理差异
> - **超时**：cwnd → 1 → 慢开始（网络拥塞严重，从头开始探）
> - **3 重 ACK**：cwnd → cwnd/2 → 拥塞避免（网络仍然通畅，不需要过度反应）

## AIMD 原则

TCP 拥塞避免中的窗口调整遵循 **AIMD（Additive Increase / Multiplicative Decrease）** 原则：

| 阶段 | 调整方式 | 速率变化 |
|------|:--------:|----------|
| **加法增大（AI）** | 每 RTT 线性 +1 个 MSS | 缓慢提高发送速率，探测额外带宽 |
| **乘法减小（MD）** | 发生丢包时，cwnd 减半 | 快速降低发送速率，缓解拥塞 |

> AIMD 的良好特性：
> - **收敛性**：多个竞争的 TCP 流最终会公平共享瓶颈链路带宽
> - **公平性**：乘法减小不会给"抢占"留下空间——丢包时大家都减半

## TCP 拥塞控制算法使用总结

| 发生事件 | ssthresh | cwnd | 下一阶段 |
|----------|:--------:|:----:|:--------:|
| **连接建立** | 初始值（较大） | 1 MSS | 慢开始 |
| **慢开始中收到新 ACK** | 不变 | cwnd += 1 MSS（每 RTT 翻倍） | 慢开始，直到 cwnd ≥ ssthresh 时切为拥塞避免 |
| **拥塞避免中收到新 ACK** | 不变 | 每 RTT +1 MSS（线性增长） | 继续拥塞避免 |
| **超时重传** | ssthresh = cwnd / 2 | **cwnd = 1** | 重新慢开始 |
| **收到 3 个重复 ACK** | ssthresh = cwnd / 2 | **cwnd = ssthresh**（不减到 1） | 快恢复 → 拥塞避免 |

> 核心记忆：**超时 → 归零重来；3 重 ACK → 减半继续**。两种丢包的处理差异体现了"网络拥塞程度不同，反应力度不同"的原则。

# ❗ 易错点总结

1. **TCP 面向字节流，UDP 面向报文**——TCP 不保留消息边界，存在粘包问题，需应用层自行界定消息边界
2. **TCP 仅支持单播（一对一）**，不支持多播和广播
3. **序号字段**：本报文段数据部分第一个字节的序号；**确认号字段**：期望收到的下一个字节序号，同时也是累积确认；SYN 和 FIN 即使不携带数据也消耗一个序号
4. **数据偏移（首部长度）以 4 字节为单位**——值 5 表示 20 字节，最大 15 表示 60 字节
5. **TCP 检验和强制计算**（UDP 在 IPv4 中可选），覆盖首部 + 数据 + 伪首部
6. **三次握手的核心目的**：防止已失效的连接请求报文段造成服务器资源浪费；第三次握手可携带数据，前两次不可以
7. **客户端最早在 1RTT 后发送数据**，服务器最早在 1.5RTT 后发送数据
8. **四次挥手比三次握手多一次**：TCP 全双工，服务器收到 FIN 后可能还有未发完的数据，ACK 和 FIN 必须分开发送；此时 TCP 处于**半关闭**状态
9. **TIME_WAIT 等待 2MSL**：①确保最后 ACK 被收到 ②使本连接所有报文段从网络中消失；只有**主动关闭方**进入 TIME_WAIT
10. **累积确认**：确认号 n 表示 n-1 之前全部已收；**SACK 是累积确认的补充而非替代**
11. **重传分两种**：超时重传（计时器超时，RTO 动态计算 + 指数退避）和冗余确认重传（收到 3 个重复 ACK 不等超时立即重传）
12. **发送窗口 = min(cwnd, rwnd)**——既不超过接收方容量，也不超过网络容量
13. **流量控制 vs 拥塞控制**：流量控制关注接收方（rwnd），端到端；拥塞控制关注网络（cwnd），全局
14. **传输层流量控制 vs 数据链路层流量控制**：传输层端到端、rwnd 动态变化、需配合拥塞控制；链路层点到点、窗口固定、无拥塞
15. **零窗口死锁**：rwnd=0 后窗口更新报文段丢失会导致死锁，由**持续计时器**周期性发送零窗口探测解决
16. **拥塞控制四算法**：慢开始（cwnd 指数增长）→ 拥塞避免（线性增长）→ 快重传（3 重 ACK 触发）→ 快恢复（跳过慢开始）
17. **AIMD 原则**：加法增大（AI，每 RTT +1 MSS）探测带宽，乘法减小（MD，cwnd 减半）缓解拥塞——保证收敛性和公平性
18. **超时 → ssthresh=cwnd/2, cwnd=1 → 慢开始**（拥塞严重，归零重来）；**3 重 ACK → ssthresh=cwnd/2, cwnd=ssthresh → 快恢复 → 拥塞避免**（拥塞较轻，减半继续）
19. **慢开始阶段**：若 2cwnd > ssthresh，则下一轮 cwnd = ssthresh（不翻倍）
20. **MSS、窗口扩大因子、SACK 协商仅在 SYN 报文中进行**
