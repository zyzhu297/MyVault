# 传输层的功能

> **传输层**：OSI 参考模型中的第四层，介于**网络层**和**应用层**之间，负责为运行在不同主机上的**应用进程**提供**端到端的逻辑通信服务**。

传输层是**资源子网的最底层**，也是通信子网与资源子网的**分界线**。其主要功能可概括为以下四个方面：

| 核心功能                | 说明                                          |
| ------------------- | ------------------------------------------- |
| **应用进程之间的逻辑通信**     | 为不同主机的应用进程之间提供逻辑通信，屏蔽底层网络的细节                |
| **复用和分用**           | 发送方将多个应用层进程的数据复用到一个传输层连接；接收方将传输层数据分用给不同应用进程 |
| **差错检测**            | 对报文段的首部和数据进行差错检测（如 TCP/UDP 的检验和）            |
| **提供面向连接和无连接的传输服务** | TCP 提供面向连接的可靠服务，UDP 提供无连接的不可靠服务             |

> [!important] 传输层 ⇄ 网络层
> - **网络层**提供**主机之间**的逻辑通信（host-to-host）
> - **传输层**提供**应用进程之间**的逻辑通信（process-to-process）
> - 网络层是通信子网的最高层，传输层是资源子网的最底层——传输层是**承上启下**的关键层次

---

## 应用进程之间的逻辑通信

> **应用进程之间的逻辑通信**：传输层使不同主机上的应用进程看起来像是在**直接通信**，而无需关心中间经过了哪些路由器、用了什么链路——底层的物理细节对应用进程是**透明**的。

| 要点 | 说明 |
|------|------|
| **横向视角** | 从应用进程的角度看，传输层提供了**端到端的逻辑通信信道** |
| **纵向关系** | 传输层依赖网络层提供的**主机到主机的通信服务**（host-to-host），在其基础上增加**进程级寻址**（端口），实现了进程到进程的通信 |
| **本质** | 将网络层的主机间交付服务**增强**为进程间交付服务 |

> [!tip] 为什么需要传输层？
> 网络层（IP 协议）只负责将数据从源主机传送到目的主机，但一台主机上可能同时运行多个应用进程（浏览器、邮件客户端、QQ 等）。网络层无法区分数据应交给哪个进程——这需要传输层通过**端口号**来解决。

---

## 复用和分用

> **复用（Multiplexing）**：发送方的传输层将多个应用进程的报文**汇聚**后，使用同一个网络层协议（IP）发送出去。

> **分用（Demultiplexing）**：接收方的传输层收到网络层交付的数据后，根据**端口号**将数据**分别交付**给对应的应用进程。

| 概念     |    方向     | 核心机制                      | 发生位置    |
| ------ | :-------: | ------------------------- | ------- |
| **复用** | 发送方 → 网络层 | 多个 socket 的数据通过同一 IP 协议发出 | 源主机传输层  |
| **分用** | 网络层 → 接收方 | 根据目的端口号将数据交付给对应 socket    | 目的主机传输层 |

### 分用的两种方式

| 分用方式             | 依据                             | 说明                                                  |
| ---------------- | ------------------------------ | --------------------------------------------------- |
| **无连接分用（UDP）**   | 仅依据**目的端口号**（+ 目的 IP）          | 不同源 IP/源端口发来的 UDP 报文，只要目的 IP 和目的端口相同，都交付给同一个 socket |
| **面向连接的分用（TCP）** | 依据**四元组**（源 IP、源端口、目的 IP、目的端口） | 每条 TCP 连接由唯一的四元组标识，不同源发来的数据交付给不同的 socket            |

## 差错检测

> 传输层对报文段（segment）进行**端到端的差错检测**，通过**检验和（Checksum）** 字段来判断报文段在传输过程中是否发生了比特差错。

| 协议 | 检验和覆盖范围 | 检验和字段长度 |
|------|:------------:|:------------:|
| **UDP** | UDP 首部 + 数据部分 + **伪首部** | 16 位 |
| **TCP** | TCP 首部 + 数据部分 + **伪首部** | 16 位 |

> [!info] 伪首部（Pseudo Header）
> UDP 和 TCP 在计算检验和时，会额外拼接一个**伪首部**（包含源 IP、目的 IP、协议号、UDP/TCP 长度等），其作用是让传输层验证数据是否到达了**正确的主机和正确的协议**，而不仅仅是检测首部和数据本身的比特错误。伪首部**不向下传送也不向上递交**，仅参与检验和计算。

> [!warning] 传输层 vs 网络层的差错检测
> - **IP 首部检验和**仅覆盖 IP 首部，不检测数据部分
> - **传输层检验和**覆盖整个报文段（首部 + 数据 + 伪首部），提供了端到端的数据完整性校验
> - 两者**层次不同、覆盖范围不同**，互补而非替代

### 差错处理策略的差异

| 协议 | 检测到差错后的处理 | 原因 |
|------|:----------------:|------|
| **UDP** | **直接丢弃**，不通知发送方 | UDP 是无连接协议，不维护连接状态，无重传机制，差错报文直接丢弃 |
| **TCP** | 丢弃出错的报文段，后续通过**超时重传**机制要求发送方重传 | TCP 面向连接，通过序号 + 确认 + 重传机制保证可靠交付 |

> [!important] UDP 丢弃 ≠ TCP 重传
> 这是 TCP 和 UDP 在设计哲学上的根本差异——TCP 为可靠性设计，差错后的重传是其核心机制；UDP 为速度和简单性设计，"丢了就丢了"，不做过多的补救。

## 提供面向连接和无连接的传输服务

> TCP/IP 协议族在传输层提供了两种截然不同的传输协议，以适应不同应用的需求：

| 比较维度 | **TCP**（传输控制协议） | **UDP**（用户数据报协议） |
|----------|:-------------------:|:-------------------:|
| **服务类型** | **面向连接**的可靠传输服务 | **无连接**的不可靠传输服务 |
| **连接建立** | 需要三次握手建立连接 | 无需建立连接，直接发送 |
| **可靠性** | 通过确认、重传、序号等机制保证可靠 | 尽最大努力交付，不保证可靠性 |
| **数据顺序** | 保证按序交付 | 不保证顺序 |
| **流量控制** | 有（滑动窗口） | 无 |
| **拥塞控制** | 有（慢开始、拥塞避免、快重传、快恢复） | 无 |
| **报文首部开销** | 20~60 字节 | 8 字节（固定） |
| **适用场景** | 文件传输、电子邮件、Web 浏览等可靠性要求高的应用 | DNS、视频直播、VoIP 等实时性要求高、容忍少量丢失的应用 |
| **数据传输单位** | 报文段（Segment） | 用户数据报（Datagram） |

> [!important] "可靠传输"的含义
> 在计算机网络中，**可靠传输（Reliable Transmission）** 指传输层保证数据能够**不出差错、不丢失、不重复且按序**地从发送方交付给接收方；关键是**使用确认机制对传输的数据进行确认**

> [!tip] 可靠 ≠ 不拥塞、不延迟
> - TCP 的"可靠"指的是**数据正确性**的保证（不错、不丢、不重、不乱），而**不保证**数据传输的**实时性**（延迟）和**带宽**（速率）
> - 可靠传输的成本：确认开销 + 重传延迟 + 序号/缓冲区资源——这是 TCP 比 UDP 慢的根本原因

# 传输层的寻址与端口

## 端口的作用

> **端口（Port）**：传输层用来标识**应用进程**的地址信息，是传输层实现**进程到进程通信**的关键机制。

| 要点 | 说明 |
|------|------|
| **作用** | 让接收方传输层知道应将数据交付给**哪一个应用进程** |
| **本质** | 端口是传输层的**服务访问点（TSAP）**，相当于应用进程在传输层的"门牌号" |
| **软件实现** | 端口由软件实现，是**逻辑概念**而非物理接口 |
| **唯一性** | 一台主机上，同一时刻一个端口号只能分配给一个进程 |

> [!tip] 端口 vs IP 地址
> - **IP 地址**（网络层）：定位**主机**（送到哪个小区哪栋楼）
> - **端口号**（传输层）：定位**进程**（送到哪个房间、给谁）
> - IP 地址 + 端口号 = 网络中唯一标识一个应用进程

---

## 端口号

> **端口号**是一个 **16 位**的整数，范围 $0 \sim 65535$（共 $2^{16}=65536$ 个）。

### 端口号的分类

| 分类 | 范围 | 说明 |
|------|------|------|
| **熟知端口号（Well-known Ports）** | $0 \sim 1023$ | 由 **IANA** 统一分配，绑定到常见的应用层协议 |
| **登记端口号（Registered Ports）** | $1024 \sim 49151$ | 由 IANA 登记，供特定应用程序使用，非系统保留 |
| **短暂端口号（Ephemeral Ports）** | $49152 \sim 65535$ | 客户端进程动态、随机选择，通信结束后释放 |

### 常见熟知端口号

> 以下是 408 考试中经常涉及的应用层协议及其对应端口号。

| 端口号 | 传输层协议 | 应用层协议 | 说明 |
|:------:|:----------|:-----------|------|
| **20** | TCP | FTP（数据连接） | 文件传输时传送数据的连接 |
| **21** | TCP | FTP（控制连接） | 文件传输时传送控制命令的连接 |
| **22** | TCP | SSH | 安全远程登录 |
| **23** | TCP | Telnet | 远程终端登录（明文，不安全） |
| **25** | TCP | SMTP | 电子邮件发送（服务器→服务器或用 户→服务器） |
| **53** | UDP / TCP | DNS | 域名解析（查询主要用 UDP，区域传送用 TCP） |
| **67** | UDP | DHCP（服务器） | DHCP 服务器监听端口 |
| **68** | UDP | DHCP（客户端） | DHCP 客户端监听端口 |
| **69** | UDP | TFTP | 简单文件传输协议（小文件传输） |
| **80** | TCP | HTTP | 万维网超文本传输 |
| **110** | TCP | POP3 | 电子邮件接收（下载后从服务器删除） |
| **143** | TCP | IMAP | 电子邮件接收（邮件保留在服务器） |
| **161** | UDP | SNMP | 简单网络管理协议 |
| **162** | UDP | SNMP（Trap） | SNMP 陷阱报文（代理→管理站） |
| **443** | TCP | HTTPS | HTTP over TLS/SSL（加密） |
| **520** | UDP | RIP | 路由信息协议 |

> [!tip] 记忆技巧
> - FTP 用两个端口：21（控制，先建立）+ 20（数据，后建立）
> - 邮件三大协议：**SMTP**（25，发）+ **POP3**（110，收）+ **IMAP**（143，收）
> - DNS 是唯一同时使用 TCP 和 UDP 的常见协议——查询用 UDP（快），区域传送/超长响应用 TCP
> - 带 **T** 的协议（Telnet、TFTP）和 DHCP、SNMP、RIP 均使用 UDP 传输

> [!important] 端口号使用的关键原则
> - **服务器**使用**熟知端口号**等待客户端连接
> - **客户端**使用**短暂端口号**发起连接
> - 同一台主机上，**传输层协议不同**（TCP vs UDP），可以使用**相同的端口号**而不冲突——端口号是**协议独立的**

## 套接字

> **套接字（Socket）**：IP 地址 + 端口号的组合，是网络中**唯一标识一个通信端点**的抽象。

$$\text{Socket} = \text{IP 地址} : \text{端口号}$$

| 概念 | 说明 |
|------|------|
| **Socket 地址** | （IP 地址，端口号）二元组，如 `192.168.1.1:80` |
| **Socket 对** | 一条连接的完整标识，由源 Socket + 目的 Socket 组成 |
| **唯一标识 TCP 连接** | 四元组：（源 IP，源端口，目的 IP，目的端口） |

> [!info] Socket 的唯一性
> - 对于 **UDP**（无连接），一个 Socket 就可以同时与多个对方 Socket 通信——分用仅依据目的 IP 和目的端口
> - 对于 **TCP**（面向连接），每条连接需要唯一的四元组——服务器的同一个监听 Socket 可以为**多条连接**服务（通过 accept 返回新的 Socket）

> [!warning] Socket ≠ 端口号
> - 端口号只是一个数字（16 位整数），是 socket 的一个组成部分
> - Socket = IP + 端口号，是一个完整的通信端点标识

# 无连接服务与面向连接服务

> 传输层向应用层提供两种不同类型的服务：**面向连接服务**（TCP）和**无连接服务**（UDP）。二者在网络层提供的无连接分组传输基础上，提供不同级别的传输质量。

## 面向连接服务（Connection-Oriented Service）

> 通信前必须先**建立连接**，数据传输过程中维护连接状态，通信结束后**释放连接**。典型代表：**TCP**。

### 三阶段模型

| 阶段 | 说明 | TCP 对应机制 |
|------|------|:----------:|
| **建立连接** | 收发双方交换控制信息，协商参数，确认就绪 | 三次握手 |
| **数据传输** | 在已建立的连接上双向传输数据，保证可靠有序 | 序号、确认、重传、流量控制 |
| **释放连接** | 通信结束后释放资源 | 四次挥手 |

### 特点

1. **可靠性高**：通过确认重传、序号保证、流量控制、拥塞控制等机制，提供**可靠有序**的数据传输
2. **开销较大**：需要维护连接状态（发送/接收缓冲区、序号、确认号、窗口等），首部开销大
3. **适用场景**：对数据**准确性**要求高的应用（文件传输、电子邮件、Web 网页）
4. **不保证实时性**：TCP 的重传和拥塞控制机制可能导致传输延迟增加

## 无连接服务（Connectionless Service）

> 无需建立连接即可直接发送数据，每个报文独立处理，**尽最大努力交付**。典型代表：**UDP**。

### 特点

1. **无需建立连接**：发送方随时可以发送数据，没有连接建立的时延
2. **不保证可靠**：不保证数据一定到达、不保证顺序、不保证不重复——可靠性由应用层负责
3. **开销小**：首部仅 8 字节，无连接状态需要维护
4. **实时性好**：没有拥塞控制和重传机制，适合实时应用
5. **支持多播/广播**：UDP 支持一对多和多对多通信，TCP 仅支持一对一

## 两种服务的对比

| 比较维度         |      面向连接服务（TCP）      |       无连接服务（UDP）        |
| ------------ | :-------------------: | :---------------------: |
| **是否需要建立连接** |      ✅ 需要（三次握手）       |          ❌ 不需要          |
| **通信方式**     |       仅一对一（单播）        |       一对一、一对多、多对多       |
| **可靠性**      |    可靠，保证按序到达、不丢不重     |       不可靠，尽最大努力交付       |
| **流量控制**     |       ✅ 有（滑动窗口）       |           ❌ 无           |
| **拥塞控制**     | ✅ 有（慢开始、拥塞避免、快重传、快恢复） |           ❌ 无           |
| **首部开销**     |       20~60 字节        |          8 字节           |
| **传输效率**     |     较低（有确认、重传开销）      |           较高            |
| **实时性**      |      较差（重传增加延迟）       |           较好            |
| **连接状态维护**   |     收发双方均需维护连接状态      |         无需维护状态          |
| **数据边界**     |    面向**字节流**，无消息边界    |     面向**报文**，保留消息边界     |
> [!important] TCP 和 UDP 都不可或缺
> - **TCP** 解决了"**准确**"的问题——适合对数据完整性高要求的场景
> - **UDP** 解决了"**快**"的问题——适合对实时性高要求、容忍少量丢失的场景
> - 不能简单地认为 TCP 优于 UDP，二者适用于**不同的应用需求**

## 典型互联网应用与传输层协议对照

> 不同应用根据自身需求（可靠性优先 vs 实时性优先）选择 TCP 或 UDP 作为传输层协议。

| 互联网应用 | 应用层协议 | 传输层协议 | 选择原因 |
|:-----------|:----------|:----------:|----------|
| 万维网（Web） | HTTP / HTTPS | **TCP** | 网页内容必须完整、准确交付，不允许任何丢失 |
| 文件传输 | FTP | **TCP** | 文件传输要求 100% 准确，一位差错都可能导致文件损坏 |
| 电子邮件（发送） | SMTP | **TCP** | 邮件内容不允许丢失或出错 |
| 电子邮件（接收） | POP3 / IMAP | **TCP** | 邮件内容不允许丢失或出错 |
| 远程终端访问 | Telnet / SSH | **TCP** | 每个键入字符都不能丢失，需要可靠传输 |
| 域名解析 | DNS | **UDP**（主要）/ **TCP** | 大多数查询简短，UDP 低时延更高效；区域传送和长响应（> 512 B）使用 TCP |
| 动态主机配置 | DHCP | **UDP** | 广播通信（客户端尚无 IP），TCP 无法建立连接 |
| 路由信息交换 | RIP | **UDP** | 路由表周期性广播/多播，实时性高于可靠性 |
| 网络管理 | SNMP | **UDP** | 管理信息需要快速查询，网络故障时 TCP 可能无法建立连接 |
| 网络文件系统 | NFS | **UDP**（或 TCP） | 内部网络环境误码率较低，UDP 开销小、速度快 |
| 实时音视频通信 | RTP | **UDP** | 实时性要求高，少量丢包可接受，TCP 重传反而增加延迟，得不偿失 |
| IP 电话（VoIP） | SIP / RTP | **UDP** | 实时语音通话对延迟极度敏感，TCP 拥塞控制和重传会严重影响通话质量 |
| 流媒体 | HLS / DASH | **TCP** | 基于 HTTP，通过分段下载+缓冲来应对网络波动 |

> [!tip] 选择传输层协议的核心判断
> - **需要可靠传输、不允许丢数据的** → **TCP**（电子邮件、文件传输、Web 页面）
> - **实时性优先、能容忍少量丢包** → **UDP**（实时音视频、VoIP、在线游戏）
> - **需要广播/多播** → **UDP**（TCP 不支持）
> - **特殊情况**：DNS 主要用 UDP（查询短、低时延），但区域传送和超过 512B 的响应走 TCP

# ❗ 易错点总结

1. **传输层是资源子网的最底层**，也是通信子网与资源子网的分界线；网络层是通信子网的最高层
2. **网络层**提供**主机到主机**的逻辑通信；**传输层**提供**进程到进程**的逻辑通信——二者的服务对象不同
3. **复用**：多个应用进程共用同一个传输层协议发送数据；**分用**：接收方传输层根据端口号将数据交付给正确的进程
4. **UDP 分用**仅依据目的 IP + 目的端口；**TCP 分用**依据完整四元组（源 IP、源端口、目的 IP、目的端口）
5. **端口号是 16 位**，范围 $0 \sim 65535$，分为熟知端口（$0 \sim 1023$）、登记端口（$1024 \sim 49151$）和短暂端口（$49152 \sim 65535$）
6. **Socket = IP 地址 + 端口号**，唯一标识网络中的一个通信端点；端口号只是 Socket 的一个组成部分
7. **同一传输层协议下**，一台主机上同一时刻一个端口号只能分配给一个进程；**不同传输层协议**（TCP vs UDP）可以使用相同端口号
8. **传输层检验和**覆盖首部+数据+伪首部，不仅检测报文段自身的比特错，还验证数据是否到达了正确的主机和协议
9. **伪首部**仅参与检验和计算，不向下传送也不向上递交
10. **TCP 面向字节流**（无消息边界），**UDP 面向报文**（保留消息边界）——这是 TCP 和 UDP 的**核心语义差异**之一
11. TCP 适用于可靠性要求高的场景，UDP 适用于实时性要求高的场景——两者各有所长，无绝对优劣
12. 面向连接服务的三阶段：建立连接 → 数据传输 → 释放连接（TCP 三次握手 → 数据传输 → 四次挥手）
13. UDP 支持**多播和广播**，TCP 仅支持**单播**（一对一）
