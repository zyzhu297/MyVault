# 异常和中断的基本概念

在程序执行过程中，CPU 遇到特殊情况需要暂停当前程序转而处理紧急事务，处理完毕后再回到原程序继续执行，这种机制称为**中断系统**
- **内中断（异常）**：来自 CPU 内部的事件，与正在执行的指令相关
- **外中断（中断）**：来自 CPU 外部的事件，与当前指令无关

## 中断的作用

1. **实现 CPU 与 I/O 设备的并行工作** — 无需轮询，外设就绪时主动通知 CPU
2. **实现多道程序并发执行** — 操作系统通过时钟中断实现时间片轮转调度
3. **处理硬件故障和程序错误** — 如掉电中断、除零异常
4. **实现人机交互** — 键盘、鼠标等输入设备通过中断通知系统
5. **实现系统调用** — 用户程序通过陷阱指令（如 `int` / `syscall` / `ecall`）请求内核服务

## 异常 vs 中断的区别

| 比较维度   | 异常（Exception）          | 中断（Interrupt）         |
| ------ | ---------------------- | --------------------- |
| 产生来源   | CPU 内部，由指令执行触发         | CPU 外部，由外设或时钟等信号触发    |
| 与指令的关系 | **同步**，与当前正在执行的指令有关    | **异步**，与当前指令无关        |
| 响应方式   | 在当前指令执行过程中检测并响应        | 在每条指令执行结束时的**中断周期**检测 |
| 返回行为   | 故障→返回当前指令重试；陷阱→返回下一条指令 | 返回被中断程序的下一条指令继续执行     |
>[!tips] 按照软硬件划分
>故障和自陷属于**软件中断**，终止和外中断属于**硬件中断**
# 异常的分类

异常是 CPU 在执行指令时检测到的内部事件，分为三类：

## 故障（Fault）

- **特点**：可恢复，异常处理后**返回发生故障的指令重新执行**
- **发生时机**：指令执行过程中检测到
- **保存的 PC**：指向故障指令的地址
- **常见例子**：
  - **缺页异常**（Page Fault）— 页面不在内存中，调入后重新执行
  - **除零错误**（Divide Error）— 除数=0（有些架构归类为 Abort）
  - **对齐故障**（Alignment Fault）— 地址未对齐
  - **段不存在故障**（Segment Not Present）
  - **非法操作码**（Illegal Opcode / Undefined Instruction）— CPU 遇到未定义的指令编码

> [!tip] 可恢复 vs 不可恢复的故障
> 多数故障是可恢复的（如缺页、段不存在），操作系统处理后重新执行故障指令即可。但像**除零**、**非法操作码**这类故障，即便回到原指令重新执行，错误条件依然存在，因此操作系统通常会**终止引起故障的进程**而非重试

## 陷阱（Trap）

- **特点**：**预先安排**的"异常"，**异常处理后返回下一条指令继续执行**
- **发生时机**：指令执行结束后
- **保存的 PC**：指向陷阱指令的下一条指令地址
- **用途**：实现**系统调用**，让用户程序可控地进入内核态
- **常见例子**：
  - x86：`int n`、`int3`、`into`、`syscall`/`sysenter`
  - ARM：`SVC`（原 `SWI`）
  - RISC-V：`ecall`
- **为什么返回下一条**：陷阱指令是故意的，不需要重复执行它

## 终止（Abort）

- **特点**：**不可恢复**的致命错误，程序无法继续执行，有不可预测性
- **保存的 PC**：通常无意义
- **常见例子**：
  - **硬件故障**（Machine Check）— 内存奇偶校验错误、总线错误
  - **双重故障**（Double Fault）— 在处理一个异常时又发生异常
  - **三重故障**（Triple Fault）— 连双重故障处理都失败，直接导致 CPU 复位

## 按检测方式分类

| 分类      | 说明               | 示例                   |
| ------- | ---------------- | -------------------- |
| 硬件检测的异常 | CPU 硬件在执行指令时自动检测 | 除零、缺页、地址越界           |
| 软件设置的异常 | 通过执行特定指令主动触发     | `int 0x80`、`syscall` |

# 中断的分类

中断是来自 CPU 外部的事件通知，可按多种方式分类：

## 按是否可屏蔽

### 可屏蔽中断（Maskable Interrupt, INTR）

- 可通过设置 CPU 状态寄存器中的中断允许标志位（IF）来屏蔽
- 关中断（CLI）后，CPU 不再响应此类中断
- 由外部设备的**中断控制器**（如 8259A PIC / APIC）发送
- 典型来源：键盘、鼠标、硬盘、网卡等 I/O 设备

### 不可屏蔽中断（Non-Maskable Interrupt, NMI）

- **不能被屏蔽**，CPU 必须立即响应
- 优先级最高，用于紧急事件
- 典型来源：
  - 电源掉电警告
  - 内存奇偶校验错误
  - 看门狗定时器超时

## 按中断源分类

| 类型 | 例子 |
|------|------|
| I/O 中断 | 磁盘读写完成、网络数据到达 |
| 时钟中断 | 操作系统时间片轮转调度 |
| 硬件故障中断 | 掉电、内存 ECC 错误 |

## 按服务入口地址的获取方式

### 向量中断（Vectored Interrupt）

- 中断源通过硬件提供**中断向量号**，CPU 据此查表（IVT/IDT）获取 ISR 入口地址
- 响应速度快，无需软件轮询，被现代计算机广泛采用
- 每个中断源有自己独立的向量号和处理程序

### 非向量中断（Non-vectored Interrupt）

- 所有中断共享**一个固定的入口地址**，由软件逐个查询中断源状态寄存器来确定是谁发起的中断
- 响应速度较慢（软件轮询开销），结构简单但效率低
- 常见于早期简单系统或中断源较少的嵌入式场景

## 按是否允许嵌套

### 单重中断

- ISR 执行期间**禁止响应其他中断**（IF 保持=0）
- 中断处理流程为严格的"串行"模式，一个处理完才处理下一个
- 优点：实现简单，无需考虑重入和栈溢出
- 缺点：紧急中断可能被延迟处理

### 多重中断（中断嵌套）

- ISR 执行期间**允许响应更高优先级的中断**（执行 STI 开中断，IF=1）
- 类似函数调用的嵌套结构，高优先级 ISR 执行完后返回低优先级 ISR 继续
- 优点：高优先级中断能得到及时响应
- 缺点：需要**中断屏蔽字**配合管理嵌套，嵌套过深可能导致**栈溢出**

## 中断优先级

多个中断同时请求时，按优先级仲裁：

1. **硬件故障（NMI）** >
2. **不可屏蔽中断** >
3. **可屏蔽中断**（由中断控制器确定优先级）>
4. **内部异常**（通常有独立的固定优先级）



# 异常和中断的响应过程

当 CPU 在执行指令过程中检测到异常，或在指令边界处采样到中断请求时，启动*异常或中断响应*过程；该过程指**从事件发生到转移至相应处理程序之间**的**硬件自动操作**序列，主要包括**关中断、保存断点和程序状态**、以及**识别事件类型并转移至处理程序**三个阶段

## 中断周期与指令周期的关系

**指令周期**是 CPU 执行一条指令的完整过程，由若干子周期组成：

```
取指周期 → 间址周期(若有) → 执行周期 → 中断周期
```

**中断周期**是每条指令执行完毕后才进入的最后一个子周期。CPU 在此周期中检查是否有中断请求（查询 INTR/NMI 引脚），其位置决定了中断的**异步性**——中断一定发生在指令边界，而非指令执行中途。

> [!important] 关键区别
> - **异常**：在指令执行**过程中**检测并响应，打断当前指令，不经过中断周期
> - **中断**：在指令执行**结束后**的中断周期被采样，若 IF=1 且存在请求则启动响应，放弃进入下一条指令的取指周期
> - 中断周期存在中断请求时，CPU 执行**中断隐指令**（硬件自动操作序列），然后转入 ISR；不存在时则直接进入下一条指令的取指周期

## 详细步骤

### 1. 异常/中断的检测

- **异常**：在指令执行**过程中**由 CPU 内部硬件检测
- **中断**：在每条指令执行结束时的**中断周期**检测 INTR/NMI 信号

在 x86 的五级流水线中，异常通常在相应阶段被检测：
- 取指阶段：缺页（指令不在 TLB 中）
- 译码阶段：非法指令
- 执行阶段：除零、溢出
- 访存阶段：缺页（数据不在 TLB 中）、地址越界

### 2. 关中断（Disable Interrupt）

响应过程的第一步就是**清除 IF 标志位**（关中断），禁止响应新的可屏蔽中断，以保证后续"保存现场"和"转入处理程序"这一硬件操作序列**不被其他中断打断**。

> [!important] 关中断的要点
> - **只屏蔽 INTR**，NMI 不受影响（不可屏蔽中断始终可响应）
> - 关中断由**硬件自动完成**，无需软件执行 `CLI` 指令
> - 进入中断门（Interrupt Gate）时**自动关中断**；进入陷阱门（Trap Gate）时**保持 IF 不变**
> - 若处理程序需要支持中断嵌套，可在 ISR 中执行 `STI` **手动重新开中断**

### 3. 保存现场（保存断点和程序状态）

硬件自动完成以下保存操作（不可打断），这些信息通常被压入**内核栈**：

- **保存断点**
  - 将当前指令的地址或下一条**指令地址**（取决于异常类型）压入栈
  - x86：保存 **CS:EIP**（32位）/ **CS:RIP**（64位）
- **保存程序状态字**
  - x86：保存 **EFLAGS/RFLAGS** 寄存器，包含条件码、中断允许标志、特权级等运行状态
- **切换特权级**（如果需要）
  - 从用户态（Ring3）切换到内核态（Ring0）
  - 切换栈指针 SS:ESP / SS:RSP

### 4. 识别中断源（中断向量号）

CPU 需要知道发生了哪种异常/中断，以找到对应的处理程序。根据系统采用的机制，分为两种方式：

#### 向量机制（向量中断/异常）

中断源通过硬件直接提供向量号，CPU 据此查表获取 ISR 入口地址：

- **异常**：每个异常有**固定的中断向量号**（CPU 内部硬件预定义）
  - x86 示例：除零异常 = 0 号，调试异常 = 1 号，缺页异常 = 14 号
- **中断**：中断向量号由**中断控制器**（如 8259A PIC / APIC）提供
  - CPU 在中断响应周期发送 **INTA**（中断应答）信号
  - 中断控制器将对应的向量号放到数据总线上
  - CPU 读取该向量号

> [!info] 中断向量
> **中断向量**就是中断服务程序（ISR）的**入口地址**。每个中断源都有自己唯一的中断向量，CPU 在响应中断时通过**中断向量号**（向量在表中的索引）查表获取该地址，然后跳转执行。可以类比为"中断处理函数的指针"——向量号是数组下标，中断向量表就是函数指针数组。

#### 软件轮询（非向量中断）

当系统采用非向量中断时，没有硬件提供向量号，所有中断源共享一个固定入口。ISR 进入后由**软件**按优先级从高到低依次读取各外设的**中断状态寄存器**，检查其中断标志位，找到第一个置位的中断源即为当前请求方：

- 轮询顺序决定了设备的**响应优先级**（先被查询的设备优先被服务）
- 优点：硬件成本低，无需中断控制器配合
- 缺点：中断源越多，轮询开销越大，中断响应延迟越长


### 5. 查找并转入中断处理程序

**实模式（8086）**：中断向量表 IVT

- 位于物理地址 `0x00000`，共 1024 字节
- 每个表项 4 字节：2 字节段地址 + 2 字节偏移地址
- 向量号 n → 处理程序地址 = `IVT[n]`（地址 = `n × 4`）

**保护模式（x86/x64）**：中断描述符表 IDT

- 位置由 **IDTR 寄存器**指定
- 每个表项为 8 字节（32位）/ 16 字节（64位）的**门描述符**
- 门类型：
  - **中断门**（Interrupt Gate）：进入时自动关中断（清除 IF 标志）
  - **陷阱门**（Trap Gate）：进入时保持 IF 不变（用于系统调用）
  - **任务门**（Task Gate）：通过硬件任务切换（少用）
- CPU 通过向量号索引 IDT，取出门描述符，得到处理程序入口地址

### 6. 执行中断服务程序（ISR）

处理程序分为两部分：

- **前半部分**：保存更多寄存器（通用寄存器等），执行核心处理逻辑
- **后半部分**：恢复寄存器，执行 `IRET`/`IRETD`/`IRETQ` 指令返回

### 7. 恢复现场并返回

`IRET` 指令完成以下操作（硬件自动完成）：

1. 从栈中弹出 **EFLAGS**，恢复原程序状态字
2. 从栈中弹出 **EIP**（指令指针）
3. 从栈中弹出 **CS**（代码段选择子）
4. 如果切换了特权级，还需恢复用户态栈指针 **SS:ESP**

## 硬件 vs 软件的任务划分

中断处理的完整流程由**硬件**和**软件**协作完成，两者的职责边界是考试常考点：

| 阶段               | 负责方    | 具体操作                                |
| ---------------- | ------ | ----------------------------------- |
| 关中断              | **硬件** | 自动清除 IF 标志位                         |
| 保存断点和 PSW        | **硬件** | 将 CS:EIP、EFLAGS 自动压入内核栈             |
| 获取向量号            | **硬件** | 异常由 CPU 内部逻辑确定；中断由 INTA 总线周期读取      |
| 查表转入 ISR         | **硬件** | 以向量号为索引，从 IVT/IDT 中取出入口地址并跳转        |
| 保存通用寄存器          | **软件** | ISR 开头用 `PUSH` / `PUSHA` 指令保存 R0~Rn |
| 执行服务逻辑           | **软件** | ISR 的核心代码，如读取外设数据、处理缺页等             |
| 恢复通用寄存器          | **软件** | ISR 结尾用 `POP` / `POPA` 指令恢复 R0~Rn   |
| 执行 IRET 指令       | **软件** | ISR 最后一条指令（程序员显式写出）                 |
| 弹出 CS:EIP、EFLAGS | **硬件** | IRET 指令触发硬件自动从栈中弹出并恢复               |
| 开中断              | **硬件** | 弹出 EFLAGS 后 IF 自动恢复为中断前的值           |

> [!tip] 软硬件协作
> **硬件完成响应过程**：进入时关中断→保存断点→查表转入；退出时 IRET 自动弹出恢复。
> **软件完成处理逻辑**：保护寄存器→处理逻辑→恢复寄存器→写一条 IRET。

## 中断响应周期的数据流

1. 中断请求信号到达 → 在中断周期 CPU 查询 INTR/NMI
2. IF=1 且有中断请求 → CPU 发出 INTA 响应信号
3. 中断控制器将向量号放到数据总线
4. CPU 读取向量号
5. 保存 FLAGS、CS、IP
6. 从 IDT/IVT 获取 ISR 地址
7. 转入 ISR 执行

## 多重中断（中断嵌套）

- 在处理一个中断的过程中，CPU 可响应更高优先级的中断
- 条件：ISR 中**打开了中断**（STI / IF=1）
- 机制：ISR 执行期间发生的中断暂存，待当前指令结束或 IF 置位后响应
- 需要注意**栈溢出**风险（嵌套过深）
