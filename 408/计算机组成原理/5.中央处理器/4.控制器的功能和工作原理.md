# 控制器的结构和功能

**控制器**是 CPU 的**指挥中心**，负责协调和指挥整个计算机系统的操作。它根据指令的操作码和当前状态，按照正确的**时序**产生一系列**控制信号（微命令）**，驱动数据通路各部件完成指令规定的操作。
**CU** 即 **Control Unit**（控制单元），是**控制器的硬件实现实体**，"CU"、"控制器"、"控制单元"三者**基本等价**
## 控制器的基本功能

| 功能         | 说明                                |
| ---------- | --------------------------------- |
| **取指令**    | 根据 PC 的值从主存取出指令，送入 IR；自动形成下一条指令地址 |
| **分析指令**   | 对 IR 中的操作码进行译码，识别指令类型和寻址方式        |
| **执行指令**   | 按正确的**时间顺序**发出一组微命令，控制各部件完成指令功能   |
| **中断处理**   | 响应异常/中断请求，保存断点和现场，转入中断服务程序        |
| **时序控制**   | 产生全机统一的时序信号（节拍、脉冲），保证各操作有序进行      |
| **数据流向控制** | 管理 CPU、主存和 I/O 设备间的数据流向           |

> [!important] 控制器的本质
> 控制器本质上是一个**有限状态机**（FSM）：当前状态由指令的操作码、执行阶段（取指/间址/执行/中断）、状态标志等决定；根据当前状态产生一组控制信号输出（微操作序列），并在时钟驱动下转换到下一状态。

## 控制器的基本组成

控制器由以下核心部件构成：

| 部件                   | 功能                             | 输入                         | 输出                         |
| -------------------- | ------------------------------ | -------------------------- | -------------------------- |
| **程序计数器**（PC）        | 存放下一条指令地址；支持自增和跳转              | PC+1 结果 / 转移目标地址 / 中断向量    | 指令地址 → MAR                 |
| **指令寄存器**（IR）        | 存放当前执行的指令                      | MDR（从主存取出的指令）              | 操作码 → 译码器；地址码 → 寄存器堆 / MAR |
| **指令译码器**（ID）        | 对操作码译码，产生指令识别信号                | IR 的 OP 字段                 | 指令类型信号 → 控制信号产生电路          |
| **时序产生器**            | 产生节拍脉冲，划分机器周期                  | 时钟信号（CLK）                  | 节拍信号（T0, T1, T2...）+ 脉冲信号  |
| **微操作信号发生器**         | **核心部件**——综合所有输入信号，产生各微操作控制信号  | 指令译码信号 + 时序信号 + 状态标志       | 控制信号组（送往各执行部件）             |
| **存储器地址寄存器**（MAR）    | 存放访存地址                         | PC（取指）/ ALU 输出（LOAD/STORE） | 地址总线 → 主存                  |
| **存储器数据寄存器**（MDR）    | 暂存与主存交换的数据                     | CPU 内部总线（写）/ 主存数据总线（读）     | 主存数据总线（写）/ CPU 内部总线（读）     |


# 硬布线控制器

**硬布线控制器**（Hardwired Controller）由复杂的组合逻辑电路和触发器组成， 又称为**组合逻辑控制器**，它根据指令需求、当前时序和内部状态，按时间序列生成一系列**微操作控制信号**，以完成指令的执行；指令的操作码是决定**控制单元（CU）** 行为的关键；
为简化 CU 的设计，通常将 IR 中的 $n$ 位操作码通过译码电路转换为 $2^n$ 个输出信号，每种操作码经译码后，会唯一的激活一条输出信号线，用于触发对应的控制逻辑
硬布线控制器将微操作信号发生器实现为**组合逻辑网络**：
- 输入：
  - 指令译码信号，与节拍信号共同用于生成相应的微操作控制信号
  - **时钟脉冲**，其频率即机器主频，用于划分节拍，确保控制信号按时发出
  - 执行单元反馈的状态标志，使 CU 能根据 CPU 当前状态动态调整控制逻辑
- 输出：各控制信号的逻辑电平（高/低）
- 本质：一组**布尔逻辑函数**，$C_i = f(\text{opcode}, \text{timing}, \text{flags})$
>[!important] 微操作控制信号的形成的主要因素
>是指令译码信号和节拍信号
## 硬布线控制器的优缺点

| 优点                               | 缺点                                 |
| -------------------------------- | ---------------------------------- |
| **速度快**——控制信号由组合逻辑直接产生，主要受限于电路延迟 | **设计复杂、难以调试**——指令增多时逻辑表达式急剧膨胀，难以设计 |
| **不需要控制存储器**，硬件开销小（仅门电路）         | **不易修改**——指令集变动需要重新设计整个组合逻辑        |
| 适合**RISC 处理器**（指令少、格式规整）         | **缺乏规整性**——不同指令的控制逻辑分散在电路中         |
| 控制信号产生与微操作一一对应，控制精确              | **难以实现复杂指令**（如 CISC 的串操作、复杂寻址）     |

# 微程序控制器

**微程序控制器**（Microprogrammed Controller）采用**存储逻辑**实现，其核心思想是：将控制器所需的**微操作**控制信号组织为**微指令**，并将每条机器指令的执行所需的**微指令序列（微程序）** 预先存入一个专用的**高速存储器** 中，执行机器指令的过程就是**逐条取出微指令并执行**的过程。

> [!note] 微程序控制器 ≈ "计算机中的计算机"
> 如果把 CPU 看作第一层"计算机"，那么微程序控制器内部就是第二层"计算机"：
> - 第一层：执行**机器指令**（程序存在主存）
> - 第二层：执行**微指令**（微程序存在控存）

## 微程序控制基本概念
微程序的设计思想是将每条机器指令编写为一段**微程序**；该微程序由若干**微指令**组成，每条微指令可产生一个或多个**微命令**；因此，执行一条机器指令的过程，实际上就是顺序执行其对应微程序的过程；这些微程序预先存放在专用的**控制存储器（CM）** 中；

### 微命令与微操作

| 概念                       | 定义                          | 关系        | 示例                                            |
| ------------------------ | --------------------------- | --------- | --------------------------------------------- |
| **微命令**（Micro-command）   | 控制信号的**最小单位**，直接作用于某个部件的控制端 | 微命令是**因** | `PC_out`, `MAR_in`, `MEM_R`, `ALU_add`        |
| **微操作**（Micro-operation） | 执行部件在微命令作用下完成的**最基本的操作**    | 微操作是**果** | `MAR ← (PC)`, `MDR ← M(MAR)`, `PC ← (PC) + 1` |

> [!important] 两类微命令
> - *相容性微命令*是指同时有效，协同完成某一功能的微命令
> - *互斥性微命令*是指在同一时刻不允许同时有效的微命令

### 微指令与微周期

| 概念                         | 定义                           | 时间跨度                 |
| -------------------------- | ---------------------------- | -------------------- |
| **微指令**（Micro-instruction） | 在一个微周期内**同时执行**的一组微命令的**集合** | 存放于 CU 的一个字中         |
| **微周期**（Micro-cycle）       | 执行一条微指令所需的时间                 | 通常 = 1 个时钟周期（若控存足够快） |

一条微指令通常包含两个字段：
- **操作控制字段**（微操作码字段）：用于生成当前步骤所需的微操作控制信号
- **顺序控制字段**（下址字段）：包含下一条微指令的地址（或地址形成方式）

> [!note] 微周期与时钟周期的关系
> 在大多数教材中，**一个微周期 = 一个时钟周期**（CPU 周期），即：
> - 时钟上升沿：从控存取出微指令 → μIR
> - 微指令的操作控制字段直接输出控制信号
> - 控制信号在整个微周期内有效，驱动各部件完成微操作
> - 下一个时钟上升沿：取出下一条微指令
>
> 如果控存读取速度跟不上 CPU 时钟，则微周期可能 = 多个时钟周期。

### 主存储器与控制存储器

| 维度       | 主存储器（Main Memory, MM） | 控制存储器（Control Memory/Store, CM/CS）             |
| -------- | --------------------- | ---------------------------------------------- |
| **存放内容** | **程序**（机器指令序列）+ 数据    | **微程序**（微指令序列）                                 |
| **访问者**  | CPU（取指令、LOAD/STORE）   | 微程序控制器内部（μPC 寻址）                               |
| **物理实现** | DRAM / SRAM           | **ROM** / EPROM / Flash（微程序固件），**属于 CPU 的一部分** |
| **容量**   | 很大（GB 级别）             | 很小（KB~MB 级别）                                   |
| **可改性**  | 用户可读写                 | 通常固化（ROM），部分可改写                                |
| **速度要求** | 相对较慢                  | **必须很快**——每个微周期都要读取一次                          |

> [!important] 控存为什么用 ROM？
> 1. 微程序是 CPU 设计的一部分，出厂后不需要频繁修改
> 2. ROM 比 RAM 读取更快且功耗更低
> 3. 非易失性——断电后微程序不丢失
> 4. 现代 CPU 中有些用 **Flash** 或 **可擦写 ROM**，允许微码更新

### 程序与微程序

这是理解微程序控制器**最关键的概念区分**：

| 维度         | 程序（Program）                     | 微程序（Microprogram）           |
| ---------- | ------------------------------- | --------------------------- |
| **组成单元**   | **机器指令**（如 ADD, LOAD, JMP）的有序集合 | **微指令**（一组微命令的集合）的有序集合      |
| **存放位置**   | **主存储器**                        | **控制存储器**（CM）               |
| **编写者**    | **软件程序员**（用户）                   | **CPU 设计者**（硬件工程师）          |
| **功能粒度**   | 完成一个完整的操作（如两数相加并写回）             | 完成一个最基本的微操作（如一个寄存器的传送）      |
| **可见性**    | **对程序员可见**（汇编/C 语言操作的对象）        | **对程序员透明**——用户无法直接操作微程序     |
| **执行者**    | **CPU**（取指→译码→执行）               | **微程序控制器**（取微指令→发微命令→执行微操作） |
| **执行顺序**   | 由 PC 控制（含跳转）                    | 由 μPC 控制（含微程序转移）            |
| **一条指令对应** | —                               | **一段微程序**（通常若干条微指令）         |

> [!important] 区分以下关键寄存器
> 在微程序控制器中存在**两套寄存器**——一套服务于"程序层"（机器指令），一套服务于"微程序层"（微指令）：
> - **MAR**（存储器地址寄存器）：存放**主存**读/写地址
> - **μPC**（微地址寄存器 / CMAR）：存放**控存**地址，指向下一条微指令
> - **IR**（指令寄存器）：存放当前正在执行的**机器指令**
> - **μIR**（微指令寄存器 / CMDR）：存放当前正在执行的**微指令**
>
> 易混淆辨析：
> - **MAR ≠ μPC**：MAR 指向主存，μPC 指向控存，两者地址空间完全不同
> - **IR ≠ μIR**：IR 存机器指令，μIR 存微指令，是不同层次，没有包含关系
> - **μPC ≠ PC**：PC 指向下一条机器指令，μPC 指向下一条微指令，μPC 更新频率远高于 PC
> - **控存 ≠ 主存**：控存是独立的 ROM，与主存物理分离

## 微程序控制器的组成和工作原理

### 微程序控制器的基本组成

| 核心部件 | 功能说明 |
|----------|----------|
| **控制存储器**（CM） | 存放所有微程序（由微指令组成）；由 ROM 实现；每个微地址对应一条微指令 |
| **微指令寄存器**（μIR） | 存放当前从 CM 取出的微指令；操作控制字段直接驱动控制信号；顺序控制字段送地址转移逻辑 |
| **微地址寄存器**（μPC） | 存放下一条微指令在 CM 中的地址；可由多种来源加载（μPC+1、微地址形成逻辑、下地址字段等） |
| **微地址形成逻辑** | 根据机器指令的**操作码**，生成该指令对应微程序的**入口地址**（首条微指令在 CM 中的地址） |
| **地址转移逻辑** | 根据当前微指令的顺序控制字段和状态标志，决定下一条微指令的地址（顺序执行 / 条件转移 / 无条件转移） |
| **时序产生器** | 产生微周期节拍，控制取微指令和执行微指令的时序 |

### 微程序控制器的工作过程
是指计算机在微程序控制下执行机器指令的流程，具体如下：
1. **执行取指令的公共操作**：*系统启动或每条指令执行完毕后*，将取指微程序的入口地址（通常为 CM 的 0 号单元）送入 μPC；随后，从 CM 中读取首条取指微指令送入 μIR；完成取指微程序后，从主存中读取的机器指令即被存入指令寄存器（IR）中
2. **生成当前指令的微程序入口地址**：IR 中操作码经**微地址形成逻辑**映射为对应微程序在控存中的首地址，加载到 μPC
3. **循环执行微程序**：μPC 寻址控存 → 取微指令到 μIR → 操作控制字段直接发出微命令 → 顺序控制字段经地址转移逻辑形成下一 μPC；如此循环直到微程序结束
4. **返回循环**：微程序末尾由地址转移逻辑将 μPC 无条件置为取指微程序入口，开始下一条机器指令

### 微程序和机器指令
**一条机器指令对应一个微程序**；由于所有指令的取指操作相同，可将取指操作的微命令 统一编写为一个**公共的取指微程序**，专门负责从主存读取指令送入 IR；此外，还可以为间址、中断处理等公共操作分别编写独立程序；因此，CU 中存储的**微程序总数 = 机器指令条数 + 公共微程序数**（如取指、间址、中断等）
## 微指令的编码方式

又称*微指令的控制方式*，微指令的**操作控制字段**需要编码表示所有控制信号。编码方式直接影响微指令字长、控存容量和控制信号产生速度

### 直接编码（Direct Encoding / Unencoded）

操作控制字段中**每一位直接对应一个微命令**，该位为 1 表示发出该微命令，为 0 表示不发出

| 特点       | 说明                                                                |
| -------- | ----------------------------------------------------------------- |
| **优点**   | **速度最快**，简单直观，控制信号直接从 μIR 的位输出，无译码延迟；<br>**并行性好**所有不冲突的微命令可以同时发出； |
| **缺点**   | 微指令字长**极长**，控存容量随控制信号数量线性增长                                       |
| **适用场景** | 控制信号数量较少的简单 CPU；对速度有极端要求的场景                                       |

> [!warning] 直接编码的限制
> 虽然每位独立控制，但仍需满足**微命令的相容性约束**——不能在同一条微指令中同时发出冲突的微命令（如同时 `PC_out` 和 `MDR_out` 驱动同一条总线），这需要通过**微程序设计规范**来保证，而非编码方式本身。

### 字段直接编码（Field Direct Encoding）

将操作控制字段划分为若干**互斥段**；**互斥性命令被归入同一字段，相容性命令则分配到不同字段**；每个字段独立编码，经译码后在其所属的互斥微命令集中激活一个微命令；各字段之间相互独立，编码互不影响；

> [!important] 互斥性微命令
> **互斥性微命令**是指在**同一个微周期内不可能同时有效**的微命令，即它们在逻辑上互相排斥，发出其中一个就必然不能发出另一个。例如：
> - `PC_out`、`MDR_out`、`Z_out`、`GPR_out` 等**总线输出使能**信号——同一时刻最多只能有一个部件驱动 CPU 内部总线
> - `ALU_add`、`ALU_sub`、`ALU_and`、`ALU_or` 等**ALU 操作选择**信号——ALU 一次只能做一种运算
> - `MEM_R` 和 `MEM_W`——不能同时对主存既读又写
>
> 与之相反，**相容性微命令**是可以同时有效的微命令，例如 `MDR_out`（驱动总线）和 `IR_in`（锁存总线数据）就是相容的——它们协同完成 `IR ← (MDR)` 这个微操作。
>
> 字段直接编码正是利用了这个性质：将互斥的微命令归入同一字段，通过一个译码器保证**任何时候该字段最多只激活一个微命令**，从而天然避免了冲突。

微命令字段分段须遵循以下原则：
- 互斥性微命令应分配在同一字段，而相容性微命令应分布在不同字段
- 每个字段的位数不宜过多，以避免译码电路过于复杂，引入较大延迟
- 每个字段通常还需**预留一个编码表示本字段无操作**，例如，字段长度为 3 位时，最多只能表示 7 个互斥微命令，通常用 000 表示误操作

| 特点       | 说明                                                      |
| -------- | ------------------------------------------------------- |
| **优点**   | 微指令字长大幅缩减；互斥微命令不会同时发出，避免总线冲突                            |
| **缺点**   | 需要**译码电路**，每个字段需要一个小型译码器，增加了控制信号<br>产生的一级门延迟；灵活性略低于直接编码 |
| **适用场景** | **大多数实际微程序控制器**的首选编码方式（速度和面积的折中）                        |


### 字段间接编码（Field Indirect Encoding）

一个字段的**含义**由另一个字段的取值决定，相当于字段间存在"编码依赖"。

```
示例：
字段A (操作类型):
  00 = 数据传送类  →  字段B 被解释为"传送方向"
  01 = ALU运算类   →  字段B 被解释为"ALU操作选择"
  10 = 移位类      →  字段B 被解释为"移位类型"
  11 = 访存类      →  字段B 被解释为"访存类型"
```

| 特点 | 说明 |
|------|------|
| **优点** | 进一步压缩微指令字长——同几位可以表示更多种微命令 |
| **缺点** | 需要**两级译码**（先译字段A确定含义，再按此含义译字段B），延迟更大；控制逻辑复杂，设计易出错 |
| **适用场景** | 控制信号非常多且编码空间极为紧张的场景；实际中**较少单独使用**，更多是在某些字段中局部使用 |

### 三种编码方式的对比

| 维度 | 直接编码 | 字段直接编码 | 字段间接编码 |
|------|---------|-------------|-------------|
| **微指令字长** | 最长 | 中等 | 最短 |
| **译码级数** | 0（无需译码） | 1 级（段内译码） | 2 级（段间译码 + 段内译码） |
| **控制信号产生速度** | 最快 | 较快 | 较慢 |
| **并行性** | 最高 | 高（互斥信号不会同时） | 较高 |
| **设计复杂度** | 最简单 | 中等 | 最复杂 |
| **灵活性** | 最低（每位固定含义） | 中等 | 最高 |
| **实际应用** | 很少（字长太长） | **主流方案** | 较少单独使用 |
| **控存容量需求** | 最大 | 中等 | 最小 |

## 微指令的地址形成方式

微程序执行过程中，需要确定**下一条微指令的地址**。地址形成方式有以下几种：

### 增量方式（计数器方式）

类似 CPU 中的 PC，下条微地址由**微程序计数器**（μPC）自动加 1 生成，指向下一条微指令，适用于顺序执行段

| 特点 | 说明 |
|------|------|
| 优点 | 简单，类似 CPU 的 PC 机制；顺序执行时微指令无需携带下地址字段 |
| 缺点 | 转移时需要额外的地址加载电路 |
| 适用 | 垂直型微指令（微指令较短，不希望占用下地址字段） |

### 断定方式（下地址字段方式）

每条微指令的**顺序控制字段**中**显式指定**下一条微地址

### 下条微地址的确定
- **微程序入口地址的形成**——由 IR 的操作码经**微地址形成逻辑**映射产生，加载到 μPC，指向该机器指令对应微程序的第一条微指令
- **顺序执行**——当前微指令执行完毕后，μPC 自动 **+ 1**，指向控存中紧接着的下一条微指令（后续微指令）
- **条件分支**——根据状态标志（如 ZF、CF）和微指令中顺序控制字段的转移条件，经**地址转移逻辑**判定后，将分支目标地址加载到 μPC

## 微指令的格式

根据微指令的结构特点，分为**水平型**和**垂直型**两种

### 水平型微指令（Horizontal Micro-instruction）

从编码方式看，直接编码、字段直接编码和字段间接编码都属于水平型微指令；操作控制字段**很宽**，**每一位（或每一字段）直接对应一个微命令**；置 1 表示有效，置 0 表示无效；一条水平型微指令可同时定义并执行多个并行微操作；
![[水平型微指令的基本格式.png]]

| 特征        | 说明                             |
| --------- | ------------------------------ |
| **字长**    | 很长（几十到几百位）                     |
| **并行性**   | **极高**——一条微指令可同时发出多种互不冲突的微命令   |
| **编码程度**  | 低（直接编码或字段数很多的字段直接编码）           |
| **微程序长度** | **短**——一条水平微指令能完成多个并行操作，微程序条数少 |
| **执行速度**  | 快——并行度高，微周期短                   |
| **控存容量**  | 宽度大，但深度（微指令条数）小                |

### 垂直型微指令（Vertical Micro-instruction）

类似机器指令格式，在微指令中设置**微操作码字段**，通过译码产生控制信号；一条垂直型微指令通常显式定义一个基本微操作
![[垂直型微指令的基本格式.png]]

| 特征 | 说明 |
|------|------|
| **字长** | 短（十几到几十位） |
| **并行性** | **低**——一条微指令通常只完成一个基本传送或运算 |
| **编码程度** | 高（高度编码，需要多级译码） |
| **微程序长度** | **长**——需要更多条微指令才能完成同一功能 |
| **执行速度** | 慢——微程序条数多 |
| **控存容量** | 宽度小，但深度（微指令条数）大 |

### 水平型 vs 垂直型对比

| 维度           | 水平型微指令                  | 垂直型微指令            |
| ------------ | ----------------------- | ----------------- |
| **微指令字长**    | **长**（几十~几百位）           | **短**（十几~几十位）     |
| **并行微操作数**   | **多**（一条发多个微命令）         | **少**（一条通常只做一个操作） |
| **灵活性**      | 强                       | 弱                 |
| **编码程度**     | 低（接近直接控制）               | 高（类似机器指令的编码）      |
| **微程序长度**    | **短**（微指令条数少）           | **长**（微指令条数多）     |
| **执行速度**     | **快**                   | **慢**             |
| **控存容量**     | 宽而浅（横向大，纵向小）            | 窄而深（横向小，纵向大）      |
| **设计难度**     | **较难**（需考虑微命令间的相容性和并行性） | 较易（逐条编写，类似汇编编程）   |
| **可读性/可维护性** | **差**（位串难以理解）           | 好（有操作码，类似机器指令）    |
| **类比**       | 大规模并行控制                 | 类似简化的机器语言         |
| **实际应用**     | 追求性能的场景                 | 追求规整性和可维护性的场景     |

> [!important] 408 考点：水平 vs 垂直的取舍
> - **水平型** = "一条做很多事" = 快但宽 → 适合对速度要求高的场合
> - **垂直型** = "一条只做一件事" = 慢但窄 → 适合对控存容量有限制的场合
> - 实际的商用微程序控制器往往采用**混合型**（大多数是偏水平的）——操作控制字段用字段直接编码压缩宽度，同时保持较高的并行度

# 硬布线控制器和微程序控制器的特点

硬布线控制器和微程序控制器是控制器的两种根本不同的实现方式，它们在设计方法、性能特征和适用场景上有显著差异。

## 两者的全面对比

| 对比维度       | 硬布线控制器                         | 微程序控制器                          |
| ---------- | ------------------------------ | ------------------------------- |
| **实现原理**   | 组合逻辑电路（门电路 + 译码器）              | 存储逻辑（控存 ROM + μIR + 时序）         |
| **控制信号产生** | 组合逻辑根据输入**实时产生**               | 从控存**读出**微指令，由操作控制字段产生          |
| **设计方法**   | 列出微操作时间表 → 写逻辑表达式 → 化简 → 门电路实现 | 编写微程序 → 编码 → 固化到控存 ROM          |
| **规整性**    | **差**——控制逻辑分散在电路中，不同指令差异大      | **好**——所有指令统一为微程序，格式规整统一        |
| **可修改性**   | **困难**——修改指令功能需重新设计并布线         | **容易**——只需修改控存中对应的微程序（若控存可写）    |
| **执行速度**   | **快**——控制信号直接由组合逻辑产生，延迟仅为门延迟   | **较慢**——每条微指令需从控存读取，增加了控存访问时间   |
| **成本**     | 门电路数量随指令复杂度**指数增长**            | 控存容量随微程序规模**线性增长**              |
| **指令集复杂度** | 适合**简单指令集**（RISC）              | 适合**复杂指令集**（CISC）               |
| **设计周期**   | 长（逻辑设计、化简、验证）                  | 较短（微程序设计类似编程，有系统性方法）            |
| **错误修复**   | 困难（需重新制版/流片）                   | 容易（修改微码 ROM 即可）                 |


> [!important] 硬布线 vs 微程序的核心判断
> | 判别标准 | 硬布线 | 微程序 |
> |----------|--------|--------|
> | 控制信号产生方式 | 组合逻辑运算 | 控存读取 |
> | 是否有控制存储器 | **否** | **是** |
> | 微操作时序控制 | 由节拍信号直接参与逻辑表达式 | 由微指令序列决定 |
> | 修改指令功能 | 修改电路 | 修改微程序 |
> | 执行速度 | **硬布线更快**（无控存访问延迟） | 微程序较慢 |
> | 应用场景 | CISC CPU | RISC CPU |


# 总结

> [!summary] 本章重点
> 1. **控制器 = CU**（控制单元）：CPU 的指挥中心，三者等价；本质是一个**有限状态机**，综合 opcode + 时序 + 标志产生控制信号
> 2. **控制器的基本功能**：取指、分析指令、执行指令、中断处理、时序控制、数据流向控制
> 3. **控制器的基本组成**：PC、IR、ID、时序产生器、**微操作信号发生器**（核心）、MAR、MDR、PSW
> 4. **硬布线控制器**：由**组合逻辑电路**实现，$C_i = f(\text{opcode}, \text{timing}, \text{flags})$；速度快但不灵活，适合 RISC
> 5. **微程序控制器**：采用**存储逻辑**，微程序存入控存 ROM，执行机器指令 = 逐条取微指令执行；规整但较慢，适合 CISC
> 6. **微程序基本概念层级**：微命令（单个控制信号）→ 微操作（基本硬件动作）→ 微指令（一组可并行的微命令，含操作控制字段 + 顺序控制字段）→ 微程序（实现一条机器指令的微指令序列）
> 7. **相容性 vs 互斥性微命令**：相容性可同时有效，互斥性不允许同时有效——这是字段直接编码的理论基础
> 8. **主存 vs 控存**：主存存程序/数据（DRAM，大容量），控存存微程序（ROM，高速小容量），两者物理分离、地址空间独立
> 9. **程序 vs 微程序**：程序由机器指令组成、存主存、对用户可见；微程序由微指令组成、存控存、对用户透明；一条机器指令对应一段微程序
> 10. **关键寄存器辨析**：MAR（主存地址）≠ μPC（控存地址）；IR（机器指令）≠ μIR（微指令）；μPC ≠ PC（更新频率远高于 PC）
> 11. **微程序控制器的基本组成**：CM（控存）、μIR、μPC、微地址形成逻辑、地址转移逻辑、时序产生器
> 12. **微程序控制器工作过程**：执行取指公共微程序 → 操作码映射入口地址 → 逐条循环执行微程序（μPC→CM→μIR→发微命令→下一μPC）→ 末尾跳回取指
> 13. **微程序总数 = 机器指令条数 + 公共微程序数**（取指、间址、中断等公共操作各自独立编写）
> 14. **微指令三种编码方式**：直接编码（最快、字长最长）、字段直接编码（主流，互斥命令分组编码）、字段间接编码（最紧凑、两级译码最慢）
> 15. **字段直接编码原则**：互斥性微命令归入同一字段，相容性分入不同字段；每字段预留一个 NOP 编码
> 16. **微地址形成方式**：增量方式（μPC+1，简单但需额外转移电路）、断定方式（顺序控制字段显式指定下地址）
> 17. **下条微地址的三种确定**：入口地址由操作码经微地址形成逻辑映射；顺序执行 μPC+1；条件分支经地址转移逻辑判定后加载
> 18. **水平型 vs 垂直型微指令**：水平型宽而快（并行度高、微程序短），垂直型窄而慢（类似机器指令、微程序长）；实际多用混合型
> 19. **硬布线 vs 微程序全面对比**：实现原理（组合逻辑 vs 存储逻辑）、速度（快 vs 慢）、灵活性（差 vs 好）、成本规律（指数 vs 线性）、适用场景（RISC vs CISC）
> 20. **现代 CPU 多采用混合方案**：常用简单指令用硬布线追求速度，复杂指令用微程序保证规整性
