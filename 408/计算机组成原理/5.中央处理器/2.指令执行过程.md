# CPU 的时序控制

CPU 执行指令需要严格的时序控制，确保各微操作在正确的时间点完成
**时钟信号**是时序控制的基础，通常由机器内部的脉冲源（如晶振）产生，经整形和分频后形成供全机同步使用的节拍信号；**时钟周期的长度**由数据通路中相邻状态单元之间组合逻辑的最大传播延迟决定，以确保信号在下一个时钟边沿到来前稳定；
时序系统由大到小划分为三个层次：

## 三级时序体制

### 1. 时钟周期（节拍）

- **定义**：CPU 时钟信号一个完整振荡周期的时间（周期 = 1 / 主频）
- **作用**：最基础的时序单位，所有微操作的时间基准
- 必要时在节拍内插入工作脉冲，以实现更精细的时序控制

### 2. 机器周期（CPU 周期）

- **定义**：完成一次**基本操作**所需的时间
- **基本操作**：一次访存（读/写）、一次总线传输等
- **与时钟周期的关系**：一个机器周期通常包含**若干个时钟周期**（几个节拍）
- **典型划分**：
  - **取指周期**：从主存取出一条指令
  - **间址周期**：取操作数的有效地址（间接寻址时）
  - **执行周期**：完成指令规定的操作
  - **中断周期**：响应中断，保存断点

> [!important] 408 关键概念
> 一个机器周期内完成一次访存（通常如此），所以指令执行所需的机器周期数与**访存次数**密切相关。大多数指令的机器周期数 = 取指（1次访存）+ 可能的间址 + 可能的执行访存。

### 3. 指令周期

- **定义**：CPU 取出并执行**一条指令**所需的全部时间
- **组成**：由若干个机器周期组成
- **不同指令的指令周期**：
  - **MOV 寄存器间传送**：1 个取指周期 + 1 个执行周期（无访存）
  - **ADD 寄存器加法**：1 个取指周期 + 1 个执行周期
  - **LOAD 取数**：1 个取指周期 + 可能 1 个间址周期 + 1 个执行周期（含访存）
  - **STORE 存数**：同上，执行周期含写访存

## 指令周期流程（含中断）

- **取指周期**：每条指令**必须**有
- **间址周期**：操作数有**间接寻址**时才有
- **执行周期**：每条指令**必须**有
- **中断周期**：执行完一条指令后检查中断请求，有请求时进入

> [!warning] 注意
> 中断周期并不是每条指令都有，仅在当前指令执行完毕后检测到中断请求时才进入。408 

# 指令周期的基本概念

一条指令的执行过程从微操作角度来看，可以分为以下几个阶段。

## 取指（IF — Instruction Fetch）

**任务**：根据 PC 的值从主存中取出指令，放入 IR；同时 PC 被更新为下一条指令的地址：顺序执行时，PC 增加当前指令长度；若为变长指令，则需要在取指阶段初步解析其格式，以确定长度；若发生分支，则其目标地址将在后续阶段计算

### 微操作序列

| 步骤 | 微操作 | 说明 |
|------|--------|------|
| ① | `MAR ← (PC)` | PC 值送 MAR，准备访问主存 |
| ② | `MDR ← M(MAR)` | 从主存读取指令到 MDR |
| ③ | `IR ← (MDR)` | MDR 中的指令送 IR |
| ④ | `PC ← (PC) + 1` | PC 递增，指向下一条指令 |


## 译码/读寄存器（ID — Instruction Decode / Register Fetch）

**任务**：对 IR 中的操作码进行译码，识别操作码和寻址方式，同时从寄存器堆读出操作数；立即数也在本阶段提取

### 微操作序列

| 步骤 | 微操作 | 说明 |
|------|--------|------|
| ① | 操作码 → 译码器 | 产生控制信号 |
| ② | `A ← (R[rs])` | 读出源操作数 1 到暂存寄存器 A |
| ③ | `B ← (R[rt])` | 读出源操作数 2 到暂存寄存器 B |
| ④ | `IMM ← 符号扩展(立即数字段)` | 若指令含立即数，将其符号扩展至机器字长 |

### 译码阶段的并行操作
- 操作码译码与读寄存器是**并行进行**的——两者互不依赖
- 在流水线处理器中，ID 阶段同时完成：
  1. 译码产生控制信号
  2. 读寄存器堆获取操作数
  3. 对立即数进行符号扩展
  4. 判断是否需要转移（提前计算分支目标和条件）

> [!important] 译码阶段的控制信号
> 译码器根据操作码生成一组控制信号，这些信号决定了后续各阶段各部件的工作方式。控制信号的生成方式决定了控制器的类型：**硬布线控制器**（组合逻辑）或**微程序控制器**（查控存）。

## 执行/有效地址计算（EX — Execute / Address Calculation）

**任务**：根据指令类型执行相应操作：算术或逻辑指令，由 ALU 完成运算；**访存类指令，计算操作数在主存中的有效地址**；分支指令，计算目标地址并判断是否转移；此外，**运算结果状态在此阶段生成并暂存**

### 不同类型的操作

| 指令类型 | 微操作 | 说明 |
|----------|--------|------|
| **寄存器-寄存器 ALU** | `RESULT ← (A) op (B)` | ALU 对 A、B 中操作数运算 |
| **寄存器-立即数 ALU** | `RESULT ← (A) op IMM` | ALU 对寄存器和立即数运算 |
| **LOAD/STORE 地址计算** | `ALUOutput ← (A) + IMM` | 计算有效地址 = 基址 + 偏移 |
| **分支指令** | `ALUOutput ← (PC) + IMM×4` | 计算分支目标地址 |
| **条件分支判断** | `Cond ← (A) == 0` 等 | 比较并设置条件 |

### 有效地址的计算

不同寻址方式下有效地址 EA 的计算，见[[2.寻址方式#常见的数据寻址方式]]

> [!note] LOAD/STORE 的 EX 阶段
> 对于访存指令，EX 阶段并不真正访存，只计算有效地址并存入 ALUOutput 寄存器（或暂存寄存器），真正的访存发生在 MEM 阶段。

## 访存（MEM — Memory Access）

**任务**：LOAD 指令从主存读取数据，STORE 指令将数据写入主存；**寄存器-寄存器类指令不涉及此阶段**

### 微操作序列

| 指令类型 | 微操作 | 说明 |
|----------|--------|------|
| **LOAD** | `MAR ← ALUOutput` → `MDR ← M(MAR)` → `LMD ← (MDR)` | 读出的数据暂存在 Load Memory Data 寄存器 |
| **STORE** | `MAR ← ALUOutput` → `MDR ← (B)` → `M(MAR) ← (MDR)` | 将寄存器 B 的内容写入主存 |
| **其他指令** | 无访存操作 | MEM 阶段空闲（数据在 WB 阶段直通） |

- LOAD 指令读出的数据通常先放在**暂存寄存器 LMD**（或直接叫 MDR），WB 阶段再写入目标寄存器
- STORE 指令在 MEM 阶段完成写入后，无需 WB 阶段
- **对齐访问 vs 非对齐访问**：
  - 对齐访问：一次访存即可完成
  - 非对齐访问：可能需要多次访存，用移位/拼接得到最终数据

## 写回（WB — Write Back）

**任务**：将运算结果或读出的数据写入目标**寄存器**

### 微操作序列

| 指令类型 | 微操作 | 说明 |
|----------|--------|------|
| **ALU 运算** | `R[rd] ← RESULT` | ALU 结果写入目标寄存器 |
| **LOAD** | `R[rt] ← LMD` | 访存取出的数据写入寄存器 |
| **STORE** | 无需写回 | 数据已在 MEM 阶段写入主存 |
| **分支/跳转** | `PC ← ALUOutput`（若跳转） | 更新 PC 为分支目标地址 |

> [!note] WB 阶段的时机
> - 对于**ALU 指令**，结果在 EX 阶段结束时已稳定，在 WB 阶段写入寄存器堆
> - 对于**LOAD 指令**，数据在 MEM 阶段从主存取出，在 WB 阶段写入寄存器堆——这导致 LOAD 指令后面紧跟的指令如果要用这个数据，会产生**数据冒险**（见 [[6.指令流水线|指令流水线]]）

# 处理器指令执行模型

根据一条指令在几个时钟周期内完成、以及多条指令是否重叠执行，处理器可分为三种模型。

## 单周期处理器（Single-Cycle Processor）

**核心思想**：每条指令都在**一个时钟周期**内完成全部操作。

### 工作原理

- 所有指令的执行时间统一为一个（较长的）时钟周期
- 时钟周期的长度由**最慢指令**的执行时间决定
- 取指、译码、执行、访存、写回全部在一个周期内串行完成

### 数据通路特点

- 一个时钟周期内需要使用不同的功能部件，但不能有**同一部件复用**（如一个周期内不能两次使用 ALU）
- 需要多个存储器：**指令存储器**和**数据存储器**分开（哈佛结构）
- PC 在每个时钟周期更新一次

### 优缺点

| 优点 | 缺点 |
|------|------|
| 设计简单，控制逻辑直观 | 时钟周期由最慢指令决定，**严重拖慢整体性能** |
| 无冒险问题（每条指令在一个周期内完成） | 每个部件一个周期只能用一次，**资源利用率低** |
| 状态更新仅在时钟沿发生，时序清晰 | 指令存储器和数据存储器必须分开，成本高 |

### 性能分析

- **CPI = 1**（每条指令 1 个时钟周期）
- **时钟周期 = 最长指令的执行时间**（通常为 LOAD 指令：取指 + 译码 + ALU计算地址 + 访存 + 写回）
- **执行时间 = 指令数 × 时钟周期 = IC × 最慢指令时间**
- 对简单指令（如寄存器加法）来说，大量时间被浪费在等待上

> [!example] 数字举例
> 假设 LOAD 指令需要 8ns（取指 2 + 译码 1 + ALU 2 + 访存 2 + 写回 1），ADD 指令只需要 6ns（无访存阶段）。则单周期处理器的时钟周期必须设为 **8ns**，执行 ADD 时也花 8ns，浪费了 2ns。

## 多周期处理器（Multi-Cycle Processor）

**核心思想**：将指令执行过程分解为多个阶段，**每个阶段用一个时钟周期**，不同指令使用不同数量的时钟周期。

> [!warning] 仍然是串行执行！
> 多周期处理器**同一时刻只处理一条指令**，一条指令的所有阶段全部执行完毕后，才开始取下一条指令。它只是把一条指令内部拆成多段来缩短时钟周期，**并没有实现多条指令的重叠执行**。真正实现指令间重叠并行的是流水线处理器。

### 工作原理

- 每个时钟周期完成指令执行的一个阶段
- 不同指令需要的时钟周期数不同：
  - **R 型 ALU 指令**：取指（1）+ 译码/读寄存器（1）+ 执行（1）+ 写回（1）= **4 周期**
  - **LOAD 指令**：取指（1）+ 译码（1）+ 地址计算（1）+ 访存（1）+ 写回（1）= **5 周期**
  - **STORE 指令**：取指（1）+ 译码（1）+ 地址计算（1）+ 访存写（1）= **4 周期**
  - **分支指令**：取指（1）+ 译码（1）+ 地址计算与比较（1）= **3 周期**

### 数据通路特点

- 功能部件可以在**不同周期复用**（如 ALU 在 EX 周期算结果，在 MEM 周期算地址）
- 指令存储器和数据存储器可以**合并为统一主存**（冯·诺依曼结构），因为取指和访存在不同周期
- 需要增加**暂存寄存器**来保存各阶段的中间结果：
  - **IR**：保存取出的指令
  - **MDR**：保存读出的数据
  - **A、B**：保存从寄存器堆读出的操作数
  - **ALUOut**：保存 ALU 计算结果或有效地址

> [!note] 多周期处理器需要更多的寄存器
> 因为结果在不同周期产生和使用，必须用暂存寄存器保存中间值。这增加了硬件开销，但换来了时钟周期缩短和部件复用。

### 优缺点

| 优点 | 缺点 |
|------|------|
| **时钟周期短**，只需完成一个阶段的操作 | 控制逻辑更复杂（需要状态机） |
| 功能部件可在不同周期**复用** | CPI > 1，不同指令的 CPI 不同 |
| 指令和数据可共享同一主存 | 增加了暂存寄存器的硬件开销 |
| 不同指令周期的指令可只花需要的周期数 | 设计复杂度高于单周期 |

### 性能分析

- **CPI 不是常数**：每种指令有不同的 CPI
- **平均 CPI** = Σ(各指令比例 × 各指令周期数)
- **时钟周期** = 最慢**单个阶段**的时间（远小于单周期的时钟周期）

> [!important] 多周期 vs 单周期的小结
> 多周期处理器**并没有比单周期快很多**。它的最大价值不是提升单条指令速度，而是**让不同指令不再被最慢指令束缚**，同时允许**部件复用**和**统一主存**。真正的性能飞跃来自流水线。

## 流水线处理器（Pipelined Processor）

**核心思想**：多条指令**同时**处于执行的不同阶段，像工厂流水线一样重叠执行。

### 基本原理

- 将指令执行过程划分为若干个阶段（通常 5 个：IF/ID/EX/MEM/WB）
- 每个阶段用独立的硬件资源处理
- 每个时钟周期，各阶段同时工作，处理不同的指令
- 每个时钟周期可**启动一条新指令**，也可**完成一条指令**

### 流水线的时空图

```
时钟周期:   1    2    3    4    5    6    7    8    9
指令i:     IF   ID   EX   MEM  WB
指令i+1:        IF   ID   EX   MEM  WB
指令i+2:             IF   ID   EX   MEM  WB
指令i+3:                  IF   ID   EX   MEM  WB
指令i+4:                       IF   ID   EX   MEM  WB
```

### 关键指标

| 指标 | 定义 | 理想值 | 实际值 |
|------|------|--------|--------|
| **吞吐率（TP）** | 单位时间完成的指令数 | 1 条/时钟周期 | < 1（有冒险和停顿） |
| **加速比（S）** | 非流水时间 / 流水时间 | = 流水线级数 k | < k（有装入/排空开销 + 冒险） |
| **效率（E）** | 流水线各段利用率 | 100% | < 100%（有气泡和冲突） |

### 流水线的性能问题——冒险（Hazards）

流水线处理器面临三类冒险，详见 [[6.指令流水线|指令流水线]]：

| 冒险类型     | 产生原因                       | 解决方案                       |
| -------- | -------------------------- | -------------------------- |
| **结构冒险** | 硬件资源冲突（同一时钟周期多个指令争用同一部件）   | 增加硬件资源 / 分离指令与数据 Cache     |
| **数据冒险** | 指令间存在数据依赖（前一条结果后一条就要用）     | 转发（Forwarding）/ 旁路 + 必要时停顿 |
| **控制冒险** | 分支/跳转指令导致 PC 改变，流水线中已取指令作废 | 分支预测 / 延迟分支 / 提前判断         |

### 三种模型的全面对比

| 特性 | 单周期 | 多周期 | 流水线 |
|------|--------|--------|--------|
| **CPI** | 1 | > 1（各指令不同） | ≈ 1（理想情况下） |
| **时钟周期** | 最长指令时间 | 最长阶段时间 | 最长阶段时间 + 流水线寄存器延迟 |
| **指令存储器** | 必须独立 | 可统一 | 独立 Cache（通常） |
| **数据存储器** | 必须独立 | 可统一 | 独立 Cache（通常） |
| **硬件利用率** | 低 | 中 | 高 |
| **控制复杂度** | 低 | 中 | 高 |
| **吞吐率** | 低 | 中 | 高（接近 1 条/周期） |
| **需要处理冒险** | 否 | 否 | 是 |
| **设计思想** | 简单直接 | 分时复用 | 重叠执行 |

# 总结

> [!summary] 本章重点
> 1. 指令执行过程可以分解为 **IF → ID → EX → MEM → WB** 五个基本阶段，不同指令可能跳过某些阶段
> 2. CPU 时序体制分为三级：**指令周期 > 机器周期 > 时钟周期（节拍）**，机器周期通常以一次访存时间为基准
> 3. **取指周期**和**执行周期**是每条指令必须的；**间址周期**仅间接寻址时有；**中断周期**仅在检测到中断请求时进入
> 4. 取指阶段核心微操作：`MAR←(PC), MDR←M(MAR), IR←(MDR), PC←(PC)+1`
> 5. 译码阶段：操作码译码与读寄存器**并行进行**，同时产生控制信号
> 6. EX 阶段对不同指令含义不同：ALU 指令做运算，LOAD/STORE 计算有效地址，分支指令计算目标地址并判断条件
> 7. MEM 阶段：LOAD 从主存读数据到 LMD，STORE 将数据写入主存
> 8. WB 阶段：结果写入目标寄存器（ALU 结果或 LOAD 读出的数据）
> 9. **单周期处理器**：CPI = 1，但时钟周期由最慢指令决定，效率低
> 10. **多周期处理器**：不同指令花不同周期数，部件可复用，但 CPI > 1
> 11. **流水线处理器**：指令重叠执行，理想吞吐率 = 1 条/周期，但面临结构/数据/控制三类冒险
> 12. 流水线加速比 $S = \frac{nk}{k+n-1}$，当指令数足够大时接近流水线级数 k
