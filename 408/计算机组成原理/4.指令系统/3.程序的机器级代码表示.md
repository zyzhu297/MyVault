# 常用汇编指令

> 本节以 **x86（IA-32）架构**为主，介绍程序的机器级代码表示。汇编语言是机器指令的文本助记符，一条汇编指令对应一条机器指令。

## 相关寄存器

x86-32 架构中的主要寄存器：

### 通用寄存器（8个，32位）

| 寄存器 | 名称 | 常用用途 |
|:---:|:---|:---|
| **EAX** | 累加器（Accumulator） | 算术运算、函数返回值 |
| **EBX** | 基址寄存器（Base） | 基址寻址 |
| **ECX** | 计数器（Counter） | 循环计数、字符串操作 |
| **EDX** | 数据寄存器（Data） | I/O 操作、乘除法扩展 |
| **ESI** | 源变址寄存器（Source Index） | 字符串/数组源地址 |
| **EDI** | 目的变址寄存器（Dest Index） | 字符串/数组目的地址 |
| **EBP** | 基址指针（Base Pointer） | 栈帧基址 |
| **ESP** | 堆栈指针（Stack Pointer） | 栈顶指针 |

> 八个通用寄存器的低 16 位可分别访问为 AX、BX、CX、DX、SI、DI、BP、SP；其中 AX~DX 的高/低 8 位又可分别访问为 AH/AL、BH/BL、CH/CL、DH/DL。

### 专用寄存器

| 寄存器 | 说明 |
|:---|:---|
| **EIP / PC** | 程序计数器，存放下一条指令地址 |
| **EFLAGS** | 标志寄存器（条件码），记录运算结果的状态 |

### 段寄存器

| 寄存器 | 用途 |
|:---|:---|
| CS | 代码段 |
| DS | 数据段 |
| SS | 堆栈段 |
| ES、FS、GS | 附加数据段 |

### EFLAGS 标志位（关键）

| 标志位 | 名称 | 含义 |
|:---:|:---|:---|
| **ZF** | 零标志 | 结果为 0 时置 1 |
| **SF** | 符号标志 | 结果为负时置 1（最高位的值） |
| **CF** | 进位/借位标志 | 无符号运算进位/借位时置 1 |
| **OF** | 溢出标志 | 有符号运算溢出时置 1 |
| **PF** | 奇偶标志 | 低 8 位中 1 的个数为偶数时置 1 |

## 常用指令

指令中的符号约定：

| 符号 | 含义 |
|:---|:---|
| `<reg>` | 任意寄存器，如 `eax`、`ebx` |
| `<mem>` | 内存地址，如 `[ebx]`、`[ebp-4]` |
| `<con>` | 立即数（常数），如 `5`、`0x100` |
| `D` | 目的操作数（Destination） |
| `S` | 源操作数（Source） |

### 数据传送指令

#### MOV — 数据传送

```
MOV D, S    ; D ← S
```

- 将源操作数 S 的值复制到目的操作数 D
- **不允许两个操作数都是内存**（x86 限制：`MOV [mem1], [mem2]` 非法），若需要在内存之间复制，可先将源内存内容加载到寄存器，再从该寄存器写入目标内存
- 不影响标志位

**允许的传送方向**：

| 源 \ 目的 | 寄存器 | 内存 | 立即数 |
|:---:|:---:|:---:|:---:|
| 寄存器 | ✓ | ✓ | ✗ |
| 内存 | ✓ | ✗ | ✗ |
| 立即数 | ✓ | ✓ | ✗ |

#### PUSH — 压栈

```
PUSH S    ; ESP ← ESP - 4, [ESP] ← S
```

- 将操作数压入栈顶
- x86 堆栈向**低地址**方向生长（满递减栈）
- `ESP` 先减 4，再写入数据

#### POP — 出栈

```
POP D    ; D ← [ESP], ESP ← ESP + 4
```

- 将栈顶数据弹出到目的操作数
- 先读数据，`ESP` 再加 4


### 算术和逻辑运算指令

#### 算术运算

| 指令       | 格式         | 含义        | 说明                           |
| :------- | :--------- | :-------- | :--------------------------- |
| **ADD**  | `ADD D, S` | D ← D + S | 加法，影响 OF/SF/ZF/CF            |
| **SUB**  | `SUB D, S` | D ← D - S | 减法，影响 OF/SF/ZF/CF            |
| **INC**  | `INC D`    | D ← D + 1 | 加 1，影响 OF/SF/ZF，**不影响 CF**   |
| **DEC**  | `DEC D`    | D ← D - 1 | 减 1，影响 OF/SF/ZF，**不影响 CF**   |
| **NEG**  | `NEG D`    | D ← -D    | 取负（**补码**，按位取反+1）            |
| **MUL**  | `MUL S`    | 无符号乘法     | EAX × S → EDX:EAX（64位结果）     |
| **IMUL** | 见下方详解      | 有符号乘法     | 有单/双/三操作数三种形式                |
| **DIV**  | `DIV S`    | 无符号除法     | EDX:EAX ÷ S → 商在 EAX，余数在 EDX |
| **IDIV** | `IDIV S`   | 有符号除法     | 同 DIV，有符号版本                  |

> [!important] INC/DEC 不影响 CF
> `INC` 和 `DEC` 指令**不改变 CF（进位标志）**，这是常见考点。原因是这两条指令在硬件上通过专用加 1/减 1 电路实现，不经过完整的加法器。为什么这样设计？—— 因为 LOOP 循环中常用 `INC` 修改计数器，若 `INC` 改变 CF 则会破坏此前 `ADD/SUB` 留下的进位标志，干扰循环中可能的进位判断逻辑。

> [!warning] MUL/DIV 隐含操作数
> - **乘法**：被乘数**隐含在 EAX** 中，结果存入 **EDX:EAX**（高 32 位在 EDX，低 32 位在 EAX）
> - **除法**：被除数**隐含在 EDX:EAX**（64 位，高 32 位在 EDX），商在 **EAX**，余数在 **EDX**
> - 除法前需用 `CDQ`（有符号扩展 EAX→EDX:EAX）或 `XOR EDX, EDX`（无符号清空 EDX）准备被除数

#### IMUL 的三种操作数形式

> [!important] IMUL 三种形式
> x86 中 **IMUL**（有符号乘法）有三种操作数形式，灵活性逐级递增；而无符号 **MUL** 只有单操作数形式：
>
> **① 单操作数形式**（同 MUL）
> ```
> IMUL S    ; EAX × S → EDX:EAX（64 位结果）
> ```
> - 被乘数隐含在 EAX，结果扩展到 EDX:EAX
> - 这是最基本的乘法形式，MUL 只有这一种形式
>
> **② 双操作数形式**
> ```
> IMUL D, S    ; D ← D × S
> ```
> - D 必须是**寄存器**（16/32 位通用寄存器）
> - S 可以是寄存器、内存或立即数
> - 结果**截断**到 D 的宽度（只保留低 16/32 位），高位置接丢弃
> - **可能丢失精度**：若结果超出 D 的表示范围，OF/CF 置 1（不精确标志），但结果仍保留低位
> - 例：`IMUL ebx, ecx` → ebx ← ebx × ecx
> - 例：`IMUL ebx, [mem]` → ebx ← ebx × [mem]
> - 例：`IMUL ebx, 10` → ebx ← ebx × 10
>
> **③ 三操作数形式**
> ```
> IMUL D, S1, S2    ; D ← S1 × S2
> ```
> - D 必须是**寄存器**
> - S1 可以是寄存器或内存
> - S2 必须是**立即数**
> - 同样存在**截断**风险，溢出时 OF/CF 置 1
> - 例：`IMUL ebx, ecx, 10` → ebx ← ecx × 10（S1 的原始值不变）
> - 例：`IMUL ebx, [mem], 5` → ebx ← [mem] × 5
>
> **三者关系与对比**：
>
> | 对比维度 | 单操作数 IMUL | 双操作数 IMUL | 三操作数 IMUL |
> |:---|:---|:---|:---|
> | 结果位宽 | 64 位（EDX:EAX） | 32 位（截断） | 32 位（截断） |
> | OF/CF | 高 32 位非全 0（无符号）时置 1 | 结果被截断时置 1 | 结果被截断时置 1 |
> | 操作数破坏 | 覆盖 EDX | 覆盖 D | 只写入 D（S1 不变） |
> | 是否保留源操作数 | EAX 不变 | 否（D 被改写） | 是（S1 不变） |
>
> 实际使用场景：
> - 需要 64 位完整积（如大数运算）→ 用单操作数形式
> - 计算 `a = a * b`（自修改）→ 用双操作数形式
> - 计算 `a = b * 10`（不破坏 b 且 10 是常数）→ 用三操作数形式

#### 逻辑运算

| 指令 | 格式 | 含义 | 说明 |
|:---|:---|:---|:---|
| **AND** | `AND D, S` | D ← D & S | 按位与，CF/OF 清零 |
| **OR** | `OR D, S` | D ← D \| S | 按位或，CF/OF 清零 |
| **XOR** | `XOR D, S` | D ← D ^ S | 按位异或，CF/OF 清零 |
| **NOT** | `NOT D` | D ← ~D | 按位取反，不影响标志位 |
| **TEST** | `TEST D, S` | D & S（只设标志位） | 同 AND 但不保存结果 |
| **SHL** | `SHL D, count` | 逻辑左移，低位补 0 | 移出位 → CF |
| **SHR** | `SHR D, count` | 逻辑右移，高位补 0 | 移出位 → CF |
| **SAL** | `SAL D, count` | 算术左移（同 SHL） | 移出位 → CF |
| **SAR** | `SAR D, count` | 算术右移，高位补符号位 | 移出位 → CF |
| **ROL** | `ROL D, count` | 循环左移 | CF 参与或不参与（视指令变体） |
| **ROR** | `ROR D, count` | 循环右移 | 同上 |

> [!important] CMP vs TEST
> - **CMP D, S** = `SUB D, S` 但不保存结果，只影响标志位
> - **TEST D, S** = `AND D, S` 但不保存结果，只影响标志位
> - 两者都用于设置条件码，后跟条件跳转指令

#### CMP 配合条件跳转

`CMP a, b` 执行 `a - b`，根据结果设置标志位：

| 关系 | 条件 | 标志位依据 |
|:---|:---|:---|
| `a == b` | ZF = 1 | 差为 0 |
| `a != b` | ZF = 0 | 差不为 0 |
| **无符号比较** | | |
| `a > b` (above) | CF=0 且 ZF=0 | 无借位且不等 |
| `a ≥ b` (above/equal) | CF = 0 | 无借位 |
| `a < b` (below) | CF = 1 | 有借位 |
| `a ≤ b` (below/equal) | CF=1 或 ZF=1 | 有借位或相等 |
| **有符号比较** | | |
| `a > b` (greater) | SF=OF 且 ZF=0 | 符号与溢出相同且不等 |
| `a ≥ b` (greater/equal) | SF = OF | 符号与溢出相同 |
| `a < b` (less) | SF != OF | 符号与溢出不同 |
| `a ≤ b` (less/equal) | SF!=OF 或 ZF=1 | 符号与溢出不同或相等 |

> [!important] 有符号比较为何要看 OF？
> 单独用 SF 判断有符号数可能出错：例如 `-100 - 100 = -200`，结果溢出为正（OF=1），此时 SF=0 看似"正"，但实际是"负数更小"。因此有符号比较必须结合 SF 和 OF：
> - **SF = OF** → 正确反映符号 → 被减数 ≥ 减数（大于等于成立）
> - **SF ≠ OF** → 符号被溢出反转 → 被减数 < 减数

### 控制流指令

#### JMP — 无条件跳转

```
JMP target    ; EIP ← target
```

- 无条件转移到目标地址
- 目标地址可以是指令标号（汇编器自动计算偏移量）

#### 条件跳转指令（Jcc）

以 `CMP a, b` 的后续跳转为例：

| 指令 | 全称 | 跳转条件 | 配合 CMP 含义 |
|:---|:---|:---|:---|
| **JE / JZ** | Jump if Equal / Zero | ZF = 1 | a == b |
| **JNE / JNZ** | Jump if Not Equal / Not Zero | ZF = 0 | a != b |
| **无符号跳转** | | | |
| **JA** | Jump if Above | CF=0 且 ZF=0 | a > b（无符号） |
| **JAE / JNB** | Jump if Above or Equal | CF = 0 | a ≥ b（无符号） |
| **JB / JNAE** | Jump if Below | CF = 1 | a < b（无符号） |
| **JBE / JNA** | Jump if Below or Equal | CF=1 或 ZF=1 | a ≤ b（无符号） |
| **有符号跳转** | | | |
| **JG** | Jump if Greater | SF=OF 且 ZF=0 | a > b（有符号） |
| **JGE** | Jump if Greater or Equal | SF = OF | a ≥ b（有符号） |
| **JL** | Jump if Less | SF ≠ OF | a < b（有符号） |
| **JLE** | Jump if Less or Equal | SF≠OF 或 ZF=1 | a ≤ b（有符号） |

> 此外还有 **JC**（CF=1 跳转）、**JNC**（CF=0）、**JO**（OF=1 跳转）、**JNO**（OF=0）、**JS**（SF=1）、**JNS**（SF=0）、**JP/JPE**（PF=1）、**JNP/JPO**（PF=0）等以单个标志位为条件的跳转。

#### CALL / RET — 过程调用与返回

```
CALL proc    ; 将返回地址压栈，然后跳转到 proc
RET          ; 从栈中弹出返回地址，然后跳转回去
```

- **返回地址** = CALL 指令的下一条指令的地址（即 EIP 的当前值）
- `CALL` 等价于：`PUSH EIP; JMP proc`
- `RET` 等价于：`POP EIP`

#### LOOP — 循环指令

```
LOOP label    ; ECX ← ECX - 1，如果 ECX ≠ 0 则跳转到 label
```

- 隐含使用 **ECX** 作为循环计数器
- 等价于：`DEC ECX; JNZ label`（但 LOOP 不影响标志位）
- 变体：`LOOPE`/`LOOPZ`（ECX≠0 且 ZF=1 时循环）、`LOOPNE`/`LOOPNZ`（ECX≠0 且 ZF=0 时循环）

## 汇编指令格式

### AT&T 格式 vs Intel 格式

| 对比项       | AT&T 格式（GCC 默认）              | Intel 格式（MASM/Windows）        |
| :-------- | :--------------------------- | :---------------------------- |
| **大小写**   | 必须小写                         | 对大小写不敏感                       |
| **操作数顺序** | `源, 目的`                      | `目的, 源`                       |
| **寄存器前缀** | `%eax`, `%ebx`               | `eax`, `ebx`                  |
| **立即数前缀** | `$5`, `$0x10`                | `5`, `10h`                    |
| **内存寻址**  | `disp(%base, %index, scale)` | `[base + index*scale + disp]` |
| **指令后缀**  | 有（`movl`, `addl`）            | 无（`mov`, `add`）               |
| **示例**    | `movl $5, %eax`              | `mov eax, 5`                  |
| **示例**    | `addl (%ebx), %eax`          | `add eax, [ebx]`              |



### 常见指令后缀（AT&T 格式）

| 后缀 | 操作数大小 | 含义 |
|:---:|:---|:---|
| `b` | 8 位（byte） | `movb` |
| `w` | 16 位（word） | `movw` |
| `s` | 32 位（short 或 single） | `movs`（浮点） |
| `l` | 32 位（long） | `movl` |
| `q` | 64 位（quad） | `movq` |

### Intel 格式指定操作长度

Intel 格式没有指令后缀，操作长度通过以下方式确定：

#### 1. 寄存器名暗示长度

寄存器名称本身就携带了长度信息，汇编器自动识别：

| 寄存器 | 长度 | 示例 |
|:---|:---:|:---|
| `al`, `ah`, `bl`, `cl`, `dl` 等 | 8 位 | `MOV al, 5` — 8 位传送 |
| `ax`, `bx`, `cx`, `dx`, `si`, `di` 等 | 16 位 | `MOV ax, bx` — 16 位传送 |
| `eax`, `ebx`, `ecx`, `esi`, `ebp` 等 | 32 位 | `MOV eax, 100` — 32 位传送 |
| `rax`, `rbx`, `rcx`, `rsi`, `rbp` 等 | 64 位 | `MOV rax, rbx` — 64 位传送 |

> 凡是指令中有寄存器操作数时，操作长度由该寄存器决定，无需额外标记。例如 `MOV eax, [ebx]` 中，`eax` 是 32 位，因此内存读取也是 32 位（从 `[ebx]` 起读 4 字节）。

#### 2. PTR 伪指令（内存操作数长度显式指定）

当两个操作数都不是寄存器（例如 `MOV [addr], 5`），或需要明确内存访问宽度时，使用 **PTR** 进行显式标注：

| 伪指令           | 含义    |  长度  | 示例                                                |
| :------------ | :---- | :--: | :------------------------------------------------ |
| **BYTE PTR**  | 按字节访问 | 1 字节 | `MOV BYTE PTR [ebx], 0` — 向 [ebx] 写入 1 字节的 0      |
| **WORD PTR**  | 按字访问  | 2 字节 | `MOV WORD PTR [ebx], 5` — 向 [ebx] 写入 2 字节的 5      |
| **DWORD PTR** | 按双字访问 | 4 字节 | `MOV DWORD PTR [ebx], 100` — 向 [ebx] 写入 4 字节的 100 |
| **QWORD PTR** | 按四字访问 | 8 字节 | `MOV QWORD PTR [ebx], rax` — 向 [ebx] 写入 8 字节      |

**何时必须使用 PTR**：

1. **两个操作数均不暗示长度**：如 `MOV [ebx], 5` 非法，因为只看 `5` 不知道它是 8 位、16 位还是 32 位 → 必须写成 `MOV DWORD PTR [ebx], 5`
2. **需要非默认宽度的内存访问**：如向前/向后读写部分字节，或从内存读取一个字节扩展写入 32 位寄存器：

> [!important] PTR 的大小与寄存器匹配
> 当涉及寄存器时，PTR 的大小应与寄存器大小保持一致。例：`MOV eax, DWORD PTR [ebx]` 中 `eax` 是 32 位，对应 `DWORD PTR`。若写 `MOV eax, BYTE PTR [ebx]` 则是错误（源 8 位、目标 32 位，不匹配），应改用 `MOVZX eax, BYTE PTR [ebx]` 先零扩展再传送。

#### 3. 三种指定方式的对比

| 方式 | 适用场景 | 示例 |
|:---|:---|:---|
| **寄存器暗示** | 至少有一个操作数是寄存器 | `MOV eax, [ebx]`（eax 暗示 32 位） |
| **PTR 伪指令** | 无不暗示长度的操作数，或需要非默认宽度 | `MOV BYTE PTR [ebx], 5`、`MOVSX eax, WORD PTR [esi]` |
| **指令自身决定** | 特定指令有固定操作宽度（如 `PUSH` 在 32 位模式下固定压 4 字节） | `PUSH eax`（32 位模式下固定 32 位） |

# 选择语句的机器级表示

## if-else 语句

高级语言的 if-else 结构在机器级通过**条件跳转**实现。

### 通用模板

```c
if (condition)
    then_statement;
else
    else_statement;
```

编译为：

```asm
    计算 condition
    CMP / TEST (设置标志位)
    Jcc  else_label       ; 条件不满足时跳过 then 部分
    ; --- then_statement ---
    JMP  end_label        ; 跳过 else 部分
else_label:
    ; --- else_statement ---
end_label:
```

### 示例：if-else

C 代码：
```c
if (a > b)
    max = a;
else
    max = b;
```

对应汇编（假设 a 在 eax，b 在 ebx，max 在 ecx）：

```asm
    CMP  eax, ebx        ; 计算 eax - ebx，设置标志位
    JLE  else_label      ; 若 a ≤ b 则跳到 else
    MOV  ecx, eax        ; max = a
    JMP  end_label
else_label:
    MOV  ecx, ebx        ; max = b
end_label:
```

### 多分支 if-else if-else

C 代码：
```c
if (score >= 90)
    grade = 'A';
else if (score >= 80)
    grade = 'B';
else if (score >= 70)
    grade = 'C';
else
    grade = 'D';
```

对应汇编：

```asm
    CMP  eax, 90         ; score >= 90?
    JL   check_B          ; 不满足，检查下一个条件
    MOV  ebx, 'A'
    JMP  end_label
check_B:
    CMP  eax, 80
    JL   check_C
    MOV  ebx, 'B'
    JMP  end_label
check_C:
    CMP  eax, 70
    JL   set_D
    MOV  ebx, 'C'
    JMP  end_label
set_D:
    MOV  ebx, 'D'
end_label:
```

## switch-case 语句

switch-case 有两种机器级实现方式：

### 1. 逐条件比较（条件过多时退化为 if-else 链）

```asm
    CMP  eax, case1_val
    JE   case1
    CMP  eax, case2_val
    JE   case2
    ...
    JMP  default_case
```

### 2. 跳转表（Jump Table）

当 case 值**分布密集且连续**时，编译器会构造跳转表：

```c
switch (x) {
    case 0: ... break;
    case 1: ... break;
    case 2: ... break;
    case 3: ... break;
    default: ...
}
```

编译为：

```asm
    ; 先判断 x 是否在 [0, 3] 范围内
    CMP  eax, 3
    JA   default_case     ; 超出范围 → default

    ; 以 x 为索引查跳转表
    JMP  [jump_table + eax * 4]   ; 跳转到对应的 case 代码

jump_table:
    .long case_0    ; 偏移 0：case 0 的地址
    .long case_1    ; 偏移 4：case 1 的地址
    .long case_2    ; 偏移 8：case 2 的地址
    .long case_3    ; 偏移 12：case 3 的地址
```

> [!note] 跳转表的优势
> - **O(1) 时间复杂度**：无论多少 case，一次索引即定位
> - **条件**：case 值必须连续/密集，否则跳转表会因"空洞"太大而浪费空间，此时编译器退化为逐条件比较

# 循环语句的机器级表示

## do-while 循环（最简形式）

```c
do {
    loop_body;
} while (condition);
```

编译为：

```asm
loop_start:
    ; --- loop_body ---
    计算 condition
    CMP / TEST
    Jcc  loop_start     ; 条件满足则继续循环
```

> do-while 效率最高：循环体至少执行一次，**只需一个条件跳转**（跳回循环开始），无额外分支。

## while 循环

```c
while (condition) {
    loop_body;
}
```

编译方式一（先判断条件——含中间跳转）：

```asm
    JMP  test_cond      ; 先跳到条件判断处
loop_start:
    ; --- loop_body ---
test_cond:
    计算 condition
    CMP / TEST
    Jcc  loop_start     ; 条件满足则跳回循环体
```

编译方式二（先判断条件——含条件分支再循环体——优化后，-O1 常见）：

```asm
test_cond:
    计算 condition
    CMP / TEST
    Jcc_n  exit_loop    ; 条件不满足时跳出
    ; --- loop_body ---
    JMP  test_cond      ; 无条件跳回条件判断处
exit_loop:
```

> 方式二更自然（"条件不满足就退出"），但有**两次跳转**在每次循环中（Jcc_n 不出时通过 JMP 跳回条件判断处）。而 do-while 每次循环只需一次条件跳转。GCC 在 `-O1` 及以上会将 while 先转换为 do-while 形式后优化（"先判断是否触发第一次执行，然后改成 do-while"），消除第二个无条件跳转。

## for 循环

```c
for (init; condition; update) {
    loop_body;
}
```

等价于：

```c
init;
while (condition) {
    loop_body;
    update;
}
```

编译为：

```asm
    ; --- init ---
    JMP  test_cond
loop_start:
    ; --- loop_body ---
    ; --- update ---
test_cond:
    计算 condition
    CMP / TEST
    Jcc  loop_start
```

> for 循环的机器级实现与 while 几乎完全相同，仅多了 update 部分。

## 循环优化

### 将 while/for 转换为 do-while

编译器优化时，常将 while 循环转换为 do-while 形式，减少无条件跳转指令：

原始 while：
```asm
test_cond:
    CMP ...
    JE  exit
    ; body
    JMP test_cond      ← 额外的无条件跳转
exit:
```

优化的 do-while 形式（-O1）：
```asm
    ; 首先：若初始条件就不满足，直接结束（guard 一次判断）
    CMP ...; JE exit
loop_start:
    ; body
    CMP ...; JNE loop_start   ← 只有一次条件跳转
exit:
```

# 过程调用的机器级表示

过程（函数）调用是高级语言最重要的抽象之一，在机器级涉及**栈帧管理、参数传递、寄存器保存、返回地址维护**等机制。

## 栈帧（Stack Frame）

每次函数调用都会在栈上分配一个**栈帧**（Activation Record），用于存储：

```
         高地址
    ┌──────────────┐
    │  调用者栈帧    │
    ├──────────────┤
    │  参数 n       │  ← 入栈顺序：从右向左（cdecl 约定）
    │  ...         │
    │  参数 1       │
    ├──────────────┤
    │  返回地址     │  ← CALL 指令自动压入
    ├──────────────┤  ← EBP 指向此处（callee 的栈帧基址）
    │  旧 EBP 值   │  ← 由 callee 保存（PUSH EBP）
    ├──────────────┤
    │  局部变量     │  ← 通过 SUB ESP, N 分配空间
    │  ...         │
    ├──────────────┤
    │  保存的寄存器  │  ← callee-save 寄存器（EBX, ESI, EDI 等）
    ├──────────────┤  ← ESP 指向此处（栈顶）
    │              │
         低地址
```

## 调用约定（Calling Convention）

调用约定（Calling Convention）是调用者与被调用者之间的**协议**，规定了三件事：**参数怎么传**、**栈由谁清理**、**返回值放哪里**。如果双方调用约定不一致（例如 caller 按 cdecl 清理、callee 却按 stdcall 用 `RET n` 清理），会导致栈指针错乱，程序崩溃。

> 三种调用约定的共同点：**返回值统一放在 EAX 中**（64 位返回值则 EDX:EAX）。

### 1. cdecl（C Declaration，C 语言默认）

```
int add(int x, int y)   // 右侧参数 y 先压栈，左侧 x 后压栈
```

| 维度 | 规则 | 具体表现 |
|:---|:---|:---|
| **参数传递顺序** | 从右向左依次压栈 | `add(a, b)` → 先 push b，再 push a |
| **栈由谁清理** | **调用者清理** | `CALL` 返回后，由 caller 执行 `ADD ESP, N` 弹出参数 |
| **返回值位置** | EAX | callee 将返回值放入 EAX |
| **适用场景** | C 语言默认约定 | 适合**可变参数函数**（如 `printf`），因为只有调用者知道到底传了几个参数 |

> cdecl 的核心特征是 **"谁传参谁清理"**——调用者知道参数个数，负责恢复栈指针；这是支持 `printf(fmt, a, b, c, ...)` 可变参数的唯一方式。

### 2. stdcall（Standard Call，Win32 API 标准）

| 维度 | 规则 | 具体表现 |
|:---|:---|:---|
| **参数传递顺序** | 从右向左依次压栈 | 同 cdecl |
| **栈由谁清理** | **被调用者清理** | callee 返回时用 `RET N`（N = 参数总字节数）自动释放参数 |
| **返回值位置** | EAX | 同 cdecl |
| **适用场景** | Win32 API | 被调用者个数固定，清理一次即可；减少代码体积（每条调用处省去 `ADD ESP, N`） |

> stdcall 的关键：`RET 8` 等价于 `RET` + `ADD ESP, 8`，在返回的同时弹出栈参数。优点是调用点代码更短（省去了每个 call site 的 `ADD ESP, N`），缺点是不支持可变参数——因为 callee 在编译时就固定了 `RET N` 中的 N。

### 3. fastcall（Fast Call，寄存器传参优化）

| 维度 | 规则 | 具体表现 |
|:---|:---|:---|
| **参数传递顺序** | 前两个参数用 ECX、EDX，其余从右向左压栈 | `add(a, b, c)` → ECX=a, EDX=b, push c |
| **栈由谁清理** | **被调用者清理** | 同 stdcall（仅针对栈上的参数部分） |
| **返回值位置** | EAX | 同上 |
| **适用场景** | 性能敏感场景 | 减少访存，前两个参数在寄存器中直接获取 |

> fastcall 的核心优势：前两个参数不进栈，省去 push 和内存访问，速度最快。缺点是寄存器个数有限，超过两个参数时依然需要压栈。

### 三种约定对比

| 对比维度 | cdecl | stdcall | fastcall |
|:---|:---|:---|:---|
| 参数压栈顺序 | 右→左 | 右→左 | 前两个进寄存器(ECX,EDX)，其余右→左压栈 |
| 栈清理方 | Caller | Callee（`RET N`） | Callee |
| 支持可变参数 | ✓（唯一支持） | ✗ | ✗ |
| 代码体积 | 调用点大（每条 call 后需 `ADD ESP, N`） | 调用点小（清理在函数内） | 调用点较小 |
| 执行速度 | 一般 | 一般 | 快（减少访存） |
| 典型场景 | C 程序、printf 系列 | Windows DLL 导出函数 | 性能关键路径 |

> [!important] 传参顺序为什么是从右向左？
> cdecl 和 stdcall 都是**从右向左压栈**（先压最右边的参数）。根本原因是支持**可变参数函数**：`printf(fmt, a, b)` 中，调用者按右→左压栈，最终 `fmt`（第一个参数）在**栈顶**（低地址）固定位置 `[EBP+8]`。被调用者无需知道参数个数，只需通过第一个固定参数（如格式串）来推导后续参数的位置。无论传了多少个变参，`fmt` 总在同一个偏移处。

> [!note] x86-64 的调用约定变化
> 以上是 IA-32 的传统约定。在 x86-64 下，主流约定改为**寄存器传参为主**（System V AMD64：前 6 个整型参数用 RDI/RSI/RDX/RCX/R8/R9，浮点参数用 XMM0~XMM7），栈传参为辅，且不再有 cdecl/stdcall/fastcall 之分。408 考试以 IA-32 为主，了解这一演进即可。

## 过程调用的完整步骤

### 调用者（Caller）的职责

1. **保存调用者寄存器**（caller-save）：`EAX`、`ECX`、`EDX` 的值可能在调用中被覆盖，若需保留先在当前栈帧中保存到内存
2. **参数压栈**：从右向左将实参依次压入栈中
3. **执行 CALL 指令**：将返回地址压栈，跳转到被调用函数
4. **清理栈参数**（cdecl 约定）：`ADD ESP, N`（N = 参数总字节数）
5. **获取返回值**：从 `EAX` 中读取

### 被调用者（Callee）的职责

callee 被 CALL 进入后，负责完成以下工作：

1. **保存调用者的 EBP**（`PUSH EBP`）— 旧栈帧基址是 caller 在栈上的"锚点"，不保存会导致调用返回后 caller 无法恢复其局部变量寻址
2. **建立自己的栈帧**（`MOV EBP, ESP`）— 将 EBP 设为当前栈顶，CEBP 在函数体中保持**只读不变**，作为访问参数和局部变量的基准
3. **为局部变量分配空间**（`SUB ESP, N`）— 在栈上开辟 N 字节空间给局部变量使用，通过 `[EBP - offset]` 访问
4. **保存 callee-save 寄存器**（`PUSH EBX/ESI/EDI`）— 若函数体要使用这些"跨调用长寿命"寄存器，必须先保存原值
5. **执行函数体** — 参数从 `[EBP + 8]` 起递增读取，局部变量向 `[EBP - offset]` 读写，最终值放入 EAX
6. **恢复 callee-save 寄存器**（`POP EDI/ESI/EBX`）— **严格逆序**恢复，保证原值不变
7. **释放栈帧并返回** — `MOV ESP, EBP` + `POP EBP` + `RET`（或 `RET N`）

下面详细拆解每一步的汇编，即函数的**序言**和**尾声**：

**序言（Prologue）——进入函数时执行**：

```asm
PUSH EBP            ; ① 将调用者的 EBP 压栈保存（进入后第一个动作）
MOV  EBP, ESP       ; ② EBP ← ESP，自己的栈帧基址建立完毕
SUB  ESP, N         ; ③ ESP 下移 N 字节，为局部变量预留空间
PUSH EBX            ; ④ 依次保存 callee-save 寄存器
PUSH ESI            ;    若函数体不用这些寄存器，可省略
PUSH EDI
```

> 序言完成后，栈布局如上节栈帧图所示：`[EBP+8]` 起是参数，`[EBP+4]` 是返回地址，`[EBP]` 是旧 EBP，`[EBP-4]` 起是局部变量。

**尾声（Epilogue）——退出函数时执行**：

```asm
POP  EDI            ; ⑤ 恢复 callee-save 寄存器（与保存顺序严格相反）
POP  ESI
POP  EBX
MOV  ESP, EBP       ; ⑥ 释放局部变量空间：ESP 回到 EBP 处
POP  EBP            ; ⑦ 恢复调用者的 EBP（caller 的栈帧锚点恢复）
RET                 ; ⑧ 弹出返回地址并跳转
```

> [!important] 尾声的几个要点
> - callee-save 寄存器的 POP 顺序与 PUSH **严格相反**（栈是 LIFO：先 push EBX → 后 push ESI → 先 pop ESI → 后 pop EBX）
> - `MOV ESP, EBP` 一步释放所有局部变量，无需逐一 `ADD ESP, N`（N 由序言中的 `SUB ESP, N` 决定，编译时已确定）
> - `POP EBP` 后，ESP 指向返回地址，EBP 恢复到调用者的值——caller 现在可以用 `[EBP-xx]` 重新访问自己的局部变量

### 函数内部对参数和局部变量的访问

EBP 在函数体中固定不变（硬件上的栈帧指针），通过 EBP + 偏移来访存：

```
[EBP + 8]   →  第一个参数（往上跳过返回地址 4 字节 + 旧 EBP 4 字节）
[EBP + 12]  →  第二个参数
[EBP - 4]   →  第一个局部变量
[EBP - 8]   →  第二个局部变量
```

### CALL 和 RET 的硬件执行过程

**CALL proc**（假设 32 位模式）：

1. `ESP ← ESP - 4`（栈指针下移 4 字节）
2. `M[ESP] ← EIP`（当前指令地址压栈，即返回地址）
3. `EIP ← proc`（跳转到被调用函数）

**RET**：

1. `EIP ← M[ESP]`（从栈中取出返回地址）
2. `ESP ← ESP + 4`（栈指针上移）

**RET N**（stdcall 约定，被调用者清理参数）：

1. `EIP ← M[ESP]`
2. `ESP ← ESP + 4 + N`（额外跳过 N 字节参数）

## 寄存器保存策略

| 类型 | 寄存器 | 含义 |
|:---|:---|:---|
| **caller-save** | EAX, ECX, EDX | 调用者负责保存。这些寄存器可能在调用过程中被 callee 随意使用和修改；若 caller 希望保留它们的值，必须自己在调用前保存（push），调用后恢复（pop） |
| **callee-save** | EBX, ESI, EDI, EBP | 被调用者负责保存。callee 使用这些寄存器前必须先保存原值（push），返回前恢复（pop），确保 caller 看到的值不变 |

> [!important] 保存策略的意义
> caller-save 适用于"随时可丢弃"的临时值（如计算结果进入 eax 后马上就要传给函数），callee-save 适用于"跨函数调用需要保持"的长寿命值（如循环变量在 esi 中、基址在 ebx 中）。这样既避免了不必要的大量 push/pop，又确保了调用后长寿命变量不被破坏。

## 完整示例

C 代码：

```c
int add(int x, int y) {
    int sum = x + y;
    return sum;
}

int main() {
    int a = 3, b = 5;
    int result = add(a, b);
    return 0;
}
```

对应汇编：

```asm
; ===== add 函数 =====
add:
    PUSH EBP
    MOV  EBP, ESP
    SUB  ESP, 4         ; 为局部变量 sum 分配空间
    MOV  EAX, [EBP+8]   ; 取参数 x
    ADD  EAX, [EBP+12]  ; 加上参数 y
    MOV  [EBP-4], EAX   ; sum = x + y
    ; 返回值已在 EAX 中
    MOV  ESP, EBP
    POP  EBP
    RET

; ===== main 函数 =====
main:
    PUSH EBP
    MOV  EBP, ESP
    SUB  ESP, 8         ; 为 a, b 分配空间（或为 result 等）
    MOV  DWORD PTR [EBP-4], 3   ; a = 3
    MOV  DWORD PTR [EBP-8], 5   ; b = 5

    ; 准备调用 add(a, b)
    PUSH DWORD PTR [EBP-8]      ; 参数 b（从右向左先压）
    PUSH DWORD PTR [EBP-4]      ; 参数 a
    CALL add
    ADD  ESP, 8                 ; 调用者清理栈（cdecl 约定，两个 int = 8 字节）

    MOV  [EBP-12], EAX          ; result = 返回值（在 EAX 中）
    MOV  EAX, 0                 ; return 0
    MOV  ESP, EBP
    POP  EBP
    RET
```

调用 add 前栈的变化：

```
调用前（main 已 push 参数后）：
    ┌──────────┐
    │  b = 5   │  ← ESP+4（第二个参数，先压入的在高地址）
    │  a = 3   │  ← ESP（第一个参数，后压入的在低地址，栈顶）
    └──────────┘

CALL add 后（返回地址已自动压栈）：
    ┌──────────┐
    │  b = 5   │
    │  a = 3   │
    │ 返回地址  │  ← ESP（CALL 压入）
    └──────────┘

add 序言后：
    ┌──────────┐
    │  b = 5   │  ← [EBP+12]
    │  a = 3   │  ← [EBP+8]
    │ 返回地址  │  ← [EBP+4]
    │ 旧 EBP   │  ← EBP 指向此处
    │ sum      │  ← [EBP-4], ESP 指向此处
    └──────────┘
```

# 总结

> [!summary] 本章重点
> 1. **常用寄存器**：8 个通用寄存器（EAX/EBX/ECX/EDX/ESI/EDI/EBP/ESP）+ EIP + EFLAGS；关键标志位 ZF/SF/CF/OF
> 2. **CF vs OF**：CF 服务于无符号数（进位/借位），OF 服务于有符号数（补码溢出）；二者独立设置
> 3. **数据传送指令**：MOV（不允许 mem→mem）、PUSH/POP（x86 满递减栈）、LEA（取地址非取值）、XCHG（交换）
> 4. **算术指令**：ADD/SUB/INC/DEC/NEG/MUL/DIV；INC/DEC **不影响 CF**（408 常考点）；MUL/DIV 使用**隐含操作数**在 EAX/EDX 中；IMUL 有单/双/三操作数三种形式，双操作数和三操作数形式结果会被截断（可能丢精度，溢出时 OF/CF=1），三操作数形式还保留源操作数 S1 不变
> 5. **逻辑指令**：AND/OR/XOR/NOT/TEST + 移位（SHL/SHR/SAL/SAR/ROL/ROR）；CMP 做减法不存结果，TEST 做与运算不存结果——都只影响标志位配条件跳转
> 6. **有符号比较 vs 无符号比较**：无符号（CF 判决，above/below），有符号（SF⊕OF 判决，greater/less）；有符号比较必须同时检查 SF 和 OF，因为溢出会反转 SF 的含义
> 7. **条件跳转**：JMP 无条件 → JE/JNE（相等），JA/JB（无符号），JG/JL（有符号），以及单标志位跳转（JC/JO/JS 等）
> 8. **汇编格式**：AT&T（源→目的，`%eax`，`$5`）vs Intel（目的←源，`eax`，`5`）；408 默认 Intel 格式
> 9. **if-else → 条件跳转**：计算条件 → CMP → Jcc 跳过 then → 执行 else → JMP 跳过 else 部分
> 10. **switch → 跳转表**（case 密集时）：边界检查 + `JMP [table + index*4]`，O(1) 时间复杂度
> 11. **循环语句对比**：
>     | 循环类型 | 条件判断位置 | 最少执行次数 | 跳转数/次迭代 |
>     |:---|:---|:---:|:---:|
>     | do-while | 循环体之后 | 1 次 | 1（条件跳转） |
>     | while | 循环体之前 | 0 次 | 2（条件跳转 + 无条件跳转） |
>     | for | 循环体之前 | 0 次 | 2（同 while） |
> 12. **循环优化**：编译器将 while/for 转为 do-while + guard（先判断零次），消除循环内无条件跳转
> 13. **过程调用栈帧**：CALL 压入返回地址 → 序言（PUSH EBP; MOV EBP,ESP; SUB ESP,N）→ 函数体（[EBP+8] 起访问参数，[EBP-N] 访问局部变量）→ 尾声（MOV ESP,EBP; POP EBP; RET）
> 14. **调用约定**：cdecl（参数右→左压栈，调用者清理）、stdcall（被调用者清理，RET N）、fastcall（寄存器传参）；408 主要考 cdecl
> 15. **寄存器保存策略**：caller-save（EAX/ECX/EDX，调用者负责保存，适用于临时值）vs callee-save（EBX/ESI/EDI/EBP，被调用者负责保存，适用于跨函数调用的长寿命值）
> 16. **参数寻址**：[EBP+8] 即第一个参数（往上跳过返回地址 4B + 旧 EBP 4B）；[EBP-4] 即第一个局部变量
