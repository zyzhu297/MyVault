# 程序查询方式

## 基本概念

- CPU 通过**不断查询**外设的状态来"主动"控制数据传送，是最简单的 I/O 方式
- 外设的速度远慢于 CPU，因此 CPU 在查询过程中大量时间在**踏步等待**，效率低下

## 工作流程

1. CPU 执行初始化程序，预置传送参数（如起始地址、数据量等）
2. 向 I/O 接口发送命令字，启动外设
3. CPU 循环向 I/O 接口发出查询命令，读取**状态寄存器**
4. 若外设**未就绪**，则继续查询，若外设**已就绪**，CPU 执行一次数据传送
5. 修改地址和计数器参数
6. 判断传送是否完成，若未完成返回步骤 3，直至计数器归零
>[!note] 查询策略
>- **独占查询**：CPU 一旦启动外设，便**全程反复查询**该设备状态，直到数据传送全部完成才转去执行其他任务。此期间 CPU 不能处理其他任何事务，与程序查询的基本流程一致。
>- **定时查询**：CPU 周期性地（如每隔固定时钟周期）检查设备状态。若设备未就绪，CPU **暂时转去执行其他任务**，等待下一个查询时刻再次检查。该策略兼顾了对外设的响应和对 CPU 的利用，但需要额外的**定时器**硬件支持。

| 优点 | 缺点 |
|------|------|
| 硬件结构简单，控制容易 | CPU 与外设只能**串行**工作 |
| 无需额外硬件支持 | CPU 大量时间处于**忙等**（Busy Waiting） |
| 适用于慢速外设 | 系统效率极低，无法响应紧急事件 |

# 程序中断方式

## 基本概念

- CPU 启动外设后**继续执行现行程序**，外设准备好后**主动**向 CPU 发出中断请求
- CPU 与外设可以**并行**工作，大幅提升系统效率
- 中断是 I/O 系统中最核心的机制之一

## 中断的功能

### 基本功能

1. **实现 CPU 与外设并行工作**：CPU 启动外设后可继续执行其他程序，外设准备就绪时主动通知 CPU，实现了 CPU 与 I/O 设备之间的异步协同。
2. **实时响应**：外设可在任意时刻向 CPU 发出请求，CPU 在指令执行间隙响应，适用于对时间敏感的 I/O 场景。
3. **故障处理与异常检测**：能检测和响应硬件故障、电源掉电等异常事件，提高系统的可靠性和容错能力。
4. **程序调试**：支持断点中断、单步中断等调试功能，便于程序开发与错误排查。
5. **多任务调度**：为操作系统进程调度提供时钟中断支持，是实现分时多任务的基础。
6. **软中断与陷阱（Trap）**：支持系统调用（SVC）、软件中断（SWI）等由程序主动触发的中断，用于请求操作系统服务或捕获异常指令。
7. **支持人机交互**：键盘、鼠标、触摸屏等输入设备的每一次操作都以中断方式通知 CPU，使得系统能够即时响应用户动作，实现流畅的人机交互体验。
8. **满足实时系统对快速响应的需求**：实时系统要求对外部事件在严格的时间约束内做出反应。中断机制使得 CPU 能够在事件发生的极短时间内（指令执行结束时）立刻转入处理，满足实时系统中**确定性**和**低延迟**的关键需求。

## 中断技术的基本思想

（1）**被动响应**：外设主动发出请求，CPU 被动响应，体现了"请求-响应"的异步模式。
（2）**程序切换**：通过保存/恢复现场（PC、PSW、通用寄存器），实现程序执行流的上下文切换。
（3）**向量化入口**：通过中断向量表，将不同中断源映射到对应的服务程序入口，实现中断的快速分发

## 程序中断的工作流程

### 中断请求
*中断源*是指能够向 CPU 发出中断请求的设备或事件；一台计算机允许多个中断源同时存在，且各中断源发出请求的时间具有随机性
- 外设通过 I/O 接口向 CPU 发出中断请求信号
- 每个中断源对应一个**中断请求触发器（INTR）**，当外设需要服务时置 1
- CPU 在每个指令周期结束时采样中断请求线，检测是否有中断请求
- 多个中断源的中断请求触发器构成**中断请求寄存器**，CPU 可通过读该寄存器获知哪些中断源发出了请求

>[!note] 中断请求的触发方式
> - **边沿触发**：以请求信号的跳变沿（上升沿或下降沿）作为中断请求标志，请求信号消失后触发器仍保持
> - **电平触发**：以请求信号的高/低电平表示中断请求，电平消失则请求也消失

### 中断判优（优先级排队）
中断响应优先级是指 CPU 响应**多个同时发生的中断请求**的先后顺序
- **硬件判优**：链式排队器、中断控制器（如 8259A）
- **软件判优**：CPU 逐个查询各中断源
- 优先级原则：**非屏蔽 > 可屏蔽**；**硬件故障 > I/O 设备**；**高速 > 低速**；**实时设备** > **普通设备**；**输入设备** > **输出设备**
>[!important] 中断优先级
>中断优先级实际上包含两个不同层面的概念：
>### 响应优先级
>- 由**硬件排队电路**（如链式排队器、中断控制器）固定决定
>- 决定当**多个中断同时请求**时，CPU 最先响应哪一个
>- 优先级由 CPU 与外设的连接顺序或中断类型号决定，**用户不可修改**
>
>### 处理优先级
>- 由**中断屏蔽字**（软件设置）动态决定
>- 决定当前正在处理某个中断时，是否允许被另一个中断打断（中断嵌套）
>- 用户可通过设置屏蔽寄存器**灵活调整**，实现动态优先级调度
>
> **核心区别**：响应优先级决定"多个请求同时到达时谁先被响应"；处理优先级决定"正在服务时谁可以打断谁"。二者可以不一致——例如某设备响应优先级低，但通过屏蔽字可使其**处理优先级**提高，不被其他设备打断。

### CPU 响应中断的条件

1. **中断源有请求**（中断请求触发器 INTR = 1）
2. **CPU 允许中断**（中断允许触发器 EINT / IF = 1），异常和不可屏蔽中断不受此限制
3. **一条指令执行结束**，异常不受此限制
4. **无更高优先级的中断正在处理**

### 中断响应过程
由**硬件自动完成**一系列操作，称为**中断隐指令**；随后转入中断服务程序；中断隐指令并非指令系统中的真实指令，而是对硬件自动操作的统称

1. **关中断**：首先关闭中断允许表示，防止保存现场时被再次打断
2. **保存断点**：PC 和 PSW 压入堆栈或专用寄存器
3. **取中断向量**：根据中断类型号查中断向量表，获取中断服务程序入口地址

## 中断向量 & 中断向量表

- **中断向量**：中断服务程序的**入口地址**，送入程序计数器 PC
- **中断向量表**：存放所有中断向量的内存区域
- **中断类型号**：每个中断的唯一编号，用于在向量表中索引
> 中断请求和响应信号通过 I/O 总线的控制线传输，而中断类型号则在中断响应阶段由中断控制器经数据总线提供给 CPU，用于定位中断向量表的对应表项

## 中断服务程序流程

1. **关中断**：进入中断服务程序后首先关闭中断，防止在保护现场的过程中被新的中断请求打断
2. **保护断点**：将程序计数器 PC 和程序状态字 PSW 压入堆栈，保存被中断程序的返回地址和执行状态
3. **中断服务程序寻址**：通过中断类型号查找中断向量表，获取对应中断服务程序的入口地址，送入 PC

4. **保存现场和中断屏蔽字**：将**通用寄存器**的内容压栈保存（防止中断服务程序修改后无法恢复）；同时保存当前的中断屏蔽字，并根据需要设置新的屏蔽字（允许更高级中断嵌套，屏蔽同级或低级中断）
>[!warning]
>断点与现场信息均不可被中断服务程序破坏；由于现场信息用指令可直接访问，因此由**软件**在进入中断服务程序时**显式保存**；而断点信息由**硬件**在中断响应阶段自动保存
5. **开中断**：现场保护完毕后重新开中断，允许更高优先级的中断请求在本次服务期间得到响应，是实现**中断嵌套**的关键
6. **服务处理**：执行具体的中断服务内容，针对不同中断类型进行相应处理（如数据传送、异常处理等），**由软件处理**
7. **关中断**：服务处理完毕后再次关中断，防止在恢复现场的过程中被打断，保证恢复操作的原子性
8. **恢复现场**：将堆栈中保存的 PSW、PC 及通用寄存器弹出，恢复被中断程序执行前的完整状态
9. **开中断**：现场恢复完毕后重新开中断，使 CPU 能够继续响应新的中断请求
10. **中断返回（IRET）**：执行中断返回指令，CPU 从原程序的断点处继续执行

⭐**其中 1~3 由中断隐指令（硬件自动）完成；4~10 由中断服务程序（软件）完成**

> 两次"关中断→开中断"的包围结构：**第一次包围保护现场**（确保保存完整），**第二次包围恢复现场**（确保恢复完整），中间开中断的窗口则为**中断嵌套**提供了可能；若为单重中断，则可省略中间的 5 和 7

## 多重中断和中断屏蔽技术

### 多重中断（中断嵌套）

CPU 正在执行某个中断服务程序时，又被新的中断请求打断，转而执行更高优先级的中断服务程序，处理完毕后再返回原中断继续执行。

**与单重中断的区别**：
- **单重中断**：整个中断服务期间不开中断，所有新请求必须等待当前中断执行完毕
- **多重中断**：保护现场后即开中断，允许更高优先级请求抢占

**实现条件**：
1. 中断服务程序中**提前开中断**（保护现场后立即开中断）
2. 硬件支持**优先级排队**和**中断嵌套**机制
3. 通过**中断屏蔽字**控制嵌套范围

### 中断屏蔽技术

通过设置**中断屏蔽寄存器**中的**屏蔽字（Mask Word）**，允许或禁止特定中断源的中断请求信号到达 CPU；若不使用屏蔽技术，则**处理优先级**和**响应优先级**一致

- 屏蔽字的每一位对应一个中断源：**0 = 开放（允许）**，**1 = 屏蔽（禁止）**
- CPU 响应中断后，硬件自动设置屏蔽字（亦可由软件动态修改）

**屏蔽字的设置原则**：当中断源 i 被响应后，应屏蔽**与 i 同级及更低级**的中断源，开放比 i 更高级的中断源，从而实现"高可打断低，低不可打断高"的嵌套策略。

>[!warning] 中断屏蔽字在进入中断服务程序时由软件加载
>用于控制后续中断的嵌套行为；因此**中断屏蔽仅在 CPU 执行中断服务期间生效**；而主程序运行阶段，中断响应仍由主程序所设置的屏蔽状态决定；此外在中断服务程序执行期间，若有多个未屏蔽中断同时到达，其处理顺序仍由中断判优电路的响应优先级决定

**实例**：设 4 个中断源，处理优先级为 **A > B > C > D**（A 最高），每个中断源的屏蔽字如下：

| 中断源 | 屏蔽字（D C B A） | 说明 |
|--------|-------------------|------|
| A（最高） | 1 1 1 1 | 屏蔽所有（含自身），不允许任何中断打断 |
| B | 1 1 1 0 | 屏蔽 C、D 和自身，仅开放 A |
| C | 1 1 0 0 | 屏蔽 D 和自身，开放 A、B |
| D（最低） | 1 0 0 0 | 仅屏蔽自身，开放 A、B、C |

> **规律**：优先级越高，屏蔽字中 1 越多——因为能打断它的中断越少。自身的位始终为 1（防止自身重入）。

**嵌套过程中的屏蔽字切换**：
```
主程序（屏蔽字 = 0000，全部开放）
   │
   ├── 响应中断 C → 设置屏蔽字 = 1100（屏蔽 D 和自身，开放 A、B）
   │    │
   │    ├── A 请求到来，优先级 A > C → 允许嵌套
   │    │   响应中断 A → 设置屏蔽字 = 1111（屏蔽所有）
   │    │   A 服务完成 → 恢复 C 的屏蔽字 1100
   │    │
   │    └── C 服务完成 → 恢复主程序屏蔽字 0000
```

> **关键**：屏蔽字在嵌套时压栈保存，退出时逐层恢复，保证各级中断的屏蔽策略互不干扰。

### 总结

| 技术         | 作用            | 实现方式            |
| ---------- | ------------- | --------------- |
| 多重中断（中断嵌套） | 优先级高的可打断低的    | 提前开中断 + 硬件优先级判决 |
| 中断屏蔽       | 动态控制允许/禁止哪些中断 | 设置屏蔽寄存器中的屏蔽字    |

两者配合使用：**中断屏蔽是实现可控嵌套的手段，多重中断是屏蔽技术发挥作用的场景**

## 特点

| 优点 | 缺点 |
|------|------|
| CPU 与外设部分并行工作 | 仍需要 CPU 介入数据传送 |
| 可响应实时事件 | 每次中断有**保存/恢复现场**的开销 |
| 支持中断嵌套 | 批量数据传送时效率仍不够高 |

# DMA 方式

## 基本概念

- **Direct Memory Access**：直接存储器访问
- 在主存与外设之间建立**直接数据通路**，**无需 CPU 干预，完全由硬件控制的数据传输方法**
- ⭐适用于**高速、大批量**数据传送（磁盘、网络、显卡等）

## 主要特征

- 打破主存与 CPU 的固定关联，主存既可被 CPU 访问，又可被外设直接访问
- 在块数据传输过程中，主存地址的生成与传输字数的计数均有**硬件电路自动完成**
- 主存中设置专用缓冲区，以支持高效数据交换
- 支持高速数据传输，且 CPU 与外设可并行工作，显著提升系统效率
- 传输开始前需由**软件**进行预处理，结束后则通过**中断机制**完成后处理

## DMA 控制器（DMAC）的组成

对数据传送过程中进行控制的硬件称为 DMA 控制器（DMA 接口），主要功能如下：
- 接受外设发出的 DMA 请求，并向 CPU 发出总线请求信号
- 在 CPU 发出总线响应信号后，DMA 控制器接管总线控制权，进入 DMA 操作周期
- 确定数据传送的主存起始地址及长度，并自动更新主存地址计数器和传送长度计数器
- 指定数据在主存与外设间的传送方向，发出读/写等控制信号，完成数据传送
- 在 DMA 操作结束后，向 CPU 报告传送完成

DMA 控制器是负责在 I/O 设备与主存之间建立直接数据通路的专用硬件，其核心部件包括：

| 部件                  | 功能                                       |
| ------------------- | ---------------------------------------- |
| **主存地址计数器（MAR/AR）** | 存放本次 DMA 传送的主存首地址，每传送一个字自动 +1            |
| **传送长度计数器（WC）**     | 存放本次传送的剩余字数，每传送一个字自动 -1，减至 0 时表示传送完成     |
| **数据缓冲寄存器（DBR/DR）** | 暂存 DMA 传送过程中每次读/写的数据（外设与主存之间的中转站）        |
| **DMA 请求触发器**       | 当外设准备好进行 DMA 传送时置 1，向控制/状态逻辑发出请求，进而向 CPU 申请总线控制权 |
| **控制/状态寄存器（CSR）**   | 存放 CPU 发来的控制命令（传送方向、操作模式等）及 DMA 控制器的当前状态 |
| **中断机构**            | 数据块传送完成后向 CPU 发出中断请求，请求进行后处理             |

> DMA 控制器接管总线时的角色相当于一个"微型 CPU"——它能发出主存地址和读/写控制信号，但不具备指令执行能力，只做数据搬运。

## DMA 的传送方式

解决 **DMA 控制器与 CPU 同时访问主存** 的冲突问题：

### 停止 CPU 访存

- DMA 控制器向 CPU 发出 HOLD 信号申请总线控制权
- CPU 在当前总线周期结束后释放总线（交出地址线、数据线和控制线），并回送 HLDA 响应信号
- DMA 获得总线后连续传送一整块数据，传送期间 CPU **完全停止**主存访问
- 传送完成后 DMA 释放总线，CPU 恢复运行

**优点**：控制简单，硬件开销小，适用于数据传输速率很高的 I/O 设备进行成组数据传输；
**缺点**：CPU 利用率低，DMA 传送期间 CPU 基本处于空闲状态

### 周期挪用

对于高速 I/O 设备，若不及时访存可能导致数据丢失，因此其访存请求具有**较高优先级**周期挪用允许 I/O 设备挪用一个主存存储周期，传送完**一个数据字**House立即释放总线；**属于单字传送方式**
当 I/O 设备发起 DMA 请求时，可能出现以下三种情况：
- CPU 当前不在访存，I/O 无冲突，可直接使用总线
- CPU 正在访存，则需等待当前存储周期结束，再释放总线控制权
- I/O 与 CPU 同时请求访存，发生冲突，此时 CPU 暂时放弃总线控制权

**优点**：兼顾了 DMA 传输效率与 CPU 利用率；CPU 不会长期停摆，DMA 传送期间仍可执行不访存的指令；既适用于单字传送，也能通过反复挪用实现成块传送
**缺点**：每次传送都需要申请和释放总线，控制开销较大；若 DMA 请求频繁，CPU 频繁让出总线会导致 CPU 执行速度下降；需要复杂的总线仲裁电路

### DMA 与 CPU 交替访存

将总线时间划分为固定的时间片，CPU 与 DMA **轮流使用**总线，互不冲突。

- 总线周期分为 C1（CPU 使用）和 C2（DMA 使用），交替运行
- CPU 和 DMA 各自拥有独立的时间窗口，无需总线申请/释放握手过程

**优点**：无需总线请求和仲裁，传输流畅
**缺点**：硬件复杂，需要精确的时间同步机制，CPU 访存带宽减半。

## DMA 的传送过程

### 预处理

由 **CPU 执行程序** 完成，为 DMA 传送做好准备工作：

1. 向 DMA 控制器的 **主存地址寄存器（AR）** 写入数据块在主存中的首地址
2. 向 DMA 控制器的 **字计数器（WC）** 写入本次传送的字数
3. 向 DMA 控制器的 **控制/状态寄存器** 写入传送方向（读/写）及操作模式
4. 向 I/O 接口发送启动命令，启动外设
5. CPU 继续执行现行程序，将数据传送任务完全交给 DMA 控制器

### 数据传送

由 **DMA 控制器硬件** 自动完成，CPU 不参与：

1. 外设准备好（输入：数据进入数据缓冲寄存器；输出：数据已被取走）
2. DMA 控制器向 CPU 发出 **总线请求（HOLD/BR）**
3. CPU 在当前**总线周期结束**后释放总线控制权，回送**总线响应（HLDA/BG）**
4. DMA 控制器获得总线控制权，执行 **一次数据传送**：
   - 输入：DBR → 主存（AR 指向的单元）
   - 输出：主存（AR 指向的单元） → DBR
5. 释放总线控制权，AR + 1，WC - 1
6. 若 WC ≠ 0，返回步骤 1 继续传送；若 WC = 0，转入后处理

> **注意**：步骤 4 每次只传送一个字（单字传送模式），传完后归还总线，让 CPU 可以利用总线访存，避免长期占用，**整个过程以数据块为单位传输**

### 后处理

由 **CPU 执行中断服务程序** 完成：

1. 数据块传送完毕，WC 减至 0，DMA 控制器向 CPU 发出 **中断请求**
2. CPU 响应中断，转入 DMA 后处理中断服务程序
3. 校验数据（奇偶校验、CRC 校验等），判断传送是否有错
4. 若需继续传送（多块数据），重新设置 AR 和 WC，再次启动 DMA
5. 若传送全部完成，中断返回，CPU 继续执行原程序

## DMA 方式与中断方式的区别

| 维度 | 程序中断方式 | DMA 方式 |
|------|-------------|----------|
| **数据传送执行者** | 由 CPU 执行中断服务程序完成 | 由 DMA 控制器硬件自动完成 |
| **CPU 介入时机** | 每传送一个字，外设就中断 CPU 一次 | 一整块数据全部传送完毕后，才向 CPU 发一次中断 |
| **是否修改 CPU 现场** | 需要保存/恢复 PC、PSW 及通用寄存器 | 无需保存 CPU 现场（DMA 不涉及 CPU 内部状态） |
| **请求性质** | 请求 CPU 处理（执行中断服务程序） | 请求总线控制权（使 CPU 暂停访存） |
| **并行能力** | CPU 与外设部分并行（中断服务时 CPU 仍在工作） | CPU 与外设高度并行（传送期间 CPU 可执行不访存的指令） |
| **适用场景** | 慢速、随机、少量数据（键盘、鼠标、打印机） | 高速、批量、连续数据（磁盘、以太网、显卡） |
| **硬件开销** | 较低（需中断控制器） | 较高（需 DMA 控制器 + 总线仲裁逻辑） |
| **异常处理能力** | 强（可灵活处理各类事件） | 弱（仅完成数据搬运，异常需后处理中转）

# 三种方式对比总结

| 维度 | 程序查询 | 程序中断 | DMA |
|------|---------|----------|-----|
| 数据传送单位 | 字/字节 | 字/字节 | 块（多字） |
| CPU 参与 | 全程参与 | 每字传送时参与 | 仅预处理和后处理 |
| 并行能力 | 串行 | 部分并行 | 高度并行 |
| 硬件成本 | 最低 | 中等 | 最高 |
| 适用速度 | 极慢速 | 中低速 | 高速 |
| 典型应用 | 早期简单系统 | 键盘、鼠标、打印机 | 磁盘、以太网、显卡 |
| 响应速度 | 取决于查询频率 | 较快（指令结束时） | 最快（周期级） |

> **核心演进逻辑**：CPU 越来越"放手"——从全程控制 → 部分委托 → 完全由专用硬件接管，CPU 专注于运算，I/O 交给专用控制器。这就是 I/O 方式发展的主线。
