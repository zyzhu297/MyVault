# 程序访问的局部性原理

详见[[1.存储器概述#程序访问的局部性原理]]

> [!important] 局部性原理是 Cache 有效的前提
> 程序访问的**时间局部性**和**空间局部性**使得将少量高速 Cache 作为主存的"缓冲"成为可能——大部分访存命中 Cache，整体呈现接近 Cache 的速度和主存的容量。

# Cache 的结构

Cache 由**存储体**（存放 Cache 行）和**控制逻辑**（地址映射、命中判断、替换控制等）两部分组成。其核心是 **Cache 行**（Cache Line / Cache Block），每个 Cache 行包含若干字段，构成一条 **Cache 表项**。

## Cache 表项的组成

每个 Cache 行（即一条表项）通常包含以下字段：

| 字段 | 功能 | 说明 |
|------|------|------|
| **有效位**（Valid Bit，$V$） | 标识该 Cache 行中的数据是否**有效** | 初始或替换后置 0；从主存调入数据后置 1 |
| **标记位**（Tag） | 存放主存块地址的**高位部分** | 与 CPU 访存地址的高位比较，判断是否命中 |
| **数据块**（Data Block） | 存放从主存调入的**数据副本** | 即主存块的内容，大小为一个 Cache 块 |
| **脏位**（Dirty Bit / Modified Bit） | 标识该块**是否被修改过** | 仅**写回策略**需要；脏位=1 表示与主存不一致，替换时必须写回 |
| **替换信息位** | 为替换算法提供依据 | 如 LRU 的计数器位、FIFO 的入队顺序位等 |

> [!important] 有效位 vs 脏位
> - **有效位**：**无论何种写策略都需要**，标记该行数据是否可用
> - **脏位**：仅在**写回策略**（Write Back）下需要，写直达策略下每次写都同时更新主存，无需脏位

### Cache 表项的结构示意图

```
┌──────┬──────────────────────┬──────────┬─────────────────────────────┐
│  V   │         Tag          │    D     │         Data Block          │
│(1位) │     (标记字段)        │  (脏位)   │       (数据块，如 64B)       │
└──────┴──────────────────────┴──────────┴─────────────────────────────┘
                                          ← 可选的 LRU/替换信息位 →
```

### 有效位（Valid Bit）

- 占 **1 位**
- 上电/复位时所有行的有效位清 0（无效）
- 当主存块被调入该 Cache 行后，有效位置 1
- 命中判断时，**必须同时满足 $V=1$ 且 Tag 匹配**，否则视为未命中

> [!note] 有效位的必要性
> 上电时 Cache 中内容是随机的，若不设有效位，可能将"垃圾"数据误判为命中。

### 标记位（Tag）

- 位数取决于**主存地址空间大小**、**Cache 容量**和**映射方式**
- **直接映射**：Tag 位数 = 主存地址位数 − 行号位数 − 块内偏移位数
- **全相联映射**：Tag 位数 = 主存地址位数 − 块内偏移位数（即**主存块号**的全部位数）
- **组相联映射**：Tag 位数 = 主存地址位数 − 组号位数 − 块内偏移位数

Tag 位数的计算是 Cache 容量反推地址格式的**关键**，也是 408 考试中的高频考点。

### 数据块（Data Block）

- 存放从主存调入的一个**完整主存块**
- 块大小（Block Size / Line Size）通常为 $2^n$ 字节（如 16B、32B、64B）
- **块大小的选择折中**：
  - 块过大 → 调入时间长，Cache 行数减少（相同总容量下），命中率可能下降
  - 块过小 → 空间局部性利用不充分，命中率可能下降

> [!note] 例题常考数据
> 当题目给出 Cache 总容量时，通常说的是**包含所有表项字段（V + Tag + Dirty + Data + 替换信息）的总容量**，而不仅仅是 Data Block 的总和。计算时需注意区分「Cache 总容量」与「有效数据容量」。

### 脏位（Dirty Bit）

- 占 **1 位**
- 仅在**写回策略**下使用；写直达策略**不需要**脏位
- 该行中的数据被 CPU **写修改**后置 1
- 该行被替换时，若脏位=1，则必须先将数据块**写回主存**（或下一级 Cache），再调入新块

### 替换信息位（Replacement Info）

- 位数因替换算法而异：

| 替换算法 | 每组（或每行）所需信息位 | 说明 |
|----------|------------------------|------|
| **随机替换** | 0 | 无需额外信息位 |
| **FIFO** | $\log_2(\text{路数})$ 位（或几位的计数器） | 记录入队顺序 |
| **LRU（计数器法）** | 每路 $\log_2(\text{路数})$ 位 | 如 4 路组相联，每路需 2 位计数器，一组共 $4 \times 2 = 8$ 位 |
| **伪 LRU**（Tree-PLRU） | $\text{路数} - 1$ 位 | 如 4 路用 3 位，8 路用 7 位 |

# Cache 的基本工作原理

Cache 的功能**完全由硬件实现**，为了便于 Cache 和主存交换信息，Cache 和主存都被划分为**大小相等的块**，Cache 块也称为 Cache 行；Cache 中仅保存主存中最活跃的若干块的副本，其核心目标是**缓解 CPU 与主存之间的速度矛盾**。

## Cache 的访问过程

1. CPU 将访问**地址**发送给 Cache 和主存
2. Cache 控制逻辑将地址拆分为三部分（映射方式不同，拆分方式不同）：
   - **标记位**（Tag）：用于与 Cache 中的 Tag 比较
   - **索引位**（Index）：用于选择 Cache 中的某一行/组
   - **块内偏移**（Block Offset）：用于在块内定位具体字节
3. **并行比较**：将地址的 Tag 与选中行/组中各路的 Tag 比较，同时检查**有效位**
4. 若 Tag 匹配且有效位 = 1 → **命中**，从 Cache 读取
5. 若 Tag 不匹配或有效位 = 0 → **未命中**，**则需从主存中将该地址所在的主存块整体调入 Cache，并将该块写入一个 Cache 行；此后，CPU 再从 Cache 中获取所需数据**

## Cache 的命中率分析

### 命中率公式

设 $N_c$ 为 Cache 完成存取的总次数，$N_m$ 为主存完成存取的总次数，$h$ 为命中率：

$$
h = \frac{N_c}{N_c + N_m}
$$

**缺失率**（Miss Rate）$= 1 - h$

### 平均访问时间

设 $t_c$ 为命中时的 Cache 访问时间（即 Cache 存取周期），$t_m$ 为未命中时的主存访问时间（含将块调入 Cache 的开销）：

$$
t_a = h \cdot t_c + (1 - h) \cdot t_m
$$

### 访问效率

$$
e = \frac{t_c}{t_a} = \frac{t_c}{h \cdot t_c + (1 - h) \cdot t_m}
$$

### 命中率的影响因素

| 因素 | 对命中率的影响 |
|------|---------------|
| **Cache 容量** | 容量越大，命中率越高（但成本越高、速度可能下降） |
| **块大小** | 过小则空间局部性利用不足；过大则块数减少、调入时间长 |
| **映射方式** | 全相联 > 组相联 > 直接映射（在相同容量下） |
| **替换算法** | LRU 等优化算法可提高命中率 |
| **程序局部性** | 程序本身的时间/空间局部性越好，命中率越高 |

# Cache 和主存的映射方式

Cache 与主存之间的**地址映射**（Address Mapping）是 Cache 设计的核心问题。映射方式决定了主存块可以放入 Cache 中的**哪些位置**。

> [!note] 地址映射的核心问题
> 主存的容量远大于 Cache，主存中的一个块可以映射到 Cache 中的哪个（哪些）位置？这决定了地址如何拆分以及命中如何判断。

## 三种基本映射方式

| 映射方式                         | 基本原理              | 主存块可放入 Cache 的位置 | 特点          |
| ---------------------------- | ----------------- | ---------------- | ----------- |
| **直接映射**（Direct Mapped）      | 主存块号 mod Cache 行数 | **唯一**一行         | 简单、快、冲突多    |
| **全相联映射**（Fully Associative） | 主存块可放入            | **任意**一行         | 灵活、命中率高、成本高 |
| **组相联映射**（Set Associative）   | 主存块号 mod 组数；组内任意  | **特定组**中的任意行     | 折中方案，最常用    |

## 直接映射（Direct Mapped）

### 映射规则

每个主存块只能映射到 Cache 中的一个**固定位置**；若该位置已有内容，则发生块冲突，原块将**被无条件替换**：

$$
\text{Cache 行号} = (\text{主存块号}) \bmod (\text{Cache 总行数})
$$
**主存块号的低 $c$ 位即为对应的 Cache 行号**，然后将该 Cache 行的标记与主存地址的高 $t$ 位进行比较，若标记相等且有效位位 1，则 Cache 命中；否则未命中
### 地址结构

| Tag（标记） | Line（行号/索引） | Offset（块内偏移） |
|------------|------------------|-------------------|

- **行号位**：对应 Cache 的行数，用于定位 Cache 中的唯一一行
- **Tag 位**：主存块号中除行号外的剩余高位部分，用于判断是否命中
- **块内偏移**：在块内定位具体字节/字

### 特点分析

| 优点                | 缺点                                    |
| ----------------- | ------------------------------------- |
| 硬件实现简单，成本低        | **冲突概率最高**——多个映射到同一 Cache 行的主存块无法同时驻留 |
| 命中判断快（只需比较一个 Tag） | 即使在 Cache 其他位置有空闲，也无法利用，**空间利用率最低**   |
| 功耗低               | 命中率相对较低                               |

## 全相联映射（Fully Associative）

### 映射规则

主存中的**任意块**可以放入 Cache 中的**任意行**，没有任何位置限制。

### 地址结构

| Tag（标记） | Offset（块内偏移） |
|------------|-------------------|

- 没有行号/索引字段——需要同时与 Cache 中**所有行的 Tag** 进行比较
- Tag 位数 = 主存块号的全部位数

### 特点分析

| 优点        | 缺点                              |
| --------- | ------------------------------- |
| **冲突概率低** | 需要大量比较器（通常采用按内容寻址的相联存储器），硬件成本极高 |
| 空间利用率最高   | 速度较慢（比较器多，延迟大）                  |
| 命中率最高     | 仅适用于**小容量 Cache**（如 TLB）        |

## 组相联映射（Set Associative）

### 映射规则

将 Cache 分为 $Q$ 个**组**（Set），每组包含 $r$ **路**（Way）。主存块先映射到**固定的组**，在组内可以放在**任意一路**。

$$
\text{Cache 组号} = (\text{主存块号}) \bmod \text{Cache 组数}(Q)
$$
即**组间采用直接映射，组内采用全相联映射**；当 $Q=1$ 时退化为全相联映射，当 $Q$ = Cache 总行数 时，退化为直接映射；**设每组包含 $r$ 个 Cache 行，则称为 $r$ 路组相联映射**
### 地址结构

| Tag（标记） | Set（组号/索引） | Offset（块内偏移） |
|------------|-----------------|-------------------|

### 组相联中的命中判断流程

1. 根据地址中的**组号**，选中 Cache 中的某一组
2. 将该组内**所有 $W$ 路**的 Tag 与地址中的 Tag 同时进行比较
3. 若某一路 Tag 匹配且有效位 = 1 → 命中
4. 若 $W$ 路都不匹配 → 未命中

## 三种映射方式的地址拆分对比

| 地址字段 | 直接映射 | 全相联映射 | 组相联映射 |
|----------|----------|------------|------------|
| **Tag** | 主存块号的高位（= 总块号位数 − 行号位数） | 主存块号的**全部**位数 | 主存块号的高位（= 总块号位数 − 组号位数） |
| **Index** | 有（行号），位数 = $\log_2(\text{Cache 行数})$ | **无**（全比较） | 有（组号），位数 = $\log_2(\text{组数})$ |
| **Offset** | 有，位数 = $\log_2(\text{块大小/字节})$ | 有 | 有 |

## 映射方式选择的影响

| 评价维度 | 直接映射 | 组相联 | 全相联 |
|----------|----------|--------|--------|
| **命中率** | 低 | 中 | 高 |
| **判断速度** | 快 | 中 | 慢 |
| **硬件成本** | 低 | 中 | 高 |
| **冲突缺失** | 严重 | 少 | 无 |

# Cache 中主存块的替换算法

当 Cache 未命中且目标位置（直接映射的唯一行 / 组相联的组内所有路）已满时，需要**替换**（Evict）一个现有的块，为新块腾出空间。替换算法的选择直接影响 **Cache 的命中率**。

## 常用替换算法

| 算法 | 英文 | 替换策略 | 硬件实现难度 |
|------|------|----------|-------------|
| **随机替换** | Random（RAND） | 随机选择被替换的块 | 最简单 |
| **先进先出** | First In First Out（FIFO） | 替换最早进入 Cache 的块 | 较简单 |
| **近期最少使用** | Least Recently Used（LRU） | 替换最久未被访问的块 | 较复杂 |
| **最不经常使用** | Least Frequently Used（LFU） | 替换被访问次数最少的块 | 复杂 |

## 随机替换（RAND）

- 使用一个**伪随机数发生器**选择被淘汰的块
- 实现最简单，不需记录任何历史信息
- **缺点**：未利用局部性原理，命中率不稳定

## 先进先出（FIFO）

- 记录每个块进入 Cache 的**先后顺序**（用计数器或循环队列）
- 替换**最早进入**的块，不论它是否被频繁访问
- **缺点**：未利用局部性原理，最早进入的块可能是最常被访问的"热"块（如循环核心代码）

## 近期最少使用（LRU）

基于**时间局部性**：最近被访问过的块，在未来很可能再次被访问；反之，近期一直未被访问的块，未来也不太可能被访问。所以替换**近期最久未被使用**的那个块。

> [!important] 计数器法规则
> - **命中时**：命中块的计数器**清零**，其他比它小的计数器 **+1**，比它大的**不变**
> - **未命中且有空位时**：新块计数器置 0，其他所有计数器 **+1**
> - **未命中且已满时**：计数器值**最大**的块被替换，新块计数器置 0，其他计数器 **+1**

## 最不经常使用（LFU）

- 为每个块维护一个**访问次数计数器**
- 替换**访问次数最少**的块
- **缺点**：无法反映访问的"时效性"——一个很久以前被频繁访问、但此后不再使用的块，计数器仍很大，不会被替换

## 替换算法对比

| 算法 | 命中率 | 硬件开销 | 适用场景 |
|------|--------|----------|----------|
| Random | 低 | 极小 | 对功耗/面积极度敏感的嵌入式场景 |
| FIFO | 较低 | 小 | 简单系统 |
| LRU | **高** | 较大 | **通用处理器 Cache（最常用）** |
| LFU | 中 | 大 | 特定访问模式下可能优于 LRU |

# Cache 的一致性问题

Cache 一致性（Cache Coherence / Consistency）问题源于：当 CPU 执行**写操作**时，Cache 中的数据被修改，但**主存中对应的数据尚未更新**。如何保证 Cache 与主存之间的数据一致性，是 Cache 设计的又一关键问题。

## 写命中（Write Hit）时的处理策略

当写操作命中 Cache 时（目标数据在 Cache 中），有两种处理方式：

| 策略                | 全称        | 写入方式                 | 特点                  |
| ----------------- | --------- | -------------------- | ------------------- |
| **Write Through** | 全写法 / 直写法 | 同时写入 Cache 和**主存**   | 一致性好，但写操作慢（每次都要等主存） |
| **Write Back**    | 回写法       | 只写入 Cache，被替换出时才写回主存 | 写操作快，但一致性维护复杂       |

### 全写法（Write Through）

- **做法**：CPU 的每次写操作**同时更新** Cache 和主存
- **优点**：Cache 与主存始终保持一致，**可靠性高**，实现简单
- **缺点**：每次写操作都要访问主存，**写速度受限于主存速度**，总线流量大

> [!note] 写直达 + 写缓冲
> 为缓解写直达的性能问题，常在 Cache 与主存之间加入一个**写缓冲**（Write Buffer）：
> - CPU 将数据写入 Cache 和写缓冲后即认为写完成
> - 写缓冲在后台慢慢将数据写入主存
> - **问题**：若下次读取同一地址，可能读到主存中的旧数据——需要额外检查写缓冲

### 回写法（Write Back）

- **做法**：CPU 写操作**只更新 Cache**，并在该 Cache 行设置**脏位**（Dirty Bit / Modified Bit）
- 当该 Cache 块需要被替换出时，若脏位 = 1，则将其**写回主存**
- **优点**：写操作速度快（只写 Cache），总线流量小
- **缺点**：存在一段时间内 Cache 与主存**不一致**；控制逻辑复杂

| 对比        | 全写法             | 回写法          |
| --------- | --------------- | ------------ |
| **写操作延迟** | 长（等主存）          | 短（只写 Cache）  |
| **总线占用**  | 每次写都占总线         | 仅替换时写回       |
| **一致性**   | 始终一致            | 暂时不一致（需脏位管理） |
| **实现复杂度** | 低               | 较高           |
| **典型应用**  | L1→L2 到下级 Cache | L1 内部、多处理器系统 |

## 写不命中（Write Miss）时的处理策略

当写操作**不命中** Cache（目标数据不在 Cache 中），有两种处理方式：

| 策略                          | 说明                                    |
| --------------------------- | ------------------------------------- |
| **写分配**（Write Allocate）     | 先将数据写入主存的对应单元，然后将该主存块调入 Cache 的一个空闲行中 |
| **非写分配**（No-Write Allocate） | **绕过** Cache，直接写入主存，不调入 Cache         |

### 常见组合

| 组合 | 写命中策略 | 写不命中策略 | 使用场景 |
|------|----------|-------------|----------|
| **组合一** | 写回（Write Back） | 写分配（Write Allocate） | **最常用**——利用局部性，后续对该块的读/写都可能命中 |
| **组合二** | 写直达（Write Through） | 非写分配（No-Write Allocate） | 实现简单，一致性要求高的场景 |

> [!important] 策略配对逻辑
> - **写回 + 写分配**：既然选择了写回（追求写性能），就应利用即将被写的块很可能再被访问的局部性
> - **写直达 + 非写分配**：既然每次写都直达主存（不需要 Cache 暂存），就没必要为此块占用 Cache 空间

# Cache 的应用

## 多级 Cache 层次结构

现代处理器普遍采用**多级 Cache** 结构，以在容量、速度和功耗之间取得最佳平衡：

| Cache 级别 | 典型容量 | 典型延迟 | 构成 | 特点 |
|-----------|---------|---------|------|------|
| **L1 Cache** | 32KB ~ 64KB（每核） | 2~4 个时钟周期 | 高速 SRAM | 速度最快，容量最小，**指令/数据分立** |
| **L2 Cache** | 256KB ~ 512KB（每核） | 10~20 个时钟周期 | 高速 SRAM | 速度和容量的折中 |
| **L3 Cache** | 4MB ~ 32MB+（共享） | 30~50 个时钟周期 | SRAM | 多核共享，容量大 |
| **L4 Cache**（少数） | 可达 128MB+ | 更慢 | eDRAM 等 | 仅部分处理器采用（如 Intel 部分型号的 eDRAM） |

> [!note] 多级 Cache 的"金字塔"结构
> L1 速度最快但容量最小，L2 次之，L3 容量最大但速度最慢。每一级 Cache 都是下一级的"Cache"，上一层内容是下一层内容的子集。


> [!important] 为什么各级都倾向写回？
> - 越靠近 CPU 的 Cache，速度要求越高，**写回**可以避免每次写都穿透到下一级
> - 写回大幅**减少下级带宽压力**——只有被替换的脏块才需要写下去
> - 现代处理器中，L1/L2/L3 之间几乎一律使用**写回 + 写分配**

### 三种典型配置

| 配置 | L1→L2 | L2→L3 | L3→主存 | 特点 | 典型应用 |
|------|--------|--------|---------|------|----------|
| **全写回** | 写回 | 写回 | 写回 | 写性能最优，总线流量最小 | **现代通用处理器主流方案** |
| **L1 写直达 + 下级写回** | 写直达 | 写回 | 写回 | L1 无需脏位，替换开销小；L2 吸收写流量 | 部分 ARM 处理器、低功耗设计 |
| **全写直达** | 写直达 | 写直达 | 写直达 | 一致性最强，但性能最差 | 对可靠性要求极高的特殊系统 |

## 分离 Cache

现代处理器中，**L1 Cache 通常分为两个独立的部分**：

| Cache 类型 | 存储内容 | 特点 |
|-----------|---------|------|
| **L1 指令 Cache**（I-Cache） | 仅存储**指令** | 只读，无需处理写一致性问题，设计简单 |
| **L1 数据 Cache**（D-Cache） | 仅存储**数据** | 可读写，需要处理写策略和一致性 |

### 分立的好处

1. **并行访问**：取指令和读/写数据可以**同时进行**（流水线处理器必须）
2. **避免相互干扰**：指令和数据不会互相挤占 Cache 空间
3. **支持不同访问模式优化**：指令 Cache 几乎只读（顺序访问），数据 Cache 读写混合

> [!important] 哈佛架构 vs 冯·诺依曼架构
> L1 的分立 Cache 在 Cache 层面体现了**哈佛架构**的思想（指令和数据分存），而在主存层面通常仍保持**冯·诺依曼架构**（指令和数据混合存储）。这种混合设计被称为**改进的哈佛架构**。


## Cache 与虚拟存储器的关系

Cache 和虚拟存储器共同构成现代存储系统的核心，Cache 解决**速度问题**，虚拟存储器解决**容量问题**。

| 对比维度 | Cache | 虚拟存储器 |
|----------|-------|-----------|
| **目标** | 弥补 CPU 与主存的**速度差距** | 弥补主存与辅存的**容量差距** |
| **实现方式** | 由**硬件**实现 | 由**硬件 + OS** 共同实现 |
| **对程序员** | **透明** | 对应用程序员透明 |
| **基本单位** | **块**（Block / Line） | **页**（Page）或**段**（Segment） |
| **替换策略** | LRU 等硬件实现 | 页置换算法（LRU、Clock 等）|
| **映射方式** | 直接/组相联/全相联 | 页表映射（全相联）|
