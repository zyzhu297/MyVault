

> [!info] 前情回顾
> 传统的存储管理方式要求一次性将作业**全部装入内存**才能运行，这导致两个问题：
> 1. **一次性**：作业大小受物理内存限制，大作业无法运行
> 2. **驻留性**：大量作业驻留内存，降低了内存利用率和系统并发度

# 虚拟内存的基本概念

## 局部性原理

> [!note] 局部性原理是虚拟内存技术的理论基础
> - **时间局部性**：某条指令或数据被访问后，**不久后**可能再次被访问（典型：循环、子程序）
> - **空间局部性**：某存储单元被访问后，其**附近**的存储单元不久后也可能被访问（典型：顺序执行、数组遍历）

## 虚拟内存的定义

基于局部性原理，在程序**装入**时仅将即将运行的部分装入内存（其余留在外存），在程序**运行**过程中根据需要动态地调入/调出页面。从用户角度看，系统提供了一个比实际物理内存大得多的存储器。

>[!note] 虚拟的解释
>所谓*虚拟*存储器，是因为该存储器**并非真实存在的物理实体**，而是操作系统通过**部分装入**、**请求调入**和**置换功能**（均对用户**透明**）所构造的逻辑抽象
## 虚拟内存的三个特征

| 特征 | 含义 |
|------|------|
| **多次性** | 作业无需一次性全部装入内存，可分为多次调入 |
| **对换性** | 作业运行时允许在内存与外存之间换入换出 |
| **虚拟性** | 从逻辑上扩充了内存容量，使用户感觉到的内存远大于实际物理内存 |

### 虚拟内存的实现方式

虚拟内存必须建立在**离散**分配的内存管理基础之上，有请求分页存储管理、请求分段存储管理和请求段页式存储管理三种实现方式；
无论采用哪种方式，均需一定的**硬件支持**：
- 足够容量的**内存**和**外存**，用于存放程序的当前部分和后备部分
- **页表机制**（或**段表机制**），作为地址映射的关键数据结构
- **中断机构**，用于在用户程序访问尚未调入内存的部分时，触发**缺页（或缺段）中断**
- **地址变换机构**，负责在程序运行过程中**动态的**将逻辑地址转换为物理地址

# 请求分页存储管理

请求分页 = 基本分页 + **请求调页** + **页面置换**

> [!tip] 与基本分页的区别
> - 基本分页：所有页必须**一次性全部调入**内存
> - 请求分页：只需要调入**当前需要的部分页面**，根据运行情况动态调入/换出

## 页表机制的扩充

请求分页的页表项增加了四个字段：
![[请求分页系统的页表项|900]]

| 字段             | 含义                            |
| -------------- | ----------------------------- |
| **状态位 P（存在位）** | 该页是否已调入内存（1=在内存，0=在外存）        |
| **访问位 A**      | 该页最近的访问次数或上次访问的时间间隔（供置换算法参考）  |
| **修改位 M（脏位）**  | 该页调入内存后是否被修改过（若未修改，换出时无需写回外存） |
| **外存地址**       | 该页在外存上的存放地址（通常为交换区）           |

## 缺页中断机构

当要访问的页面不在内存中（状态位 = 0），**硬件**便产生一次**缺页中断**，请求 OS 将所缺页面调入内存。

> [!note] 缺页中断与普通中断的区别
> - 缺页中断在**指令执行期间**产生并处理，属于**内部异常（内中断）**
> - 普通 I/O 中断在**指令执行完毕后**（指令周期末尾）才响应，属于**外部中断**
> - 一条指令可能产生**多次**缺页中断（如指令跨页、操作数跨页）

### 地址变换过程（请求分页）

> [!tip] 请求分页的完整地址变换流程
> ```
> 逻辑地址 A
>   → 分离页号 P 和页内偏移 W
>   → 越界检查（P ≥ 页表长度？）
>   → 查快表 TLB
>       ├─ 命中 → 拼接物理地址，访问内存
>       └─ 未命中 → 查内存页表
>            ├─ 状态位P=1（在内存）→ 得到块号，更新 TLB，拼接物理地址
>            └─ 状态位P=0（不在内存）→ 缺页中断！
>                 → 检查是否有空闲页框
>                     ├─ 有 → 从外存调入页面
>                     └─ 无 → 执行页面置换算法，换出一页后再调入
>                 → 更新页表和 TLB
>                 → 重新执行指令
> ```

>[!warning] 越界检查
>越界检查在查表之前
# 页框分配

## 最小页框数

为保证进程能**正常运行**所需的最少物理页框数量

- 下限取决于**指令格式**、**寻址方式**等硬件因素
- 若分配的页框数小于该下限，进程将无法执行（如一条指令横跨多页、操作数横跨多页时，所需页都必须同时在内存中）
- 极端情况：一条指令可能引发**多次缺页中断**
>[!warning] 最坏情况
>若分配给进程的页框数少于其指令所需的最少页面数，该进程将**永远无法执行完成**——刚调出的一页马上又被访问，调入后又触发新的缺页，陷入**死循环**。例如一条跨页指令本身占 2 页、两个操作数又各跨页，最坏情况下一条指令就需 6 个页框同时在内存。
## 驻留集大小

OS 分配给一个进程的**物理页框数量**。驻留集大小直接影响系统的缺页率。
- **驻留集过小**：内存可容纳的进程数量增多，有助于提高并发度；单缺页频繁，系统吞吐量下降，进程大部分时间花在等待调页 I/O 上，严重时引发**抖动（Thrashing）**
- **驻留集过大**：浪费内存资源，系统中能同时驻留的进程数减少，**系统并发度降低**；且单个进程的额外页框对降低缺页率的边际收益递减

## 内存分配策略

| 策略           | 含义                            | 特点                |
| ------------ | ----------------------------- | ----------------- |
| **固定分配局部置换** | 每个进程分配固定数量页框，缺页时**只能置换自己的页面** | 页框数不变，过于死板        |
| **可变分配全局置换** | 允许从所有进程的页框中置换，且可动态调整各进程的页框数   | 灵活，活跃进程的驻留集可能持续膨胀 |
| **可变分配局部置换** | 缺页时只置换自己的页面，系统定期评估并调整各进程页框数   | **最常用**，兼顾灵活与可控   |
>[!important] 不存在 "固定分配 + 全局置换"
>如果分配是固定的，那就没有"全局"一说——固定意味着每个进程的页框数定死了，不可能从别人那里抢。所以固定分配天然只能是局部置换。

### 置换范围 × 分配灵活性

|维度|选项|含义|
|---|---|---|
|**置换范围**|**局部置换**|缺页时只能从**自己**的页框中选一页换出，不影响其他进程|
||**全局置换**|缺页时可以从**所有进程**的页框中选一页换出（包括抢别人的）|
|**分配灵活性**|**固定分配**|进程运行期间页框数量**保持不变**，一开始分多少就多少|
||**可变分配**|系统可**动态调整**各进程的页框数，缺页多的多给、浪费的回收|

## 页框分配算法

采用**固定分配策略**时，系统需将空闲页框分配给各进程，常见算法如下：

| 算法 | 说明 | 特点 |
|------|------|------|
| **平均分配法** | 将系统中所有空闲物理块**平均**分配给各个进程 | 简单公平，但未考虑进程大小和实际需求的差异；小进程浪费页框，大进程不够用 |
| **按比例分配法** | 根据各进程的**虚拟地址空间大小**（页面总数）按比例分配物理块 | 比平均分配更合理，考虑了进程规模差异；但未考虑优先级和实际访存行为 |
| **优先权分配法** | 根据进程的**优先级**分配物理块，高优先级进程获得更多页框 | 有利于重要/紧急进程快速完成；但可能导致低优先级进程"饥饿"，需要配合其他策略使用 |

> [!tip] 分配粒度
> 实际系统中常将**按比例分配**与**优先权分配**结合：先按比例计算出基准值，再根据优先级进行微调。Linux 等现代 OS 更多采用**动态调整**策略而非静态分配。


## 调入页面的时机

| 策略       | 说明                                                       | 特点                                |
| -------- | -------------------------------------------------------- | --------------------------------- |
| **预调页**  | 根据局部性原理，一次调入若干相邻页，在进程**首次调入之前**，由程序员显式指定，预先将预计会使用的页面调入内存 | 若预测准确可减少缺页中断；但预测不准会浪费 I/O 带宽和内存空间 |
| **请求调页** | 只在**缺页中断发生时才**将所缺的那**一个**页面调入内存                          | 按需调入，不浪费；但启动阶段缺页频繁，I/O 次数多        |

> [!tip] 实际策略
> 现代 OS 通常采用**请求调页为主、预调页为辅**的策略。例如：进程启动时预调工作集相关的页面，运行中按需请求调页

## 从何处调入页面


| 来源           | 适用场景                                                             |
| ------------ | ---------------------------------------------------------------- |
| **交换区（对换区）** | 进程运行前将**全部**映像从文件区复制到交换区，以后从交换区调入（速度快）                           |
| **文件区**      | 对于**不会被修改**的文件（如程序代码），直接从文件区调入，换出时无需写回；对于可能修改的页面，换出时**必须保存到对换区** |
| **UNIX 方式**  | 未运行过的页从文件区调入；运行过且被换出的页从交换区调入                                     |

> [!note] 交换区 vs 文件区
> **交换区（对换区 / Swap Area）**
> - 磁盘上的一块**连续**区域（独立分区或 swap 文件），专门用于存放从内存换出的进程映像（数据段、堆、栈）
> - 以**连续分配**方式管理，读写速度快（无文件系统开销，直接按盘块号寻址）
> - 数据是**临时**的——进程结束时即释放，因此无需持久化
> 
> **文件区（File Area / File System）**
> - 磁盘上用于存放**持久文件**（可执行程序、共享库、数据文件等）的区域
> - 以**离散分配**方式管理，按文件名/路径访问，存在文件系统元数据开销
> - 数据是**持久**的——进程结束后仍然保留在磁盘上
> 
> **为什么需要交换区？**
> 文件区采用离散分配，访问一个页面可能需要多次磁盘寻道；交换区采用连续分配，顺序读写快得多。因此将可能被换出换入的页面（如数据段、堆栈）放在交换区，性能远优于直接从文件区调入。
> 

## 如何调入页面

当缺页中断发生、系统决定调入一个页面时，需要执行以下步骤：

1. **定位外存**：根据页表中的**外存地址**字段找到该页面在外存（交换区或文件区）的位置
2. **分配页框**：若内存有**空闲页框**则直接使用；否则通过**页面置换算法**选出一页换出
3. **启动 I/O**：将所需页面从外存**读入**分配到的页框中
4. **更新页表**：将该页表项的**状态位 P 置 1**，填入物理块号
5. **重新执行**：重新执行触发缺页中断的那条指令

整个页面调入过程对用户**完全透明**，由操作系统完成

> [!warning] 写回检查
> 若步骤 2 中被换出的页面其**修改位 M = 1**（即被修改过），则必须先将其**写回外存**再读入新页；若 M = 0，可直接覆盖，省去一次磁盘写操作。

# 页面置换算法

> [!info] 页面置换
> 当需要调入新页面而内存已满时，必须选择一个页面换出到外存。选择**哪个**页面换出就是页面置换算法要解决的问题，一个好的页面置换算法应致力于**降低缺页率**

## 最佳置换算法（OPT）

> [!note] Optimal Page Replacement
> 选择**以后永不使用**或**在最长时间内不再被访问**的页面淘汰。

- 需要预知未来的页面访问序列，实际系统中**无法实现**
- 理论上的最优算法，用于**衡量其他算法的性能**

## 先进先出置换算法（FIFO）

> [!note] First In First Out
> 淘汰**最先进入内存**的页面（即驻留时间最长的页面）。

- 实现简单，利用队列记录页面调入顺序
- 与进程实际运行规律可能不一致（先进入的不一定以后不用）

### Belady 异常

> [!warning] Belady 异常（Belady's Anomaly）
> 对 FIFO 算法，**增加物理块数反而使缺页次数增多**的现象。

- 只有 **FIFO 算法**可能出现 Belady 异常
- LRU 和 OPT 算法永远不会出现 Belady 异常
- 原因：FIFO 不考虑页面的使用频率（**未利用局部性原理**），增加块数后置换顺序可能"更差"

## 最近最久未使用置换算法（LRU）

> [!note] Least Recently Used
> 淘汰**最近最长时间未被访问**的页面（利用局部性原理中的时间局部性）。

- 性能接近 OPT，但需要**硬件支持**，开销较大
- 是**栈类算法**，不会出现 Belady 异常

### LRU 的实现方式

| 方式 | 操作 | 特点 |
|------|------|------|
| **寄存器移位法** | 每个页面关联一个移位寄存器，定时右移一位；被访问时最高位置 1 | 寄存器值最小的页面即为 LRU 页 |
| **栈** | 维护一个特殊栈，被访问页面移到栈顶 | 栈底始终是 LRU 页 |
| **计数器法** | 每个页关联一个计数器，被访问时计数器加 1 并记录时间 | 选择计数值最小或时间最早的页 |

> [!failure] LRU 的缺点
> 实现复杂，需要较多硬件支持，实际系统中常用 LRU 的近似算法

## 时钟置换算法（CLOCK / 最近未用NRU）

> [!note] CLOCK Algorithm（Not Recently Used）
> 给每个页框关联一个**访问位 A**，将所有页框组织成**循环队列**，用指针扫描。

### 简单 CLOCK 算法

> [!note] 数据结构
> - **访问位 A**：每个页框关联 1 bit，由硬件自动维护
> - **循环队列**：所有页框逻辑上首尾相连
> - **扫描指针**：指向下一个待考察的页框，初始指向**第一个装入内存的页面**


- 页面**首次装入内存**时，其访问位 **A = 1**（视为刚被访问）
- 装入完成后，指针指向该页的**下一个**页框
- 空闲页框的 A 视为 0（或被跳过不参与扫描）

| 场景 | 指针是否移动 | A 变化 |
|------|------------|--------|
| **访问命中** | ❌ 不动 | 该页 A ← 1（硬件置位） |
| **扫描到 A = 1** | ✅ 前移一格 | 该页 A ← 0（OS 清零） |
| **扫描到 A = 0（淘汰）** | ✅ 前移一格 | 新页 A ← 1 |

> [!tip] 关键理解
> - 指针**只在缺页时扫描**，命中时完全不参与
> - 置换完成后指针停在**被淘汰页框的下一个位置**，这意味着刚调入的新页（A=1）在下一轮扫描中不会被立即考察——相当于一个"保护期"
> - 被清零的 A 位给了页面**一个完整扫描周期**作为"最后机会"：若在此期间被再次访问，A 会被硬件重新置 1；若整个周期都未被访问，下一轮 A = 0 被淘汰


## 改进型 CLOCK 算法

> [!note] Enhanced CLOCK
> 由于**修改过的页面置换代价更高**，为降低 I/O 开销，同时考虑 **访问位 A** 和 **修改位 M**，将页面分为四类。

| 类 | (A, M) | 含义 | 淘汰优先级 |
|----|--------|------|-----------|
| 1 | (0, 0) | 最近未访问、未修改 | **最高**（最佳淘汰对象） |
| 2 | (0, 1) | 最近未访问、但已修改 | 次高（淘汰前需写回外存） |
| 3 | (1, 0) | 最近访问过、未修改 | 较低（可能再次被访问） |
| 4 | (1, 1) | 最近访问过、已修改 | **最低**（最不适合淘汰） |

> [!tip] 改进型 CLOCK 扫描过程（四轮）
> ```
> 第一轮：找 (0,0)，不修改任何位
> 第二轮：找 (0,1)，将所有经过的访问位清零
> 第三轮：重新找 (0,0)，不修改任何位
> 第四轮：重新找 (0,1)，此时 A 必定为 0
> ```

## 页面置换算法总结对比

> [!summary] 算法对比
>
> | 算法 | 思想 | 实现难度 | 缺页率 | Belady异常 |
> |------|------|---------|--------|-----------|
> | OPT | 淘汰最远将来不用的页 | — | 最低（理论） | 无 |
> | FIFO | 淘汰最先进入的页 | 简单 | 较高 | **有** |
> | LRU | 淘汰最久未使用的页 | 困难 | 低 | 无 |
> | CLOCK | 淘汰访问位 = 0 的页 | 简单 | 中等 | 无 |
> | 改进 CLOCK | 优先淘汰 (0,0) 页 | 中等 | 更低 | 无 |


# 抖动（Thrashing）

## 抖动的概念

> [!warning] 抖动（颠簸）
> 进程频繁地进行页面换入换出，以致**大部分时间都花在页面置换上**而非真正执行任务，导致 CPU 利用率急剧下降的现象。

> [!tip] 抖动产生的根本原因
> - 分配给进程的物理页框数**小于其工作集**大小
> - 驻留集太小 → 频繁缺页 → 页面置换频繁 → CPU 大部分时间等待 I/O → 系统"颠簸"

## 工作集

> [!note] 工作集（Working Set）
> 进程在**最近 Δ 次访问**中所引用的页面集合，记作 $W(t,\Delta)$

- 工作集大小反映了进程对内存的需求：工作集越大，需要的物理页框越多
- 监控每个进程的工作集，**确保驻留集 ≥ 工作集大小** → 可避免抖动

## 抖动解决方法

| 方法             | 核心思想                 | 具体操作                                                                                                       |
| -------------- | -------------------- | ---------------------------------------------------------------------------------------------------------- |
| **工作集策略**      | 动态调整驻留集，使其始终覆盖工作集    | 跟踪每个进程最近 $\Delta$ 次访问的页面集合，若进程缺页率过高则**扩大驻留集**，若长期不缺页则**回收多余页框**                                            |
| **缺页率控制（PFF）** | 将缺页率控制在预设的上下界之间      | 设定缺页率**上界**（如 10 次/ms）和**下界**（如 1 次/ms）：缺页率超过上界 → 缺页过于频繁 → **增加页框**；缺页率低于下界 → 页框富余 → **回收页框**；缺页率在区间内则不做调整 |
| **挂起部分进程**     | 减少多道程序度，腾出物理页框       | 当内存严重紧缺、多个进程同时抖动时，选择一个或几个进程**挂起**，将其全部页面换出到外存，释放的页框分给其他活跃进程；待内存压力缓解后再**激活**被挂起的进程                          |
| **L=S 准则**     | 控制多道程序度，使 CPU 利用率最大化 | 设 L 为多道程序度、S 为缺页处理平均时间，当 **L 增大到使 L × S 接近于进程平均执行时间**时，CPU 利用率达到峰值；继续增加 L → 每个进程页框数过少 → 抖动 → CPU 利用率反而下降   |

# 页框回收

> [!note] 页框回收（Page Frame Reclamation）
> OS 将分配给进程的物理页框收回，使其重新成为**空闲页框**的过程。分配与回收是页框管理的两个基本操作，共同维持系统的内存供需平衡

## 页面缓冲算法（Page Buffering Algorithm）

**页面换入/换出的开销**对系统性能影响显著，主要取决于以下三个因素：
- **页面置换算法的选择**：决定了**缺页率**的高低。若算法选择不当（如 FIFO），频繁选中"即将被访问"的页面换出，导致刚换出又立刻缺页，产生大量不必要的读 I/O
- **已修改页面写回磁盘的频率**：决定了**写 I/O 开销**。若每个脏页都在被淘汰时立即同步写回，进程必须等待写完成才能读入新页；而若能将多个脏页**批量、异步**写回，则可将写操作从关键路径中剥离
- **磁盘内容读入内存的频率**：即**缺页中断发生的频次**。每次缺页都是一次磁盘读 I/O，频繁缺页会使进程大部分时间阻塞在 I/O 等待上，CPU 利用率急剧下降

> [!note] 页面缓冲算法
> 传统页面置换的瓶颈在于**同步 I/O**：选出被淘汰页 → 若脏则写回磁盘 → 再从磁盘读入新页。整个过程是**串行**的，进程必须等待两次磁盘 I/O 完成。
>
> 页面缓冲算法通过引入**两个专用链表**对**置换算法**进行**优化**，核心思想是：
> 1. **维护一个空闲帧池**，使新页总能立刻获得一个空闲页框（无需等待换出）
> 2. **脏页批量、异步写回**，将写回操作从"关键路径"中剥离

### 两个专用链表

| 链表         | 存放内容                  | 来源                                                        | 用途                                                |
| ---------- | --------------------- | --------------------------------------------------------- | ------------------------------------------------- |
| **空闲页面链表** | 干净页框（可**直接覆盖使用**的物理块） | ① 系统启动时的空闲帧 ② 被淘汰的**干净页**（M=0；未修改） ③ 已写回完成的脏页（从已修改链表迁移而来） | 缺页时直接从该链表**取一个页框**装入新页，无需等待                       |
| **修改页面链表** | 脏页框（内容已被修改、需写回外存的物理块） | 被淘汰的**脏页**（M=1；已修改）                                       | 当链表长度达到**阈值**或系统**空闲**时，**批量**将这些页面写回磁盘，写回后移入空闲链表 |

## 回收时机

| 时机        | 触发条件                         | 说明                                      |
| --------- | ---------------------------- | --------------------------------------- |
| **进程终止**  | 进程正常退出或被 kill                | 回收该进程占用的**所有**页框，并释放交换区中对应的备份空间         |
| **内存紧缺**  | 空闲页框数低于阈值（如 `pages_low`）     | OS 主动回收：通过页面置换换出、缩减进程驻留集（工作集修剪）等方式腾出页框  |
| **周期性回收** | 守护进程定时唤醒（如 Linux `kswapd`）   | 周期性扫描内存使用情况，提前回收长时间未访问的页面，避免"平时不管、急时恐慌" |
| **显式释放**  | 进程调用 `munmap()` / `free()` 等 | 进程主动告知 OS 某段虚拟地址空间不再需要，由 OS 回收对应的物理页框   |

## 回收时的处理

回收一个页框时，需要根据页面的**类型**和**修改位 M**区别对待：

| 页面类型                    | M = 0（干净）         | M = 1（脏）      |
| ----------------------- | ----------------- | ------------- |
| **文件映射页**（如代码段、mmap 文件） | 直接回收，该页可随时从文件重新读入 | 先**写回文件**再回收  |
| **匿名页**（如堆、栈，无文件后备）     | 直接回收，但该页内容就此丢失    | 先**写回交换区**再回收 |

> [!tip] 回收优先级
> OS 通常按以下顺序回收页框，以最小化 I/O 开销：
> 1. **干净的（M=0）文件映射页** —— 无需任何写回，最快
> 2. **干净的匿名页** —— 无需写回，但内容丢失
> 3. **脏的（M=1）文件映射页** —— 需写回文件
> 4. **脏的匿名页** —— 需写回交换区，最慢

## 空闲页框的管理

回收后的页框进入空闲池，由 OS 统一管理：

- **空闲页框链表**：将所有空闲页框串成链表，分配时取链头，回收时插入链尾，简单但仅适合单页分配
- **伙伴系统（Buddy System）**：将空闲页框按 $2^k$ 大小分组管理，支持快速合并与拆分，用于满足**连续多页**的分配需求
- 现代 OS 通常结合两种方式：单页分配走链表，多页分配走伙伴系统

## 与页面置换的关系

> [!tip] 页面置换 vs 页框回收
> | 对比 | 页面置换 | 页框回收 |
> |------|----------|----------|
> | 触发条件 | 缺页中断，需要调入新页 | 回收时机中的任一种 |
> | 目的 | 选一页换出，**为新页腾出空间** | 回收页框，**扩充空闲池** |
> | 是否调入新页 | 是（换出旧页 + 调入新页） | 不一定（可能仅换出而不调入） |
> | 粒度 | 每次一个页框 | 可以批量回收多个页框 |
>
> **页面置换是页框回收的一种特殊形式**——在缺页时为了给新页"腾位置"而触发的针对性回收。

### 工作流程

> [!tip] 页面缓冲算法的完整流程
> ```
> 缺页中断发生，需要调入新页
>   → 若空闲页面链表非空：
>       直接从空闲链表取一个页框，将新页读入该页框
>   → 若空闲页面链表为空：
>       从已修改链表取出若干页，批量写回磁盘后移入空闲链表
>       再从空闲链表取页框装入新页
>   
> 淘汰旧页（被置换算法选中的页）：
>   ├─ M = 0（干净）→ 直接挂入空闲页面链表
>   └─ M = 1（脏）  → 挂入已修改页面链表，等待批量写回
> ```
>
> 被淘汰的页面**并不立即写回**，而是先挂入对应链表。这带来了一个额外好处：**若该页在被覆盖之前又被访问，可直接从链表中"捡回"而不需要从磁盘重新读入。**

### 与传统置换的对比

| 对比维度            | 传统置换流程          | 页面缓冲算法        |
| --------------- | --------------- | ------------- |
| **脏页写回时机**      | 立即同步写回          | 批量异步写回        |
| **新页装入等待**      | 需等脏页写回完成        | 从空闲链表直接取，无需等待 |
| **I/O 模式**      | 串行（写旧页 → 读新页）   | 并行（写入和读入分离）   |
| **被淘汰页的"后悔"成本** | 已写回或已覆盖，必须从磁盘重读 | 仍在链表中，可直接恢复   |
| **磁盘 I/O 次数**   | 每个脏页单独写回        | 批量写回，减少寻道次数   |

### 优点总结

1. **降低缺页处理延迟**：新页可以立刻从空闲链表拿到页框，无需等待脏页写回
2. **减少磁盘 I/O**：脏页批量写回，减少磁头寻道次数；干净页直接回收，完全不产生写 I/O
3. **支持"后悔"机制**：被淘汰但尚未覆盖的页面，若被再次访问可直接从链表中恢复，避免一次磁盘读
4. **适合多进程环境**：各进程的淘汰页统一管理，空闲帧池由系统全局共享

# 内存映射文件

> [!note] 内存映射文件（Memory-Mapped Files）
> 是操作系统向应用程序提供的一种**系统调用机制**，将文件的一部分或全部映射到进程的虚拟地址空间，使得**访问文件如同访问内存**一样。

>[!success] 内存映射文件的好处
>- 使程序员的编程更为简单，已建立映射的文件，可以直接按内存方式进行读/写，对映射区域的读写自动反映到磁盘文件上
>- 便于多个进程共享同一磁盘文件，实现[[1.进程#共享存储（Shared Memory）|共享内存]]，即同一文件映射到多个通信进程的虚拟地址空间来实现；尽管各进程虚拟地址空间**相对独立**，但操作系统会通过页表将它们对应的虚拟页映射到**相同的物理页**

# 虚拟存储器性能影响因素

> [!tip] 缺页率是影响虚拟存储器性能的和性因素

> [!note] 核心指标：有效访问时间（EAT）
> 衡量虚拟存储器性能的关键指标是**有效访问时间**（Effective Access Time），即 CPU 从发出逻辑地址到实际取得数据所花的平均时间：
> $$EAT = (1 - p) \times T_{mem} + p \times T_{fault}$$
> 其中 $p$ 为**缺页率**，$T_{mem}$ 为内存访问时间，$T_{fault}$ 为缺页处理时间（主要由磁盘 I/O 决定）。
>
> 由于 $T_{fault}$ 通常比 $T_{mem}$ 大 **5~6 个数量级**（毫秒 vs 纳秒），即使缺页率很低，有效访问时间也会被严重拉高。因此缺页率 $p$ 是性能的**决定性因素**。

## 缺页率的主要影响因素

| 因素 | 说明 | 对性能的影响 |
|------|------|-------------|
| **页面大小** | 每页的字节数 | 页面**过大**→ 页内碎片增加，I/O 时间变长，但缺页率降低、页表减小；页面**过小**→ 缺页率升高、页表膨胀，但碎片少、I/O 快。需在碎片与缺页率之间权衡 |
| **物理块数量** | 分配给进程的页框数（驻留集大小） | 物理块**越多**→ 缺页率越低，但边际收益递减，且并发度下降；物理块**过少**→ 工作集装不下 → 频繁缺页 → **抖动** |
| **置换算法** | 缺页时选择哪个页面换出 | **LRU / 改进 CLOCK** 缺页率低、性能好；**FIFO** 简单但可能命中"刚需"的页，甚至出现 **Belady 异常**（块数增、缺页反增） |
| **已修改页面处理** | 脏页写回磁盘的时机与方式 | **同步立即写回**→ 进程阻塞等待 I/O，缺页处理延迟大；**批量异步写回**（页面缓冲算法）→ 写操作从关键路径剥离，缺页处理更快 |
| **局部化程度** | 程序访问模式的时空聚集性 | **局部性好**（顺序遍历、循环）→ 缺页率极低；**局部性差**（随机跳跃、按列遍历行优先矩阵）→ 缺页率极高，任何算法都难以挽救 |

## 页面大小的选择

> [!tip] 页面大小是一个需要精心权衡的系统参数

| 页面偏大 | 页面偏小 |
|----------|----------|
| ✅ 缺页率更低（预取了更多内容，利用空间局部性） | ✅ 页内碎片更少 |
| ✅ 页表更小（页数少 → 页表项少） | ✅ 调入调出更快（单次 I/O 量小） |
| ✅ 磁盘 I/O 效率更高（一次传输更多数据） | ✅ 局部性更精细（只调入真正需要的部分） |
| ❌ 页内碎片更大 | ❌ 缺页率更高 |
| ❌ 调入调出耗时更长 | ❌ 页表更大（需要更多页表项） |

## 程序局部性与性能

> [!important] 虚拟存储器的性能依赖于程序的局部性


| 局部性类型     | 良好的访问模式              | 不良的访问模式                |
| --------- | -------------------- | ---------------------- |
| **时间局部性** | 循环中的指令、频繁调用的子程序      | 代码只执行一次，数据从不复用         |
| **空间局部性** | 数组顺序遍历（步长=1）、连续结构体访问 | 按列遍历行优先存储的矩阵、跳跃访问大跨度数组 |

# 虚拟内存管理知识体系总结

> [!summary] 本章重点
> 1. **虚拟内存的基本概念**：局部性原理（时间 + 空间）、多次性/对换性/虚拟性三大特征
> 2. **请求分页存储管理**：页表扩充（P/A/M/外存地址）、缺页中断（指令执行期间产生，属于内中断）、地址变换流程（TLB → 页表 → 缺页处理）
> 3. **页框分配**：最小页框数、驻留集大小、三种分配策略（固定局部/可变全局/可变局部）、预调页与请求调页、交换区与文件区
> 4. **页面置换算法**：OPT/FIFO/LRU/CLOCK/改进CLOCK，核心是降低缺页率；FIFO 有 Belady 异常；LRU 和 OPT 是栈类算法
> 5. **抖动与工作集**：驻留集 < 工作集 → 频繁缺页 → CPU 利用率下降；解决方法：工作集策略、缺页率控制（PFF）、挂起进程、L=S 准则
> 6. **页框回收**：四种回收时机、按页面类型与修改位区别处理、页面缓冲算法（空闲链表 + 已修改链表，批量异步写回）
> 7. **虚拟存储器性能影响因素**：EAT 公式、页面大小/物理块数/置换算法/脏页处理/局部化程度五维分析
