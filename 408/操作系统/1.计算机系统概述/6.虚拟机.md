# 基本概念

**虚拟机（Virtual Machine）** 是指通过**虚拟化（Virtualization）** 技术，将一台物理计算机抽象为多台逻辑计算机的技术。每台逻辑计算机称为虚拟机，它们拥有各自独立的处理器、内存、I/O 设备等虚拟硬件，可运行各自独立的操作系统。

>[!info] 虚拟化的核心思想
>虚拟化是**将物理资源映射为逻辑资源**的技术。它与操作系统的"虚拟"特征（详见 [[1.操作系统的基本概念#虚拟(Virtual)]]）一脉相承，但虚拟机层面的虚拟化更彻底——它虚拟的是**整台计算机系统**，而不仅仅是某个资源

## 虚拟机管理程序（Hypervisor / VMM）

**虚拟机管理程序（Virtual Machine Monitor, VMM）**，也称 **Hypervisor**，是运行在物理硬件与操作系统之间的**中间软件层**，负责：
- **创建和管理虚拟机**：为每个虚拟机分配虚拟资源
- **处理器虚拟化**：截获并模拟虚拟机的特权指令执行
- **内存虚拟化**：管理虚拟机物理地址到机器物理地址的映射
- **设备虚拟化**：为虚拟机提供虚拟 I/O 设备
>[!important] VMM特权级
>VMM 运行在高于虚拟机的操作系统特权级上，才能有效拦截和模拟敏感指令、管理物理资源并维持**虚拟机之间的隔离**
# 虚拟化的分类

## 按虚拟化层次分类

| 类型           | 代表技术              | 说明                   |
| ------------ | ----------------- | -------------------- |
| **指令集虚拟化**   | Bochs, QEMU（纯软件）  | 通过软件模拟指令集，可运行不同架构的系统 |
| **硬件虚拟化**    | Intel VT-x, AMD-V | 借助 CPU 硬件扩展实现高效虚拟化   |
| **操作系统级虚拟化** | Docker, LXC       | 共享宿主机内核，实现轻量级隔离（容器）  |
| **编程语言虚拟化**  | JVM, CLR          | 屏蔽底层差异，提供跨平台运行环境     |

## 按替代程度分类

- **全虚拟化（Full Virtualization）**：虚拟机运行**未经修改**的操作系统，VMM 负责截获所有敏感指令，完全模拟底层硬件
- **半虚拟化（Para-virtualization）**：虚拟机运行**经过修改**的客户操作系统，通过超调用（Hypercall）直接与 VMM 通信，性能更优

# 虚拟机管理程序的类型

## 第一类（Type 1，裸机型 / Native / Bare-metal）

VMM **直接**运行在**物理硬件**之上，具有最高特权级别（类似操作系统的内核态），负责管理所有硬件资源，同时向上层提供虚拟化接口

```
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  虚拟机 A    │  │  虚拟机 B    │  │  虚拟机 C    │
│  (Guest OS)  │  │  (Guest OS)  │  │  (Guest OS)  │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       │                 │                 │
       └─────────────────┼─────────────────┘
                         │
              ┌──────────▼──────────┐
              │   VMM (Hypervisor)  │
              │  (Ring 0 / Root)    │
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │    物理硬件         │
              │  (CPU, 内存, 设备)  │
              └─────────────────────┘
```

**特点**：
- VMM 运行在最高特权级（Ring 0 / Root Mode）
- **性能开销小**，接近物理机性能
- 通常用于**服务器虚拟化**和**数据中心**
- 典型代表：**VMware ESXi**、**Xen**、**KVM**（Linux 内核模块）、**Hyper-V**

## 第二类（Type 2，宿主型 / Hosted）

VMM 作为**普通应用程序**运行在**宿主操作系统（Host OS）** 之上，通过宿主操作系统来管理硬件资源

```
┌──────────────┐  ┌──────────────┐
│  虚拟机 A    │  │  虚拟机 B    │
│  (Guest OS)  │  │  (Guest OS)  │
└──────┬───────┘  └──────┬───────┘
       │                 │
       └────────┬────────┘
                │
    ┌───────────▼───────────┐
    │  VMM (应用程序)       │
    └───────────┬───────────┘
                │
    ┌───────────▼───────────┐
    │  宿主操作系统 (Host OS)│
    └───────────┬───────────┘
                │
    ┌───────────▼───────────┐
    │    物理硬件           │
    └───────────────────────┘
```

**特点**：
- VMM 运行在用户态，作为普通进程被调度
- 需要经过宿主操作系统间接访问硬件，**性能开销较大**
- 安装部署**方便灵活**，适合桌面开发和测试环境
- 典型代表：**VMware Workstation**、**Oracle VirtualBox**、**QEMU**

>[!summary] 两类 VMM 对比
>
| 对比项 | 第一类（Type 1） | 第二类（Type 2） |
|-------|----------------|----------------|
| 运行位置 | 直接运行在**裸机**上 | 运行在**宿主操作系统**上 |
| 特权级别 | 最高特权级（Ring 0 / Root Mode） | 用户态（Ring 3） |
| 硬件访问 | **直接管理**硬件资源 | 通过宿主操作系统间接访问 |
| 性能 | **高**（接近物理机） | 较低（需经过宿主 OS 中转） |
| 适用场景 | 服务器、数据中心、云计算 | 桌面开发、测试、个人使用 |
| 典型代表 | VMware ESXi, Xen, KVM, Hyper-V | VMware Workstation, VirtualBox |

# CPU 虚拟化

## 敏感指令与特权指令

虚拟化的关键难题在于如何让 VMM 保持对物理资源的**完全控制**，同时为虚拟机提供执行特权指令的能力：

| 概念                               | 定义                               | 关系              |
| -------------------------------- | -------------------------------- | --------------- |
| **特权指令（Privileged Instruction）** | 只能在**最高特权级**执行的指令（如开/关中断、修改页表基址） | 特权指令 ⊆ 敏感指令     |
| **敏感指令（Sensitive Instruction）**  | 会**影响系统状态**或**暴露硬件状态**的指令        | 敏感指令 ⊇ 特权指令     |
| **陷入指令（Trap Instruction）**       | 执行后**自动陷入内核**的指令                 | 特权指令在非特权级执行时会陷入 |

>[!important] Popek & Goldberg 虚拟化条件
>若某指令集架构满足：**敏感指令 ⊆ 特权指令**（即所有敏感指令都是特权指令），则该架构是**可虚拟化**的。此时 VMM 只需将虚拟机运行在非特权级，虚拟机的敏感指令执行时会自动陷入 VMM 处理。
>
>x86 指令集在硬件虚拟化扩展出现之前**不满足**此条件（如 `POPF` 指令在低特权级修改 IF 标志位时**静默忽略**而不陷入），因此需要通过**二进制翻译**等额外技术解决。

# 容器技术（Containers）

**容器**是一种**操作系统级虚拟化**技术，它和传统虚拟机的本质区别在于：

```
┌─────────────────────────────────────────────────┐
│                   传统虚拟机                      │
├──────────────┬──────────────┬──────────────────┤
│  应用程序    │  应用程序    │   应用程序       │
├──────────────┼──────────────┼──────────────────┤
│  客户 OS A   │  客户 OS B   │   客户 OS C      │
├──────────────┴──────────────┴──────────────────┤
│              VMM (Hypervisor)                  │
├────────────────────────────────────────────────┤
│                  宿主机硬件                     │
└────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│                     容器                         │
├──────────────┬──────────────┬──────────────────┤
│  应用程序    │  应用程序    │   应用程序       │
├──────────────┼──────────────┼──────────────────┤
│  Container   │  Container   │   Container      │
│  Runtime     │  Runtime     │   Runtime        │
├──────────────┴──────────────┴──────────────────┤
│               宿主机 OS（共享内核）              │
├────────────────────────────────────────────────┤
│                  宿主机硬件                     │
└────────────────────────────────────────────────┘
```

| 对比项 | 虚拟机 | 容器 |
|-------|--------|------|
| **内核** | 每个虚拟机有**独立内核** | **共享宿主机内核** |
| **隔离性** | **强**（硬件级隔离） | **弱**（进程级隔离，共享内核） |
| **启动速度** | 慢（需启动完整 OS） | **快**（秒级启动） |
| **资源占用** | 大（每个虚拟机需完整 OS） | **小**（仅应用及其依赖） |
| **性能** | 有虚拟化开销 | 接近原生（无硬件模拟开销） |
| **典型代表** | VMware ESXi, KVM, Hyper-V | Docker, Kubernetes, LXC |

>[!tip] 容器和虚拟机的关系
>容器**不是虚拟机**，它们共享宿主机内核，属于**操作系统级虚拟化**。Docker 等容器技术通常依赖 Linux 内核的 **cgroups（资源隔离）** 和 **Namespace（命名空间隔离）** 实现。

# 关键知识点

## 虚拟化的核心条件

虚拟机管理程序（VMM）必须满足的三个条件（Popek & Goldberg 虚拟化定理）：

1. **等价性**：虚拟机运行环境应与物理机等价（除时间差异外）
2. **资源控制**：VMM 必须完全控制系统资源，虚拟机无法绕过 VMM 访问资源
3. **高效率**：绝大多数指令直接在硬件上执行，无需 VMM 干预

## 虚拟化的两种实现方式

| 方式 | 原理 | 特点 |
|-----|------|------|
| **直接执行 + 陷入模拟** | 用户态指令直接执行，特权指令通过陷入由 VMM 模拟 | 效率高，但需硬件支持 |
| **二进制翻译（Binary Translation）** | VMM 在运行时将虚拟机中的敏感指令替换为等效的安全指令序列 | 兼容性好，但性能开销大 |

## 第一类与第二类 VMM 的区分

- **第一类**：VMM 直接运行在硬件上 → "裸机型" → 代表：ESXi, Xen, KVM, Hyper-V
- **第二类**：VMM 运行在宿主操作系统上 → "宿主型" → 代表：VirtualBox, VMware Workstation
