# 调度的基本概念

在多道程序系统中，进程的数量往往多于 CPU 的数量，因此必然会产生多个进程竞争 CPU 的情况。**CPU 调度**（CPU Scheduling）就是操作系统按照一定的策略，从就绪队列中选择一个进程，将 CPU 分配给它运行。

> [!important] 调度的本质
> 进程切换必然涉及 CPU 调度，CPU 调度是操作系统**进程管理**的核心功能之一，其目的是**提高 CPU 的利用率**和**系统的并发性**。

# 调度的层次

操作系统中的调度可以划分为 **三个层次**，从宏观到微观分别是：

| 层次       | 名称   | 调度对象                     | 调度频率 | 主要目标           |
| -------- | ---- | ------------------------ | ---- | -------------- |
| **高级调度** | 作业调度 | 外存中的作业 → 内存中的进程          | 低    | 控制多道程序度，平衡系统负载 |
| **中级调度** | 内存调度 | **挂起**的进程 ↔ 就绪的进程（换入/换出） | 中等   | 提高内存利用率和系统吞吐量  |
| **低级调度** | 进程调度 | 就绪队列中的进程 → CPU           | 高    | 公平高效地分配 CPU 时间 |

## 高级调度（作业调度）

- 在外存与内存之间进行调度：从后备队列中选择一个或多个**作业**，将它们调入内存，**创建进程并分配资源**，从创建态到就绪态
- 主要存在于**批处理系统**中，分时系统和实时系统通常无此层次
- 每个作业在整个生命周期中**仅**被调入一次、调出一次

## 中级调度（内存调度）

- 将暂时不能运行的进程从内存**换出**（Swap Out）到外存，进入[[1.进程#挂起状态|挂起状态]]，将已具备运行条件的进程从外存**换入**（Swap In）到内存，恢复为**就绪态**
- 提高**内存利用率**和**系统吞吐量**

## 低级调度（CPU 调度、进程调度）

- 按某种算法从就绪队列中选取一个进程，将 CPU 分配给它
- 是操作系统中最**基本**、最**频繁**的调度，调度频率通常为每 10~100ms 一次

## 三级调度的联系

三级调度虽然处于不同层次，但**相互配合、协同工作**，共同构成操作系统的完整调度体系：

> [!important] 三者关系
> 1. **高级调度是入口**：决定哪些作业能进入内存，控制系统的**多道程序度**。只有通过高级调度的作业才能成为进程，为后续调度流程**做准备**。
> 2. **中级调度是缓冲**：在内存紧张时将部分进程暂时换出，内存宽裕时再换入，起到**调节内存压力**的缓冲作用，为高级和低级调度提供运行空间。
> 3. **低级调度是核心**：直接决定"哪个进程获得 CPU"，是唯一真正分配 CPU 的层次，其频率最高，直接影响系统性能。
> 4. **层层过滤**：高级调度筛选作业 → 中级调度调节内存驻留 → 低级调度分配 CPU，每一层都在缩小候选范围，最终由 CPU 执行。
> 5. **速度递进**：高级调度最慢（分钟级）→ 中级调度中等（秒级）→ 低级调度最快（毫秒级），频率依次递增，形成一个**漏斗形**调度体系。

> [!tip] 与 [[1.进程#进程的状态与转换|进程状态转换]] 的对应
> - **高级调度** → 创建态 → 就绪态：作业首次进入内存
> - **低级调度** → 就绪态 → 运行态：获得 CPU
> - **中级调度** → 就绪态 ↔ 就绪挂起态 / 阻塞态 ↔ 阻塞挂起态：内存换入/换出

# 进程调度

## 进程调度任务

1. **保存 CPU 现场**：当进程切换时，必须完整保存当前进程的执行上下文（PC、通用寄存器、PSW 等）
2. **按算法选择进程**：根据调度策略（如 RR、优先级等）从就绪队列中选出最适合运行的进程，将其状态由就绪态变为运行态
3. **分配 CPU**：由分派程序将选中进程 PCB 中保存到的 **CPU 现场**加载到处理器寄存器中，将 CPU 控制权移交给选中的进程，使其从从上次断点处回复执行

## 调度程序（调度器）

**调度程序**（Scheduler）是操作系统**内核中**负责实施 CPU 调度和分派的**软件组件**，由三部分组成：

- **排队器**：将系统中所有就绪进程按照特定的策略组织成一个或多个就绪队列，为后续调度选择提供基础
- **分派器**：根据调度算法选定进程，负责进行实际的 CPU 分配操作，主要任务包括：从就绪队列移出目标进程，**触发**上下文的切换，并将 CPU 控制权交给该进程
- **上下文切换器**：在 CPU 切换时会发生**两对**上下文的切换：第一对，将当前进程的上下文保存到其 PCB 中，再装入**分派程序**的上下文，以便分派程序的运行；第二对，移出分派程序的上下文，将新选进程的 CPU 现场信息装入 CPU 的各个相应寄存器

## 调度的时机

### 需要调度的时机

- **进程创建后**：创建新进程后，父进程和子进程都处于就绪态，需决定运行父进程还是子进程
- **进程终止时**：进程正常结束或异常终止后，必须从就绪队列中选择某个进程运行；若没有就绪进程，则通常运行一个系统提供的**闲逛进程**
- **进程中断时**：当进程因 I/O 请求、信号量操作或其他原因而被阻塞时，主动放弃 CPU
- **中断返回时**：当 I/O 设备准备就绪后，发出 I/O 中断，原先等待 I/O 的进程由阻塞态转变为就绪态，此时需决定让该新就绪进程投入运行，还是让中断发生时正在运行的进程继续进行
- **高优先级就绪时**：在支持抢占的系统中，更高优先级的进程进入就绪队列时，立即抢占当前进程
- **时间片用完时**：当前进程的时间片用完，被剥夺 CPU 并重新放回就绪队列

>[!tips] 闲逛进程
>当就绪队列为空，CPU 不能空转，此时操作系统会运行**闲逛进程**（Idle Process）：
>- 它的 PID 为 0
>- 优先级**最低**，只有当就绪队列为空时才被调度
>- 不占用除 CPU 外的其他系统资源，也不会因等待事件而被阻塞
>- 只要有任何其他进程就绪，闲逛进程立即被抢占

>[!warning] 处于临界区的进程也可能因中断或抢占而导致调度
>**用户态下**的进程即使进入了临界区，仍可能被时钟中断打断。中断处理完成后，若发现有更高优先级的进程就绪，调度器会执行抢占，当前进程被迫让出 CPU。此时临界区尚未执行完毕，**其他试图进入同一临界区的进程仍会被阻塞**，直到原进程下次获得 CPU 并完成临界区
>原子操作是实现临界区互斥的手段，临界区是需要互斥访问的代码区域；临界区中的操作不一定是原子的

### 不能调度的时机

- 在**中断处理程序**执行期间（中断本身不能被调度打断，但嵌套中断系统可能允许）；同时中断处理属于系统内核工作，逻辑上不隶属于任何用户进程，不应被剥夺 CPU 资源
- 在**原语执行期间**（原子操作，不可分割），需要屏蔽中断来保证原子性
- 调度程序本身执行期间

>[!note] 若在不可调度期间发生了调度条件
>系统会设置一个**调度请求标志**，待当前临界区或中断处理完成后，在根据该标志执行相应的调度和切换操作

## 调度的方式

### 非抢占式调度（Non-Preemptive）

当前运行进程**主动放弃** CPU 之前，其他进程不能抢占 CPU。调度仅发生在：
- 进程运行完毕（终止）
- 进程主动阻塞（I/O 请求等）

> [!note] 特点
> - 实现简单、系统开销小
> - 响应时间不稳定，不适合分时/实时系统
> - 早期批处理系统常用

### 抢占式调度（Preemptive）

操作系统可以**强制剥夺**当前进程的 CPU 使用权，分配给其他进程。调度可发生在：
- 时间片用完
- 更高优先级的进程就绪
- 中断处理结束后

> [!note] 特点
> - 响应时间快，适合分时系统和实时系统
> - 需要解决并发访问共享数据的问题（[[4.同步与互斥|同步与互斥]]机制）
> - **现代操作系统**（Linux、Windows、macOS）普遍采用

> [!important] 抢占式 vs 非抢占式的关键区别
> - 非抢占式：进程**自愿**让出 CPU，调度点有限
> - 抢占式：操作系统**强制**切换，不管作业是否完成，调度点更多，**系统开销增大**

## 两种线程的调度

根据线程的实现方式不同，调度涉及的范围也不同：

| 线程类型      | 调度范围 | 调度者     | 特点                   |
| --------- | ---- | ------- | -------------------- |
| **用户级线程** | 进程内部 | 线程库、用户态 | 内核不可见，仅将整个进程作为一个调度单位 |
| **内核级线程** | 系统全局 | 操作系统内核  | 内核直接调度线程，多核下可真正并行    |

> [!important] 两种线程的对比
> - 用户级线程的调度对内核**透明**，内核仍以**进程**为单位分配 CPU，运行哪个线程由用户级线程库决定，线程切换在同一个进程内完成，只需要保存少量寄存器状态
> - 内核级线程的调度单位是**线程**，同一进程的多个线程可被分配到不同 CPU 核心上，每个线程拥有**独立的**内核上下文，可被单独赋予时间片，线程切换开销远高于用户级线程
> - 两种模型的详细对比参见 [[2.线程#三种实现方式对比|两种线程的对比]]

# 调度的目标

为了定量评价调度算法的优劣，操作系统使用以下衡量标准：

## CPU 利用率

CPU 利用率指 CPU 处于**忙碌状态**的时间比例：

$$
\text{CPU 利用率} = \text{CPU 忙碌时间} \space / \space \text{总时间}
$$

多道程序系统中，CPU 利用率越高说明系统资源利用越充分。

## 系统吞吐量

系统吞吐量指**单位时间内**完成的作业（进程）数量：

$$
\text{系统吞吐量} = \text{完成的作业数} \space / \space \text{总时间}
$$

吞吐量越大，系统处理能力越强，长作业因占用较多 CPU 时间，会降低系统的吞吐量；而短作业所需的 CPU 时间较少，有助于提升吞吐量

## 周转时间

周转时间指从**作业提交**到**作业完成**的总时间：

$$
\text{周转时间} = \text{完成时间} - \text{到达时间} = \text{等待时间} + \text{运行时间}
$$

由此衍生出以下相关指标：

- **平均周转时间**：$\sum{\text{各作业周转时间}} \space / \space \text{作业数}$
- **带权周转时间**：$\text{周转时间} \space / \space \text{运行时间}$（≥ 1），值越小说明等待时间占比越低
- **平均带权周转时间**：$\sum{\text{各作业带权周转时间}} \space / \space \text{作业数}$

## 等待时间

等待时间指进程在**就绪队列**中等待 CPU 的总时间：

$$
\text{等待时间} = \text{周转时间} - \text{运行时间} - \text{I/O 时间}
$$

> [!warning] 注意
> 等待时间**只包含**在就绪队列的等待时间，**不包含**阻塞等待 I/O 的时间。

## 响应时间

响应时间指从**提交请求**到**首次产生响应**的时间：

$$
\text{响应时间} = \text{首次响应时刻} - \text{提交时刻}
$$

响应时间是评价**交互式系统**（如分时系统）的核心指标——用户关心的是"多快能开始响应"，而非"多快能完成"。

> [!tip] 关键区分
> - **周转时间**关注"整个作业何时完成"（适合批处理系统），包含等待 + 运行 + I/O 的全部时间
> - **响应时间**关注"多快开始响应"（适合交互式系统），仅衡量首次获得 CPU 之前的等待
> - **等待时间**只包含在就绪队列的等待时间，**不包含**阻塞等待 I/O 的时间
> - **带权周转时间** ≥ 1，值越大说明等待时间占比越高，越小说明作业等待越少

> [!important] 各指标的权衡
> 调度目标之间存在**矛盾**：提高吞吐量可能牺牲响应时间，保证公平性可能降低 CPU 利用率。调度算法的设计本质上是在这些目标之间找到**折中**，要满足特定应用场景的需求，也要兼顾系统整体效率，同时还需考虑调度算法自身的开销

# 进程切换

进程的切换通常紧随调度之后发生，其核心任务时保存原进程的执行现场，并恢复新进程的现场

1. **保存上下文**：内核将原进程的 CPU 现场保存到其 PCB 或内核栈中
2. **更新 PCB**：修改原进程的状态（运行 → 就绪/阻塞/终止）并移入相应队列
3. **选择新进程**：按照调度算法从就绪队列中选取一个进程
4. **更新新进程 PCB**：修改新进程状态为运行态
5. **恢复上下文**：从新进程的 PCB 或内核栈中加载 CPU 现场
6. **更新地址空间相关信息**：如页表基址
7. **重置程序计数器**：开始执行新进程

## 上下文切换

**上下文切换**（Context Switch）是指 CPU 从一个进程/线程的执行上下文切换到另一个进程/线程的执行上下文的过程，包含在进程切换中

切换时，操作系统需要完成以下核心工作：

1. 保存当前进程的 CPU 上下文（PC、通用寄存器、PSW、栈指针等）到其 PCB
2. 将当前进程状态更新（运行态 → 就绪态/阻塞态）并移入相应队列
3. 从新进程的 PCB 恢复上下文到 CPU 各寄存器
4. 更新内存管理相关结构（如页表基址寄存器），可能刷新 TLB
5. 跳转到新进程 PCB 中的程序计数器所指向的位置，开始执行该程序

> [!note] 进程上下文的组成
> - **用户级上下文**：程序段（代码）、数据段、用户栈、共享库等
> - **寄存器上下文**：PC（程序计数器）、PSW（程序状态字）、通用寄存器、栈指针等
> - **系统级上下文**：PCB（进程控制块）、页表、内核栈等

## 上下文切换的消耗

上下文切换本身是**纯开销**——切换期间 CPU 不执行任何用户进程的有用工作，主要消耗包括：

1. **直接开销**：执行大量 `load` / `store` 指令以保存和恢复寄存器内容
2. **TLB 刷新**：切换进程意味着切换地址空间，TLB（快表）中旧进程的缓存条目全部失效，后续内存访问需重新填充，命中率骤降
3. **Cache 失效**：新进程的代码和数据不在 CPU 高速缓存中，导致大量 Cache Miss
4. **流水线刷新**：切换导致 CPU 指令流水线中预取的指令作废，需重新填充流水线

> [!important] 降低上下文切换开销的硬件手段
> - 采用**两组寄存器**（内核组 / 用户组），切换时只需改变指针指向当前所需寄存器组
> - 使用**进程标识符**（ASID，Address Space ID）标记 TLB 条目，同一核心可在 TLB 中同时缓存多个进程的映射，避免每次切换都全量刷新 TLB

## 上下文切换与模式切换

| | 上下文切换 | 模式切换 |
|------|---------------|-------------|
| **含义** | 从一个进程的执行上下文切换到另一个进程 | 用户态与内核态之间的切换 |
| **触发场景** | 时间片到、进程阻塞、抢占、进程终止 | 系统调用、中断、异常 |
| **切换内容** | 全部寄存器、页表、TLB 可能刷新 | 仅切换堆栈指针和 CPU 特权级 |
| **进程是否改变** | 是（进程 A → 进程 B） | 否（同一进程进入内核态后返回） |
| **相对开销** | **大** | 较小 |

> [!tip] 关键结论
> - 一次模式切换**不一定**导致上下文切换（如系统调用后返回同一进程继续执行）
> - 一次上下文切换**必然**包含模式切换（必须陷入内核态才能完成进程切换）
> - 上下文切换**只能由内核**完成，整个切换过程运行在**内核态**
> - 中断 / 系统调用首先触发模式切换，内核处理完成后**可能**触发调度，此时才发生真正的上下文切换

## 调度与切换的区别

> [!important] 调度（Scheduling）≠ 切换（Switching）
> 
> | 维度 | 调度 | 切换 |
> |------|------|------|
> | **本质** | **决策**：选择哪个进程获得 CPU | **执行**：实施进程上下文的保存与恢复 |
> | **发生顺序** | 先调度（选定目标），后切换（执行切换） | 调度完成后立即执行 |
> | **负责组件** | 调度算法 / 调度策略 | 分派程序 / 上下文切换器 |
> 
> 调度的结果决定**切换的目标**，切换是调度的**落实手段**。两者紧密配合但职责分明：调度负责"选谁"，切换负责"怎么换"

> [!tip] 进程切换 vs 线程切换
> | 切换类型 | 需要切换的内容 | 相对开销 |
> |----------|---------------|----------|
> | **进程切换** | 页表、TLB 刷新、地址空间、寄存器、内核栈 | **大** |
> | **同一进程内线程切换** | 寄存器、内核栈（**共享**地址空间和页表，不需要切换） | **小**（无需切换地址空间） |

# CPU 调度算法

>[!warning] 注意在进程序列中，调度第一个进程也算一次调用

## 先来先服务（FCFS，First Come First Served）

按照进程到达就绪队列的**先后顺序**分配 CPU，先到达的进程先运行，直到完成或阻塞。

### 算法特点

- 调度方式：**非抢占式**
- 数据结构：FIFO [[5.队列|队列]]
- 公平性：形式上公平（先来先服务）

> [!note] 优缺点分析
> 
> | 优点 | 缺点 |
> |------|------|
> | 实现简单，算法本身开销**最小** | 对短作业不利：长作业先到达会导致后续全部短作业长时间等待（**护航效应**） |
> | 不会导致饥饿（先到的总能被服务） | 对 I/O 密集型进程不利：每次 I/O 后回到队尾，导致 I/O 设备利用率低 |
> |对长作业有利| 平均等待时间和平均周转时间通常**不是最优** |

> [!warning] 护航效应（Convoy Effect）
> 当多个 I/O 密集型作业在一个 CPU 密集型长作业后排队时，所有 I/O 密集型作业必须等待长作业完成后才能运行，导致 CPU 和设备利用率都很低的现象。

## 短作业优先（SJF / SPF，Shortest Job First）

选择**运行时间最短**的进程优先分配 CPU。对于作业调度称为 SJF（Shortest Job First），对于进程调度称为 SPF（Shortest Process First）。

### 两种变体

| 变体                | 调度方式 | 说明                                                 |
| ----------------- | ---- | -------------------------------------------------- |
| **非抢占式 SJF**      | 非抢占  | 选中运行时间最短的进程，即便运行中有更短进程到达也需等当前进程完成                  |
| **抢占式 SJF**（SRTF） | 抢占式  | 若新到达进程的运行时间**比当前进程剩余时间更短**，则抢占 CPU；也称为**最短剩余时间优先** |

### 算法特点

- 对于同时到达的进程且进程时间已知的情况，SJF 可以证明是**平均等待时间最短**的算法
- **无法准确预知**下一 CPU 区间长度（需要估计或由用户提供）

> [!note] 优缺点分析
> 
> | 优点 | 缺点 |
> |------|------|
> | 平均等待时间/平均周转时间**最优**（同时到达时） | 需要预知作业运行时间（实际中只能估计） |
> | 吞吐量高（快速完成短作业，释放资源） | 对长作业不利，可能导致**饥饿**（不断有更短作业到达时） |
> | | 不适合交互式系统（无法满足快速响应需求） |

> [!warning] 抢占点
> 每次新进程到达就绪队列时，都要比较其运行时间与当前进程**剩余时间**

## 高响应比优先（HRRN，Highest Response Ratio Next）

为解决 SJF 对长作业的饥饿问题而提出。调度时计算各进程的**响应比**，选择响应比最高的进程运行。

### 响应比公式

$$
\text{响应比 } R_p = \frac{\text{等待时间} + \text{要求运行时间}}{\text{要求运行时间}} = 1 + \frac{\text{等待时间}}{\text{要求运行时间}}
$$

### 算法特点

- 调度方式：**非抢占式**（仅在当前进程完成后重新计算响应比）
- 每次调度前需计算所有就绪进程的响应比，选择**最高者**

> [!important] 响应比的含义
> - 等待时间越长 → 响应比越大 → **长作业可获得补偿**，避免饥饿
> - 要求运行时间越短 → 响应比越大 → **短作业优先得到服务**
> - 所以 HRRN **兼顾长短作业**，是 SJF 和 FCFS 的折中

> [!note] 优缺点分析
> 
> | 优点 | 缺点 |
> |------|------|
> | 兼顾长短作业，**公平性好** | 每次调度前都要计算响应比，**开销较大** |
> | 不会导致**饥饿 **| |

## 时间片轮转（RR，Round Robin）

专为**分时系统**设计。所有就绪进程排成 FIFO [[5.队列|队列]]，CPU 依次为每个进程分配一个**时间片**（Time Quantum/Slice）。时间片到期后，当前进程重新放入队尾。

### 算法特点

- 最大特点是**公平性**
- 调度方式：**抢占式**（时间片到即抢占）
- 响应时间有上限：最多等待 $n-1$ 个时间片即可获得 CPU
- 调度时机为当前进程在时间片结束前已运行完成或时间片用尽
### 时间片大小的选择

> [!important] 时间片大小的权衡
> 
> | 时间片大小 | 效果 |
> |-----------|------|
> | **时间片过大**| 退化为 FCFS，响应时间变长 |
> | **时间片过小**| 上下文切换过于频繁，系统开销急剧增大 |
> | **合理取值** | 略大于**一次典型交互的时间长度**，使交互式进程在一个时间片内完成 |

## 优先级调度（Priority Scheduling）

为每个进程分配一个**优先级**，CPU 总是分配给**优先级最高**的就绪进程。

### 分类方式

| 维度 | 分类 | 说明 |
|------|------|------|
| **抢占方式** | 非抢占式优先级 | 当前运行进程完成后才重新选择最高优先级进程 |
| | 抢占式优先级 | 更高优先级进程就绪时立即抢占 CPU |
| **优先级类型** | 静态优先级 | 创建时确定，运行期间不变（实现简单，但不灵活） |
| | 动态优先级 | 运行期间根据情况动态调整（更灵活，如老化算法） |

### 优先级定义

通常用**数字**表示优先级，不同系统的约定不同（如 Linux 中数值越小优先级越高，Windows 则相反），优先级的设定通常遵循：

| 因素                   | 影响                              |
| -------------------- | ------------------------------- |
| **进程类型**             | 系统进程优先级 > 用户进程                  |
| **I/O 密集 vs CPU 密集** | I/O 密集型通常获得更高优先级（对外设响应快）        |
| **前后台类型**            | 交互型（前台）> 非交互型（后台），以便快速响应用户      |
| **资源需求**             | 已占用资源多的进程优先级低（防止长期占用）           |
| **等待时间**             | 等待越久，优先级动态提升（**老化 Aging**，防止饥饿） |

> [!warning] 饥饿问题
> 低优先级进程可能永远得不到 CPU（**饥饿**，Starvation）
> **解决方案**：**老化（Aging）** — 根据等待时间逐步提高进程的优先级。

## 多级队列调度（Multilevel Queue Scheduling）

将就绪队列**划分为多个独立的子队列**，每个队列有各自的调度算法和优先级。

### 队列划分依据

| 类型              | 队列属性  | 调度算法       |
| --------------- | ----- | ---------- |
| **前台队列**（交互式进程） | 高优先级  | 时间片轮转（RR）  |
| **后台队列**（批处理进程） | 低优先级  | FCFS / SJF |
| **系统进程队列**      | 最高优先级 | 优先级调度      |
| **实时进程队列**      | 最高优先级 | 抢占式优先级调度   |

### 队列间调度

各队列之间可采用以下两种方式分配 CPU：

| 方式 | 说明 |
|------|------|
| **固定优先级抢占** | 严格按队列优先级：高优先级队列非空时，优先运行其中的进程；低优先级队列只有高优先级队列为空时才能被调度 |
| **时间片分配** | 每个队列获得一定比例的 CPU 时间（如前台 80%、后台 20%） |

> [!warning] 局限性
> 多级队列调度中，进程**固定分配**到某个队列，一旦进入就无法转移。这意味着进程的行为可能随时间变化（如从交互式变为计算密集型），但队列调度策略不会随之调整。

## 多级反馈队列调度（Multilevel Feedback Queue Scheduling）

多级队列调度的改进版，**允许进程在不同队列间迁移**，是目前公认的**最优**通用调度方案之一（Linux、Windows、macOS 均采用类似机制）。

### 核心机制

1. 设置**多个不同优先级**的就绪队列，优先级从高到低排列
2. 高优先级队列的**时间片较短**，低优先级队列的**时间片较长**
3. **新进程**总是进入**最高优先级**队列
4. 若进程在当前时间片内**未完成**（用完时间片），则被**降级**到下一级队列
5. 若进程在时间片内**主动放弃 CPU**（I/O 请求等阻塞），则保持当前优先级（甚至**提升**优先级）
6. **最低优先级队列**时间片较长，其行为接近 FCFS 调度

### 算法特点

- 调度方式：**抢占式**（高优先级队列有进程时立即抢占当前进程）
- 自适应：自动识别进程类型，**无需事先直到进程的运行时间**，CPU 密集型逐步降级到低优先级长队列，I/O 密集型保持高优先级

> [!important] 多级反馈队列的精妙之处
> - **偏好短进程**：新进程进入最高优先级（小时间片），若是短进程则在最高层快速完成
> - **惩罚 CPU 密集进程**：每次用完时间片就降一级，最终到最低优先级队列以 FCFS 方式执行
> - **奖励 I/O 密集进程**：主动阻塞说的进程保持或提升优先级
> - **不需要预知运行时间**：系统通过进程的行为**自动学习**其类型

### 参数设计问题

| 参数 | 影响 |
|------|------|
| **队列数量** | 太多 → 调度开销大；太少 → 区分度不足 |
| **每级队列时间片** | 通常逐级翻倍（如 8ms × 2^k）或按固定函数增长 |
| **升降级策略** | 决定何时将进程从一个队列移到另一个队列 |
| **队列间调度方式** | 绝对优先级 vs 时间片分配 |

## 基于公平原则的调度

前面介绍的算法多聚焦于**性能指标**（如周转时间、吞吐量等），而公平原则调度则以**确保每个进程/用户获得公平的 CPU 份额**为首要目标

### 公平分享调度（Fair Share Scheduling）

传统调度以**进程**为单位分配 CPU，但同一用户可能创建多个进程，从而导致该用户抢占过多的 CPU 时间。公平分享调度将**用户或用户组**纳入考量：

- 每个用户/组被赋予一定的 CPU **份额权重**
- 调度时不仅看进程优先级，还看其所属用户**已使用的 CPU 时间与应得份额的比值**
- 已超额使用的用户，其进程会被降低调度优先级；份额不足的用户则获得补偿

> [!note] 核心思想
> 公平分享调度的关键在于：**公平是面向用户/组，而非面向进程**。一个用户运行再多进程，也不能获得超过其份额的 CPU 时间。

### 保证调度算法（Guaranteed Scheduling）

保证调度算法的核心思想是：向每个进程**承诺**一定比例的 CPU 时间，调度器跟踪实际分配情况，优先调度"落后"的进程以确保承诺兑现。

**核心机制**：

1. 系统内有 $n$ 个就绪进程，则每个进程应获得 $1/n$ 的 CPU 时间
2. 跟踪每个进程的**实际 CPU 时间**，计算其实际获得比率：$\text{比率} = \text{实际 CPU 时间} \space / \space \text{应得 CPU 时间}$
3. 调度时选择比率**最小**的进程（即实际获得 CPU 时间最少、最"亏欠"的进程）
4. 新进程加入或旧进程退出时，重新计算基准值

> [!important] 保证调度的精妙之处
> - 比率最小优先 → 自动补偿"吃亏"的进程，天然的公平性
> - 无需预知进程运行时间 → 仅根据历史 CPU 占有量做决策
> - 进程未使用 CPU（如阻塞等待 I/O）时，其应得时间会累积，恢复就绪后将获得更高的调度优先级

> [!tip] 公平分享调度 vs 保证调度
> | 维度 | 公平分享调度 | 保证调度 |
> |------|------------|----------|
> | 公平对象 | 用户 / 用户组 | 进程 |
> | 核心机制 | 按用户份额权重分配 | 按实际/应得比率分配 |
> | 份额依据 | 预先设定的用户权重 | 均等 $1/n$（或按进程权重） |
> | 典型场景 | 多用户分时系统 | 通用交互式系统 |

## 调度算法的总结与对比

### 各算法对比表

> [!important] 408 考试核心对比
> 
> | 算法 | 调度方式 | 适用场景 | 平均等待时间 | 饥饿可能 | 需要预知运行时间 |
> |------|----------|----------|-------------|----------|------------------|
> | **FCFS** | 非抢占 | 批处理 | 较差 | 无 | 否 |
> | **SJF（非抢占）** | 非抢占 | 批处理 | **最优** | 长作业可能 | 是 |
> | **SRTF（抢占 SJF）** | 抢占式 | 批处理 | **最优** | 长作业可能 | 是 |
> | **HRRN** | 非抢占 | 批处理 | 较好 | 无 | 是 |
> | **RR** | 抢占式 | 分时 | 与时间片有关 | 无 | 否 |
> | **优先级调度** | 均可 | 通用 | 取决于优先级 | 可能 | 否 |
> | **多级队列** | 均可 | 通用 | 取决于配置 | 低优先级队列可能 | 否 |
> | **多级反馈队列** | 抢占式 | **通用/实际 OS** | 较好 | 低优先级可能 | 否 |

### 算法选择参考

| 系统类型 | 推荐算法 |
|----------|----------|
| **批处理系统** | SJF（最优）、HRRN（兼顾公平） |
| **分时系统** | RR（响应快）、多级反馈队列（自适应） |
| **实时系统** | 抢占式优先级调度（确定性响应） |
| **通用操作系统** | 多级反馈队列（Linux、Windows、macOS 均用类似方案） |

# 多处理机调度

多处理机系统中，多个 CPU 核心共享就绪队列（或各自维护局部队列），调度器需在多个 CPU 之间协调，**比单处理机调度更复杂**

## 多处理机的类型

根据 CPU 之间地位是否平等，多处理机系统分为非对称多处理机（Asymmetric MultiProcessing，AMP）和对称多处理机（Symmetric MultiProcessing）两类：

|          | 非对称多处理机（AMP）                     | 对称多处理机（SMP）                         |
| -------- | -------------------------------- | ----------------------------------- |
| **结构**   | 一个**主处理器** + 多个**从处理器**          | 所有处理器**地位平等**，无主从之分                 |
| **调度方式** | 主处理器专门运行操作系统内核（包括调度），从处理器仅运行用户进程 | 每个处理器都可调度自己，自行从就绪队列选取进程             |
| **优点**   | 数据结构共享简单，实现容易，无需复杂的同步            | 资源利用率高，无单点瓶颈，可扩展性好                  |
| **缺点**   | 主处理器成为**瓶颈**和单点故障，从处理器利用率可能不足    | 需处理并发访问共享就绪队列的同步问题                  |
| **代表**   | 早期多处理器系统                         | **现代操作系统**（Linux、Windows、macOS）普遍采用 |

## 亲和性和负载平衡

### 亲和性（Processor Affinity）

进程在上次运行的 CPU 核心上会积累**缓存状态**（Cache、TLB 条目等），若被迁移到其他 CPU，这些缓存都将失效。因此调度器倾向于让进程**尽可能留在同一 CPU** 上运行，这就是**处理器亲和性**。

| 类型 | 说明 |
|------|------|
| **软亲和性**（Soft Affinity） | 调度器尽量保持进程在同一 CPU 运行，但不强制；必要时允许迁移 |
| **硬亲和性**（Hard Affinity） | 进程明确指定允许运行的 CPU 集合（如 Linux 的 `taskset`），调度器不得违反 |

> [!note] 亲和性 vs 迁移开销
> 保持亲和性 → 缓存命中率高，进程执行快；但可能导致某些 CPU 过载，其他 CPU 空闲。

### 负载平衡（Load Balancing）

负载平衡的目标是**让各 CPU 核心的工作负载尽量均衡**，避免部分 CPU 空闲而其他 CPU 过载。

| 方式 | 说明 |
|------|------|
| **推迁移**（Push Migration） | 由专门的负载检测任务定期检查各 CPU，将**过载 CPU** 上的进程"推"到空闲 CPU |
| **拉迁移**（Pull Migration） | **空闲 CPU** 主动从其他 CPU 的就绪队列中"拉"取进程来运行 |

> [!important] 亲和性与负载平衡的矛盾
> - **亲和性**倾向"留在原地"→ 缓存友好，但可能导致负载不均衡
> - **负载平衡**倾向"均匀分布"→ 提高整体 CPU 利用率，但可能导致缓存失效


## 多处理机调度方案

根据就绪队列的组织方式，多处理机调度分为**公共就绪队列**和**私有就绪队列**两大类。

### 公共就绪队列（Shared Ready Queue）

所有 CPU 共享**一个全局就绪队列**，每个 CPU 在空闲时从该队列中选取进程运行。

- 任意一个 CPU 变为空闲时，直接从全局队列取走一个进程，**负载自然均衡**
- 所有 CPU 访问同一队列，需通过**互斥锁**等同步机制保护，CPU 核心数越多，锁竞争越激烈
- **扩展性差**，不适合核心数较多的大规模多处理机系统
- **亲和性**不好
- 也称为**集中式调度**

### 私有就绪队列（Private Ready Queue / Per-CPU Queue）

每个 CPU 拥有**独立的局部就绪队列**，各 CPU 在自己的队列上独立调度，无需跨 CPU 同步。

- 无锁竞争，**可扩展性好**，适合核心数较多的大规模多处理机系统
- 负载可能不均：某 CPU 队列积压大量进程，另一 CPU 队列为空
- 需配合**负载平衡机制**（推迁移 / 拉迁移）定期在各 CPU 间重新分配进程
- 也称为**分布式调度**

> [!important] 两类方案的对比
> | | 公共就绪队列 | 私有就绪队列 |
> |------|-------------|-------------|
> | 队列数量 | 1 个全局队列 | 每 CPU 一个局部队列 |
> | 同步开销 | **大**（需加锁访问） | **无**（各自独立） |
> | 负载均衡 | 自动均衡 | 需额外负载平衡机制 |
> | 可扩展性 | 差（CPU 越多锁竞争越烈） | 好（CPU 增加无影响） |
> | 亲和性 | 差（进程可能被任意 CPU 调度） | 好（进程倾向留在同一 CPU） |
> | 代表系统 | 早期多处理机系统 | Linux、Windows、macOS |

> [!important] 重点
> - 区分**软亲和性**和**硬亲和性**
> - 理解**推迁移**和**拉迁移**的区别
> - 对比**公共就绪队列**和**私有就绪队列**：前者负载自动均衡但锁竞争严重且需要提升亲和性，后者可扩展性好但需额外负载平衡

> 
> **调度的基本概念与层次**
> - 调度的三个层次：高级（作业调度，外存→内存）、中级（内存调度，挂起↔就绪）、低级（CPU调度，就绪→CPU）
> - 三级调度的联系：层层过滤，速度递进（分钟→秒→毫秒），形成漏斗形体系
> 
> **进程调度机制**
> - 调度程序的三组件：排队器（组织就绪队列）、分派器（选定进程并分配 CPU）、上下文切换器（两对切换）
> - 调度时机：创建/终止/阻塞/I/O完成/抢占/时间片到 → 可调度；中断处理/原语/调度程序自身 → 不可调度
> - 抢占式 vs 非抢占式：抢占式响应快、开销大（现代 OS 采用）；非抢占式实现简单、响应不稳
> - 闲逛进程：PID 0，优先级最低，就绪队列为空时运行
> - 两种线程的调度：用户级线程（内核不可见，进程内调度）vs 内核级线程（内核直接调度，可真正并行）
> 
> **调度的目标与衡量标准**
> - 五大指标：CPU 利用率、系统吞吐量、周转时间（含带权/平均/平均带权）、等待时间、响应时间
> - 周转时间关注"何时完成"（批处理），响应时间关注"多快响应"（交互式），等待时间不含 I/O 阻塞
> 
> **进程切换**
> - 上下文切换的四大消耗：直接开销（load/store）、TLB 刷新、Cache 失效、流水线刷新
> - 上下文切换 vs 模式切换：一次上下文切换必然包含模式切换，反之不一定
> - 调度（决策，选谁）vs 切换（执行，怎么换）：两者紧密配合但职责分明
> 
> **CPU 调度算法**
> - 七种经典算法：FCFS（护航效应）、SJF/SRTF（平均等待最短，但需预知时间）、HRRN（折中，响应比公式）、RR（时间片权衡）、优先级调度（饥饿+老化）、多级队列（固定分配）、多级反馈队列（自适应，现代 OS 首选）
> - 基于公平原则：公平分享调度（面向用户/组）、保证调度算法（面向进程，按实际/应得比率）
> 
> **多处理机调度**
> - AMP（主从结构）vs SMP（地位平等，现代主流）
> - 亲和性（软/硬）与负载平衡（推迁移/拉迁移）的矛盾与折中
> - 公共就绪队列（自动均衡但锁竞争严重）vs 私有就绪队列（可扩展好但需负载平衡，Linux/Windows 采用）
