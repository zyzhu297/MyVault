# 进程

## 进程的概念

**进程**是程序的一次执行过程，是系统进行**资源分配和调度的一个独立单位**，操作系统引入进程，以支撑操作系统的两大基本特性：**并发性**与**共享性**

> [!important] 进程与程序的区别
> - **程序**是静态的，是存放在磁盘上的可执行文件；**进程**是动态的，是程序的一次执行实例
> - 一个程序可以对应多个进程（如多次打开同一个浏览器）
> - 进程具有**并发性**和**异步性**，程序没有

## 进程的特征

| 特征        | 说明                                  |
| --------- | ----------------------------------- |
| **动态性**   | 进程是程序的一次执行，有创建、运行、阻塞等过程（**最基本的特征**） |
| **并发性**   | 多个进程可在同时驻留内存，并在一段时间内交替或并行执行         |
| **独立性**   | 进程是独立获得资源和独立调度的基本单位                 |
| **异步性**   | 进程按各自独立的、**不可预知**的速度向前推进            |
| **结构性**   | 每个进程都配置一个 PCB（进程控制块）进行管理            |

>[!warning] 异步性和不可再现性和独立性
>进程具有独立性，因此能并发运行；并发运行具有异步性，而异步执行若共享资源又缺乏同步，就会导致结果的不可再现性
# 进程的组成

进程由三部分组成：**PCB**、**程序段**、**数据段**

## PCB（进程控制块）

PCB 是进程存在的**唯一标志**，操作系统通过 PCB 来感知进程的存在，任何未建立 PCB 的程序，均不能被操作系统调度执行。

> [!important] PCB 的作用
> - PCB 是进程在操作系统中的"身份证"，系统对进程的所有操作（创建、调度、撤销等）本质上都是对该进程 PCB 的操作
> - 当进程被中断（时间片到、I/O 请求等）时，CPU 的现场信息被保存到 PCB 中；恢复运行时，从 PCB 中恢复现场
> - 当进程正常结束或被强制终止时，系统将回收其 PCB，该进程也随之消亡

PCB 包含的信息可划分为 **四个部分**：

### 进程标识信息

用于唯一标识和区分一个进程。

| 字段               | 含义                                            |
| ---------------- | --------------------------------------------- |
| **进程 ID（PID）**   | 操作系统为每个进程分配的唯一数字标识，类似身份证号                     |
| **父进程 ID（PPID）** | 创建本进程的父进程标识（进程树：所有进程构成一棵树，根是 init/systemd 进程） |
| **用户 ID（UID）**   | 标识拥有该进程的用户，用于权限控制（如文件访问权限）                    |

### 进程调度信息

操作系统调度器依据这些信息决定哪个进程获得 CPU、何时切换等。

| 字段                  | 含义                                                                                    |
| ------------------- | ------------------------------------------------------------------------------------- |
| **进程状态**            | 标识进程当前所处状态：创建、就绪、运行、阻塞、终止（以及挂起状态）                                                     |
| **优先级（Priority）**   | 决定进程获得 CPU 的先后顺序。可分为**静态优先级**（创建时确定，保持不变）和**动态优先级**（运行期间根据情况调整，如 I/O 密集型进程可提升优先级防止饥饿） |
| **时间片（Time Slice）** | 进程一次最多能连续占用 CPU 的时间长度。时间片到 → 调度器切换进程                                                  |
| **阻塞原因/事件**         | 记录进程因何进入阻塞态（等待的 I/O 设备号、信号量、消息等），便于事件完成时**精准唤醒**                                      |
| **CPU 使用时间**        | 记录进程已占用 CPU 的总时间，用于调度决策（如保证公平性）                                                       |
| **等待时间**            | 进程在就绪队列中等待的总时间，用于**老化（Aging）算法**（等待越久优先级越高，防止饥饿）                                      |

### 进程控制信息（资源和内存信息）

记录进程拥有的资源及其在内存中的位置，是进程**独立运行**的基础。

| 字段                   | 含义                                                                        |
| -------------------- | ------------------------------------------------------------------------- |
| **内存指针**             | 记录程序代码段、数据段的**起始地址和长度**（通过基址寄存器 + 限长寄存器实现地址映射和保护）                         |
| **页表指针 / 段表指针**      | 指向该进程**页表或段表的基地址**（在支持虚拟内存的系统中，这是进程地址空间映射的核心）。进程切换时更新 CR3 寄存器（x86）即为该指针的值 |
| **资源清单**             | 列出进程已获得的系统资源：打开的文件表（文件描述符）、使用的 I/O 设备、信号量、管道等                             |
| **文件描述符表**           | 记录进程打开的所有文件及其状态（文件指针位置、访问模式等）                                             |
| **链接指针**             | 用于将处于同一状态的 PCB 链接成队列（就绪队列/阻塞队列等），是 PCB 组织方式的基础                            |
| **进程间通信（IPC）信息**     | 与进程间通信相关的控制信息，如消息队列指针、共享内存段标识等                                            |
| **记账信息（Accounting）** | 统计进程执行的累计时间、账户号等，用于性能统计和计费                                                |

### 处理机状态信息（CPU 上下文 / 处理器现场）

进程被中断时，必须保存 CPU 的"快照"，以便恢复时能**从断点处精确继续执行**。

| 字段                                   | 含义                                                                             |
| ------------------------------------ | ------------------------------------------------------------------------------ |
| **通用寄存器（General-Purpose Registers）** | 保存运算的中间结果、函数参数等。x86 架构如 EAX/EBX/ECX/EDX，x86-64 如 RAX/RBX 等                     |
| **指令计数器（PC / EIP / RIP）**            | **下一条待执行指令的地址**。恢复进程时必须首先恢复此值，确保从断点处继续                                         |
| **程序状态字（PSW / EFLAGS）**              | 包含**条件码**（零标志 ZF、进位标志 CF、溢出标志 OF、符号标志 SF）和控制标志（中断允许 IF、方向标志 DF）。由这些标志决定条件跳转的结果 |
| **栈指针（SP / ESP / RSP）**              | 指向当前栈顶，用于函数调用和局部变量访问                                                           |
| **帧指针（BP / EBP / RBP）**              | 指向当前栈帧的基址，用于访问函数参数和局部变量（在调试和栈回溯中非常重要）                                          |

> [!note] 为什么 PC 和 PSW 尤其重要？
> - **PC**：决定了进程"从哪里继续执行"，没有 PC 就不知道代码执行到了哪里
> - **PSW**：条件码决定了后续条件跳转（如 `if`、`while`）的分支方向。例如 ZF=1 时 `je` 指令会跳转，如果 ZF 被错误恢复，程序逻辑会完全错误

### PCB 的组织方式

操作系统需要对大量 PCB 进行高效管理，主要采用以下两种方式：

| 方式       | 说明                                              | 特点                         |
| -------- | ----------------------------------------------- | -------------------------- |
| **链接方式** | 将同一状态的 PCB 链接成**链表**（就绪队列链表、阻塞队列链表、空闲 PCB 链表等）  | 内存开销小，但查找 O(n)，早期 OS 常用    |
| **索引方式** | 为每种状态建立**索引表**，表中记录该状态所有 PCB 在 PCB 表（数组）中的下标/地址 | 查找 O(1)，但内存开销更大（现代 OS 更常用） |

## 程序段
存放要执行的代码（即程序的指令序列），**程序本身是静态的，可被多个进程共享**
## 数据段

存放程序运行过程中需要的各种数据，每个**进程**的数据段是**私有的**，保障了进程之间的独立性和安全性，包括原始数据、中间结果、最终输出、全局变量、局部变量、堆和栈

# 进程的内存映像

当进程被加载运行时，操作系统会为其建立一个**虚拟地址空间**，该空间的逻辑布局称为*进程的内存映像*，内存映像是**动态**的，在[[2.虚拟内存管理|虚拟内存]]机制下，进程的整个地址空间无须全部驻留物理内存，进程的虚拟地址空间由以下几个主要区域构成（地址从低到高）：

| 段              | 说明                                                                                                          |
| -------------- | ----------------------------------------------------------------------------------------------------------- |
| **代码段（Text）**  | 存放程序的机器指令，**只读**，**可共享**<br>`.init`：包含程序启动时由系统自动调用的初始化代码<br>`.text`：存放主程序的指令<br>`.rodata`：存放只读数据，如字符串**常量** |
| **数据段（Data）**  | 已初始化的**全局变量**和**静态变量**<br>`.data`：存放已初始化（显式赋值）的全局变量和静态变量<br>`.bss`：存放未初始化的全局变量和静态变量                         |
| **堆（Heap）**    | 动态分配的内存区域，向**高**地址增长（malloc/new），大小不固定                                                                      |
| **共享库的存储映射区**  | 共享库用于提供进程依赖的公共函数代码，它们在运行时**被动**映射到进程地址空间，代码是只读的，且可被多个进程共享                                                   |
| **栈（Stack）**   | 函数调用的**局部变量**、**函数参数**、**返回地址**，向**低**地址增长                                                                  |
| **进程控制块（PCB）** | **并不属于进程的虚拟地址空间**，由操作系统内核维护，存放在系统内核区，**对用户进程不可见**                                                           |

> [!note] 区分
> - **代码段和数据段**：程序编译时就确定大小，也统称为**正文段**
> - **堆**：运行时动态分配，程序员管理
> - **栈**：编译器自动管理，函数调用时分配/释放

# 进程的状态与转换

## 进程的五种基本状态

| 状态                       | 说明                                          |
| ------------------------ | ------------------------------------------- |
| **创建态（New）**             | 进程正在被创建，尚未就绪，创建态只是一个短暂的中间状态，正常情况下**不会长期停留** |
| **就绪态（Ready）**           | 进程已获得除 CPU 外的所有资源，等待 CPU                    |
| **运行态（Running）**         | 进程正在 CPU 上执行                                |
| **阻塞态（Blocked/Waiting）** | 进程等待某事件（如 I/O 完成），暂停执行                      |
| **终止态（Terminated）**      | 进程执行完毕，等待回收资源                               |

## 状态转换

| 转换      | 触发条件                            |
| ------- | ------------------------------- |
| 就绪 → 运行 | 进程被调度程序选中，获得 CPU                |
| 运行 → 就绪 | 时间片用完，或被更高优先级进程抢占               |
| 运行 → 阻塞 | 请求系统资源（如 I/O）、等待某事件发生，是**主动行为** |
| 阻塞 → 就绪 | 等待的事件发生（I/O 完成、资源获得等），是**被动行为** |
![[进程状态转换]]
> [!warning] 注意
> - **运行 → 就绪**：是被动放弃 CPU（时间片到/被抢占），不是主动放弃
> - **阻塞 → 运行**：**不能直接转换**，必须先经过就绪态
> - **就绪 → 阻塞**：**不可能发生**，因为就绪态没有 CPU，无法发起 I/O 请求

## 挂起状态

当内存不足或有其他原因时，操作系统会将某些进程**挂起**（Suspend），即将进程映像从内存换出到外存。

| 状态 | 说明 |
|------|------|
| **就绪挂起** | 进程在外存，但只要调入内存即可运行 |
| **阻塞挂起** | 进程在外存，且等待的事件尚未发生 |

# 进程控制

进程控制的职责是对系统中的所有进程实施有效管理，实现进程的创建、撤销、阻塞、唤醒和切换等操作。

进程控制通过**原语**（Primitive）来实现，原语在执行期间**不允许被中断**（关中断实现），确保关键操作的**原子性**和**一致性**

## 进程创建

允许一个进程创建另一个进程，创建者为*父进程*，被创建的进程称为*子进程*，子进程在创建时会复制父进程的当前状态，包括代码段、数据段、堆栈内容、打开的文件描述符、环境变量等，但是**子进程和父进程是相互独立的实体**，拥有各自的 PCB 和虚拟地址空间

**用户登录、作业调度、提供服务、应用请求**等，都会触发进程创建，操作系统通过**创建原语**建立新进程，步骤如下：
1. 分配唯一标识符（PID）并申请 PCB，若 PCB 资源耗尽，则创建失败，系统返回错误
2. 分配新进程运行所需的资源，如内存、文件、I/O 设备等，这些资源从操作系统或父进程获得
3. 初始化 PCB，初始化内容包括：标识信息、处理机状态、调度与控制信息和内存与资源信息；通常将新进程设置为**最低优先级**
4. 当所有必要资源都已到位，系统将该进程转入就绪态，并插入就绪队列

## 进程终止

当进程完成使命或遇到无法继续执行的情况时，操作系统将终止该进程。进程终止意味着系统要撤销该进程的所有痕迹，归还它占用的所有资源，包括其所有子进程。

引发进程终止的事件有以下几类：

| 类别 | 具体场景 |
|------|----------|
| **正常结束** | 进程执行完所有任务，主动调用 `exit()` 系统调用，自愿终止 |
| **异常结束** | 运行期间发生无法处理的错误，如：内存越界（Segmentation Fault）、非法指令（除以零）、特权指令越权、I/O 故障等 |
| **外界干预终止** | 用户通过任务管理器或 `kill` 命令强行终止；父进程主动终止子进程；操作系统因系统资源紧张而终止低优先级进程（OOM Killer） |

操作系统通过**终止原语**撤销进程，步骤如下：

1. 根据进程的 PID 在 PCB 集合中查找目标进程的 PCB，并从 PCB 中获取进程当前状态
2. **若进程正处于运行态**，则立即剥夺其 CPU，将 CPU 分配给其他进程
3. **递归终止**该进程的所有子进程（若有），防止子进程变为孤儿子进程
4. 将被终止进程所拥有的所有资源归还给父进程或操作系统，包括：释放内存空间、关闭打开的文件、释放 I/O 设备、归还共享内存段、清空信号量等
5. 将该进程的 PCB 从所在队列（就绪队列/阻塞队列等）中移除，回收 PCB 空间，该进程从此在系统中彻底消失
>[!note] 级联终止
>当一个进程终止时，其所有子进程也随之终止，这种机制被称为*级联终止*
>但并非所有操作系统都采用这种策略，如 Linux，父进程终止时，其子进程不会自动终止，而是由 init 进程（PID=1)收养，并正常运行
## 进程阻塞与唤醒

**阻塞（Block）** 与 **唤醒（Wakeup）** 是一对互为反操作的原语，用于进程在执行过程中因等待资源或事件而暂停，以及事件完成后的恢复。

引发进程阻塞的典型事件包含：**I/O 请求**，进程请求磁盘读写、网络通信、打印机等外设操作时，必须等待 I/O 完成才能继续；**等待资源**，进程等待另一个进程完成某操作后发来的信号（如父进程 `wait()` 等待子进程结束）；**等待事件**，进程等待另一个进程完成某操作后发来的信号（如父进程 `wait()` 等待子进程结束）；**等待数据**，进程调用 `read()` 但管道/套接字中暂无数据可读

### 阻塞原语的执行步骤

1. 正在运行的进程在请求某种资源或服务失败后，**主动调用**阻塞原语
2. 阻塞原语**立即停止**当前进程的执行，保存其 CPU 现场（处理机状态信息）到 PCB
3. 将进程状态从**运行态**改为**阻塞态**，并在 PCB 中记录**阻塞原因/事件**
4. 将该进程的 PCB 从运行队列移除，插入到对应事件的**阻塞队列**（等待该事件的所有进程）
5. 调用调度程序，将 CPU 分配给其他就绪进程

> [!warning] 关键：阻塞是**主动行为**
> 只有**正在运行的进程**才有资格主动调用阻塞原语，因为它持有 CPU 才能执行原语代码。就绪态和阻塞态的进程都没有 CPU，因此**无法主动阻塞**。

### 唤醒原语的执行步骤

1. 当某个事件（I/O 完成、资源释放、信号发送等）发生时，由**其他进程**或**中断处理程序**调用唤醒原语
2. 唤醒原语在**阻塞队列**中查找等待该事件的所有进程
3. 将被唤醒进程的 PCB 从阻塞队列中移出
4. 将其状态从**阻塞态**改为**就绪态**
5. 将 PCB 插入**就绪队列**，等待调度程序分配 CPU

> [!warning] 关键：唤醒是**被动行为**
> 被唤醒的进程不会立刻获得 CPU，而是回到就绪态等待调度。唤醒后是否真正运行，取决于调度程序的**调度时机**。

> [!important] 阻塞与唤醒的配对
> - **阻塞原语**与**唤醒原语**必须成对出现：有 block 必有 wakeup，否则进程将永远阻塞（死锁/饥饿）
> - 如果阻塞队列中同时有多个进程在等待同一事件，唤醒时通常**全部唤醒**（将所有等待进程移至就绪队列），由调度程序决定谁先运行；也可按优先级只唤醒一个

## 进程切换

> [!important] 进程切换的步骤（上下文切换）
> 1. 保存当前运行进程的**上下文**（寄存器、PC、栈指针等）到其 PCB
> 2. 将当前进程的 PCB 移出运行队列，修改其状态
> 3. 选择下一个要运行的进程（调度）
> 4. 从新进程的 PCB 中恢复其上下文
> 5. 将 CPU 控制权交给新进程

**上下文切换的开销**：寄存器保存/恢复、TLB 刷新、缓存失效等，切换越频繁，系统开销越大

# 进程的通信

进程通信（IPC，Inter-Process Communication）是指进程之间的信息交换，根据通信效率和数据量的不同，可分为：
- **低级通信**：主要用于传递控制信息（如[[4.同步与互斥|同步与互斥]]），典型代表为 PV 操作
- **高级通信**：用于高效传输大量数据，主要包括共享传输、消息传递和管道通信、信号等方式

## 共享存储（Shared Memory）

操作系统提供了*共享存储*机制：通过特殊的系统调用，创建一段独立的内存区域，并将其**映射**到多个进程的虚拟空间中，使这些进程都可以直接读/写该区域，是**最快**的进程间数据的传输方式

> [!warning] 互斥问题
> 共享存储必须**由用户解决互斥访问**问题，通常使用信号量等同步机制保障。操作系统仅负责分配共享区并完成地址映射，并**不会自动提供同步保护**

## 消息传递（Message Passing）

进程间的数据交换以格式化的信息（Message）为单位，进程通过操作系统提供的 `send`/`receive` 原语交换数据，这种通信的底层细节封装在内核中，对用户**透明**；是目前应用最广泛的进程间通信机制

| 方式 | 说明 |
|------|------|
| **直接通信** | 发送进程直接将消息发给接收进程：`send(P, msg)`、`receive(Q, msg)` |
| **间接通信** | 通过**信箱**（Mailbox）中转消息：`send(A, msg)`、`receive(A, msg)` |

## 管道通信（Pipe）

管道是连接两个进程（常用于父子进程之间）的**共享文件**（pipe 文件），本质上是一块**固定大小的内核缓冲区**；以**生产者—消费者模式**进行数据交换，一个进程向管道写入数据，另一个进程从管道读取数据，实现**单向的数据流**

### 管道的两种类型

| 类型                          | 说明                                                              | 生命周期              |
| --------------------------- | --------------------------------------------------------------- | ----------------- |
| **匿名管道（Anonymous Pipe）**    | 没有名字，仅存在于内存中。只能用于**具有亲缘关系**的进程间通信（如父子进程），因为只有父子进程才能共享管道文件描述符    | 随进程结束而销毁          |
| **命名管道（Named Pipe / FIFO）** | 在文件系统中有一个**文件名**（作为路径入口），任何知道该路径的进程都可以通过文件名打开管道进行通信，**不要求亲缘关系** | 持久存在于文件系统中，直到显式删除 |

### 工作机制

1. 进程通过 `pipe()` 系统调用创建管道，内核返回两个文件描述符：`fd[0]`（读端）和 `fd[1]`（写端）
2. **写进程**通过 `write(fd[1], buf, n)` 将数据写入管道缓冲区
3. **读进程**通过 `read(fd[0], buf, n)` 从管道缓冲区读取数据
4. 数据**被读出后立即从管道中消失**，管道是流式、不可重复读取的

### 关键特性

| 特性             | 说明                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------- |
| **半双工通信**      | 数据只能**单向流动**，若要双向通信，需创建两个管道                                                                       |
| **先进先出（FIFO）** | 数据按写入顺序被读出，不能随机访问                                                                                 |
| **流式传输**       | 数据以字节流形式传输，没有消息边界（不像消息传递那样有固定的消息格式），且读取数据是**一次性**操作，数据一旦被读出，其占用的缓冲区空间及被释放                         |
| **互斥与同步**      | 内核自动保证读写操作的**原子性**，任意时刻只允许一个进程对管道进行读或写操作                                                          |
| **阻塞规则**       | 管道满时 `write()` 阻塞；管道空时 `read()` 阻塞；当所有写端关闭时，`read()` 返回 0（EOF）；当所有读端关闭时，`write()` 触发 `SIGPIPE` 信号 |
| **容量有限**       | 管道缓冲区大小固定（Linux 默认 64KB，可通过 `fcntl` 调整），使用**内核内的环形缓冲区**，不能无限写入                                    |
| **判断对方存在**     | 管道必须感知对端进程是否存活：读端关闭 → 写端收到 `SIGPIPE`；写端关闭 → 读端读到 EOF                                              |

> [!example] 典型应用：Shell 管道
> ```bash
> ls -l | grep ".md" | wc -l
> ```
> 上述命令创建了两个匿名管道：`ls` 的输出连接 `grep` 的输入，`grep` 的输出连接 `wc` 的输入，三个进程协同完成数据流水线处理。


## 信号（Signal）

**信号**是操作系统提供的**异步通信机制**，用于通知进程发生了某种事件；信号是进程间通信中**唯一的一种异步方式**，接收进程无需主动等待，信号可以在任何时刻到达。
在进程 PCB 中，用一个 $n$ 位向量记录该进程当前**待处理的信号**，还维护一个 $n$ 位向量，用于记录**被阻塞（屏蔽）** 的信号，某位为 1 时，表示对应的信号**不会被立即处理**，而是暂存起来，直到解除阻塞

> [!important] 信号的特点
> - **异步性**：信号的产生与接收是异步的，进程无法预测信号何时到达
> - **少量信息**：信号本身只携带信号编号，不传输大量数据（传递的数据仅限于一个整数编号和 `siginfo_t` 中的少量附加信息）
> - **内核介入**：信号的发送、暂存、投递均由内核完成

### 常见的信号类型

信号的编号和名称在 `<signal.h>` 中定义（Linux 中有 1~64 号信号，其中 1~31 为普通信号，34~64 为实时信号）：

| 信号                    | 编号      | 含义                   | 默认行为                |
| --------------------- | ------- | -------------------- | ------------------- |
| **SIGINT**            | 2       | 来自键盘的中断（`Ctrl+C`）    | 终止进程                |
| **SIGQUIT**           | 3       | 来自键盘的退出（`Ctrl+\`）    | 终止进程 + 核心转储         |
| **SIGKILL**           | 9       | 强制终止进程               | 终止进程（**不可捕获、不可忽略**） |
| **SIGTERM**           | 15      | 终止信号（`kill` 命令的默认信号） | 终止进程                |
| **SIGSTOP**           | 19      | 暂停进程执行               | 暂停进程（**不可捕获、不可忽略**） |
| **SIGCHLD**           | 17      | 子进程状态变化（停止或终止）       | 忽略                  |
| **SIGALRM**           | 14      | `alarm()` 定时器到期      | 终止进程                |
| **SIGPIPE**           | 13      | 向无读端的管道写入数据          | 终止进程                |
| **SIGSEGV**           | 11      | 无效的内存引用（段错误）         | 终止进程 + 核心转储         |
| **SIGUSR1 / SIGUSR2** | 10 / 12 | 用户自定义信号，供应用程序自行使用    | 终止进程                |

### 信号的产生方式

| 方式       | 示例                                                            |
| -------- | ------------------------------------------------------------- |
| **硬件产生** | 除零错误 → `SIGFPE`；非法内存访问 → `SIGSEGV`                            |
| **终端按键** | `Ctrl+C` → `SIGINT`；`Ctrl+\` → `SIGQUIT`；`Ctrl+Z` → `SIGTSTP` |
| **系统调用** | `kill(pid, sig)` 向指定进程发送信号；`alarm(n)` 在 n 秒后向自己发送 `SIGALRM`   |
| **软件事件** | 管道读端关闭时向写端发 `SIGPIPE`；子进程终止时向父进程发 `SIGCHLD`                   |
### 信号的处理方式

当操作系统**从内核态返回用户态**时，会检查当前进程是否存在**未被阻塞的待处理信号**，若存在，则立即处理其中一个信号（通常优先处理序号较小的信号）

| 处理方式              | 说明                                                             |
| ----------------- | -------------------------------------------------------------- |
| **默认处理（Default）** | 执行系统预设动作：终止进程、忽略信号、终止并核心转储、暂停/继续进程                             |
| **忽略（Ignore）**    | 进程收到信号后不做任何处理，信号被丢弃（`SIGKILL` 和 `SIGSTOP` **不可忽略**）            |
| **自定义捕获（Catch）**  | 进程注册一个**信号处理函数**（Signal Handler），收到信号时自动跳转执行该函数，处理完后返回被中断处继续执行 |
>[!warning] 自定义捕获
>**不是所有**信号都能被用户自定义处理的，其只能执行操作系统默认的处理程序

信号处理程序结束后，通常会返回到原程序被中断的位置，继续执行后序指令
