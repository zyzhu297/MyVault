# 基本概念

在多道程序环境下，多个进程并发执行，它们可能**共享某些资源**或**相互协作**完成同一任务

## 临界资源（Critical Resource）

**临界资源**是指一次仅允许**一个**进程访问的共享资源；若多个进程同时访问临界资源，可能导致数据不一致或执行结果错误，所以对临界资源的访问必须以**互斥**的方式进行

常见的临界资源包括：
- **硬件资源**：打印机、磁带机等独占型 I/O 设备（磁盘虽为共享设备，但其访问由
- **软件资源**：共享变量、共享文件、共享数据结构（如队列、栈）等

> [!warning] 磁盘
> 磁盘虽为共享设备，但其访问由操作系统统一协调，不构成临界资源

进程访问临界资源必须遵守协议，访问过程被划分为四个部分：

| 概念                         | 定义                                                     |
| -------------------------- | ------------------------------------------------------ |
| **进入区（Entry Section）**     | 临界区之前，需检查是否可以，若可以，则进入并设置"正在访问"标志，以阻止其他进程同时进入临界区的**代码** |
| **临界区（Critical Section）**  | 进程中**真正访问临界资源**的那段**代码**                               |
| **退出区（Exit Section）**      | 临界区之后，清除"正在访问"标志并释放资源的**代码**                           |
| **剩余区（Remainder Section）** | 进程中**代码**除进入区、临界区和退出区之外的部分                             |

```c
while (true){
	entry section;//进入区
	critical section;//临界区
	exit section;//退出区
	remainder section;//剩余区
}
```

> [!important] 理解要点
> - **临界资源**关注的是"**是什么**"（被保护的对象）
> - **临界区**关注的是"**哪段代码**"（访问临界资源的代码区域）
> - 一个进程的临界区中可能访问多个不同的临界资源
> - 当进程进入临界区但尚未离开时，允许被迫**进入阻塞状态**，但是在该进程退出临界区之前不允许其他进程进入该临界区

## 同步（Synchronization；直接制约关系）

**同步**是指多个进程/线程在执行过程中存在**时序关系**，必须在某些关键点产生**相互等待或通知**的**制约关系**，以保证协作的正确性，同步关系源于进程之间的**相互合作**

- 同步解决的核心问题：**"按正确顺序执行"**
- 同步关注的是进程间的**协作关系**与**执行顺序**
## 互斥（Mutual Exclusion；间接制约关系）

**互斥**是指多个进程不能**同时**进入各自的临界区访问同一**临界资源**。互斥保证了临界资源访问的**排他性**

> [!important] 同步与互斥的关系
>|          | 互斥                    | 同步                  |
>| -------- | --------------------- | ------------------- |
>| **核心目标** | **排他性**，同一时刻只有一个进程 | **时序性**，按约定的顺序执行 |
>| **关注点**  | 临界资源的访问权限             | 进程间的协作顺序            |
>| **关系**   | 互斥是同步的特殊形式            | 同步涵盖了互斥，范围更广        |
>| **典型场景** | 多进程写同一文件              | 前驱关系、生产者—消费者        |

## 同步机制应遵循的准则

| 准则       | 含义                                         |
| -------- | ------------------------------------------ |
| **空闲让进** | 临界资源空闲时，应允许一个请求进入临界区的进程立即进入                |
| **忙则等待** | 临界资源正被访问时，其他试图进入临界区的进程必须等待                 |
| **有限等待** | 对请求访问临界区的进程，应在**有限时间**内允许其进入，**避免饥饿**      |
| **让权等待** | 当进程不能进入临界区时，应**立即释放 CPU**（转为阻塞态），避免**忙等待** |
> [!warning] 让权等待
> 原则上应该遵循，但非必须，而其他三条准则都必须遵循
# 互斥的实现方法

互斥的实现方法可以分为：**软件方法**、**硬件方法**和**操作系统提供的机制**（信号量、管程）

## 软件实现方法

软件方法由应用程序员在**用户态**编写代码实现互斥，不依赖硬件或操作系统支持，若已有进程在临界区，则其他进程在进入区通过**循环检查**的方式等待

### 单标志法

设置一个公共整型变量 `turn`，指示**允许进入临界区的进程编号**。

```c
进程P0:                     进程P1:                      
while (turn != 0);          while (turn != 1);           // 进入区
critical section;           critical section;            // 临界区
turn = 1;                   turn = 0;                    // 退出区
remainder section;          remaider section;            // 剩余区
```

> [!warning] 严重缺陷
> - **违背"空闲让进"**：若 `turn = 0` 但 P0 不访问临界区，P1 即使空闲也无法进入（因为 `turn = 0` 不满足 `turn == 1`）
> - 两个进程必须**交替**进入临界区，若一方不进入，另一方被永久阻塞

### 双标志法（先检查后上锁）

每个进程设置一个标志 `flag[i]`，表示该进程**想要进入**临界区。先检查对方标志，再设置自己的标志。

```c
进程P0:                     进程P1:                      
while (flag[1]);            while (flag[0]);             // 进入区（检查对方）
flag[0] = true;             flag[1] = true;              // 进入区（上锁）
critical section;           critical section;            // 临界区
flag[0] = false;            flag[1] = false;             // 退出区（解锁）
remainder section;          remainder section;           // 剩余区
```

> [!warning] 严重缺陷
> - 在**检查对方标志**和**设置自己标志**之间存在间隙，可能被中断，因为操作不是**原子**的
> - 若 P0 检查 `flag[1]` 为 false → **中断** → P1 检查 `flag[0]` 为 false → P0 设 `flag[0] = true` → P1 设 `flag[1] = true` → **两者同时进入临界区**
> - **违背"忙则等待"**

### 双标志法（先上锁后检查）

先设置自己的标志，再检查对方的标志。

```c
进程P0:                     进程P1:                      
flag[0] = true;             flag[1] = true;              // 进入区（先上锁）
while (flag[1]);            while (flag[0]);             // 进入区（后检查）
critical section;           critical section;            // 临界区
flag[0] = false;            flag[1] = false;             // 退出区（解锁）
remainder section;          remainder section;           // 剩余区
```

> [!warning] 严重缺陷
> - 若 P0 设 `flag[0] = true` → **中断** → P1 设 `flag[1] = true` → P0 检查 `flag[1]` 为 true → P1 检查 `flag[0]` 为 true → **双方互相等待，死锁**
> - **违背"空闲让进"和"有限等待"**

### Peterson 算法

**Peterson 算法**是解决两个进程互斥访问临界区的经典软件方法，它综合了**单标志法**和**双标志法先上锁后检查**的思想，同时使用 `flag[2]` 和 `turn` 变量。

```c
进程P0:                     进程P1:                      
flag[0] = true;             flag[1] = true;             // 进入区（表示想进入）
turn = 1;                   turn = 0;                   // 进入区（谦让对方）
while (flag[1] && turn==1); while (flag[0] && turn==0); // 进入区（检查等待）
critical section;           critical section;           // 临界区
flag[0] = false;            flag[1] = false;            // 退出区（放弃标志）
remainder section;          remainder section;          // 剩余区
```

> [!important] Peterson 算法的正确性分析

| 准则       | 满足情况                                                     |
| -------- | -------------------------------------------------------- |
| **空闲让进** | ✅ 满足。若对方不想进入（`flag[j] == false`），则 `while` 条件为假，Pi 直接进入  |
| **忙则等待** | ✅ 满足。若双方同时想进入，`turn` 变量保证了只有一方能通过 `while` 循环             |
| **有限等待** | ✅ 满足。对方退出后将 `flag[j]` 置为 false，等待方立即可以进入，不会无限等待，**不会饥饿** |
| **让权等待** | ❌ **不满足**。等待的进程在 `while` 循环中持续占用 CPU（**忙等待**）            |

> [!note] 关键设计思想
> - `flag[i]` 表示进程 Pi 想要进入临界区
> - `turn = j` 的"**谦让**"机制：表示轮到 Pj 进入临界区
> - `flag[i] = true` + `turn = j`：既表达了自己想进入的意愿，又主动礼让对方

### Dekker 算法（了解）

Dekker 算法是更早的两进程互斥算法，实现较复杂，考试不作要求，但了解其存在即可。

## 硬件实现方法

软件方法复杂且容易出错，现代计算机提供**硬件指令**来简化互斥实现，能够以**原子**的方式对一个字进行监测和修改，或是交换两个字的内容

### 中断屏蔽（关中断）

进入临界区前**关闭中断**，退出临界区后**开启中断**。

```c
关中断;
临界区;//（此时不会被中断，不会发生进程切换）
开中断;
```

> [!warning] 缺点
> - 关中断期间无法响应外部事件，**限制并发能力**，影响系统整体性能
> - **不适用于多 CPU 系统**：关中断只影响本 CPU，其他 CPU 仍可访问临界区
> - **用户态程序不能使用**关中断指令（特权指令），只能在内核中短暂使用
> - 若临界区执行时间过长或关中断后未及时开中断，可能导致系统死锁或崩溃

### 硬件指令方法

#### Test-and-Set 指令（TS 指令 / TSL 指令）

由硬件提供的**原子操作**，读出指定标志的**旧值**并设置为 true，整个过程不可被中断
TS 指令在执行时会向系统总线发送 **LOCK# 信号**，锁定内存总线，阻止其他 CPU 在此时访问同一块内存，直到指令执行完毕才释放总线。

```c
// 原子操作（由硬件保证，整个过程不可被中断）
bool TestAndSet(bool *lock) {
    bool old = *lock;       // 读取 lock 的当前值，存入局部变量 old
    *lock = true;           // 将 lock 强制设为 true（上锁）
    return old;             // 返回 old 值（上锁前的状态——"现场快照"）
}

// 使用 TS 指令实现互斥
while (TestAndSet(&lock));  // 进入区：若 lock 为 true 则忙等待
critical section;           // 临界区
lock = false;               // 退出区：释放锁
remainder section;          // 剩余区：其他代码
```

> [!note] TS 指令功能解释
> **核心变量：**
> - `lock`：**共享全局布尔变量**，所有进程可见，初值为 `false` 表示临界资源空闲，`true` 表示已被占用
> - `old`：**函数局部变量**，保存 `lock` 在修改前的旧值，作为返回值传递给调用者——相当于"**上锁前的现场快照**"
> 
> **互斥实现逻辑：**
> - 若 `lock` 原为 `false`（空闲）→ 返回 `false`，同时将 `lock` 置为 `true` → **进程成功获取锁，进入临界区**
> - 若 `lock` 原为 `true`（已被占用）→ 返回 `true`，`lock` 保持 `true` → **进程在 `while` 中循环等待（自旋）**
> - 退出临界区时手动将 `lock` 置为 `false`，**释放锁**，使其他进程的 `TestAndSet` 能够返回 `false`
> 
> **关键保证**：`TestAndSet` 的"**读—改—写**"全过程由**硬件保证原子性**，即这三步要么全部执行、要么都不执行，执行期间不会被中断或抢占。这从根本上避免"检查—上锁"之间的**间隙问题**。

> [!important] TS 指令优缺点
> 
> **优点：**
> - **实现简单**：只需一条硬件指令 + 一个 `while` 循环即可实现互斥，逻辑清晰
> - **适用多 CPU / 多核系统**：原子性由硬件保证，多个 CPU 同时执行 TS 指令也不会冲突；相比之下，**关中断仅对当前 CPU 有效**
> - **适用于任意数量进程**：不像 Peterson 算法只能用于两个进程，TS 指令对进程数无限制
> - **不会出现"检查—上锁"间隙**：软件方法（双标志法）的致命缺陷被硬件原子操作彻底解决
> 
> **缺点：**
> - **忙等待（自旋锁），违背"让权等待"**：等待进程在 `while` 循环中持续占用 CPU 空转，浪费处理器时间
> - **可能饥饿**：等待进程没有排队机制，某些进程理论上可能长期无法获得锁
> - **临界区较长时效率极低**：若临界区执行时间远大于进程切换开销，自旋等待白白消耗 CPU
> 
> **适用场景**：临界区极短（几条指令内完成），自旋等待的开销小于进程上下文切换的开销。典型例子：操作系统内核中保护极短小的共享数据结构的访问。

#### Swap 指令（Exchange 指令 / XCHG 指令）

由硬件提供的另一个原子操作，交换两个字的内容，通过为每个临界资源设置一个**共享布尔变量** `lock` ，且在每个进程中定义一个**局部布尔变量** `key`，每次尝试进入临界区前将其初始化为 `true`

```c
// 原子操作（由硬件保证）
void Swap(bool *a, bool *b) {
    bool temp = *a;
    *a = *b;
    *b = temp;
}

// 使用 Swap 指令实现互斥
bool key = true;
while (key) {                // 进入区
    Swap(&lock, &key);       // 若lock为false则key变false；lock变true，退出循环
}
critical section;            // 临界区
lock = false;                // 退出区
```

> [!note] Swap 指令功能解释
> **核心变量：**
> - `lock`：**共享全局变量**，所有进程可见，初值为 `false` 表示临界资源空闲，`true` 表示已被占用
> - `key`：**局部私有变量**，每个进程独立持有，进入临界区前初始化为 `true`
> 
> **互斥实现逻辑：**
> - 若 `lock` 原为 `false`（空闲）→ Swap 后 `key` 变为 `false`、`lock` 变为 `true` → `while(key)` 条件为假，**退出循环，进入临界区**
> - 若 `lock` 原为 `true`（已被占用）→ Swap 后 `key` 仍为 `true`、`lock` 仍为 `true`（两者交换的均为 `true`）→ `while(key)` 条件为真，**继续循环等待（自旋）**
> - 退出临界区时手动将 `lock` 置为 `false`，**释放锁**，使其他进程下次 Swap 时能够获得 `false`
> 
> **关键理解**："检查"和"上锁"被封装在一条**原子指令**中——不会像双标志法那样出现"检查完对方还没上锁就被中断"的间隙。

> [!important] Swap 指令优缺点
> 
> **优点：**
> - **实现简单**：只需一条硬件 Swap 指令 + 一个 `while` 循环即可实现互斥
> - **适用多 CPU / 多核系统**：原子性由硬件保证，所有 CPU 上的 Swap 操作被硬件串行化
> - **适用于任意数量进程**：不受进程数量限制，每个进程只需各自维护一个局部 `key` 变量
> 
> **缺点：**
> - **忙等待（自旋锁），违背"让权等待"**：与 TS 指令相同，等待进程持续占用 CPU
> - **需要额外局部变量**：每个进程必须维护一个私有的 `key` 变量，相比 TS 指令多了一个变量的开销
> - **可能饥饿**：无排队机制，等待顺序不确定
> - **临界区较长时效率极低**：自旋空转浪费 CPU 时间
> 
> **Swap vs TS 对比：**
> 
> | 维度 | TS 指令 | Swap 指令 |
> |------|---------|-----------|
> | 操作方式 | 读旧值 → 写 true → 返回旧值 | 交换两个变量的值 |
> | 额外变量 | 无（直接返回值） | 需要局部 `key` 变量 |
> | 循环条件 | `while (TestAndSet(&lock))` | `while (key)` |
> | 逻辑本质 | 完全相同——原子的"检查并设置" |
> 
> **适用场景**：与 TS 指令相同——临界区极短时，自旋开销小于进程切换开销。

### 硬件方法的优缺点总结

|               | 优点            | 缺点                   |
| ------------- | ------------- | -------------------- |
| **中断屏蔽**      | 实现简单          | 仅适用于单 CPU 内核态；影响系统响应 |
| **TS / Swap** | 适用于多 CPU；逻辑简单 | 忙等待，浪费 CPU；可能饥饿      |

# 互斥锁（mutex lock）

**互斥锁**（也称*自旋锁*）是操作系统提供的最简单的互斥工具，本质上是一个**布尔变量** `available`，表示临界资源是否可用
互斥锁只提供两个**原子**操作：
- **`acquire()`**（获取锁 / 上锁）：若锁空闲则占用；否则阻塞或自旋等待
- **`release()`**（释放锁 / 解锁）：将锁置为空闲，唤醒等待者（如有）

```c
// 互斥锁的基本结构
bool available = true;  // true = 空闲可用

// 获取锁
void acquire() {
    while (!available);      // 忙等待（自旋锁实现）
    available = false;       // 上锁
}

// 释放锁
void release() {
    available = true;        // 解锁
}
```

> [!warning] 注意
> 上面的 `acquire()` 在 **单 CPU 且无抢占** 的环境下，若进程在 `while` 中被切换，其他进程无法执行 `release()`，会导致**永久等待**。因此实际系统中 `acquire()` 必须依赖硬件原子指令（如 TS、Swap）来实现。

> [!note] 理解
> 互斥锁是硬件指令的**上层封装**——将 TS/Swap 等底层原子操作包装成 `acquire()` / `release()` 这对简洁的接口，供程序员直接调用。

>[!success] **优点**
> - **实现高效**：基于硬件原子指令，，且在等待期间没有上下文切换，开销极低
> - **接口极简**：只有 `acquire()` 和 `release()` 两个操作，比信号量的 P/V 更直观
> - **语义单一**：专用于互斥，概念清晰，不容易像信号量那样因误用而死锁

> [!failure] **缺点**
> - **仅能解决互斥，不能解决同步**：互斥锁只有 0/1 两个状态，无法像信号量那样维护计数器来实现前驱关系、生产者-消费者等同步问题
> - **自旋锁违背"让权等待"**：**忙等待**浪费 CPU
> - **可能死锁**：若一个进程获取锁后未释放（如代码异常退出），其他进程永远等待

# 信号量（Semaphore）

**信号量**是操作系统提供的用于解决**互斥**和**同步**问题的**核心工具**，信号量本质上是一个**受保护的整型变量**，只能通过两个标准的**低级的进程通信原子操作**（原语）来访问：
- **P 操作**（Proberen，荷兰语"测试"）：申请资源 / 等待信号，也称为 `wait()`
- **V 操作**（Verhogen，荷兰语"增加"）：释放资源 / 发送信号，也称为 `signal()`

## 互斥锁 vs 信号量 对比

| 维度       | 互斥锁（mutex）                | 信号量（semaphore）              |
| -------- | ------------------------- | --------------------------- |
| **状态**   | 二值（locked / unlocked）     | 整型（任意非负整数）                  |
| **功能**   | **仅互斥**                   | 互斥 **+ 同步**                 |
| **操作**   | `acquire()` / `release()` | `wait()`(P) / `signal()`(V) |
| **复杂度**  | 简单                        | 较复杂，容易出错                    |
| **使用场景** | 保护临界区（单一互斥需求）             | 生产者-消费者、前驱关系等复杂同步           |
| **比喻**   | **门锁**：要么锁着要么开着           | **计数器**：记录可用资源的数量           |

> [!tip] 总结
> **互斥锁是功能精简的"二值信号量"**（`value` 只为 0 或 1），专为互斥而生，简单高效但无法处理同步问题。需要同步时仍需使用信号量。

## 整型信号量

最简单的信号量形式，用一个整型变量 `S` 表示资源数量，P/V 操作如下：

```c
int S = 1;  // 初始资源数为 1（互斥信号量）

// P 操作（原子）
void wait(int S) {
    while (S <= 0);  // 忙等待！
    S--;
}

// V 操作（原子）
void signal(int S) {
    S++;
}
```

> [!warning] 整型信号量的致命缺陷
> - `while (S <= 0)` 是**忙等待**，**违背"让权等待"原则**

## 记录型信号量

为解决整型信号量的"忙等待"问题，记录型信号量引入了一个**等待队列**，让无法获取资源的进程**主动阻塞**

```c
typedef struct {
    int value;           // 资源数量（剩余可用的资源数目）
    Queue *wait_queue;   // 等待该信号量的进程阻塞队列
} semaphore;

// P 操作（原子操作）
void wait(semaphore *S) {
    S->value--;
    if (S->value < 0) {          // 资源不够！
        block(S->wait_queue);    // 将当前进程加入等待队列，并阻塞该进程
    }
}

// V 操作（原子操作）
void signal(semaphore *S) {
    S->value++;
    if (S->value <= 0) {         // 还有进程在等待该资源
        wakeup(S->wait_queue);   // 从等待队列中唤醒一个进程，将其转为就绪态
    }
}
```

> [!important] 关键理解：`S->value` 的含义
> 
> | `S->value` 的值 | 含义 |
> |-----------------|------|
> | `S->value > 0` | 表示还有 `value` 个**可用资源**，没有进程在等待 |
> | `S->value = 0` | 表示资源刚好用完，但**没有进程在等待** |
> | `S->value < 0` | 表示资源已用完，且有 \|value\| 个进程正**在等待队列中阻塞** |

> [!tip] 关键
> - `wait()` 中先 `S->value--` 再判断；`signal()` 中先 `S->value++` 再判断
> - `wait()` 的阻塞条件是 `S->value < 0`（已经 -- 之后）
> - `signal()` 的唤醒条件是 `S->value <= 0`（已经 ++ 之后）
> - P 操作**可能阻塞**调用者；V 操作**不会阻塞**调用者，但可能唤醒等待者

## 信号量的分类

| 类型        | 初始值              | 含义             | 用途          |
| --------- | ---------------- | -------------- | ----------- |
| **互斥信号量** | `mutex = 1`      | 一次只允许一个进程访问    | 实现**临界区互斥** |
| **同步信号量** | `S = 0`（**或其他**） | 控制进程间的**执行顺序** | 实现**前驱关系**等 |

## 信号量的应用

### 利用信号量实现互斥

```c
进程P0:                     进程P1:                      
wait(&mutex);               wait(&mutex);                // 进入区（P操作）
critical section;           critical section;            // 临界区
signal(&mutex);             signal(&mutex);              // 退出区（V操作）
remainder section;          remainder section;           // 剩余区
```

> [!note] 不同 `mutex` 取值的含义
> 互斥信号量 `mutex` 初值为 **1**，经过 P/V 操作后，不同取值对应不同的系统状态：
> 
> | `mutex` 的值 | 临界区状态 | 等待队列 |
> |-------------|-----------|---------|
> | **1** | 空闲，无进程在临界区 | 空 |
> | **0** | 有 1 个进程在临界区 | 空（无人等待） |
> | **-1** | 有 1 个进程在临界区 | 有 1 个进程等待 |

> [!warning] 互斥信号量的使用原则
> - 不同的临界资源应配置**独立**的互斥信号量，避免相互干扰
> - 互斥使用的临界资源初值只能为 1
> - 若系统中存在多个**同类资源**（不是**临界资源**），应将信号量初值设为**资源总数**
> - 在互斥问题中，P、V 操作必须**夹紧**使用临界资源的那个行为
> - 对**同一个信号量**的 P/V 操作必须**成对出现**，且**在同一进程中**
> - P 操作遗漏 → 无法保证互斥；V 操作遗漏 → 资源永不释放，其他进程永远等待

### 利用信号量实现同步

设 P1 中的语句 S1 必须在 P2 中的语句 S2 **之前**执行：

```c
进程P1:                     进程P2:                      
S1;                         wait(&S);                  // P2等待P1完成S1
signal(&S);                 S2;                        // P1通知P2：S1已完成
```

> [!note] 不同 `S` 取值的含义
> 同步信号量 `S` 初值为 **0**，经过 P/V 操作后，不同取值对应不同的系统状态：
> 
> | `S` 的值 | 前驱事件状态 | 等待队列 |
> |---------|------------|---------|
> | **1** | 前驱事件已发生（signal 已执行），后续 wait 可直接通过 | 空 |
> | **0** | 前驱事件尚未发生，也无进程在等待（初始状态） | 空 |
> | **-1** | 前驱事件尚未发生 | 有 1 个进程在等待 |
> | **-n**（n > 0） | 前驱事件尚未发生 | 有 n 个进程在等待 |

### 利用信号量实现前驱关系

> [!important] 前驱关系的一般规则
> - 对**一对前驱关系** `S1 → S2`（S1 必须在 S2 之前执行）：
>   - 在 S1 **之后**执行 `signal(S)`
>   - 在 S2 **之前**执行 `wait(S)`
>   - 信号量 `S` 初始值为 **0**
> - 多个前驱关系：需要多个同步信号量，每个前驱关系一个

对于如下的前驱图

共 7 个前驱关系，需要为每个前驱关系设定共 7 个同步信号量，每个初值为 0：

```c
进程P1:              进程P2:              进程P3:
S1;                  wait(&a);            wait(&b);
signal(&a);          S2;                  S3;
signal(&b);          signal(&c);          signal(&e);
                     signal(&d);

进程P4:              进程P5:              进程P6:
wait(&c);            wait(&d);            wait(&e);
S4;                  S5;                  wait(&f);
signal(&f);          signal(&g);          wait(&g);
                                          S6;
```

> [!note] 信号量说明
> - `a`：S1 → S2，`b`：S1 → S3
> - `c`：S2 → S4，`d`：S2 → S5
> - `e`：S3 → S6
> - `f`：S4 → S6，`g`：S5 → S6
> - 每个信号量初值为 **0**，共需要 **7** 个同步信号量

# 分析同步与互斥问题的方法步骤

解决同步与互斥问题时，按照以下四步系统性地分析和编写代码：

## 步骤一：关系分析（找约束）

1. **找互斥关系**：哪些资源是临界资源（一次只能被一个进程访问）？→ 需要互斥信号量
2. **找同步关系**：进程间是否存在时序依赖（"A 必须在 B 之前执行"）？→ 需要同步信号量
3. 画**前驱图**辅助理解：将同步关系可视化，每个箭头就是一个同步信号量

## 步骤二：确定信号量

1. **互斥信号量** `mutex`：初值为 **1**（同一时刻只允许一个进程进入临界区）
2. **同步信号量**：初值为 **0**（表示"事件尚未发生"），**每个前驱关系一个**
3. 资源计数型信号量（如生产者-消费者中的 `empty`、`full`）：初值为**缓冲区容量**或 **0**

> [!tip] 信号量命名建议
> 互斥信号量用 `mutex`；同步信号量用有意义的名字（如 `empty`、`full`、`a`、`b` 等），便于检查和验证。

## 步骤三：写代码框架

对每个进程，按以下顺序编写：

1. ⭐⭐**同步 P 在先，互斥 P 在后**：先 `wait(同步)` 检测资源是否足够，再 `wait(mutex)` 申请进入临界区。**顺序绝不能颠倒**，否则可能死锁
2. ⭐在互斥关系中，P、V 操作必须**夹紧**使用临界资源的那个行为
3. **V 操作顺序无限制**：V 操作不会阻塞，顺序自由，但通常保持与 P 操作对称
4. **P/V 成对出现**：在一个进程中，对每个信号量的 P 操作必须对应一个 V 操作

```c
// 标准模板
void Process() {
    while (true) {
        wait(&同步信号量);   //  先检测同步条件
        wait(&mutex);       //  再申请互斥访问
        // 临界区操作
        signal(&mutex);     //  释放互斥
        signal(&同步信号量); //  通知其他进程
    }
}
```

## 步骤四：检查验证

写完代码后逐一检查四项准则：

| 检查项 | 验证方法 |
|--------|---------|
| **P/V 是否配对** | 每个信号量的 P 和 V 在同一个进程中是否成对出现 |
| **同步 P 是否在互斥 P 之前** | 若颠倒在满/空时会导致死锁 |
| **初始值是否正确** | `mutex = 1`，同步信号量 `= 0` |
| **是否可能死锁** | 检查是否存在"占有并等待"的循环 |

# 经典同步问题

## 生产者—消费者问题（Producer-Consumer Problem）

### 问题描述

- 一组**生产者**进程生产产品放入缓冲区，一组**消费者**进程从缓冲区取出产品消费
- 缓冲区大小为 `n`（有限缓冲）
- 生产者不能向**满**缓冲区放入产品；消费者不能从**空**缓冲区取出产品
- 生产者和消费者对缓冲区的访问必须**互斥**

### 同步与互斥关系分析

按照四步法分析生产者—消费者问题：

**1. 互斥关系：**
- 缓冲区是临界资源，多个生产者之间、多个消费者之间、生产者与消费者之间对缓冲区的访问必须互斥
- → 需要 **1 个互斥信号量** `mutex`

**2. 同步关系（两个）：**
- 生产者不能向满缓冲区放入产品 → 需要有空位才能生产 → 同步信号量 `empty`（初值 `n`，一开始全是空位）
- 消费者不能从空缓冲区取出产品 → 需要有产品才能消费 → 同步信号量 `full`（初值 `0`，一开始没有产品）

> [!note] 同步关系口诀
> **"生产者关心空位，消费者关心产品"**——`empty` 和 `full` 恰好互补，`empty + full = n`（缓冲区总容量）

### 信号量设计

| 信号量 | 类型 | 初值 | 含义 |
|--------|------|------|------|
| `mutex` | 互斥信号量 | `1` | 保证对缓冲区的互斥访问 |
| `empty` | 同步信号量 | `n` | 缓冲区中**空位**的数量（可生产的空间） |
| `full` | 同步信号量 | `0` | 缓冲区中**产品**的数量（可消费的产品） |

### 代码实现

```c
semaphore mutex = 1;   // 互斥访问缓冲区
semaphore empty = n;   // 空缓冲区的数量
semaphore full  = 0;   // 满缓冲区的数量

// 生产者
void Producer() {
    while (true) {
        生产一个产品;
        P(&empty);      // P1: 申请一个空位
        P(&mutex);      // P2: 申请访问缓冲区
        将产品放入缓冲区;
        V(&mutex);    // V2: 释放缓冲区
        V(&full);     // V1: 增加一个产品
    }
}

// 消费者
void Consumer() {
    while (true) {
        P(&full);       // P3: 申请一个产品
        P(&mutex);      // P4: 申请访问缓冲区
        从缓冲区取出产品;
        V(&mutex);    // V4: 释放缓冲区
        V(&empty);    // V3: 增加一个空位
        消费产品;
    }
}
```

> [!important] P/V 操作的顺序问题
> 
> **互斥 P 操作必须在同步 P 操作之后！**
> 
> 如果交换生产者中的 `wait(empty)` 和 `wait(mutex)`：
> ```c
> P(&mutex);   // 先占坑
> P(&empty);   // 若缓冲区满，在此阻塞！
> // 死锁！因为 mutex 已被占用，消费者无法进入
> ```
> 
> **正确顺序**：先对**资源信号量**做 P 操作（检测资源是否足够），再做**互斥 P 操作**
> 
> **V 操作的顺序没有限制**（V 操作不会阻塞），但通常保持与 P 操作对称

### 单生产者—单消费者

**问题描述**：只有一个生产者进程和一个消费者进程共享一个大小为 `n` 的缓冲区。

#### 同步与互斥关系分析

- **互斥关系**：仅有单生产者和单消费者，两者不会同时访问缓冲区的同一位置（一个只放、一个只取），**不需要互斥信号量**
- **同步关系（两个）**：与多生产者—消费者相同
  - 生产者需要空位 → `empty`（初值 `n`）
  - 消费者需要产品 → `full`（初值 `0`）
> [!note] 不需要 `mutex` 的原因
> 单生产者和单消费者分别操作缓冲区的**不同位置**（`in` 和 `out` 指针各自由一方独占修改），不存在对同一共享变量的并发写入，因此**天然互斥**。

### 多生产者—多消费者

**问题描述**：多个生产者进程和多个消费者进程共享一个大小为 `n` 的缓冲区。

#### 同步与互斥关系分析

- **互斥关系**：多个生产者之间会竞争 `in` 指针，多个消费者之间会竞争 `out` 指针，生产者与消费者可能同时访问缓冲区 → **需要互斥信号量 `mutex`**（初值 `1`）
- **同步关系（两个）**：与单生产者—消费者完全相同
  - 生产者需要空位 → `empty`（初值 `n`）
  - 消费者需要产品 → `full`（初值 `0`）
> [!note] 为什么必须加 `mutex`
> 多个生产者并发执行 `P(&empty)` 都通过后，会同时向缓冲区放入产品——此时 `in` 指针和缓冲区内容被并发修改，数据可能被覆盖或错乱。`mutex` 保证每次只有一个进程操作缓冲区。

### 单 vs 多 对比

| 维度 | 单生产者—单消费者 | 多生产者—多消费者 |
|------|------------------|-------------------|
| **互斥关系** | ❌ 不需要（天然互斥） | ✅ 需要 `mutex` |
| **同步关系** | 2 个（`empty` + `full`） | 2 个（`empty` + `full`） |
| **信号量总数** | **2** | **3** |
| **P 操作顺序** | 只有同步 P | **同步 P 先，互斥 P 后** |

> [!tip] 小结论
> - **缓冲区大小为 1 时**，即使多生产者—多消费者，`mutex` 也可以省略（因为 `empty` 和 `full` 已天然互斥）
> - 缓冲区大小 ≥ 2 时，多生产/多消费场景下 `mutex` **必须保留**

## 读者—写者问题（Readers-Writers Problem）

### 问题描述

- 一个数据对象（文件、数据库记录等）被多个进程共享
- **读者（Reader）**：只读数据，不修改，允许多个读者**同时**访问
- **写者（Writer）**：写数据（读+写），写者与其他读者/写者**互斥**
- 核心挑战：如何处理**读者优先** vs **写者优先** vs **公平竞争**

### 同步与互斥关系分析

按照四步法分析读者—写者问题：

**1. 互斥关系（两个）：**
- 写者与其他读者/写者之间必须互斥 → 需要互斥信号量 `rw_mutex`（初值 `1`）
- 多个读者对 `read_count`（读者计数器）的修改必须互斥 → 需要互斥信号量 `mutex`（初值 `1`）

**2. 同步关系：**
- 读者—写者问题**没有同步（前驱/时序）关系**，只有互斥关系
- 但写优先策略中的 `w` 信号量实际上起到了同步作用——写者到达后"阻断"后续读者，让写者有机会先执行

> [!note] 读者—写者问题的特殊性
> 与生产者—消费者不同，读者—写者问题中**读者之间不需要互斥**（可以同时读），只有"写者 vs 读者"和"写者 vs 写者"是互斥的。核心在于**如何控制读者与写者的优先顺序**。

### 信号量设计

| 信号量        | 初值  | 含义                     |
| ---------- | --- | ---------------------- |
| `rw_mutex` | `1` | 写者与读者/写者间的互斥信号量        |
| `mutex`    | `1` | 对 `read_count` 变量的互斥访问 |
| `w`        | `1` | 写者优先时，防止读者在写者等待后又插队    |

### 读优先（Reader-Priority）

```c
semaphore rw_mutex = 1;  // 写者与读者/写者间的互斥（数据访问锁）
semaphore mutex = 1;     // 读者计数器 read_count 的互斥锁
int read_count = 0;      // 当前正在读的读者数量

// 读者
void Reader() {
    while (true) {
        P(&mutex);               // 互斥修改 read_count
        read_count++;            // 读者+1
        if (read_count == 1)     // 第一个读者
            P(&rw_mutex);        // 锁门：阻止写者进入
        V(&mutex);               // 释放 read_count 锁

        读数据;

        P(&mutex);               // 互斥修改 read_count
        read_count--;            // 读者-1
        if (read_count == 0)     // 最后一个读者
            V(&rw_mutex);        // 开门：唤醒写者
        V(&mutex);               // 释放 read_count 锁
    }
}

// 写者
void Writer() {
    while (true) {
        P(&rw_mutex);            // 申请独占数据访问
        写数据;
        V(&rw_mutex);            // 释放数据访问锁
    }
}
```

> [!warning] 读优先的缺陷
> - 只要有一个读者在读，后续读者可以不断进入，写者可能**永远无法获得访问权**（写者饥饿）
> - 适用于**读多写少**且对写延迟不敏感的场景

### 写优先（Writer-Priority）

增加信号量 `w`，写者到达后"占位"，阻止后续读者插队：

```c
semaphore rw_mutex = 1;  // 写者与读者/写者间的互斥（数据访问锁）
semaphore mutex = 1;     // 读者计数器 read_count 的互斥锁
semaphore w = 1;         // 写优先控制：写者到达后占位，阻断后续读者插队
int read_count = 0;      // 当前正在读的读者数量

// 读者
void Reader() {
    while (true) {
        P(&w);                   // 在 w 上排队；若写者已占位 w 则在此阻塞
        P(&mutex);               // 互斥修改 read_count
        read_count++;
        if (read_count == 1)
            P(&rw_mutex);        // 第一个读者锁门，阻塞写者
        V(&mutex);               // 释放 read_count 锁
        V(&w);                   // 立刻释放 w，让后续读者/写者可以竞争 w
        读数据;
        P(&mutex);               // 互斥修改 read_count
        read_count--;
        if (read_count == 0)
            V(&rw_mutex);        // 最后一个读者开门，唤醒写者
        V(&mutex);               // 释放 read_count 锁
    }
}

// 写者
void Writer() {
    while (true) {
        P(&w);                   // 先占位 w，阻断后续读者在 w 上排队
        P(&rw_mutex);            // 等待当前读者/写者完成，获取数据访问锁
        写数据;
        V(&rw_mutex);            // 释放数据访问锁
        V(&w);                   // 释放 w，让读者/写者可以再次竞争
    }
}
```

> [!note] 写优先分析
> - 写者到达后，`P(&w)` 阻塞后续读者（但已在读的读者不受影响）
> - 写者等待当前读者完成 → 获得 `rw_mutex` → 执行写操作 → 释放 `w`
> - 减少写者饥饿，但**不能完全消除**（如果读者源源不断，第一个读者永远不退出）

### 公平竞争（读写公平 / 无饥饿）

不偏向任何一方，按到达顺序排队：

```c
semaphore rw_mutex = 1;  // 写者与读者/写者间的互斥（数据访问锁）
semaphore mutex = 1;     // 读者计数器 read_count 的互斥锁
semaphore queue = 1;     // 公平排队信号量：所有进程按到达顺序排队
int read_count = 0;      // 当前正在读的读者数量

// 读者
void Reader() {
    P(&queue);                 // 在 queue 上排队（FIFO）
    P(&mutex);                 // 互斥修改 read_count
    read_count++;
    if (read_count == 1)
        P(&rw_mutex);          // 第一个读者锁门，阻塞写者
    V(&mutex);                 // 释放 read_count 锁
    V(&queue);                 // 立即释放 queue，让下一个进程可以排队
    读数据;
    P(&mutex);                 // 互斥修改 read_count
    read_count--;
    if (read_count == 0)
        V(&rw_mutex);          // 最后一个读者开门，唤醒写者
    V(&mutex);                 // 释放 read_count 锁
}

// 写者
void Writer() {
    P(&queue);                 // 在 queue 上排队（FIFO）
    P(&rw_mutex);              // 获取数据访问锁
    V(&queue);                 // 立即释放 queue，让下一个进程可以排队
    写数据;
    V(&rw_mutex);              // 释放数据访问锁
}
```

> [!note] 公平竞争分析
> - 每个进程在真正操作前先在 `queue` 上排队
> - 若同时有读者和写者到达，按**到达顺序**获得服务
> - **不会出现饥饿**，但**并发度降低**（因为第一个读者也要排队，不能和其他读者一起进入）

## 哲学家就餐问题（Dining Philosophers Problem）

### 问题描述

- 5 位哲学家围坐圆桌，每两人之间放一根筷子（共 5 根）
- 哲学家要么**思考**，要么**吃饭**
- 吃饭需要**同时拿到左右两根筷子**（每根筷子是临界资源）
- 吃完后放下筷子继续思考

### 同步与互斥关系分析

按照四步法分析哲学家就餐问题：

**1. 互斥关系：**
- 每根筷子是临界资源（一次只能被一位哲学家持有）→ 需要 **5 个互斥信号量** `chopstick[5]`，每个初值 `1`

**2. 同步关系：**
- 哲学家就餐问题**没有同步（前驱/时序）关系**，只有互斥关系
- 但问题的关键在于**同时需要两根筷子**（同时持有两个互斥资源），这和普通的单资源互斥不同

**3. 死锁风险：**
- 若 5 位哲学家同时拿起左边筷子，则每个人持有一根、等待另一根 → **死锁**（循环等待）
- 这是经典的同时持有多个互斥资源导致的死锁问题

> [!note] 哲学家就餐 vs 生产者—消费者
> 生产者—消费者既有互斥也有同步；哲学家就餐**只有互斥**，难点在于**如何避免同时获取多个互斥资源时产生死锁**。

### 问题分析（错误实现）

若每位哲学家先拿左边筷子再拿右边筷子：

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};

// 哲学家 i（错误的实现）
void Philosopher(int i) {
    while (true) {
        P(&chopstick[i]);            // 拿左边筷子
        P(&chopstick[(i + 1) % 5]);  // 拿右边筷子
        吃饭;
        V(&chopstick[i]);            // 放左边
        V(&chopstick[(i + 1) % 5]);  // 放右边
        思考;
    }
}
```

> [!warning] 死锁风险
> 若 5 位哲学家**同时**拿起左边筷子，则每个人都只有一根筷子，且都在等待右边筷子 → **死锁**

### 解决方案

#### 方案一：限制就餐人数

最多允许 4 位哲学家同时尝试拿筷子：

```c
semaphore chopstick[5] = {1, 1, 1, 1, 1};
semaphore limit = 4;  // 最多 4 人同时就餐

void Philosopher(int i) {
    while (true) {
        P(&limit);                    // 申请就餐名额
        P(&chopstick[i]);
        P(&chopstick[(i + 1) % 5]);
        吃饭;
        V(&chopstick[i]);
        V(&chopstick[(i + 1) % 5]);
        V(&limit);
        思考;
    }
}
```

> [!note] 为什么有效？
> 只要不超过 4 人同时就餐，就**至少有一位**哲学家可以拿到两根筷子并完成就餐。

#### 方案二：奇数/偶数哲学家不同拿筷顺序

奇数先左后右，偶数先右后左（避免循环等待）：

```c
void Philosopher(int i) {
    while (true) {
        if (i % 2 == 1) {  // 奇数
            P(&chopstick[i]);            // 先左
            P(&chopstick[(i + 1) % 5]);  // 后右
        } else {            // 偶数
            P(&chopstick[(i + 1) % 5]);  // 先右
            P(&chopstick[i]);            // 后左
        }
        吃饭;
        V(&chopstick[i]);
        V(&chopstick[(i + 1) % 5]);
        思考;
    }
}
```

#### 方案三：只有两根筷子都可用时才拿（AND 型信号量）

使用 `Swait(chopstick[i], chopstick[(i+1)%5])` 同时申请两根筷子，或者用互斥锁确保同时申请两侧筷子是原子操作

# 管程（Monitor）

## 管程的引入动机

信号量机制功能强大，但存在以下问题：
- P/V 操作散布在程序中，大量信号量容易导致死锁
- 正确性难以验证，编写/调试/维护困难
- 要求程序员极强的逻辑思维，容易出错

**管程**将共享资源及对资源的操作**封装**成一个模块，减少程序员负担，使同步机制更**结构化**。

## 管程的基本概念

**管程（Monitor）** 是由一组数据结构和在这组数据结构上操作的**过程（函数）** 组成的软件模块，具有以下特性：

| 特性       | 说明                                         |
| -------- | ------------------------------------------ |
| **封装性**  | 管程内的共享变量只能由管程内定义的过程访问，**外部不可直接访问**         |
| **互斥性**  | 任一时刻，**最多只有一个**进程可以在管程内执行（由编译器/解释器保证）      |
| **条件变量** | 提供条件变量（Condition Variables）来支持进程间的**同步等待** |
| **共享性**  | 所有调用管程过程的进程共享管程中的变量                        |

## 管程的组成

```c
monitor Demo {
    // ============ 共享数据结构（私有变量）============
    int buffer[N];           // 临界资源，仅管程内过程可访问
    int count = 0;           // 共享变量状态
    // ... 外部进程无法直接访问这些变量，局限于管程内部
	
    // ============ 条件变量（同步等待机制）============
    condition not_full;      // 条件变量：缓冲区不满
    condition not_empty;     // 条件变量：缓冲区不空
    // 操作：x.wait() → 在条件 x 上阻塞，释放管程互斥权
    //       x.signal() → 唤醒在条件 x 上等待的一个进程

    // ============ 一组操作过程（公有函数）============
    // 外部进程只能通过这些接口访问共享资源
    void put(int item) {     // 公有过程：生产者使用的接口
        if (count == N)
            not_full.wait(); // 缓冲区满则阻塞等待
        buffer[in] = item;
        in = (in + 1) % N;
        count++;
        not_empty.signal();  // 唤醒等待的消费者
    }

    int get() {              // 公有过程：消费者使用的接口
        if (count == 0)
            not_empty.wait();// 缓冲区空则阻塞等待
        int item = buffer[out];
        out = (out + 1) % N;
        count--;
        not_full.signal();   // 唤醒等待的生产者
        return item;
    }

    // ============ 初始化代码 ============
    // 在管程创建时自动执行，设置共享变量和条件变量的初始状态
}
```

## 条件变量（Condition Variable）

管程的互斥性只能保证同一时刻仅有一个进程在管程内执行，但无法处理"某个条件不满足时需要等待"的情况。**条件变量**就是为管程提供**同步等待**能力的机制。

**每个条件变量维护一个等待队列**，提供两个操作：
- **`x.wait()`**：当条件变量 `x` 所代表的条件不满足时，当前执行管程过程的进程调用`x.wait()`，并将其插入 `x` 的等待队列，同时**释放管程互斥权**，让其他进程可以进入管程
- **`x.signal()`**：当条件变量 `x` 所代表的条件发生变化（可能已满足）时，调用`x.signal()`从条件 `x` 的等待队列中唤醒一个阻塞进程；若队列为空，则什么也不发生

> [!note] 条件变量 vs 信号量
> 
> **相似点：**
> - 都是用于进程/线程间**同步**的机制
> - 都有"等待"和"唤醒"两种操作（`wait`/`signal` 和 P/V）
> - 内部都维护**等待队列**，阻塞的进程/线程在其中排队
> 
> **不同点：**
> 
> | | 条件变量 | 信号量 |
> |------|---------|--------|
> | `wait` 行为 | 无条件阻塞当前进程 | 若 `value > 0` 则减一并继续，否则阻塞 |
> | `signal` 行为 | 若队列空则无效果（**信号丢失**） | 必定 `value++` 并可能唤醒等待者（**信号保留**） |
> | 互斥释放 | `wait` 时自动释放管程互斥权 | 不涉及互斥释放（需程序员额外处理） |
> | 所属 | 必须配合管程使用，无独立存在意义 | 独立存在，是一种通用的同步原语 |
> | 计数能力 | 无计数，仅维护等待队列 | 有整型计数器，可表示资源数量 |
> 
> 条件变量的 `signal` 若无人等待则**信号丢失**，这是与信号量 V 操作**最关键的差异**。

> [!important] `signal()` 后的执行权问题
> 当进程 P 执行 `x.signal()` 唤醒进程 Q 后，管程内有两个就绪进程。需要决定谁继续执行：
> 
> | 策略 | 说明 | 采用者 |
> |------|------|--------|
> | **Hoare 语义** | P 挂起，Q 立即执行（Q 执行完后 P 恢复） | 理论模型 |
> | **Mesa 语义** | P 继续执行，Q 等待 P 退出管程后才执行 | Java、C#、Pthreads |
> | **Hansen 语义** | `signal()` 必须是管程过程的**最后一条语句**（执行完 signal 后 P 立即退出） | 并发 Pascal |
> 
> > 408 考试中，若无特别说明，通常采用 **Hoare 语义**

> [!note] 管程 vs 信号量
> 
> | 维度 | 管程 | 信号量 |
> |------|------|--------|
> | 易用性 | ✅ 高（封装良好） | ❌ 低（容易出错） |
> | 结构化 | ✅ 面向对象思想 | ❌ 分散在各处 |
> | 效率 | 略低（封装开销） | ✅ 高 |
> | 语言支持 | 需要语言/编译器支持 | 只需要原子操作 |
> | 实现难度 | 编译器复杂 | 简单 |
> 
> 管程是**信号量的高级抽象**，本质上底层仍可用信号量实现。

> [!note] 管程 vs 进程
> 
> | 维度 | 管程（Monitor） | 进程（Process） |
> |------|----------------|-----------------|
> | **本质** | 一种**同步机制**（语言/编译器层面的抽象） | 一种**资源分配和调度的基本单位** |
> | **目的** | 封装共享资源，保证互斥与同步 | 执行程序，完成计算任务 |
> | **并发性** | 同一时刻最多一个调用者在管程内执行（编译器保证） | 多个进程可以并发执行 |
> | **创建** | 由应用程序在代码中声明（静态结构） | 由操作系统通过 `fork()` 等系统调用动态创建 |
> | **拥有资源** | 封装共享变量和条件变量 | 拥有独立的地址空间、文件描述符等 |
> | **生命周期** | 与程序模块同生命周期（代码级） | 创建 → 就绪 → 运行 → 阻塞 → 终止 |
> | **互斥保证** | 由编译器/解释器**自动插入** P/V 操作，程序员无需手写 | 程序员必须手动使用信号量/互斥锁等机制 |
> 
> 管程是**静态的代码模块**，用来"管理"共享资源的访问规则；进程是**动态的执行实体**，是共享资源的"使用者"。管程为进程提供了一套**安全的访问接口**，让进程无需自己操心互斥细节。
