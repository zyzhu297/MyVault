# 线程的概念

**线程**是进程中的一个执行单元，是**程序执行流的最小单位**，也是 **CPU 调度的基本单位**

> [!important] 引入线程的目的
> 减少程序并发执行时所付出的时空开销，使 OS 具有更好的**并发性**。

# 线程的属性

1. 线程是轻型实体，基本不拥有系统资源，但拥有运行所必须的私有数据结构，包括**线程 ID**、**CPU 现场**（包括**程序计数器**、**寄存器集合**、**栈**），每个线程都有唯一的标识符和一个**线程控制块**（TCB），用于记录其执行上下文
2. 同一进程的线程共享代码段、数据段、**打开的文件**等
3. 线程是**处理机的独立调度单位**，可以独立执行程序
4. 线程也有三种基本状态：就绪、运行、阻塞

# 线程与进程的比较

| 维度       | 进程             | 线程                       |
| -------- | -------------- | ------------------------ |
| **资源拥有** | 资源分配的基本单位      | 不拥有系统资源，共享进程资源           |
| **调度**   | 传统 OS 中调度的基本单位 | **CPU 调度的基本单位**          |
| **并发性**  | 进程间可并发         | 同一进程的线程间可并发              |
| **系统开销** | 创建/切换开销大       | 创建/切换开销小（共享地址空间）         |
| **独立性**  | 进程间相互独立        | 同一进程内线程相互影响，不同进程间独立      |
| **地址空间** | 各进程拥有独立地址空间    | **共享所属进程的地址空间**，没有独立地址空间 |
| **通信**   | 需要 IPC 机制      | 可直接读写共享数据                |
# 线程的状态和生命周期

线程作为 CPU 调度的基本单位，也有自己的生命周期和状态转换。线程的状态与进程类似，但**更为简化**，因为线程共享进程的资源，不存在独立的资源申请/释放状态

## 线程的基本状态

| 状态                  | 说明                        |
| ------------------- | ------------------------- |
| **就绪态（Ready）**      | 线程已具备运行条件，只等获得 CPU        |
| **运行态（Running）**    | 线程正在 CPU 上执行              |
| **阻塞态（Blocked）**    | 线程等待某事件（I/O、锁、条件变量等），暂停执行 |
| **终止态（Terminated）** | 线程执行完毕或被强制取消，等待回收 TCB     |

> [!note] 与进程状态的区别
> - 线程**没有"创建态"和"挂起态"** — 创建态由所属进程承担，挂起是进程级概念（整个进程被换出时其所有线程随之挂起）
> - 线程的**状态转换更简单**，因为不需要管理资源分配（资源共享进程的）

## 线程的生命周期

| 阶段 | 说明 |
|------|------|
| **创建** | 由同一进程内的其他线程（或主线程）调用线程库函数（如 `pthread_create()`）创建新线程。内核为新线程分配 **TCB** 和独立的**线程栈**，设置初始 PC 指向线程入口函数 |
| **就绪** | 新线程创建完毕后进入就绪态，等待被调度 |
| **运行** | 调度器选中该线程，加载其上下文（PC、SP、寄存器），开始执行。线程在运行态时可能被**抢占**（时间片到）回就绪态，或主动**阻塞**（等待 I/O / 获取锁） |
| **阻塞** | 线程等待的事件发生（I/O 完成、锁释放、条件变量通知）后，转入就绪态等待重新调度 |
| **终止** | 线程函数返回、调用 `pthread_exit()`、或被其他线程 `pthread_cancel()`。终止后释放线程私有资源（栈、TCB），但共享资源仍归进程所有 |

> [!warning] 线程终止的特殊影响
> - 如果**主线程**终止（`main()` 中 `return` 或 `exit()`），**整个进程的所有线程都会终止**
> - 如果普通线程终止，仅其自身 TCB 和栈被回收，**不影响其他线程**

# 线程控制块（TCB）

**线程控制块**（TCB，Thread Control Block）是操作系统管理线程的核心数据结构，类似于 PCB 之于进程。每个线程都有唯一对应的 TCB，操作系统通过 TCB 来感知 和控制线程。

> [!important] TCB 与 PCB 的关系
> - PCB 管理**进程级别**的信息（资源、地址空间、文件描述符表等）
> - TCB 管理**线程级别**的信息（CPU 执行现场、调度参数等），且 TCB **归属于**所属进程的 PCB
> - 同一进程的多个 TCB 共享一份 PCB 中的资源信息

## TCB 包含的信息

| 类别          | 字段          | 含义                                      |
| ----------- | ----------- | --------------------------------------- |
| **线程标识**    | 线程 ID（TID）  | 在进程范围内唯一标识一个线程                          |
| **CPU 上下文** | 通用寄存器组      | 保存线程被中断时的运算中间结果                         |
|             | 指令计数器（PC）   | 线程下一条待执行指令的地址                           |
|             | 栈指针（SP）     | ⭐指向该线程**独立栈**的栈顶，是线程私有的，但是位于**进程的地址空间** |
|             | 程序状态字（PSW）  | 条件码和控制标志                                |
| **调度信息**    | 线程状态        | 就绪 / 运行 / 阻塞 / 终止                       |
|             | 线程优先级       | 调度器据此决定执行顺序（可继承自进程，也可独立设置）              |
|             | 阻塞原因        | 记录线程因何阻塞（等待的锁、条件变量、I/O 等）               |
| **栈信息**     | 栈基址与大小      | 每个线程有自己独立的调用栈，TCB 记录其位置和边界              |
| **关联信息**    | 所属进程 PCB 指针 | 链接到所属进程的 PCB，以便访问共享资源                   |
| **同步信息**    | 线程私有数据      | 如 `pthread_key_t` 线程局部存储（TLS）键值         |

> [!note] TCB vs PCB：精简的"轻量级 PCB"
> TCB 比 PCB **精简得多**，因为不需要记录页表指针、文件描述符表、资源清单等进程级信息 — 这些通过"所属进程 PCB 指针"即可间接获取，所有线程共享一份。

# 线程的实现方式

线程的实现方式根据**线程的调度和管理是在用户空间还是内核空间**，分为三种：

## 用户级线程（ULT，User-Level Thread）

线程的创建、调度、切换、销毁等全部在**用户空间**完成，内核完全感知不到线程的存在，只看到一个进程

**工作机制：**
- 由一个**线程库**（如 POSIX Pthreads 的用户级实现）在用户空间维护线程表，管理 TCB
- 线程切换**不需要内核陷入**（不涉及模式切换），完全由线程库在用户态完成
- **内核只按进程为单位调度**：该进程获得 CPU 时，线程库再在内部分配给某个线程

> [!note] 优缺点
> 
> | 优点 | 缺点 |
> |------|------|
> | 线程切换**不陷入内核**，开销极小（仅保存/恢复寄存器，无需模式切换） | 一个用户线程发起阻塞系统调用，**内核将整个进程阻塞**，所有其他线程都无法运行 |
> | 不依赖内核，可在任何操作系统上实现，具有良好的可移植性 | 内核无法感知多个线程，不能将同一进程的多个线程**分配到多个 CPU 核心** |
> |调度策略由应用程序制定，可根据需求采用不同线程调度算法 | 线程调度由用户级线程库决定，内核完全无法介入（无优先级反转保护等机制） |

## 内核级线程（KLT，Kernel-Level Thread）

线程的创建、调度、切换、销毁等全部由**内核**完成，内核为每个内核线程维护独立的 TCB

**工作机制：**
- 每次线程操作（创建、切换、销毁）都通过**系统调用**进入内核
- 内核调度器以**线程为单位**分配 CPU 时间片，同一进程的多个线程可以同时在不同 CPU 核心上运行
- 这是现代主流操作系统的实现方式：**Windows、Linux**（`clone()` 创建轻量级进程，实际上就是内核线程）

> [!note] 优缺点
> 
> | 优点 | 缺点 |
> |------|------|
> | 一个线程阻塞不影响同进程其他线程（内核知道有多个线程） | 线程创建和切换需要**模式切换**（用户态→内核态），开销比用户级线程大 |
> | 内核可将多个线程**调度到多个 CPU** 上并行执行 | 每次线程操作都需系统调用，可创建的线程总数受内核限制 |
> | 内核可感知所有线程，可提供更细粒度的调度和保护 | 内核需要维护大量 TCB，内存开销和调度开销相对较大 |

## 组合方式（用户级线程与内核级线程的映射）

组合方式描述**用户级线程（ULT）与内核级线程（KLT）之间的映射关系**。根据映射比例的不同，分为三种模型：

### 多对一模型（Many-to-One）

多个用户级线程映射到**一个**内核级线程。这本质上是**纯用户级线程**的实现方式：线程管理完全在用户空间，内核只看到一个进程。

> [!note] 优缺点
> - **优点**：线程切换不陷入内核，开销极小；调度策略可由应用程序自定义
> - **缺点**：一个线程发起阻塞系统调用导致**整个进程阻塞**；内核无法将线程分配到多核，**不能利用多核 CPU**

### 一对一模型（One-to-One）

每个用户级线程映射到**一个独立**的内核级线程。这本质上是**纯内核级线程**的实现方式：每个线程都由内核直接管理。

> [!note] 优缺点
> - **优点**：一个线程阻塞**不影响**其他线程（内核感知所有线程）；可充分利用**多核 CPU** 并行执行
> - **缺点**：每创建一个用户线程就要创建一个内核线程，**开销大**（需系统调用 + 模式切换）；系统支持的线程总数受内核限制
> - **实现**：**Windows、Linux**（使用 `clone()` 系统调用创建轻量级进程，本质为内核线程）

### 多对多模型（Many-to-Many）

n 个用户级线程**动态映射**到 m 个内核级线程（n ≥ m），是真正意义上的"组合"方式。

**工作机制：**
1. 用户在代码中使用用户级线程库创建线程（轻量、快速）
2. 线程库在底层将用户线程**动态映射**到若干内核级线程上（如通过 LWP，轻量级进程）
3. 内核调度器只调度内核级线程；线程库再将 LWP 分配给用户线程执行

> [!note] 优缺点
> - **优点**：兼顾了多对一的**低开销创建/切换**和一对一的**真正并行**能力；一个线程阻塞不影响其他线程（可通过其他 KLT 继续执行）；映射关系可灵活调整
> - **缺点**：实现复杂度**最高**（需要用户态调度器 + 内核态调度器协同工作）
> - **变体**：两级模型（Two-level Model），允许将部分关键线程**绑定**到专属内核线程（类似一对一），其余线程多对多映射
> - **实际应用**：Solaris 的 LWP 机制、Go 语言的 Goroutine（Go 调度器 + 内核线程）

### 三种模型对比总结

> [!important]
> 
> | 特性 | 多对一 | 一对一 | 多对多 |
> |------|--------|--------|--------|
> | 并发性 | 不能利用多核 | 最好 | 好 |
> | 阻塞影响 | 全部阻塞 | 无影响 | 无影响 |
> | 系统开销 | 最小 | 最大 | 中等 |
> | 实现复杂度 | 简单 | 简单 | **复杂** |

### 三种实现方式对比

> [!note] 从实现层面对比 ULT、KLT 和组合方式
> 
> | 维度 | 用户级线程（ULT） | 内核级线程（KLT） | 组合方式 |
> |------|-------------------|-------------------|----------|
> | **对应模型** | 多对一 | 一对一 | 多对多 |
> | **调度单位** | 进程 — 内核以进程为单位分配 CPU，运行哪个线程由用户级线程库决定 | 线程 — 内核以线程为单位调度，每个线程可被单独赋予时间片 | 内核以 KLT 为单位调度，线程库再将 KLT 分配给 ULT |
> | **调度透明性** | 对内核**完全透明**，内核只看到一个进程 | 内核**直接感知**每个线程，维护独立 TCB | 内核感知 KLT，ULT 对内核透明 |
> | **线程切换涉及** | 同进程内完成，仅需保存/恢复**少量寄存器状态**，不涉及模式切换 | 完整上下文切换；跨进程线程间还需**更新地址空间**、可能的 **TLB 刷新**和**缓存失效**，开销远高于 ULT | ULT 间切换同左；KLT 间切换同中 |
> | **独立的内核上下文** | ❌ 无，线程上下文仅由线程库维护 | ✅ 每个线程拥有独立的内核上下文（PC、SP、寄存器等） | KLT 有独立内核上下文，ULT 无 |
> | **创建/切换开销** | 极小（用户态） | 较大（系统调用 + 模式切换） | 大部分在用户态 |
> | **多核并行** | ❌ 不能 — 同一进程的线程无法被分配到不同 CPU 核心 | ✅ 可以 — 同一进程的多个线程可被分配到不同 CPU 核心上并行执行 | ✅ 可以 — 通过多个 KLT 分配到不同核心 |
> | **阻塞影响** | 一个线程阻塞 → **全进程阻塞**（内核不知道有其他线程） | 互不影响（内核感知所有线程，可调度其他线程） | 互不影响（可通过其他 KLT 继续执行） |
> | **内核支持** | 不需要 | 需要 | 需要 |
> | **线程数量** | 几乎无限制 | 受内核限制 | 用户线程无限制 |
> | **实现复杂度** | 低 | 中 | **高** |



## 线程库（thread library）的实现

线程库的代码和数据全部位于**用户空间**，不依赖内核支持

- 调用线程库函数只是用户态内的**本地函数调用**，不涉及系统调用
- 库内部维护线程表，自行完成线程切换（如通过 `setjmp` / `longjmp` 保存和恢复上下文）
- 对应**多对一模型**，内核只看到一个进程

### 内核级线程库

线程库是对内核系统调用的**封装**（wrapper），每个库函数最终通过系统调用进入内核。

- 调用 `pthread_create()` → 内核执行线程创建 → 内核维护 TCB
- 线程切换由内核调度器完成，涉及**模式切换**（用户态 ↔ 内核态）
- 对应**一对一模型**或**多对多模型**（取决于内核是否支持 LWP 等中间层）

### 常见线程库

| 线程库 | 实现方式 | 说明 |
|--------|----------|------|
| **POSIX Pthreads** | 用户空间或内核级均可 | 规范定义了接口，具体实现取决于 OS（Linux 上为内核级 `libpthread`） |
| **Windows Threads** | 内核级（一对一） | Win32 API，每个用户线程对应一个内核线程 |
| **Java 线程** | 依赖底层 OS 线程 | JVM 将 Java 线程映射到宿主 OS 的原生线程（Windows/Linux 上为一对一） |
| **Go Goroutine** | 用户空间 + 多对多 | Go 运行时调度器将 goroutine 映射到少量内核线程，实现 M:N 调度 |
