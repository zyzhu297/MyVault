# 目录的基本概念

**目录（Directory）** 是文件系统中用于**组织和管理文件**的关键数据结构，本质上是一种特殊的**目录文件**；目录文件是一种数据结构，用于记录每个文件的属性、存储位置等元数据，以便快速定位其在外存中的物理地址

> [!note] 目录管理的要求
> 1. **按名存取**：用户只需提供文件名，系统即可自动定位该文件在外存中的位置，用户无需关心物理存储位置
> 2. **提高检索速度**：通过合理组织目录结构，加快查找过程，从而提升文件存取效率
> 3. **支持文件共享**：在多用户系统中，允许多个用户共享同一个文件
> 4. **允许文件重名**：不同用户可对各自所属文件使用相同名称

## 目录项（Directory Entry）

目录项是目录文件的最小组成单元。在[[1.文件系统基础#索引结点（inode）|UNIX/Linux 系统中]]，目录项的结构非常精简：

$$\text{目录项} = (\text{文件名},\ \text{inode 号})$$

- **文件名**：用户可见的文件标识，支持变长名称
- **inode 号**：指向文件元数据（inode）的整数索引

目录项极小（通常 16~32 字节），这种瘦身设计使得目录查找时只需读入少量数据即可完成文件名比对，大幅减少了磁盘 I/O。详见[[1.文件系统基础#目录项与 inode 的协作|目录项与 inode 的协作]]中的逐步查找过程。

# 目录结构

## 单级目录结构

整个文件系统中仅有一张**全局目录表**，所有文件均在同一目录下，文件名必须唯一；创建新文件时，系统首先遍历整张目录表，确保无同名文件存在；确认无冲突后，方可新增一个 FCB，并填入该文件的属性信息

| 维度 | 评价 |
|:---|:---|
| **查找** | 需遍历整个目录表，文件多时效率低 |
| **命名** | 全局唯一，多用户环境下**命名冲突**严重 |
| **共享** | 无法控制访问权限，所有用户看到同一视图 |
| **组织** | 无层次结构，大量文件混杂，管理困难 |

> [!failure] 致命缺陷
> 在多用户系统中，用户 A 的文件名可能与用户 B 冲突（如都叫 `test.c`），单级目录无法解决此问题。仅适用于极简单的嵌入式系统或单用户环境。

## 两级目录结构

为每个用户建立独立的**用户文件目录（UFD, User File Directory）**，再通过一张**主文件目录（MFD, Master File Directory）** 记录各 UFD 的信息。

| 维度      | 评价                             |
| :------ | :----------------------------- |
| **查找**  | 先查 MFD 找到用户，再查 UFD，两级查找，效率优于单级 |
| **命名**  | 不同用户可以有同名文件，⭐**解决了命名冲突**       |
| **隔离性** | 各用户目录独立，无法访问其他用户的文件（除非特殊授权）    |
| **共享**  | 仍不完善：用户之间无法方便地共享文件             |

> [!note] 访问路径
> 两级目录下，文件名由 **(用户名, 文件名)** 二元组唯一标识。用户登录时系统记录当前用户名，用户只需提供文件名，系统自动在对应的 UFD 中查找。

## 树形目录结构

将两级目录推广为**多级层次结构**：根目录下可创建子目录，子目录可再包含子目录或文件，形成一棵[[1.树|树]]
在该结构中，用户通过**文件路径名**来标识文件，路径名是一个字符串，由从根目录出发到目标文件路径上所有目录名和文件名，用分隔符 "/" 连接而成
**路径名**自然地分为两类：

| 类型                      | 定义                 |
| :---------------------- | :----------------- |
| **绝对路径（Absolute Path）** | 从**根目录 `/`** 开始的路径 |
| **相对路径（Relative Path）** | 从**当前工作目录**开始的路径   |

> [!note] 当前目录（Current Directory）
> 每个进程维护一个**当前工作目录（cwd）**，相对路径均以此为起点解析。`chdir(path)` 系统调用可更改当前目录。当前目录的存在使得用户无需总是输入冗长的绝对路径。

| 维度     | 评价                                      |
| :----- | :-------------------------------------- |
| **查找** | 按路径名**逐级查找**，每级只比对当前目录的目录项，效率远高于单级      |
| **命名** | 不同目录下允许同名文件，命名灵活性极强                     |
| **组织** | 层次清晰，用户可按逻辑分类组织文件                       |
| **共享** | 树结构下每个文件仅有一个父目录，严格限制共享——两个不同路径不能指向同一个文件 |
| **权限** | 能针对**各节点**设置独立的存取权限，实现精细化控制             |

> [!warning] 树形目录的局限
> 在严格的树结构中，每个文件（叶子节点）仅有**唯一**的父目录。若用户 A 想与用户 B 共享某文件，文件只能属于 A 的目录树，B 无法直接从自己的目录树访问。**链接机制**正是为解决此问题而引入的。
> 查找一个文件按路径名逐级访问中间目录节点，导致磁盘 I/O 次数增加，可能影响查询速度

## 无环图目录结构

在树形目录基础上，允许一个文件（或子目录）**被多个目录同时引用**，形成**有向无环图（DAG）**，可以共享节点，但**禁止出现环路**。

> [!note] 实现方式
> 无环图结构通过 **[[3.文件#基于索引节点的共享（硬链接）|硬链接]]** 和 **[[3.文件#基于符号链接的共享（软链接 / 符号链接）|软链接]]** 来实现文件共享。基于 inode 的设计使得多目录项指向同一 inode 成为可能。

**无环图的约束**：为防止目录遍历陷入无限循环（如 `du` 命令递归统计磁盘用量时沿环无限递归），无环图结构**禁止**形成目录环。具体策略：
- 硬链接**不可链接到目录**（仅超级用户在极特殊情况下可创建目录硬链接）
- 符号链接可以指向目录，但系统在遍历时需检测并跳过已访问的 inode

| 维度 | 评价 |
|:---|:---|
| **共享** | 不同用户可以各自拥有指向同一文件的不同路径名，共享灵活 |
| **删除** | 需引入**引用计数**机制：当链接计数 > 0 时，即使某用户删除其目录项，文件数据仍然存在 |
| **遍历** | 需防止重复访问（如 `find` 遍历时遇到硬链接可能多次看到同一文件），通常通过记录已访问 inode 来去重 |
无环图目录结构虽有效支持灵活文件共享，但也因此增加了管理开销，需额外维护引用计数等共享状态

### 四种目录结构对比总结

| 结构      |  查找效率   | 命名冲突  | 层次组织 |   文件共享   |   删除复杂度   | 典型应用             |
| :------ | :-----: | :---: | :--: | :------: | :-------: | :--------------- |
| **单级**  | 低（全表扫描） |  严重   |  无   |    无     |    简单     | 简单嵌入式系统          |
| **两级**  |   较低    | 用户间解决 |  浅   |    弱     |    较简单    | 早期多用户系统          |
| **树形**  | 高（逐级匹配） | 完全解决  |  强   | 无（严格一父）  | 简单（直接删除）  | 大多数现代 OS         |
| **无环图** |    高    | 完全解决  |  强   | 好（硬/软链接） |   需引用计数   | UNIX/Linux（主流方案） |

# 目录的操作

操作系统为用户和进程提供了一组目录操作的系统调用，下表列出核心操作：

| 操作       | 说明                                                                         |
| :------- | -------------------------------------------------------------------------- |
| **搜索**   | 在指定目录中**逐项遍历**目录条目，按文件名或通配符筛选匹配的目标文件；允许使用精确匹配或部分文件名进行检索                    |
| **创建文件** | 在检查无重名前后才新增目录项，并初始化其属性                                                     |
| **删除文件** | 从目录中**移除目标文件的目录项**，并将对应 inode 的链接计数 −1；当链接计数归零且无进程打开该文件时，系统**释放元数据和数据块**   |
| **创建目录** | 新建一个空目录文件                                                                  |
| **删除目录** | 删除一个**空目录**（不含任何文件或子目录）；非空目录需先递归删除内部文件                                     |
| **移动目录** | 将目录（或文件）从旧路径迁移至新路径：**同一文件系统**内仅需修改源目录和目标目录中的目录项，不移动数据；**跨文件系统**时需先复制数据再删除源 |
| **显示目录** | 列出目录中所有文件和子目录的名称及属性（权限、大小、时间戳等），对应 `ls` 命令                                 |
| **改变目录** | 将当前进程的**当前工作目录（cwd）** 切换为指定路径；之后所有相对路径均以新 cwd 为起点解析，影响进程后续的全部文件操作          |

# 目录的实现

目录文件可以有多种内部实现方式，直接影响查找、插入和删除操作的效率。

## 线性表

目录项以**线性数组**形式顺序存储在目录文件中，每个目录项包含文件名和对应的 inode 号（或其他文件标识信息）。

```
目录文件（线性表）:
┌────────────────┬────────────────┬────────────────┬─────┐
│ (name1, ino1)  │ (name2, ino2)  │ (name3, ino3)  │ ... │
└────────────────┴────────────────┴────────────────┴─────┘
```

| 操作     | 说明                             |
| :----- | ------------------------------ |
| **查找** | 需从头**线性扫描**，逐一比对文件名，文件数量多时效率低下 |
| **插入** | 需要先遍历目录以确保无同名文件，然后新增目录项        |
| **删除** | 删除一个目录项后，可标记为"无效"惰性删除          |

> [!success] 优点
> - 实现简单，无需复杂数据结构
> - 适合文件数量较少的目录

> [!failure] 缺点
> - 查找效率随文件数量线性下降，大目录（如 `/usr/bin` 含数千个可执行文件）下查找缓慢

## 哈希表

在目录文件中维护一个**哈希表**，以**文件名为键**，通过哈希函数直接映射到目录项的位置。

| 操作     |   时间复杂度   | 说明                                 |
| :----- | :-------: | ---------------------------------- |
| **查找** | 理想 $O(1)$ | 直接计算哈希值定位，**避免线性扫描**；冲突时在冲突链上小范围查找 |
| **插入** | 理想 $O(1)$ | 计算哈希，插入对应桶或冲突链                     |
| **删除** | 理想 $O(1)$ | 同查找，定位后删除                          |

> [!success] 优点
> - 查找效率不受文件数量影响（取决于哈希冲突程度）
> - 适合文件数量多的目录

> [!failure] 缺点
> - **无法高效顺序遍历**（如 `ls` 列出目录中所有文件）——必须遍历整个哈希表
> - 哈希冲突处理增加实现复杂度
> - 哈希表扩容代价大（需重新哈希所有目录项）
> - 固定大小的哈希表可能导致空间浪费（多数桶为空）或不够用

# 目录查询技术

当用户通过路径名访问文件时，操作系统需经过**路径解析（Path Resolution）** 过程，逐级将路径分量转换为 inode，最终获得目标文件。本节系统总结该过程及优化手段。

## 路径解析过程

以绝对路径 `/home/alice/docs/report.txt` 为例：

```
Step 1: 根目录 inode (固定编号，通常为 2)
        │
        ▼
Step 2: 读入根目录的目录项列表，搜索文件名 "home"
        → 获得 home 的 inode 号
        │
        ▼
Step 3: 读入 home 的 inode 及其数据块，搜索 "alice"
        → 获得 alice 的 inode 号
        │
        ▼
Step 4: 读入 alice 的 inode 及其数据块，搜索 "docs"
        → 获得 docs 的 inode 号
        │
        ▼
Step 5: 读入 docs 的 inode 及其数据块，搜索 "report.txt"
        → 获得 report.txt 的 inode 号
        │
        ▼
Step 6: 读入 report.txt 的 inode，获取文件属性和数据块位置
        → 完成路径解析，可进行文件操作
```

> [!warning] 符号链接的截获
> 若路径中任一分量为**符号链接**（如 `/home/link → /home/alice`），系统需暂停当前路径解析，读取符号链接文件的内容（目标路径），然后**从目标路径重新开始**解析。为防止恶意循环，系统通常限制符号链接的**最大嵌套深度**（如 Linux 默认限制 40 次重定向）。

## 绝对路径 vs 相对路径

|          | 绝对路径                          | 相对路径                                     |
| :------- | :---------------------------- | :--------------------------------------- |
| **起点**   | 根目录 `/`                       | 进程的**当前工作目录**                            |
| **查找方式** | 直接从根目录 inode 开始逐级解析           | 从当前目录 inode 开始逐级解析                       |
| **路径长度** | 通常较长，但总是指向同一文件                | 较短，但依赖于当前目录                              |
| **适用场景** | 系统配置文件、脚本（保证不受工作目录影响）         | 用户交互、命令行日常操作                             |

## 目录项缓存（Dentry Cache / dcache）

路径解析是文件系统最频繁的操作之一，如果每次都从磁盘读取逐级目录文件，开销极大。为此，现代操作系统引入**目录项缓存（dentry cache）**：

> [!note] dcache 的工作原理
> 1. **缓存对象**：将已解析的目录项 `(目录inode, 文件名) → 子inode` 的映射关系缓存在内存中
> 2. **查找流程**：路径解析时，首先在 dcache 中查找当前级目录项，**命中则直接获得 inode**，跳过磁盘 I/O
> 3. **LRU 淘汰**：dcache 空间有限，采用 LRU（最近最少使用）策略淘汰不常访问的目录项
> 4. **一致性**：当目录内容被修改（如创建/删除文件），对应的 dcache 条目需失效或更新



# 总结

> [!summary] 本章重点
> **一、目录的基本概念**
> 1. **目录定义**：目录是特殊的**目录文件**，内容为目录项列表，每个目录项建立「文件名 → FCB/inode 号」映射，实现按名存取；目录管理需满足按名存取、提高检索速度、文件共享和允许文件重名四项要求
> 2. **目录项**：`(文件名, inode号)` 的精简结构（16~32 字节），大幅减少目录查找时的磁盘 I/O（详见 [[1.文件系统基础#目录项与 inode 的协作]]）
>
> **二、目录操作**
> 3. 八种核心操作：**搜索**（逐项遍历目录条目，按文件名或通配符匹配）、**创建文件**（检查无重名后新增目录项并初始化属性）、**删除文件**（移除目录项，链接计数 −1，归零且无进程打开时释放）、**创建目录**（新建空目录文件）、**删除目录**（仅删空目录，非空需递归）、**移动目录**（同文件系统仅改目录项、跨文件系统需复制再删除）、**显示目录**（列出文件和子目录名称及属性，对应 `ls`）、**改变目录**（`chdir` 切换当前工作目录，影响后续相对路径解析）
>
> **三、目录结构的四种演变**
> 4. **单级目录**：单一全局表、文件名全局唯一、命名冲突严重 → 仅适用极简场景
> 5. **两级目录**：MFD + 各用户 UFD、不同用户可同名 → 解决冲突但共享弱
> 6. **树形目录**：多级层次、绝对/相对路径、当前目录（cwd）→ 命名灵活、层次清晰，可对各节点独立设置权限；局限是每个文件仅一个父目录、共享严格受限
> 7. **无环图目录**：通过硬/软链接实现共享、多目录指向同一文件、禁止目录环以防无限遍历 → UNIX/Linux 的主流方案，需引用计数管理删除
> 8. **结构对比**：单级（简单低效）→ 两级（解决命名冲突）→ 树形（层次组织 + 权限控制）→ 无环图（支持灵活共享 + 引用计数删除）
>
> **四、目录的实现**
> 9. **线性表**：目录项以线性数组顺序存储，查找需线性扫描（$O(n)$），大目录效率低；优点是实现简单、适合小目录
> 10. **哈希表**：以文件名为键通过哈希函数直接定位，理想查找 $O(1)$；缺点是顺序遍历效率低、哈希冲突处理及扩容代价大
>
> **五、目录查询技术**
> 11. **路径解析**：从根目录或当前目录开始，逐级读目录文件→比对文件名→获得下一级 inode，共 6 步完成；符号链接中途截获需重新定向，并限制嵌套深度防恶意循环
> 12. **绝对 vs 相对路径**：绝对路径从根 `/` 起与当前目录无关，适合配置/脚本；相对路径从 cwd 起简捷但依赖上下文，适合日常交互
> 13. **目录项缓存（dcache）**：内存中缓存已解析的 `(目录inode, 文件名) → 子inode` 映射，命中直接获得 inode 跳过磁盘 I/O；LRU 淘汰策略，目录修改时同步失效
