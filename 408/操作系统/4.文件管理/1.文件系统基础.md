# 文件的基本概念

**文件(File)** 存储在计算机**持久性设备**上的**信息集合**，在用户输入/输出操作时，则通常将 *文件*作为**逻辑操作的基本对象**；为支持用户对文件的创建、读取、修改、删除及安全保护等需求，操作系统提供了专门的**文件系统（File System）**，用于统一管理文件资源

## 文件的定义

文件包含实际的数据，即存储空间中所保存的信息；都关联有用于分类和索引的信息；还有相应的访问控制信息
文件系统通过对二级存储相关资源的抽象，**屏蔽了文件的内部属性，存储介质的物理特性及其在设备上的具体底层细节**，使用户无需关心这些细节，便能方便、高效的使用文件
为了解文件的内部结构，我们采用自底向上的方式定义：

- **数据项**：文件系统中**最低级**的数据组织形式，是描述一个对象某种属性的**最小命名单位**
	- *基本数据项*：用于描述一个对象某种属性的**单一值**，是数据中**不可再分**的最小逻辑单元（原子数据），如学号、姓名、年龄。基本数据项通常有**名字**和**类型**两个特征
	- *组合数据项*：由若干**基本数据项**组合而成，描述对象的某一**复合属性**，如"出生日期"由年/月/日三个基本数据项组成、"地址"由省/市/区/街道组成
- **记录**：一组**相关数据项**的集合，用于描述一个对象在某方面的**完整属性**。例如一条学生记录包含学号、姓名、年龄、性别等数据项。在有结构文件中，记录是**存取（读/写）的基本单位**。记录可分为**定长记录**（所有记录长度相同）和**变长记录**（各记录长度可不同）
- **文件**：由**创建者**定义的、具有**文件名**的一组相关元素的集合，是文件系统中**最大的数据组织单位**。根据元素类型可分为：
	- **有结构文件**：由若干**相关记录**组成，如数据库文件
	- **无结构文件**：视为**字节流**，无记录边界，如源程序、可执行文件
	
文件由**文件体**（数据本身）和**文件控制块 FCB**（元数据）两部分构成，是用户进行 I/O 操作的**逻辑基本单位**


## 文件属性（文件元数据）

文件通过 **FCB（文件控制块）** 中的文件属性来描述和管理，主要属性包括：

| 属性                  | 说明                                              |
| ------------------- | ----------------------------------------------- |
| **文件名（Name）**       | 用户创建文件时指定，是用户可见的**唯一标识**，用户通过文件名进行**按名存取**      |
| **类型（Type）**        | 指示文件的类型（普通文件、目录文件、设备文件等），用于系统判断如何处理             |
| **创建者**             | 创建该文件的用户标识（ID）                                  |
| **所有者**             | 当前拥有该文件的用户标识（ID），通常有权修改文件属性或访问权限                |
| **标识符（Identifier）** | 文件系统内部的**唯一标识符**（如 inode 号），对用户**不可见**          |
| **位置（Location）**    | 指向文件在**外存**设备上的**物理位置**（如磁盘块号）                  |
| **大小（Size）**        | 文件的**当前大小**（以字节为单位）                             |
| **保护（Protection）**  | **访问控制信息**，规定哪些用户/进程能够以何种方式（读/写/执行）访问该文件        |
| **时间/日期**           | 文件的**创建时间**、**最后修改时间**、**最后访问时间**等，用于审计、备份或缓存管理 |

> [!tip] 属性存储在 FCB 中
> FCB 就是这些属性的**容器**。在 UNIX/Linux 中，FCB 的核心部分就是 **inode**，文件名与 inode 分离存储在**目录项**中。详见[[2.目录]]

>[!warning] FCB 必须连续存放
## 文件的分类

| 分类维度 | 类型 | 说明 | 示例 |
|:---:|------|------|------|
| **按性质和用途** | **系统文件** | 操作系统本身及其相关文件 | 内核文件、驱动程序、系统配置文件 |
| | **用户文件** | 用户创建和使用的文件 | 文档、源代码、数据文件 |
| | **库文件** | 为用户提供函数库和标准子程序 | C 标准库（libc）、数学库（libm） |
| **按数据形式** | **源文件** | 由源程序和数据构成，是人类可读的文本文件 | .c、.java、.py 等 |
| | **目标文件** | 源程序经编译后但**尚未链接**的中间文件 | .o、.obj 等 |
| | **可执行文件** | 经编译和链接后，可直接装入内存运行的文件 | Windows 的 .exe、Linux 的 ELF |
| **按存取控制属性** | **只执行文件** | 只能被加载执行，不能读取或修改 | 部分受保护的可执行程序 |
| | **只读文件** | 可读但不可修改 | 只读配置文件、共享库 |
| | **读写文件** | 可读可写可修改 | 用户普通文档 |
| **按组织形式和处理方式** | **普通文件** | 存储用户或系统的实际数据，占据实际的存储空间 | 文本文件、二进制文件 |
| | **目录文件** | 由若干**目录项**组成，每个目录项记录文件**文件名**及其 **FCB/inode 编号** | 文件夹 |
| | **特殊文件（设备文件）** | 将 I/O 设备抽象为文件，通过文件 I/O 操作即可访问设备 | 字符设备文件、块设备文件 |

> [!note] UNIX 的三组权限
> UNIX/Linux 系统中，每个文件对**文件所有者（Owner）**、**同组用户（Group）**、**其他用户（Others）** 三组分别设置 **r（读）、w（写）、x（执行）** 权限

# 文件系统结构

文件系统采用**层次化结构**设计，自底向上各层功能逐渐抽象。这种分层设计使得每层只需关注自身的职责，降低了系统的复杂性。
其设计涉及两个核心问题：
- 定义*用户接口*：包括文件及其属性、允许的操作类型以及目录结构的组织方式
- 相应的*数据结构和算法*：将逻辑上的文件系统映射到物理外存设备上

> [!note] 文件系统的元数据
> **元数据（Metadata）** 即"关于数据的数据"，是文件系统中用于**描述和管理文件及文件系统自身**的辅助信息，而非用户文件的**实际内容**。元数据是文件系统正常运行的核心支撑，主要包括以下几类：
>
> | 类别 | 内容 | 存储位置 | 说明 |
> |------|------|:---:|------|
> | **文件元数据** | 文件名、大小、类型、所有者、访问权限、时间戳（创建/修改/访问）、物理存储位置（磁盘块指针）等 | **FCB / inode** | 每个文件一份，随文件创建而建立、随文件删除而释放；用户通过 `ls -l`（Linux）或文件属性（Windows）可查看 |
> | **目录元数据** | 目录项列表，每个目录项包含「文件名 → inode 号」的映射 | **目录文件** | 目录本身也是文件，其数据内容就是目录项的集合 |
> | **文件系统元数据** | 文件系统整体信息：块大小、总块数、空闲块数、inode 总数、空闲 inode 数、挂载状态等 | **超级块（Superblock）** | 文件系统创建时生成，挂载时读入内存；超级块损坏将导致文件系统无法识别 |
> | **空闲空间管理元数据** | 哪些磁盘块空闲、哪些已分配 | **空闲块位图 / 空闲链表** | 分配新块时查询，回收块时更新 |
>
> 元数据虽然不包含用户的实际数据，但其**完整性至关重要**——元数据损坏可能导致文件"丢失"（即使数据仍在磁盘上）、目录结构混乱甚至整个文件系统崩溃。因此文件系统通常对元数据采用**冗余备份**（如超级块的多个副本）和**日志机制**（如 ext4 的日志、NTFS 的日志）来保证一致性。

## 文件系统的层次结构

> [!note] 自底向上的层次模型

|       层次        | 名称          | 功能                                                                                                                                    |
| :-------------: | ----------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **最<br>底<br>层** | **I/O 控制层** | 由**设备驱动程序**和**中断处理程序**组成，负责在内存与磁盘之间传输信息。**设备驱动程序**将高层命令转换为底层硬件可识别的特定指令，由**硬件控制器**执行，并向 I/O 控制器指明对设备的具体**位置**执行何种操作，并在 I/O 完成后通过中断通知上层 |
|        ↑        | **基本文件系统层** | 向设备驱动程序发送**读取/写入物理块**的通用命令，不关心数据内容。该层还管理内存中的**磁盘缓冲区**，缓存经常访问的磁盘块以提高性能                                                                 |
|        ↑        | **文件组织模块层** | 负责将**逻辑块地址**转换为**物理块地址**（管理磁盘空间分配），同时管理**空闲空间**（空闲块的管理与分配）                                                                            |
|        ↑        | **逻辑文件系统层** | 管理文件系统的**元数据**、**目录结构**和**文件保护**与**安全性**。该层根据路径名查找文件对应的 FCB/inode，并提供存取权限检查                                                           |
|        ↑        | **最顶层**     | **用户接口**（应用程序层），为用户提供文件操作的系统调用接口（如 open、read、write、close 等）                                                                           |

> [!important] 层次间的调用关系
> 用户程序→系统调用接口→逻辑文件系统（目录查找、权限检查）→文件组织模块（逻辑块→物理块映射）→基本文件系统（缓冲区管理）→ I/O 控制（磁盘 I/O）→设备
> 
> 上层只调用相邻下层，各层之间的接口清晰，便于系统的维护和扩展。
### 系统打开文件表

为管理正在使用的文件，操作系统维护两张打开文件表：

| 表           | 所属     | 内容                            | 说明                |
| ----------- | ------ | ----------------------------- | ----------------- |
| **进程打开文件表** | 每个进程一个 | 该进程已打开文件的 FCB 指针、当前读写位置、访问模式等 | 进程 PCB 中指向此表      |
| **系统打开文件表** | 整个系统一个 | 所有已被打开文件的 FCB 指针、打开计数（引用计数）   | 多个进程打开同一文件时共享 FCB |


> [!tip] 打开计数的意义
> 当打开计数降为 **0** 时，说明没有进程再使用该文件，系统可以真正关闭文件并释放相关资源。

# 文件控制块（FCB）

文件控制块(File Control Block, FCB，**也称为文件目录项**)是文件系统管理文件的**核心数据结构**，包含文件的所有属性信息，以实现**按名存取**，每个文件对应一个 FCB，所有 FCB  的集合构成**文件目录（也称目录文件）**

> [!note] FCB 的内容
> - **基本信息**：文件名、文件标识符、文件类型、文件大小、文件在外存上的物理位置（磁盘块号列表）、逻辑结构、物理结构、文件长度等
> - **使用信息**：创建时间、最后修改时间、最后访问时间、文件打开计数
> - **访问控制信息**：文件所有者、访问权限列表

# 索引结点

## 引入 inode 的动机

回顾传统的 FCB：FCB 包含了文件的所有属性（包括文件名和物理位置），目录文件由大量 FCB 组成。当系统按路径名查找文件时，需要将整个 FCB（通常很大，如 64~256 字节）从磁盘逐块读入内存进行逐一比对。这样做的问题是：
- 磁盘 I/O 开销大——查找 `/usr/share/doc/README` 这样的深层路径，每一级目录都要读入大量 FCB 数据
- 实际比对时只需要**文件名**，但每次都把不相关的属性数据一并读入
于是 UNIX 设计者将 FCB **一分为二**：把"瘦"的部分（文件名 + inode 号）放在目录项中用于快速查找，把"胖"的部分（其余所有属性）放在 inode 中。

## 索引结点（inode）
**索引结点（Index Node, inode）** ，简称 **i 结点**是 UNIX/Linux 文件系统中用于**描述文件属性的核心数据结构**，每个文件（包括目录）在磁盘上对应**唯一一个 inode**，由系统分配的 **inode 号**唯一标识，**而目录项则仅由文件名及其对应的索引节点号构成**

inode 存储了除**文件名**之外的所有文件信息，为兼顾**持久存储**与**高效访问**，文件系统将索引节点设计为两种形态：

### 磁盘索引节点
存储在磁盘上，作为文件的“永久档案”，每个文件有唯一磁盘索引节点，主要包含：

|    字段类别    | 具体内容                                                           | 说明                                                                             |
| :--------: | -------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| **文件主标识符** | UID（用户 ID）和 GID（组 ID）                                          | 标识文件的**所有者**和**所属用户组**，在进程访问文件时用于**权限校验**；超级用户（root, UID=0）通常不受权限限制            |
|  **文件类型**  | 普通文件、目录文件、字符设备文件、块设备文件、符号链接、命名管道（FIFO）、套接字（Socket）             | 决定系统对该文件执行的**操作类型**和**处理方式**；(1) 普通文件存储用户数据 (2) 目录文件存储目录项 (3) 设备文件用于 I/O 抽象    |
| **文件存取权限** | r（读）、w（写）、x（执行），按**三组**分别设置：文件所有者、同组用户、其他用户                    | 实现**访问控制**：读允许查看内容，写允许修改内容，执行允许运行程序或进入目录                                       |
| **文件物理地址** | 通过 13 个地址项以直接或间接的方式记录数据所在盘块编号                                  | 指向文件数据所在的**磁盘块号**，是 inode 最核心的字段；小文件只用直接块，大文件逐级使用间接块；间接块使单个 inode 能管理极大文件      |
|  **文件长度**  | 当前文件的**字节数**                                                   | 记录文件的实际大小，读写时据此判断**是否到达文件末尾（EOF）**；与所占磁盘空间不同——可能存在内部碎片（最后一块未填满）                |
| **文件链接计数** | 指向该 inode 的**目录项总数**（硬链接数）                                     | 记录有多少个"文件名"指向此 inode；创建硬链接时 +1，删除目录项时 −1；**归零时**说明没有文件名引用，系统回收 inode 和数据块      |
| **文件存取时间** | **atime**（最后访问时间）、**mtime**（最后数据修改时间）、**ctime**（最后 inode 变更时间） | 三个时间戳各有用途：atime 用于缓存淘汰、mtime 用于增量备份、ctime 用于审计追踪；注意：修改文件内容时 mtime 和 ctime 同时更新 |

### 内存索引结点
当文件被打开时，系统将对应的磁盘索引节点载入内存，并**附加运行时状态信息**，以支持高效并发访问与快速操作，因此额外增加了以下内容：

|   字段类别    | 具体内容                                                  | 说明                                                           |
| :-------: | ----------------------------------------------------- | ------------------------------------------------------------ |
| **索引节点号** | 对应磁盘 inode 的**编号**（整数）                                | 用于在内存中唯一标识该节点                                                |
|  **状态**   | **脏位**（是否被修改）、**锁定标志**（是否被某进程独占）、**等待队列**（是否有进程在等待解锁） | 脏位 = 1 表示内存 inode 已被修改，关闭或 sync 时需写回磁盘；锁保证多进程并发访问时的**数据一致性** |
| **访问计数**  | 当前**打开**该文件的进程数（引用计数）                                 | 判断结点是否可释放；当访问计数归零且链接计数也为零时，系统释放内存 inode 并回收磁盘资源              |
| **逻辑设备号** | 文件所在**设备（分区）的标识符**                                    | 标识文件属于哪个文件系统，**在多文件系统中尤为关键**，用于跨设备操作时的判断（如硬链接不可跨设备）          |
| **链接指针**  | **哈希链表指针**（用于快速按 inode 号查找）+ **空闲链表指针**（用于回收管理）       | 所有活跃的内存 inode 通过哈希表组织，加速已打开文件的查找；不活跃的 inode 链入空闲链表供回收        |
## 目录项与 inode 的协作

在 UNIX/Linux 中，**目录**本质上是一个特殊文件，其数据内容由若干**目录项（dentry）** 顺序组成。每个目录项只包含两样东西：

$$目录项 = (\text{文件名},\ \text{inode 号})$$

> [!note] 按名查找的过程
> 以查找 `/home/user/note.txt` 为例：
>
> 1. 根目录 `/` 的 inode 号固定（通常为 2），读入根目录的 inode
> 2. 从 inode 的数据块指针找到根目录的数据内容（目录项列表）
> 3. 在目录项列表中比对文件名 `home`，获得其 inode 号
> 4. 读入 `home` 的 inode，重复上述过程
> 5. 在 `home` 的目录项中找到 `user` 的 inode 号
> 6. 在 `user` 的目录项中找到 `note.txt` 的 inode 号
> 7. 读入 `note.txt` 的 inode，获得文件的所有属性和数据块位置

这种设计的核心优势在于：**目录项非常小**（通常仅 16~32 字节，只有文件名 + inode 号），目录查找时只需读入少量数据即可完成比对，大幅减少了磁盘 I/O。

> [!tip] inode 号耗尽问题
> 一个文件系统创建时即确定了 inode 总数（由超级块记录）。如果大量创建小文件，可能磁盘空间未满但 **inode 已耗尽**，导致无法创建新文件。可用 `df -i` 查看 inode 使用情况。

# 文件的逻辑结构

文件的**逻辑结构**是指从**用户视角**看到的文件组织形式，即文件中数据记录之间的逻辑关系。与之相对，文件的**物理结构**是指文件在存储介质上的实际存放方式，对用户**透明**。文件系统的一项重要任务就是将**逻辑地址**映射为**物理地址**。

文件的逻辑结构分为两大类：**无结构文件**和**有结构文件**。

## 无结构文件（流式文件）

无结构文件中，数据以**字节流**的形式存在，文件内部**没有结构**，只有**字节序列**。这类文件将数据看作一个**有序的字节集合**，文件的长度就是所含字节的数量。

> [!note] 流式文件的特点
> - 对文件的访问以**字节**为单位，若需**检索**特定内容，**只能顺序查找**，不适用于频繁随机访问或按记录检索的应用场景
> - 文件内部无记录概念，由应用程序自行理解和解释文件内容
> - 操作系统**只按字节流**进行读写，不对数据做任何结构化解释

> [!example] 典型应用
> - 源程序文件（.c、.java）
> - 可执行文件、库函数
> - 文本文件
> - 多媒体文件（图片、音频、视频）
> 
> UNIX、Linux、Windows 等现代操作系统均采用流式文件作为基本文件模型。

## 有结构文件（记录式文件）

有结构文件中，数据以**记录**为单位进行组织。每条**记录**是文件中一个具有特定含义的数据项集合，由一组相关的**数据项（字段）** 组成。根据记录长度是否一致，可分为两类：

- **定长记录**：文件中所有记录的**长度完全相同**，每条记录占据固定的字节数。系统可通过 $R \times L$（$R$ 为记录号，$L$ 为记录长度）**直接计算**任意记录的起始位置，从而支持高效的**随机访问**。缺点是存储空间可能浪费——若实际数据短于记录长度，未填满部分即为内部碎片
- **变长记录**：文件中各记录的**长度可以不相等**，通常用特殊分隔符（如换行符）或记录头部的长度字段来标识每条记录的边界。优点是**空间利用率高**，无内部碎片；缺点是**无法直接计算**第 $i$ 条记录的位置，必须**从头顺序扫描**前面的记录才能定位，不支持真正意义上的随机访问

根据记录的组织方式，还可以分为以下几种：
### 顺序文件

文件中的记录按照线性顺序依次排列，可采用**定长记录**或**变长记录**，根据记录是否按键字排序，可分类两种结构：
- **串结构**：记录按**到达（写入）的先后顺序**依次排列，各记录之间的顺序与关键字无关。查找时只能**从头顺序扫描**，效率较低
- **顺序结构**：所有记录按**某个关键字（Key）升序或降序排列**，记录之间的顺序由关键字值决定。对于定长且按键字有序的顺序文件，可以通过偏移量直接定位记录，由于已排序，查找时可借助**二分查找**甚至**折半查找**，效率远高于串结构；但插入新记录时需保持有序性，可能需要**移动大量记录**
此外对于顺序存储设备（如磁带），顺序文件是**唯一**能够有效工作的组织形式
>[!note] 批处理场景下表现最优
>正因记录连续存储、访问局部性强——当需要连续读取或写入大量记录时，其 I/O 效率在所有逻辑文件类型中最高

### 索引文件

为变长记录文件建立一张**索引表**，索引表的每条表项记录**记录的键值**、**记录的长度**与**记录指针**（指向该记录在文件中的位置）。

> [!note] 工作原理
> 索引表本身是**定长记录的顺序文件**，查找时先在索引表中按关键字快速找到目标（如用二分查找），再通过指针读取实际记录。

> [!success] 优点
> - 支持**快速随机访问**（通过索引直接定位）
> - 插入和删除只需修改索引表，无需移动数据记录

> [!failure] 缺点
> - 需要**额外存储空间**存放索引表，索引表本身可能很大，需要额外的管理开销

### 索引顺序文件

在顺序文件的基础上为**每组记录**建立索引项，结合了顺序文件的**批量处理效率**和索引文件的**随机访问能力**

> [!note] 组织方式
> - 先将所有记录分为若干**组（逻辑块）**
> - 索引表为**每组**建一个索引项，记录该组的**最大关键字**和**该组的起始位置**
> - **组内可无序，但组间必须有序**
> - 查找时先查索引表确定目标记录所在组，再在组内顺序查找

对于含 $N$ 条记录的普通顺序文件，平均查找需要 $N/2$ 次；而在索引顺序文件中，若将记录分为 $m$ 组，则每组约含 $s=N/m$ 条记录；查找过程分为两步：

1. **查索引表**：在 $m$ 个索引项中定位目标组，索引表本身也是顺序表，平均比较 $m/2$ 次；
2. **组内查找**：在目标组内的 $s$ 条记录中顺序扫描，平均比较 $s/2 = N/(2m)$ 次

则总平均查找次数为：

$$\text{总次数} = \frac{m}{2} + \frac{N}{2m}$$

当 $m = \sqrt{N}$ 时，上式取得**最小值** $\sqrt{N}$（由 $\frac{d}{dm}\left(\frac{m}{2} + \frac{N}{2m}\right) = \frac{1}{2} - \frac{N}{2m^2} = 0$ 解得）。这意味着索引顺序文件的查找复杂度由 $O(N)$ 降至 $O(\sqrt{N})$，效率提升显著。

#### 多级索引顺序文件

当文件规模进一步增大（$N$ 极大），即使 $\sqrt{N}$ 次比较也仍然可观。此时可以对**索引表本身再建索引**，形成多级索引结构，进一步降低查找开销——这正是*分块查找*（也称索引顺序查找）的基本思想

> [!success] 优点
> - 查找效率显著高于纯顺序文件
> - 索引表大小远小于完整索引文件（仅需按组索引）
> - 既可批量顺序处理，也可随机访问

### 直接文件 / 散列文件

通过[[4.散列表#散列函数的构造方法|哈希函数]]将记录的**关键字**直接映射为记录在文件中的**存储地址**。

> [!note] 工作原理
> - 给定关键字 $K$，通过哈希函数 $H(K)$ 计算得到记录地址
> - 查找时直接计算地址并访问，理想情况下无需比较

> [!success] 优点
> - 查找效率**极高**——理想情况下 O(1)
> - 不需要索引表，节省空间

> [!failure] 缺点
> - **哈希冲突**：不同关键字可能映射到同一地址，需要**冲突处理机制**（如链地址法、开放寻址法）
> - **不支持顺序访问**：记录按哈希值分散存放，无法高效遍历
> - 哈希函数的选择对性能影响很大

> [!important] 有结构文件对比总结
>
> | 类型 | 记录特性 | 随机访问 | 顺序访问 | 增删效率 | 额外空间 | 适用场景 |
> |------|:---:|:---:|:---:|:---:|------|------|
> | **顺序文件** | 定长/变长 | 定长支持 | 高效 | 低（需移动） | 无 | 批量处理、日志 |
> | **索引文件** | 变长 | 高效 | 支持 | 高 | 索引表 | 需要快速直接访问 |
> | **索引顺序文件** | 变长 | 较高效 | 高效 | 较高 | 组索引表 | 兼顾批量和随机 |
> | **直接文件** | 定长 | 极高 O(1) | 不支持 | 取决于冲突处理 | 无（但需额外冲突处理空间） | 关键字精准查询 |

# 总结

> [!summary] 本章重点
> **一、文件的基本概念**
> 1. **文件定义**：持久性设备上的信息集合，用户以文件为逻辑操作的基本对象，通过**按名存取**访问数据。自底向上四级结构：**基本数据项**（不可再分的原子数据）→ **组合数据项**（基本数据项的组合）→ **记录**（有结构文件读/写的基本单位，分定长和变长）→ **文件**（最大组织单位，由文件体 + FCB 构成）
> 2. **文件属性**：九大属性——文件名、类型、创建者、所有者、标识符（inode号）、位置（磁盘块号）、大小、保护（访问控制）、时间/日期——均存储在 **FCB** 中；UNIX/Linux 中由 **inode + 目录项** 分离存储
> 3. **文件分类**四个维度：按性质和用途（系统/用户/库）、按数据形式（源/目标/可执行）、按存取控制属性（只执行/只读/读写）、按组织形式（普通/目录/特殊设备文件）；UNIX 三组权限（Owner/Group/Others × rwx）
>
> **二、文件系统结构**
> 4. **元数据**四类：文件元数据（FCB/inode）、目录元数据（目录项）、文件系统元数据（超级块）、空闲空间管理元数据（位图/链表）；完整性至关重要，依赖冗余备份和日志机制
> 5. **五层层次结构**（自底向上）：I/O 控制层（驱动程序+中断）→ 基本文件系统层（缓冲区管理）→ 文件组织模块层（逻辑↔物理映射、空闲空间管理）→ 逻辑文件系统层（目录、权限、FCB/inode）→ 用户接口层（系统调用 open/read/write/close）
> 6. **系统打开文件表**：**进程打开文件表**（每进程一个，存 FCB 指针+读写位置+访问模式）vs **系统打开文件表**（全局一个，存 FCB 指针+打开计数）；打开计数归零时真正关闭文件
>
> **三、文件控制块与索引结点**
> 7. **FCB** 是文件管理核心数据结构，存文件名、标识符、类型、大小、物理位置、时间、权限等，所有 FCB 集合构成文件目录
> 8. **引入 inode 的动机**：传统 FCB 过大导致目录查找 I/O 开销高，UNIX 将其一分为二——"瘦"目录项（文件名+inode号）快速查找，"胖"inode 存其余属性
> 9. **磁盘索引节点**七个核心字段：文件主标识符（UID/GID）、文件类型（7种）、存取权限（rwx三组）、物理地址（13个地址项，直接+间接块）、文件长度（字节数）、链接计数（硬链接数）、存取时间（atime/mtime/ctime）
> 10. **内存索引结点**五个附加字段：索引节点号（关联磁盘inode）、状态（脏位+锁+等待队列）、访问计数（打开进程数）、逻辑设备号（所属分区）、链接指针（哈希链表+空闲链表）
> 11. **目录项与 inode 协作**：目录项 = (文件名, inode号)，目录本身是存储目录项的特殊文件；按名查找逐级读目录项比对文件名，获得inode号后读inode获取数据位置；目录项极小（16~32B），大幅减少磁盘I/O
> **四、文件的逻辑结构**
> 12. **无结构文件（流式文件）**：以字节为单位的字节流，结构简单、灵活性高，现代 OS 主流；源程序、可执行文件、多媒体文件均采用此模型
> 13. **有结构文件（记录式文件）**：以记录为单位的四种组织方式：
>     - **定长记录**和**变长记录**
>     - **顺序文件**：记录按线性排列，分**串结构**（按到达顺序，仅顺序扫描）和**顺序结构**（按关键字排序，可二分查找）；定长支持随机访问，变长空间利用率高但必须顺序扫描
>     - **索引文件**：为变长记录建索引表（键值→指针），支持快速随机访问，增删只需改索引
>     - **索引顺序文件**：分组建索引，索引表查目标组（$m/2$）+ 组内顺序查（$N/(2m)$），总次数 $\frac{m}{2}+\frac{N}{2m}$，最优 $m=\sqrt{N}$ 时达 $O(\sqrt{N})$
>     - **多级索引顺序文件**：对索引表再建索引，$L$ 级索引查找降至 $O(\log_k N)$，是 B树/B+树的理论基础
>     - **直接/散列文件**：哈希函数映射关键字→地址，理想 O(1) 查找，需处理冲突，不支持顺序访问
> 14. **四种有结构文件对比**：顺序（批量高效，增删需移动）→ 索引（随机高效，需额外索引空间）→ 索引顺序（兼顾批量和随机，$O(\sqrt{N})$）→ 直接（O(1) 但不支持顺序）
> 15. **逻辑结构 vs 物理结构**：逻辑结构是**用户视角**的组织形式，物理结构是**存储介质**上的存放方式，文件系统的核心任务之一是完成两者之间的映射
