# 文件系统布局

## 文件系统在磁盘中的结构

文件系统存放在**磁盘**上，磁盘通常被划分为一个或多个分区，每个分区中由一个**独立的文件系统**；文件系统一般包含以下关键信息：引导操作系统的机制、磁盘总块数、空闲块的数量与位置、目录结构，以及所有具体文件的数据；
磁盘中的文件系统结构通常包含以下几个区域：

- **主引导记录（MBR，Master Boot Record）**：位于磁盘的 0 号扇区，负责引导计算机；MBR **后紧随（内部）**一个分区表，记录每个分区的起始地址和结束地址，并标记其中一个为**活动分区**；系统加电后，BIOS 将 MBR 读入内存并执行；MBR 首先识别活动分区，随后读取该分区的第一个扇区，即**引导快** 
- **引导块（Boot Block）**：MBR 将控制权移交至引导块中的程序，尤其负责加载并启动该分区中的操作系统；**每个分区均以引导块开头**，包含该分区操作系统的引导程序（即使当前未安装可启动的操作系统，也保留此区域，以便未来部署系统）
- **超级块（Super Block）**：存放文件系统的**元信息**，当系统启动或首次挂载该文件系统时，超级块会被读入内存；超级块中的典型内容包括：
	- 分区总块数
	- 块大小
	- 空闲块数量及指针（空闲块链表首指针等）
	- 空闲 inode 数量及指针
- **空闲块信息**：记录未分配的磁盘块，通常以**位示图**或**空闲链表**实现；
- **i  节点**：每个文件对应一个 i 节点，其中完整描述了文件的各种属性
- **根目录**：作为整个目录树的起点
- 磁盘的其余空间则用于存放所有其他文件和目录的实际数据

## 文件系统在内存中的结构

为高效管理文件系统并提升性能，系统在内存维护一组数据结构；这些信息在文件系统**挂载时加载**，在**运行期间动态更新**，并在**卸载时释放**；典型的内存结构包括

- 内存中的**挂载表**(mount table)：记录每个已挂载文件系统分区的有关信息，如设备标识、挂载点、文件系统类型及挂载选项等
- 内存中的**目录项的缓存（dentry cache）**：缓存最近使用的目录项，加速路径解析。
- **打开文件表**：
	- **系统级打开文件表**：为每个当前打开的文件维护一个 FCB 的副本，记录文件状态、访问权限、读写指针、打开计数等全局信息。
	- **进程级打开文件表**：每个进程拥有独立的打开文件表，包含该进程所使用的**文件描述符**及指向系统级打开文件表中对应表项指针，从而实现进程与全局资源的关联
>[!warning] 读写指针
>读写指针（文件偏移量）保存在系统打开文件表项中，因此多个文件描述符只有在指向同一系统打开文件表项时（如 `dup`、`fork`）才共享读写指针；每次 `open` 都会创建新的系统打开文件表项，因此读写指针独立。
# 文件存储空间管理

文件的物理结构包含两个方面：**文件的分配方式**和**文件存储空间管理**；文件存储空间管理专门负责记录磁盘中哪些块是空闲的，并提供对这些块进行分配与回收的机制；存储空间管理的核心任务是**高效组织和管理空闲块**

## 空闲表法

- 属于*连续分配*方式，类似于内存管理中的[[1.内存管理概念#动态分区分配（可变分区分配）|动态分区分配]]，为每个文件分配一块物理上连续的磁盘区域，系统为外存中的所有空闲区维护一张空闲块表，每个空闲区对应一个表项，包含**序号**、**起始空闲块号**和**连续空闲块数**，并按起始盘号递增排序
- **分配**：可采用首次适应、最佳适应、最差适应等算法。
- **回收**：回收时检查是否与相邻空闲区间邻接，若邻接则合并成更大的区间。
- **优点**：分配效率较高，能够显著降低磁盘 I/O 次数；尤其对于小型文件，连续分配不仅实现简单，还能充分发挥磁盘顺序读写的性能优势
- **缺点**：随着磁盘使用，空闲表会变得非常零散，效率下降。

## 空闲链表法

### 空闲盘块链

- 将磁盘上所有**空闲盘块**链接成一条链表
- 分配时从链首取下一块；回收时将释放的盘块插入链尾
- **优点**：单个盘块的分配与回收操作极为简单
- **缺点**：分配多块时需多次遍历链表，效率低；由于管理粒度细，空间盘块链通常较长，管理开销大

### 空闲盘区链

- 将磁盘上所有**空闲盘区**（若干连续空闲盘块）链接成一条链表。
- 每个盘区包含本盘区的盘块数和指向下一空闲盘区的指针
- **分配**：分配时类似于内存管理中的[[1.内存管理概念#动态分区分配（可变分区分配）|动态分区分配]]
- **回收**：需判断回收区是否与链表中相邻的空闲盘区在物理上连续，若相邻则合并，从而有效减少外部碎片
- **优点**：相比空闲盘块链，链表长度更短，效率更高。
- **缺点**：分配和回收较复杂

## 位示图法

- **原理**：用一个**位图（bitmap）** 来表示磁盘块的使用情况。每位对应一个盘块：
	- `0` 表示空闲
	- `1` 表示已分配
	- $m \times n$ 位组成的位示图，可以描述 $m \times n$ 个盘块的使用情况
- **盘块号与位示图位置的换算**：
	设位示图每个字有 $n$ 位（字长），字号为 $i$，位号为 $j$，盘块号为 $b$：
	**盘块号 → 位示图位置（回收时用）**：
	$$i = \lfloor b \mathbin{/} n \rfloor \quad\text{(字号)}$$ $$j = b \bmod n \quad\text{(位号)}$$
	**位示图位置 → 盘块号（分配时用）**：$$b = n \times i + j$$
- **分配**：
	1. 顺序扫描位示图，找到第一个为 $0$ 的位，记录其字号 $i$ 和位号 $j$
	2. 由公式 $b = n \times i + j$ 求得对应盘块号
	3. 将该位置 $1$，完成分配
- **回收**：
	1. 已知回收的盘块号 $b$
	2. 由公式 $i = \lfloor b \mathbin{/} n \rfloor$，$j = b \bmod n$ 求得位示图中对应位置
	3. 将该位从 $1$ 重置为 $0$，完成回收
- **优点**：
	- 分配和回收非常快（位操作）。
	- 占用空间小（每个盘块仅占 1 bit）。
	- 容易找到连续空闲块。
- **缺点**：位示图需常驻内存，对于超大磁盘仍需要较大内存（但通常可以接受）。

## 成组链接法

- **原理**：将空闲表法和空闲链表法结合，以有效控制管理结构的规模，是 **UNIX 系统**采用的方法
- **结构**：
	- 将空闲盘块分组（如每 100 个盘块为一组）。
	- 除最后一组外，每组第一个盘块作为**索引块**，用于记录下一组中所有空闲盘块的**块号及其数量**
	- 在文件系统挂载时，**第一组**的空闲盘块信息（包括块号列表和数量）被预先加载到内存的一个**专用栈结构**，称为**空闲盘块号栈（也称超级块）**
	![[成组链接法示意图.png|900]]
>[!warning] 最后一组
最后一组的块号由前一组的最后一个盘块记录，而最后一组的盘块中存放的第一个盘块号为 0，用作整条空闲链的结束标志
- **分配**：
	1. 从空闲盘块号栈顶去除一个盘块号分配给用户，栈指针相应的上移，栈中空闲块数减 1
	2. 若此时栈指针到达栈底，则需要读取该栈底盘块号所对应的物理盘块内容——其中**保存的是下一组的空闲盘块号列表**；此时先将该内容整体读入栈以更新空闲盘块号栈，**再将原栈底所指盘块(其索引作用已完成）分配出去**
- **回收**：
	1. 将回收的盘块号压入空闲盘块号栈顶部，栈指针对应下移，栈中空闲盘块数加 1
	2. 若栈已满，则将当前栈中内容写入该新回收的盘块，使其成为新的索引块；随后将该盘块号设为新的栈底，并将栈中空闲盘块数重置为 1
- **优点**：兼顾空闲表的高效分配和空闲链表的灵活性，且空闲盘块栈可在内存中缓存，并在运行过程中持续维护内存副本与磁盘副本的一致性，从而确保空闲空间管理的正确性和可靠性

# 虚拟文件系统

**虚拟文件系统（VFS，Virtual File System）** 是 内核中的一层抽象，为用户空间提供统一的文件操作接口，屏蔽不同文件系统的差异，有以下功能：

- 为用户程序提供统一的**系统调用接口**（open、read、write、close 等）。
- 支持多种文件系统共存（ext4、NTFS、FAT32、NFS 等）。
- 实现"一切皆文件"的理念。

VFS 采用**面向对象**的设计思想，抽象出一个通用的文件系统模型，并将其中的公共特性**封装**为四个核心对象：
## 超级块对象

- **含义**：代表一个已挂载的文件系统实例，记录该文件系统的全局信息
- **存储内容**：
	- 文件系统的设备标识
	- 文件系统类型
	- 块大小和总数
	- 空闲块数量
	- 文件系统的根目录项
	- 超级块操作函数集合（super_operations），如 `write_super()`、`put_super()` 等
- **对应**：磁盘上的**超级块**结构。
- **生命周期**：文件系统挂载时，VFS 将其超级块读入内存，创建对应的超级块对象，并设置好相关操作函数（如分配 inode、同步元数据等），卸载时销毁。

## V 节点对象（vnode 对象）

- **含义**：**是最重要的抽象对象**，用于在**内存中**表示一个具体的文件或目录；每个 vnode 对象关联底层某个具体文件系统的索引节点（inode），本质上是对各类文件系统的元数据的统一封装：向上为系统调用提供统一的文件属性，向下通过指针指向该文件系统私有的原数据结构
- **存储内容**：
	- 文件类型（普通文件、目录、设备文件等）
	- 文件权限（rwx）
	- 文件大小
	- 访问时间、修改时间、状态改变时间（atime、mtime、ctime）
	- 引用计数
	- ⭐**标准操作函数指针**，如 `read()`、`mkdir()`、`write()` 等，VFS 通过这些指针将上层请求分发至相应文件系统的具体实现，以**屏蔽不同文件系统的差异**
- **关键点**：inode 代表的是**文件本身**，而非文件名。文件名在目录项中。
- **生命周期**：**vnode 对象仅在文件首次被访问时动态创建于内存**，引用计数器归零后自动释放
>[!warning] inode 和 vnode
>V 节点并**不等同**于磁盘上的 inode；V 节点是 VFS 为统一管理各类文件而创建的内存对象，它与 inode 通过指针关联，并协同完成文件操作：**Vnode 提供统一接口，inode 提供实际存储信息**
## 目录项对象

- **含义**：代表路径中的一个**分量**（目录或文件名的映射）, 多个目录项组合形成逻辑上的目录树
- **存储内容**：
	- 文件名称
	- 指向对应 vnode 的指针
	- 指向父目录项的指针
- **对应**：磁盘上没有直接的对应结构，是 **VFS 的内存缓存（dentry cache）**
- **作用**：加速路径名解析，避免频繁读取磁盘目录。

## 文件对象

- **含义**：代表**进程**打开的一个文件，是**文件在进程中的运行实例**
- **存储内容**：
	- 当前文件读/写指针（文件指针）
	- 文件的打开模式（只读、只写、读写等）
	- 引用计数器
	- 指向对应目录项和 vnode 的指针
	- 文件操作函数集合（file_operations），如 `read()`、`write()`、`llseek()`、`mmap()` 等，其最终由 vnode 转发至底层文件系统执行
- **对应**：磁盘上无对应结构，是**运行时内存对象**。
- **生命周期**：从 `open()` 创建到 `close()` 销毁。

>[!summary] VFS 总结
>VFS 借助这四类对象，屏蔽不同文件系统的实现细节，使其在系统中呈现出一致的行为；VFS 本身并不是一种真正的文件系统，是**软件抽象**，它仅存在于内存中，不占用磁盘空间，随系统启动而建立，随系统关闭而释放

# 文件系统挂载

- **挂载（Mounting）**：也称**安装**，将一个文件系统关联到现有目录树中的某个目录（**挂载点**），可通过该目录访问该设备上的文件，此处的设备指**逻辑上的设备**，如同一个磁盘的不同分区可视为多个独立设备
- **Linux 挂载过程**：
	以单一根文件系统为基础，该文件系统在系统启动时由**内核**直接挂载，通常包含内核映像等关键文件；除根文件系统外，所有其他文件系统都必须挂载到根文件系统中的某个目录下才能被访问；这些文件系统可在系统初始化阶段自动挂载，也可由用户手动挂载；用于挂载的目录称为**挂载点（mount point）**

>[!warning] 挂载点
>同一个设备可被挂载到多个不同的挂载点；但同一个挂载点同一时刻只能挂载一个设备

- **Windows 挂载过程**（以分配盘符为例）：
	采用驱动器号（如 C:、D:)标识分区（也称卷），每个分区拥有独立的目录树结构，并关联一个已挂载的文件系统；访问时，操作系统根据驱动器号定位对应的文件系统，再在其目录结构中查找目标文件；在较新的版本中，也支持将文件系统挂载到目录树下的任意目录（称为**装入点**）；系统启动时会自动探测所有存储设备，并挂载其所识别的文件系统

> [!warning] 装入点
> 若为**装入点方式**，则在宿主 NTFS 卷的目标空目录上创建重分析点（Reparse Point），写入目标卷的 GUID。此后遍历路径时，NTFS 驱动识别重分析点，自动跳转至目标卷的根目录，对上层透明。

- **卸载（Umount）**：将文件系统从目录树中分离。要求该文件系统没有任何打开的文件或进程在使用。

- Linux 遵循 **"一切皆文件"** 和 **"单根树"** 哲学——所有存储设备、网络文件系统都被挂载到唯一的 `/` 目录树下，用户感知不到设备边界。
- Windows 传统采用 **盘符分区** 模式——每个分区是独立的根，用户显式地通过盘符切换不同存储设备。这种设计的优点是简单直观，但缺乏统一命名空间的灵活性。

# 总结

> [!summary] 本章重点
> 
> **一、文件系统布局**
> 
> 1. **磁盘中的文件系统结构**（七个区域）：**MBR**（0 号扇区，含分区表，识别活动分区）→ **引导块**（每个分区第一个扇区，加载该分区 OS）→ **超级块**（文件系统元信息：总块数、块大小、空闲块数/指针、空闲 inode 数/指针）→ **空闲块信息**（位示图或空闲链表）→ **i 节点区**（每个文件一个 inode）→ **根目录**（目录树起点）→ **数据区**（文件和目录的实际数据）
> 2. **内存中的文件系统结构**：**挂载表**（设备标识、挂载点、FS 类型）→ **dentry cache**（加速路径解析）→ **系统级打开文件表**（全局，FCB 副本 + 打开计数）→ **进程级打开文件表**（每进程，fd → 系统表项指针）；信息在挂载时加载、运行中动态更新、卸载时释放
> 
> **二、文件存储空间管理（四种方法）**
> 
> 1. **空闲表法**：连续分配方式，类似可变分区管理；表项含起始块号 + 连续块数，按起始盘号递增排序；分配用首次/最佳/最差适应，回收时判断邻接合并；简单高效但随时间碎片化
> 2. **空闲链表法**：分 **空闲盘块链**（逐块链接，分配取链首、回收插链尾，简单但多块分配需多次遍历）和 **空闲盘区链**（按连续盘区链接，需判断物理邻接合并，链表更短效率更高）
> 3. **位示图法**：bitmap 每位对应一个盘块（0 空闲 / 1 已分配）；核心换算公式：盘块号 → 位示图位置 $i=\lfloor b/n\rfloor$、$j=b \bmod n$（回收用），位示图位置 → 盘块号 $b=n\times i+j$（分配用）；分配时扫描找 0 并置 1，回收时计算位置并置 0；速度快、空间小、易找连续块，位示图需常驻内存
> 4. **成组链接法**：UNIX 采用，结合空闲表和空闲链表；将空闲盘块分组（每组首块为索引块，记录下一组块号）；第一组信息预加载到内存 **空闲盘块号栈（超级块）**；分配时从栈顶取出，**栈底**触发读入下一组；回收时压入栈顶，**栈满**触发写入新索引块；兼顾高效分配与灵活管理，需维护内存/磁盘一致性
> 
> **三、虚拟文件系统（VFS）**
> 
> 1. **VFS 定义与功能**：内核抽象层，提供统一系统调用接口（open/read/write/close），支持多种 FS 共存，实现"一切皆文件"；VFS 本身不是真正的文件系统，仅存在于内存，随系统启动建立、关闭释放
> 2. **超级块对象**：代表已挂载文件系统实例，存设备标识、FS 类型、块大小/总数、空闲块数、根目录项、`super_operations` 函数集；对应磁盘超级块，挂载时创建、卸载时销毁
> 3. **vnode 对象**（最重要）：文件/目录的内存表示，关联底层 inode；存文件类型、权限、大小、时间戳、引用计数、⭐**标准操作函数指针**（`read()`/`write()`/`mkdir()` 等，屏蔽 FS 差异）；**vnode ≠ inode**——vnode 是内存统一封装，inode 是磁盘实际数据结构；首次访问时创建，引用计数归零释放
> 4. **目录项对象**：路径分量，实现 **文件名 → vnode** 映射；存文件名、vnode 指针、父目录项指针；磁盘无对应结构，纯内存缓存（dentry cache），LRU 管理，加速路径解析
> 5. **文件对象**：进程打开文件的运行实例，存文件指针、打开模式、引用计数、`file_operations` 函数集（最终由 vnode 转发至底层 FS）；`open()` 创建、`close()` 销毁；一个 inode 可对应多个文件对象
> 
> **四、文件系统挂载**
> 
> 1. **挂载基本概念**：将文件系统关联到目录树中的挂载点（空目录），挂载后原目录内容被隐藏；设备指逻辑设备（同一磁盘不同分区可视为独立设备）；同一设备可挂载到多个挂载点，同一挂载点同时只能挂一个设备
> 2. **Linux 挂载**：遵循**单根树** + "一切皆文件"哲学；内核先挂载根文件系统（含内核映像），其余 FS 挂载到根下某目录；通过 `mount()`/`umount()` 系统调用，用户感知不到设备边界
> 3. **Windows 挂载**：传统采用 **盘符分区** 模式（C: D: 等），每个分区独立根目录树；较新版本支持 **卷装入点（Reparse Point）**——将卷挂载到 NTFS 目录下，通过重分析点写入目标卷 GUID 实现透明跳转，突破盘符 26 字母限制
> 4. **Linux vs Windows 核心差异**：Linux 统一命名空间（`/` 起点，任意目录均可为挂载点）vs Windows 多根命名空间（盘符为根，装入点虽可挂在目录下但仍属盘符子树）；单根透明灵活 vs 多根直观但受限
