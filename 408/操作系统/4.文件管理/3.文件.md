# 文件的物理结构

文件的物理结构研究的是**文件数据在物理存储设备（磁盘）上的分布与组织**，包括**文件分配方式**，解决如何为文件分配已使用的磁盘块，属于对**非空闲块的管理**；**文件存储空间管理**，解决如何跟踪和分配空闲磁盘块，属于对**空闲块的管理**

## 连续分配

- **定义**：每个文件在磁盘上占用**一组连续的物理块**
- **目录项**：记录文件名 + **起始块号** + **文件长度**（块数）
- **优点**：
  - 实现简单，记录文件只需起始块号和长度
  - **顺序访问速度快**，因为磁头移动距离少
  - 支持**随机存取**（Direct Access），第 i 块可直接定位
- **缺点**：
  - 要求文件创建时确定最大长度，**不利于文件动态增长**
  - 产生**外部碎片**，需紧凑处理（开销大），浪费外存空间
  - 为维持文件的逻辑顺序，插入或删除记录时，需要对后续记录进行物理移动，开销较大

## 链接分配

采用离散分配方式，主要优点有：消除了外部碎片，提高磁盘空间利用率；支持动态分配盘块，无须预先知道文件大小；文件插入、删除和修改操作十分便捷

### 隐式链接

- **定义**：除最后一块外，每个物理块中设有**指向下一块的指针**（块号），这些指针对用户完全**透明**
- **目录项**：记录文件名 + 首块指针 + 末块指针
- **缺点**：
  - **不支持随机访问**，只能顺着指针链顺序查找（如访问第 i 块，需从头遍历 i 次）
  - 指针占用额外空间
  - 若指针损坏/丢失，**可靠性差**，可能丢失整条链

### 显式链接 (FAT)

- **定义**：把用于链接的指针**从物理块中提取出来**，统一存放在一张**文件分配表（FAT，File Allocation Table）** 中
  - 每个磁盘块对应 FAT 中一个表项，表项存放指向下一块的指针（或 EOF）
  - FAT 在系统启动时常驻内存
- **目录项**：记录文件名 + 首块号
- **工作原理**：
  - 后续块号通过查 FAT 表获取：FAT\[首块号\] → 下一块号 → ...
  - 最后一块表项标记为 EOF（End of File，如 -1）
- **优点**：
  - **支持顺序访问和随机访问**（指针在内存的 FAT 中，无需读盘）
  - FAT 常驻内存，查找速度快，大幅减少磁盘 I/O 次数
  - 系统可以直接利用 FAT **管理磁盘空闲空间**
- **缺点**：
  - FAT 表占用一定内存空间，磁盘容量越大，开销越显著

## 索引分配

为每个文件建立一张**索引表**，记录该文件所有块的物理块号，索引表本身占用磁盘块。

### 单级索引分配

- **定义**：每个文件对应一张索引表（一个索引块），索引表中每个表项存放文件的物理块号
- **目录项**：记录文件名 + **索引块所在块号**
- **访问过程**：读目录 → 获取索引块地址 → 读索引块（1 次磁盘 I/O）→ 查得目标数据块号 → 读数据块（1 次磁盘 I/O）
- **优点**：
  - **支持随机访问**，且无外部碎片
  - **容易实现文件的扩展**：新增数据块时只需分配一个空闲块并将其块号追加到索引表中，无需像连续分配那样要求相邻空间或搬迁整个文件，也无需像隐式链接那样遍历整条链才能定位末尾
- **缺点**：
  - 访问需多一次读索引表的 I/O，小文件也有此开销，导致利用率极低
  - 若索引表大于一个块，则存不下 → **引出多级索引**
  - 文件大小受限于一个索引块能放的表项数 × 块大小
- **访问第 i 块的磁盘 I/O 次数**：读索引块 1 次 + 读数据块 1 次 = **2 次**（首次读索引块可能需读盘 $m$ 次）

### 多级索引分配

- **定义**：将索引表本身作为文件，再进行索引
- **工作原理**：
  - 目录项指向**一级索引块**，称为**主索引**
  - 一级索引块中表项指向**二级索引块**，二级索引块中表项才是数据块号
  - 可扩展至三级、四级……
- **优点**：
  - 极大地**扩展了文件的最大容量**，提升可扩展性和查找效率
- **缺点**：
  - 层次越多，访问数据所需的**磁盘 I/O 次数越多**
    - 二级索引：读一级索引块 1 次 + 读二级索引块 1 次 + 读数据块 1 次 = **$k + 1$ 次**（k 为级数）
  - 小文件也要多次 I/O，浪费
- **最大文件长度**：
  - 设块大小 = 4KB，每表项 4B → 每索引块可存 1024 个表项
  - 二级索引最大文件：$1024 \times 1024 \times 4\text{KB} = 4\text{GB}$
  - 三级索引最大文件：$1024^3 \times 4\text{KB} = 4\text{TB}$

### 混合索引分配 (Unix / Linux inode)

- **定义**：综合多种索引方式的策略，兼顾**小文件的高效访问**和**大文件的容量支持**
- **典型实现**：Unix System V / Linux ext2 的 **inode（索引节点）结构**
  - 每个 inode 中有 **13 个地址项**（以 ext2 为例，不固定）：
    - **直接地址**（前 10 个）：直接存数据块的物理块号（适合小文件，0~48KB）
    - **一级间接地址**（第 11 个）：指向一个一级索引块（中文件）
    - **二级间接地址**（第 12 个）：指向二级索引块（较大文件）
    - **三级间接地址**（第 13 个）：指向三级索引块（巨型文件）
- **优点**：
  - **小文件零开销**：前 10 个块直接访问，无额外索引 I/O
  - **中等文件一次间接**：开销可控
  - **大文件多级间接**：容量巨大
  - 综合了连续分配的效率和索引分配的灵活性
- **最大文件容量计算**（重点计算题）：
  - 假设块大小 = 1KB，地址项 4B → 每索引块 256 个表项
  - 直接地址：$10 \times 1\text{KB} = 10\text{KB}$
  - 一级间接：$256 \times 1\text{KB} = 256\text{KB}$
  - 二级间接：$256^2 \times 1\text{KB} = 64\text{MB}$
  - 三级间接：$256^3 \times 1\text{KB} = 16\text{GB}$
  - 最大文件长度 = 16GB + 64MB + 266KB

# 文件的操作

操作系统通过**系统调用**向用户提供文件操作接口。

## 文件的基本操作

### 创建文件

- **系统调用**：`create(filename, mode)` 或 `open(filename, O_CREAT | O_WRONLY, mode)`
- **执行步骤**：
  1. **合法性和权限校验**：检查用户是否有在目标目录下创建文件的权限（写权限 + 执行权限）；检查文件名是否合法（无非法字符、不超长、不与现有文件重名）；若有一条不满足则创建立即终止
  2. **分配索引节点（inode）**：从 inode 表（或超级块管理的空闲 inode 位图）中分配一个空闲 inode，将文件类型、访问权限、所有者、时间戳等信息写入 inode；将该 inode 标记为已用
  3. **分配磁盘块**：根据文件系统策略，为文件分配初始数据块（创建空文件时可能暂不分配，延迟到首次写入）；从**空闲块管理机制**（空闲链表 / 位示图 / 成组链接法）中获取空闲块号
  4. **在目录中新建目录项**：在父目录的数据块中找到可用位置，写入一条新目录项，包含文件名和 inode 号，使文件可以被检索
  5. **更新文件系统元数据**：更新超级块中的空闲 inode 计数和空闲块计数；更新父目录的修改时间（mtime）；将目录数据写回磁盘
  6. **打开或返回文件描述符（可选）**：若创建后立即打开，则系统将在进程的打开文件表中创建表项，分配文件描述符，并初始化读/写指针

### 删除文件

- **系统调用**：`unlink(filename)` / `remove(filename)`
- **执行步骤**：
  1. **路径解析与定位**：在目标目录中按文件名查找对应的目录项，得到 inode 号；若文件不存在则返回错误（`ENOENT`）
  2. **权限校验**：检查用户对父目录是否有写权限+执行权限（删除目录项即修改目录内容）
  3. **检查使用状态与硬链接计数**：若文件正在被打开，则**部分系统将延迟物理删除**，待所有引用关闭后再回收资源；若硬链接计数大于 1，则仅将 inode 中的**硬链接计数减 1**；**不释放 inode**，只删除当前目录项；若计数等于 1，则执行完整删除流程
  4. **删除目录项**：使文件再用户视角下消失，完成**逻辑删除**
  5. **释放索引节点**：当链接计数将为 0 时，将 inode 标记为空闲，并归还至空闲管理结构，使其**元数据彻底失效**
  6. **释放文件所占磁盘块**：根据 inode 中记录的数据块指针（直接块/间接块），将所有数据块归还给空闲块管理机制（如位示图对应位置 0，或链入空闲链表）；对多级索引还需逐级释放间接索引块
  7. **释放内存临时资源**（可选）：包括对应的缓冲区、打开文件表项及内核文件对象等
  8. **更新文件系统元数据**：更新目录大小，空闲 inode 表、空闲块结构及父目录的修改时间，确保**一致性**

> [!warning] 删除文件的真正含义
> `rm` 命令删除的其实不是文件数据，而是**目录项**，并将 inode 的链接计数 −1。只有当链接计数降为 0 且没有进程打开该文件时，系统才真正释放 inode 和数据块。这也是为什么可以"删除正在被进程使用的文件"——进程持有文件的打开句柄，文件数据在进程关闭前不会被释放。
### 读文件

- **系统调用**：`read(fd, buf, count)`
- **前置条件**：文件必须先 `open()` 获得 fd，且打开模式允许读
- **执行步骤**：
  1. **定位 FCB**：根据 fd → 进程打开文件表 → 系统打开文件表，找到对应文件的 inode/FCB
  2. **权限校验**：检查 inode 中记录的权限位，确认用户/组/其他用户是否拥有读权限
  3. **计算目标物理块号**：根据**当前读写指针（文件偏移量）**，结合文件物理结构定位数据块
  4. **读取物理块到内核缓冲区**：发出磁盘 I/O 请求，将目标物理块内容读入内核的缓冲区缓存
  5. **复制到用户空间**：将所需字节从内核缓冲区**复制到用户提供的 buf**（`copy_to_user`）；若文件结束（EOF）则提前返回实际读取字节数
  6. **更新读写指针**：将进程打开文件表中的文件偏移量**向后移动实际读取的字节数**
- **磁盘 I/O 次数与物理结构密切相关**，是常考计算题（见 [[#文件操作与物理结构的关系]]）

### 写文件

- **系统调用**：`write(fd, buf, count)`
- **前置条件**：文件必须先 `open()` 获得 fd，且打开模式允许写
- **执行步骤**：
  1. **定位 FCB 并校验权限**：根据 fd 找到 inode/FCB，检查写权限
  2. **计算写入位置**：根据当前读写指针和 count，确定写入涉及的物理块范围
  3. **必要时分配新磁盘块**：若写入位置超出当前文件末尾：
     - 连续分配：检查相邻空间是否足够，**不够则需整体搬迁文件**（最坏情况）
     - 链接分配：从空闲块管理中逐个分配新块，链入文件末尾
     - 索引分配：分配新数据块，并在索引表中添加对应表项；若索引表已满，还需分配间接索引块
  4. **写入数据**：将用户 buf 中的数据复制到内核缓冲区，标记为脏（dirty），由 OS 在合适时机异步写回磁盘（**延迟写 / 回写策略**）；也可使用同步写（`O_SYNC`）直接落盘
  5. **更新读写指针**：将文件偏移量向后移动实际写入的字节数
  6. **更新文件元数据**：若文件长度增长，更新 inode 中的文件大小（size）和修改时间（mtime）；若分配了新块，还需更新超级块的空闲块计数
- **延迟写回**：现代 OS 普遍采用缓冲区缓存 + 回写策略来减少磁盘 I/O——数据先写入内存缓存即返回成功，脏页由后台 pdflush / bdflush 线程异步刷盘

## 文件的打开与关闭

用户**首次访问**必须先通过**系统调用**将其打开；系统维护一个称为**打开文件表**的数据结构，用于记录当前活跃文件的信息；
>[!note] 打开
>打开是指系统根据文件路径定位到其目录项后，将关键信息载入内存，在打开文件表中创建一个表目，并将该表目的*索引号（文件描述符）* 返回给用户；此后用户再次操作该文件时，只需要提供文件描述符，系统即可直接从内存获取所需信息，显著降低目录检索开销

为提升效率和管理灵活性，现代操作系统采用**两级打开文件表结构**：
- **系统打开文件表**：全局共享，存储与进程无关的文件属性，如磁盘位置、文件大小、访问时间等；每项都包含一个**打开计数器（Open Count）**，记录有多少进程打开了该文件
- **进程打开文件表**：每个进程独有，记录进程私有的访问状态，如当前读/写指针、访问权限及文件打开模式；**该表中的每一项⭐指向⭐系统打开文件表的对应项，实现资源共享与独立控制的同一**
当一个进程成功打开文件时，系统首先确保其在系统打开文件表中存在对应项，若不存在，则创建，随后在该进程的打开文件表中新增一项，指向该系统表项；此后若其他进程打开同一文件只需要在其各自的进程表中添加新项并指向同一系统表项，无序重复加载文件元数据，既保证了各进程访问上下文的隔离，又实现了底层资源的高效共享

### 被打开文件关键信息
- **文件指针**：记录当前读/写位置（文件偏移量），属于进程打开文件表，各进程**独立**
- **文件打开计数**：记录有多少进程打开了该文件，属于系统打开文件表，计数为 0 时方可回收资源
- **文件磁盘位置**：文件数据在磁盘上的物理位置信息，缓存于系统打开文件表中，避免重复读盘
- **访问权限**：进程打开文件时声明的访问模式（读/写/执行），用于后续操作校验，属于进程打开文件表

### 文件打开的完整过程

- **系统调用**：`open(pathname, flags, mode)`
- **执行步骤**：
  1. **路径解析**：根据路径名逐级查找目录项；将所需的目录文件从磁盘加载至内存，定位目标文件的目录项；随后从中提取其索引节点编号，并以此为关键字在系统打开文件表中查找对应项；
  2. **在系统打开文件表中建立/复用表项**：检查该文件 inode 是否已在系统打开文件表中存在；若存在则打开计数器 +1，否则**从磁盘读取该索引节点至内核缓冲区**，**在系统表中**新建表项记录 inode 指针、打开计数 = 1、文件大小等全局共享信息
  3. **在进程打开文件表中分配表项**：在当前进程 PCB 的打开文件表中分配空闲表项，初始化读写指针 = 0、访问模式，并指向系统打开文件表对应项
  4. **返回文件描述符（fd）**：将进程打开文件表中的表项索引作为 fd 返回给用户进程，fd 取当前进程可用的最小非负整数；此后进程通过 fd 直接定位，无需再次路径解析和权限校验
>[!important] 文件名和文件描述符
>文件名并非打开文件表的组成部分；一旦系统通过文件名完成对磁盘上 FCB 的定位，后续操作便不再需要该文件名；用于访问打开文件表的索引号，也称**文件描述符（Linux）或文件句柄（Windows）**；因此只要文件处于打开状态，所有文件操作均通过该索引号完成

### 文件关闭
当文件不再使用时，用户调用 `close` 关闭它；系统随即从该进程的打开文件表中删除对应项，并将系统打开文件表中相应项的打开计数器减 1；一旦减至 0，表明该文件以无任何进程引用，系统便可安全的删除其在系统表中的项，**将文件当前控制信息（即 FCB）从内存写回磁盘**，并释放所占用的相关资源

# 文件的共享

多个用户/进程同时使用同一文件，系统中仅需保留一个物理副本

## 基于索引节点的共享（硬链接）

- **原理**：多个目录项指向**同一个索引节点（inode）**
- **实现**：inode 中维护**链接计数（link count）**
- **删除处理**：仅当链接计数减至 0 时才真正删除文件和释放空间
- **特点**：
  - ✅ 共享彻底，节省空间
  - ❌ 不能跨文件系统（不同文件系统 inode 编号空间独立）

> [!note] 硬链接的本质
> 文件在文件系统中的真正"身份"是 **inode 号**，文件名只是指向 inode 的"标签"。多个标签指向同一个 inode 是完全合法的——它们**共享同一个 inode、同一份数据**。
## 基于符号链接的共享（软链接 / 符号链接）

- **原理**：创建一个 **LINK 类型的新文件**，其内容是目标文件的**路径名**
- **实现**：访问符号链接时，OS 解析路径 → 找到目标文件
- **特点**：
  - ✅ **可跨文件系统**
  - ✅ 可链接目录
  - ❌ 需额外读取 LINK 文件，访问开销大（多一次寻址）
  - ❌ 目标文件删除后符号链接失效（**悬空链接/断链**）

>[!note] 用网络文件共享
>利用符号链实现网络文件共享时，只需记录文件所在机器的网络地址及文件路径名


## 硬链接 vs 软链接 对比

| | 硬链接 | 软链接（符号链接） |
|------|------|------|
| **本质** | 多个目录项指向同一 inode | 新建文件存路径 |
| **能否跨文件系统** | ❌ 不能 | ✅ 能 |
| **能否链接目录** | ❌ 通常不能 | ✅ 能 |
| **目标删除后** | 仍在（链接计数 > 0 时） | 断链失效 |
| **inode** | 与源文件相同 | 不同（新 inode） |
| **额外磁盘 I/O** | 无 | 有（需读 LINK 文件） |

# 文件的保护

保证文件不被未授权用户访问或破坏，文件的访问通常由**用户访问权限和文件属性**共同决定

## 访问类型

| 类型 | 说明 |
|------|------|
| **读（Read）** | 查看文件内容 |
| **写（Write）** | 修改文件内容 |
| **执行（Execute）** | 将文件装入内存并执行 |
| **追加（Append）** | 只能在文件末尾添加数据 |
| **删除（Delete）** | 删除文件 |
| **列表（List）** | 列出文件名及属性 |

>[!note] 高层操作控制
>系统还可以对*重命名、复制、编辑*等高层操作进行控制，这些功能通常由系统调用实现，真正的保护机制仅在底层提供


## 保护机制

### 访问控制矩阵（Access Control Matrix）

- 行 = 域（用户/进程），列 = 对象（文件）
- 矩阵元素 = 访问权限集合
- ❌ 矩阵很大时稀疏，浪费空间

### 访问控制表（ACL，Access Control List）

- **按列划分**访问控制矩阵，每个文件维护一个 ACL
- ACL 记录：哪些用户/组对该文件拥有什么权限
- **精简用户划分**：为避免按逐用户列权限导致表项膨胀，ACL 通常将用户划分为三类——**所有者（Owner）、同组用户（Group）、其他用户（Other）**，每类分配一组 rwx 权限位，每种访问权限用一个二进制位标识，大幅压缩存储开销
  - 优点：支持灵活而复杂的访问策略
  - 缺点：列表长度不可预知，可能带来复杂的空间管理开销

### 权能表（Capability List）

- **按行划分**访问控制矩阵，每个用户持有一张权能表
- 每个权能 = (对象, 访问权限)
- 类似"钥匙"，持钥匙即可访问

### 口令保护

- 在创建文件时设定一个口令，系统将其存放在 FCB 中，并告知授权共享的其他用户，访问需要提供正确口令
  - 优点：实现简单，空间开销小
  - 缺点：口令易泄露，通常以明文或弱保护形式存储与系统结构内部，安全性较低；无法精确控制权限粒度（要么全有，要么全无）

### 加密保护（密码）

- 对文件内容进行加密存储，只有持有密钥的用户能解密访问
  - 优点：安全性高，不额外占用元数据空间
  - 缺点：加密/解密开销大

>[!note] 口令和加密
>主要用于防止文件被非法获取或窃取，并**不区分具体的访问操作类型**，因此，他们无法替代基于权限的访问控制，而应视为补充手段
