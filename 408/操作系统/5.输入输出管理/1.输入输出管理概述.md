# I/O 设备

I/O 设备管理的核心是通过**软硬件协作**，屏蔽不同设备的物理差异，使应用程序能以统一方式使用各类设备；一个 I/O 设备通常由两部分组成：
- **设备**本身是执行具体输入或输出操作的物理装置，如磁盘的盘片和磁头
- **设备控制器**则是连接到系统总线上的专用电路，如接口芯片，负责将操作系统发出的抽象命令转换为设备可识别的电信号，并协调主机与设备之间的数据传输
## I/O 设备的分类
### 按信息传输速率分类
- **低速设备**：传输速率为每秒几个字节至数百字节，如键盘、鼠标等
- **中速设备**：传输速率为每秒数千字节至数万字节，如打印机、激光打印机等
- **高速设备**：传输速率为每秒数十万字节至千兆字节，如磁盘机、光盘机等
### 按信息交换单位分类
- **块设备**：以**数据块**为单位进行信息传输，传输速率高、支持**随机访问**（可寻址），如磁盘等；常采用 DMA 方式传输
- **字符设备**：以**字符（字节）** 为单位进行信息传输，传输速率低、不可寻址，如键盘、打印机等；常采用中断驱动方式传输
### 按使用特性分类
- **人机交互设备**：用于用户与计算机之间的信息交互，如键盘、显示器、打印机等
- **存储设备**：用于持久化存储数据，如磁盘、磁带、光盘等
- **网络通信设备**：用于计算机之间通信，如网卡、调制解调器等
### 按共享属性分类
- **独占设备**：在一段时间内只能由一个进程使用的设备，多为低速设备，如打印机；一旦分配给某个进程，便由其独占使用直至释放
- **共享设备**：允许多个进程在一段时间内**交替**使用的设备，及在逻辑上**并发**访问，尽管物理访问仍需调度协调，如磁盘
- **虚拟设备**：通过 SPOOLing 技术将独占设备改造为可被多个进程共享使用的逻辑设备，使多个进程可以并发使用，其实质使利用**高速共享设备**作为缓冲区，实现**独占设备的逻辑共享**
## I/O 接口（设备控制器）
I/O 设备通常由**机械部件**与**电子部件**组成，电子部件称为**I/O 接口**或设备控制器，在小型/微型机中通常以**芯片**形式集成在主板上
### I/O 接口的功能
- **接受和识别 CPU 命令**：接收 CPU 发来的命令并识别对应控制寄存器中的命令字
- **进行数据交换**：在设备和设备控制器之间、设备控制器和 CPU/内存之间传输数据
- **记录设备和自身状态**：通过状态寄存器供 CPU 了解设备和控制器当前状态
- **数据缓冲**：用于解决速度不匹配、减少中断开销
- **地址识别**：识别所控制的每个设备的地址，区分不同的设备
- **差错检测**：检测设备或其自身在数据传输中可能出现的错误
### I/O 接口的组成
- **CPU 与设备控制器的接口**：用于实现 CPU 与控制器之间的通信，包含数据寄存器、控制寄存器和状态寄存器，还包括三类信号线：
	- *数据线*：用于传输读/写数据、控制和状态信息
	- *地址线*：用于指定 I/O 接口中的寄存器编号
	- *控制线*：用于传送读/写等控制信号
- **设备控制器与设备的接口**：一个设备控制器可连接一个或多个设备，因此内部包含一个或多个设备接口；每个接口均可传输数据、控制和状态三类信号
- **I/O 逻辑**：用于实现对设备的控制；它通过控制线与 CPU 交互，对接收的 I/O 命令进行译码，并依据目标地址识别相应设备
## I/O 接口的类型
I/O 接口可按不同维度分类：
- **按数据传送方式**：
	- **并行接口**：一个字节（或一个字）的所有位**同时传送**，传输速度快，但需要多根数据线，适用于近距离、高速设备，如打印机（并口）、磁盘
	- **串行接口**：数据**逐位依次传送**，只需一根数据线（或一对差分线），成本低、适合远距离传输，但速率相对较低，如键盘、鼠标、 USB、网口
	- 接口需要完成相应的并/串或串/并格式转换
- **按功能灵活性**：
	- **可编程接口**：其功能可通过编程设定和修改
	- **不可编程接口**：功能由硬件固化和决定，无法通过软件修改
- **按主机访问设备的控制方式**：
	- **程序查询接口**： CPU 轮询状态寄存器，等待设备就绪
	- **中断接口**：设备就绪后向 CPU 发出中断请求
	- **DMA 接口**：支持 DMA 控制器接管总线，直接在内存与设备间传送数据
- **按设备类型**：
	- **字符设备接口**
	- **块设备接口**
	- **网络设备接口**

## I/O 端口
**I/O 端口**是指 I/O 接口电路中**可被 CPU 直接访问的寄存器**，主要包括以下三类：
- **数据寄存器（端口）**：存放主机与设备之间**交换的数据**，供 CPU 读取或写入
- **控制寄存器（端口）**：存放 CPU 发给设备的**控制命令**，用于启动设备、设定工作模式等
- **状态寄存器（端口）**：存放设备的**当前状态**，供 CPU 查询（如忙/闲、就绪、出错等）

### 端口编址方式
I/O 端口需要由 CPU 进行寻址访问，常用的编址方式有两种：

#### 独立编址（I/O 映射方式）
I/O 端口使用**独立于主存的地址空间**，与内存地址空间相互独立；虽地址值可能相同，但分属不同地址空间，不会发生冲突

**优点**：
- **程序结构清晰**： I/O 操作使用专门的 I/O 指令（如 `IN`、`OUT`），与访存指令（如 `MOV`、`LOAD`）明确区分，程序**可读性好**、便于调试和维护
- **译码电路简单**： I/O 地址空间通常比内存地址空间小得多（如 64KB），所需**地址线较少**，译码电路相对简单、寻址速度快

**缺点**：
- **编程灵活性差**：专用 I/O 指令功能单一（通常仅为读/写端口），无法直接进行算术运算、逻辑运算等复杂操作；若要处理端口数据，必须先读入 CPU 寄存器再运算
- **硬件复杂度增加**： CPU 需要提供额外的控制信号来区分当前是访问内存还是 I/O 端口，总线设计更复杂

#### 统一编址（存储器映射方式）
I/O 端口与内存**统一编址**，将端口视为内存地址空间的一部分，每个 I/O 端口被分配一个唯一的内存地址。

**优点**：
- **编程灵活、功能强**：无需专用 I/O 指令，所有对内存操作的指令（如 `MOV`、`ADD`、`AND`、`TEST` 等）均可直接作用于 I/O 端口，可对端口数据直接进行算术/逻辑运算，大大增强了 I/O 处理能力
- **寻址方式丰富**：可利用内存指令的全部寻址方式（立即寻址、直接寻址、寄存器间接寻址、变址寻址等）访问外设，程序编写更便捷高效，同时也获得**较大的编址空间**
- **便于统一管理与保护**： I/O 端口与内存共享同一地址空间，可借助已有的内存管理机制（段页式管理、 MMU）对 I/O 端口实施**访问保护**，防止非法访问
- **硬件设计简化**：不需要额外的 I/O 控制信号来区分访存与访外设，总线控制逻辑更简洁

**缺点**：
- **占用内存地址空间**：部分内存地址被分配给 I/O 端口，导致**可用的物理内存地址减少**；在小型系统或地址空间紧张时影响较大
- **译码电路复杂、速度降低**： I/O 端口地址位数与内存地址位数一致，需要**全部地址线参与译码**，译码电路更复杂，延迟更大，可能降低端口访问速度
- **程序可读性下降**：端口访问与普通内存访问使用相同的指令，在源代码层面不易区分是对内存还是对设备操作，程序的可读性和可维护性受到一定影响
- **指令字长较长**：访存类指令通常比专用 I/O 指令更长，指令取指和执行周期可能更多

>[!important] 独立编址 vs 统一编址 对比
>| 对比维度 | 独立编址 | 统一编址 |
>|---------|---------|---------|
>| 地址空间 | I/O 与内存**独立** | I/O 与内存**统一** |
>| 访问指令 | 专用 I/O 指令（`IN`/`OUT`） | 普通访存指令（`MOV`等） |
>| 占用内存空间 | 不占用 | 占用部分地址空间 |
>| 编程灵活性 | 较弱，指令功能单一 | 强，指令丰富 |
>| 寻址方式 | 少 | 多（所有访存寻址方式） |
>| 程序可读性 | 好（I/O 与访存指令分明） | 差（难以区分 I/O 与访存） |
>| 译码电路 | 简单，速度快 | 复杂，速度较慢 |

# I/O 控制方式
I/O 控制的核心在于**如何在 CPU、内存与外部设备之间高效地传输数据**，四种控制方式在 CPU 的干预程度、数据传输效率和系统并发性方面逐步提升
## 程序直接控制方式（轮询方式）
CPU 在向设备控制器发出 I/O 指令后，**不断循环检查**状态寄存器，直至数据准备就绪后**亲自**完成一个字的数据传输，再继续轮询等待下一个数据就绪
- **CPU 与 I/O 串行工作**： CPU 在等待期间持续"忙等"，无法切换到其他任务
- **CPU 干预频率极高**：每个数据字都需 CPU 介入传输
- **优点**：控制简单，硬件开销小
- **缺点**： CPU 利用率极低，无法实现并发

## 中断驱动方式
CPU 发出 I/O 命令后**继续执行其他程序**，当 I/O 设备完成数据准备后，向 CPU 发出**中断信号**， CPU 响应中断后转入中断服务程序完成数据传输
### 工作流程
1. CPU 向控制器发出读/写命令，然后**切换到其他进程继续执行**
2. I/O 设备独立工作，完成数据准备
3. 设备通过中断请求信号通知 CPU 数据已就绪
4. 在每个**指令周期的末尾**， CPU 检查中断信号
5. 当有来自设备的中断请求时， CPU 响应中断，保存当前进程的上下文，转而执行中断服务程序，完成**一个字**的数据传输
6. 中断处理完成后恢复原进程执行（或进行新一轮调度）
### 特点
- **CPU 与 I/O 并行工作**：提高了 CPU 的利用率
- **CPU 干预频率仍较高**：每传输一个字仍需 CPU 介入一次
- **优点**：实现了 CPU 与 I/O 设备的并行，系统效率大幅提升
- **缺点**：每个数据字的传输都触发一次中断，中断开销大，适用于低速字符设备；设备与内存之间交换数据必须经过 CPU 中的寄存器

## DMA 方式（直接存储器访问）
在**内存**与 **I/O 设备**之间开辟**直接数据传输通路**，设置专门的**DMA 控制器**负责数据块的传输工作， CPU 仅在数据传输开始和结束时进行干预
### DMA 控制器的组成
- **命令/状态寄存器(CR)**：接收 CPU 发来的 I/O 命令或记录当前状态
- **内存地址寄存器(MAR)**：存放数据在内存中的起始地址，输入时（设备→内存），表示数据要写入内存的目标地址；输出时（内存→设备），表示数据从内存读取的源地址
- **数据寄存器(DR)**：暂存**单次传输**的数据单元（一个字）
- **数据计数器(DC)**：存放本次 DMA 传输的剩余字（节）数
### 工作流程
1. CPU **进入内核态**向 DMA 控制器写入 MAR、 DC 等参数，发出启动命令
2. CPU 将 I/O 控制权交予 DMA 控制器，转而执行其他任务，DMA 控制器接管总线后，以**突发**方式将整块数据在设备与内存之间**直接传输**，无需 CPU 干预独立完成数据块传输
3. DMA 控制器每次传送一个字， DC 减 1， MAR 加 1，直至 DC 为 0
4. DMA 控制器向 CPU 发出**中断信号**，通知数据传输完毕
5. CPU 响应中断，进行后续处理
### DMA 与主存的通信方式
- 见计算机组成原理
- **停止 CPU 访问主存（成组传送）**： DMA 传输期间 CPU 放弃总线控制权，优点控制简单，缺点 CPU 利用率受影响
- **周期挪用**：当 CPU 与 DMA 同时访存时， DMA 优先获取一个或几个总线周期，称为*周期挪用/窃取*，这是最常用的方式
- **交替访问**：将 CPU 周期分为 C1 和 C2 两部分， CPU 与 DMA 分别在不同部分交替访存，实现无冲突并行操作，但硬件复杂
### 特点
- **数据传输单位是块**：一次传输一个或多个连续的数据块
- **数据流向**：设备直接与内存交换数据（无需经过 CPU）
- **仅在块传输开始和结束时需要 CPU 干预**，传输过程中 CPU 可以执行其他程序
- **优点**：大幅减少了 CPU 干预，传输效率高，适用于高速块设备
- **缺点**：每个 DMA 控制器仅能控制同类设备，多个 DMA 同时使用时总线竞争问题需要解决
## 通道控制方式

通道是一种特殊的 **I/O 处理器**（本质是处理 I/O 指令的专用 CPU），是一种**硬件机制**；有自己的指令系统（**通道指令/通道命令字 CCW**），可独立执行由通道指令编写的**通道程序**，实现与 CPU 更高程度的并行
### 通道与 CPU 的区别
- 通道的指令类型单一，仅限于 I/O 相关操作
- 通道与 CPU **共享内存**，但通道本身没有独立内存
- CPU 通过 I/O 指令控制通道，通道通过通道指令控制设备控制器
### 通道的类型
- **字节多路通道**：以**字节**为单位交叉传输，分时为多台低速/中速字符设备服务
- **数组选择通道**：以**数据块**为单位传输，一次仅选择一个设备，待传输完成后再服务其他设备，传输速率高但通道利用率低
- **数组多路通道**：综合前两者特点，以**数据块**为单位，但允许多个设备**交替**传输，兼顾高传输速率和通道利用率

### 特点
- **CPU、通道、设备控制器和设备四级并行**工作
- 通道通过执行程序**动态确定参数**，能自主完成设计多台设备、多个阶段的复杂 I/O 任务
- CPU 干预程度最低，仅需在启动和结束时介入


>[!important] 四种 I/O 控制方式对比
>| 控制方式 | CPU 干预频率 | 数据传输单位 | 数据传输路径 | 并行程度 |
>|---------|-----------|-------------|-------------|---------|
>| 程序直接控制 | 每字节/字 | 字 | CPU↔I/O↔内存(经 CPU) | 串行 |
>| 中断驱动 | 每字节/字 | 字 | CPU↔I/O↔内存(经 CPU) | CPU 与 I/O 并行 |
>| DMA | 块开始和结束 | 块 | I/O↔内存(直接) | 更高并行 |
>| 通道 | 通道程序开始和结束 | 多块(一组块) | I/O↔内存(直接) | 四级并行 |

# I/O 软件层次结构
将 I/O 软件划分为多个层次，实现**逐层抽象**，使得上层软件可以透明地使用各类设备，具有可扩展、易移植和灵活性好的特点
## 用户层 I/O 软件
- 位于**用户态**最上层，包括库函数（如`printf`、`scanf`）和 SPOOLing 系统等
- 将用户 I/O 请求**翻译**为对应的系统调用，转入内核态执行
- 用户层库函数对系统调用进行封装，为用户提供更友好的编程接口
## 设备独立性软件（设备无关软件层）
- 设备独立性是指应用程序在使用 I/O 设备时，无需依赖具体的物理设备，为实现这一特性，系统引入了**逻辑设备**和**物理设备**的概念：应用程序使用逻辑设备名请求某类设备，系统在运行时，必须将逻辑设备名映射为实际物理设备名
- 位于**内核态**，是 I/O 软件的核心层，屏蔽不同设备的差异和向上层提供统一接口；参见[[2.设备独立性软件]]
- 主要功能：设备命名与映射、设备保护、缓冲管理、差错处理、逻辑设备到物理设备的映射、设备分配与回收
## 设备驱动程序
- 位于**内核态**，与硬件设备直接相关，**每类设备对应一个专用的驱动程序**，**多个同类设备只需要一个设备驱动程序**
- 是 I/O 软件中与设备控制器直接交互的**唯一一层**，是操作系统控制硬件的最终执行者
- 设备驱动程序的主要任务是：接受上层软件（如设备独立层）发出的抽象 I/O 请求，将其转换为**设备控制器能够识别的具体指令**，启动设备执行相应操作，并将设备控制器返回的状态信息或数据传递回上层

### 设备驱动程序的工作流程
以一次 I/O 请求（如 `read()`）为例，驱动程序按以下步骤工作：

1. **将抽象请求转换为具体操作**：上层请求通常是逻辑性的，而设备控制器只能处理物理操作；驱动程序根据设备特性，将逻辑地址转换为物理参数
2. **校验请求的合法性**：检查请求的参数是否有效——如缓冲区地址是否合法、读写范围是否越界、设备是否支持所请求的操作等。若校验失败，立即返回错误码，不启动设备
3. **检查设备状态**：读取设备控制器的状态寄存器，判断设备当前是否空闲可用。若设备正忙，则将请求**加入请求队列**并阻塞调用进程，待设备空闲后再从队列取出处理
4. **传递参数**：将转换后的命令字和参数**写入设备控制器的各寄存器**——命令字写入控制寄存器、数据缓冲区地址写入地址寄存器、传输字节数写入计数寄存器等；同时在内核空间中构造设备控制块（DCB），记录本次 I/O 请求的上下文信息（缓冲区地址、传输字节数、等待进程 ID 等）
5. **启动设备**：向控制寄存器写入启动位（或将命令字写入命令寄存器自动触发启动），设备控制器开始独立执行操作，CPU 转去调度其他进程；若为阻塞 I/O，当前进程进入阻塞态等待
6. **处理后续结果**：I/O 完成后，设备发出中断信号，**驱动程序的 ISR（中断服务例程）** 被调用——读取状态寄存器判断传输是否成功，若出错则记录错误信息并决定是否重试；若成功则将数据从设备数据寄存器**复制到用户缓冲区**（或内核缓冲区），更新 DCB 中的传输状态，最后**唤醒阻塞进程**，返回本次传输的字节数或错误码

>[!tip] 驱动程序与中断的协作
>在整个工作流程中，驱动程序的 `read()` 入口负责**发起 I/O 请求并阻塞进程**，而驱动的 ISR 负责**完成数据传输并唤醒进程**——两者通过共享的数据结构（设备控制块）传递状态和缓冲地址，形成"发起—等待—完成—唤醒"的协作闭环

## 中断处理程序
- 位于**内核态**最底层，与硬件紧密相关
- 当 I/O 设备完成操作时，硬件向 CPU 发出**中断信号**， CPU 响应并转入中断处理程序
- 中断处理程序主要任务：
	1. **检测并响应中断**：设备完成一个数据单位传输后，其控制器发出中断请求， CPU 在**每条指令执行完毕后**检查中断引脚/信号线；若有中断请求且中断使能位（IF）为开，则 CPU 发送中断应答信号（INTA），进入中断响应周期
	2. **保护被中断的进程**：**硬件自动**将关键寄存器（如程序状态字 PSW、程序计数器 PC/指令指针 IP）压入栈中；随后中断处理程序将剩余的通用寄存器、段寄存器等现场信息保存到**进程控制块（PCB）** 或内核栈中，确保中断处理完成后能正确恢复
	3. **转入相应设备的中断处理程序**： CPU 识别中断源，根据**中断向量号**查询**中断向量表**（实模式下位于内存低地址的 IVT，或保护模式下的 IDT），获取对应 ISR 的入口地址并跳转执行
	4. **处理中断**：⭐**执行该设备的 ISR**，具体包括：读取设备状态寄存器判断完成/出错、将数据从设备寄存器复制到内存缓冲区、向中断控制器发送 **EOI 信号**（End Of Interrupt）以允许接收新的中断、唤醒等待该 I/O 的进程等
	5. **恢复现场并返回**： ISR 执行 `IRET`（中断返回指令），硬件依次从栈中弹出 PC/PSW 等寄存器的值，恢复被中断进程的上下文，CPU 从被中断的指令的下一条继续执行原进程（或触发进程调度，切换到更高优先级的就绪进程）
- 为确保上述流程可靠运行，操作系统需对中断处理程序本身进行管理，主要包括三个环节：
	- **注册中断**：设备驱动程序初始化时，调用系统接口将 ISR 的入口地址**注册到中断向量表的对应表项**中，并设置触发方式（电平触发/边沿触发）；此后该中断号便被"占用"，可响应来自硬件的请求，**建立硬件中断信号与服务代码之间的映射**
	- **处理中断**：中断发生后 CPU 硬件自动走完检测→保护→查表→跳转的流程，进入 ISR 执行具体处理逻辑；处理过程中可能需要**屏蔽同级或低级中断**（通过设置中断屏蔽字）；由于运行在中断的上下文中，处理程序必须快速、非阻塞、不得调用可能引起进程睡眠或调度的函数
	- **注销中断**：设备卸载或驱动退出时，将中断向量表中的对应表项**恢复为默认值或清空**，释放该中断号；同时通知中断控制器屏蔽该中断源，防止设备移除后仍产生无效中断；若中断请求线为共享，则仅移出本设备的处理例程，该中断线需要等到所有关联设备均注销后，才会被最终禁用

>[!important] I/O 软件层次总结
>自上而下：用户层 I/O 软件 → 设备独立性软件 → 设备驱动程序 → 中断处理程序 → 硬件
>- 上层调用下层，下层为上层提供服务
>- 越上层越通用、越与设备无关；越下层越与设备特性紧密相关
>- 层次间通过**明确的接口**进行通信，实现模块化设计

# 应用程序 I/O 接口
操作系统为应用程序提供了统一的 I/O 接口，屏蔽了底层硬件差异
## 设备类型接口
- **字符流接口**：
	- 如`get()`、`put()`，用于字符设备（键盘、串口等），只能顺序读取，按字节流方式顺序访问；
	- `in-control`指令，由于字符设备种类繁多，差异较大，用于指定与具体设备相关的特定功能
- **块接口**：如`read()`、`write()`、`seek()`，用于块设备（磁盘等），支持随机定位访问
- **网络套接字接口**：如`socket()`、`bind()`、`send()`、`recv()`，用于网络通信设备
## 阻塞 I/O 与非阻塞 I/O
- **阻塞 I/O**：进程发出 I/O 请求后进入阻塞态，等待 I/O 操作完成才被唤醒
	- **优点**：编程模型简单直观，进程顺序执行，无需额外的轮询或回调机制
	- **缺点**：进程在等待期间无法做任何工作，CPU 利用率低；不适合需要同时处理多个 I/O 的场景（如网络服务器）
- **非阻塞 I/O**：进程发出 I/O 请求后**立即返回**，不阻塞进程；进程通过轮询判断 I/O 是否完成
	- **优点**：进程在等待 I/O 期间可执行其他任务，提高了进程的并发能力
	- **缺点**：需要反复轮询设备状态，轮询期间大量 CPU 时间被浪费在"检查是否完成"上；编程复杂度增加
- **异步 I/O**：进程发出 I/O 请求后立即返回继续执行，当 I/O 完成后由内核通过信号或回调机制通知进程，进程在等待期间完全不被阻塞
	- **优点**：兼具非阻塞 I/O 的并发能力，又避免了轮询的 CPU 浪费；进程与 I/O 操作完全并行
	- **缺点**：编程模型复杂（回调/信号处理），调试困难；内核支持开销较大


# 总结

> [!summary] 本章重点
> 
> **一、I/O 设备的分类**
> 
> 1. **按传输速率**：低速（键盘/鼠标）→ 中速（打印机）→ 高速（磁盘）
> 2. **按信息交换单位**：**块设备**（数据块、可寻址、DMA）vs **字符设备**（字节流、不可寻址、中断驱动）
> 3. **按使用特性**：人机交互设备 / 存储设备 / 网络通信设备
> 4. **按共享属性**：独占设备 → 共享设备（交替并发）→ 虚拟设备（SPOOLing 技术，独占设备的逻辑共享）
> 
> **二、I/O 接口（设备控制器）**
> 
> 1. **功能（6 项）**：接受识别 CPU 命令 → 数据交换（设备↔控制器↔CPU/内存）→ 记录状态 → 数据缓冲（速度匹配）→ 地址识别 → 差错检测
> 2. **组成（3 部分）**：**CPU 与控制器接口**（数据线/地址线/控制线，含数据、控制、状态三类寄存器）→ **控制器与设备接口**（可连多个设备）→ **I/O 逻辑**（命令译码、地址识别）
> 3. **类型（4 维分类）**：按传送方式（并行/串行）→ 按灵活性（可编程/不可编程）→ 按控制方式（程序查询/中断/DMA）→ 按设备类型（字符/块/网络）
> 
> **三、I/O 端口**
> 
> 1. **三类寄存器（端口）**：**数据寄存器**（交换数据）→ **控制寄存器**（CPU 发命令）→ **状态寄存器**（设备状态：忙/闲、就绪、出错）
> 2. **端口编址方式**：**独立编址**（I/O 映射，`IN`/`OUT` 指令，不占内存地址，程序清晰但指令功能单一）vs **统一编址**（存储器映射，`MOV` 等访存指令，编程灵活、可利用 MMU 保护，但占用内存地址、译码复杂）
> 
> **四、I/O 控制方式（4 种，CPU 干预递减、并行程度递增）**
> 
> 1. **程序直接控制（轮询）**：CPU 不断循环检查状态寄存器，忙等；CPU 与 I/O 串行，利用率极低；简单但仅适用低速简单系统
> 2. **中断驱动**：CPU 发命令后切换进程，设备就绪发中断，ISR 完成一个字传输；实现 CPU 与 I/O 并行，但每字一次中断开销大，适用低速字符设备
> 3. **DMA 方式**：DMA 控制器接管总线，数据在设备↔内存间直接传输，CPU 仅在开始和结束时干预；传输单位是块，大幅减少 CPU 干预，适用高速块设备；与主存通信方式：停止 CPU 访问 / **周期挪用**（最常用）/ 交替访问
> 4. **通道控制**：通道是专用 I/O 处理器，有 CCW 指令系统，可执行通道程序；CPU、通道、控制器、设备四级并行；字节多路通道（字节交叉）→ 数组选择通道（块、独占）→ 数组多路通道（块、交替）
> 
> **五、I/O 软件层次结构（自上而下逐层抽象）**
> 
> 1. **用户层 I/O 软件**：库函数（`printf`/`scanf`）和 SPOOLing 系统，翻译用户请求为系统调用
> 2. **设备独立性软件**：核心层，⭐逻辑设备→物理设备映射、设备保护、缓冲管理、差错处理、分配回收
> 3. **设备驱动程序**：内核态，每类设备一个；6 步工作流程：转换请求 → 校验合法性 → 检查状态 → 传递参数（写寄存器、构造 DCB）→ 启动设备 → ISR 处理善后；`read()` 入口与 ISR 形成"发起—等待—完成—唤醒"协作闭环
> 4. **中断处理程序**：内核态最底层；5 步流程：检测中断 → 保护现场（PSW/PC/寄存器）→ 查中断向量表 IVT/IDT 转入 ISR → 处理（读状态、传数据、发 EOI、唤醒进程）→ 恢复现场 IRET 返回；中断管理三环节：注册 → 处理 → 注销
> 
> **六、I/O 应用程序接口**
> 
> 1. **设备类型接口**：字符流接口（`get`/`put`，顺序）→ 块接口（`read`/`write`/`seek`，随机）→ 网络套接字接口（`socket`/`bind`/`send`/`recv`）
> 2. **阻塞 I/O**：进程阻塞等待完成，编程简单但 CPU 利用率低
> 3. **非阻塞 I/O**：立即返回，进程可继续工作但需轮询，CPU 仍有浪费
> 4. **异步 I/O**：内核完成后回调通知进程，完全并行无轮询开销，编程模型最复杂
