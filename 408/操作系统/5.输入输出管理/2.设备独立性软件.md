# 设备独立性软件（设备无关性软件）

设备独立性软件位于设备驱动程序之上，是**内核** I/O 子系统的最高层；该层与设备驱动程序之间的界限并非固定，常因操作系统设计或设备类型而异
设备独立性是指**应用程序独立于具体使用的物理设备**，用户在程序中使用的逻辑设备名与实际物理设备分离，由操作系统在运行时进行映射；核心职责是**实现各类设备的公共操作**，**为上层提供统一、抽象的 I/O 接口**

## 公共功能
- **提供统一的驱动程序接口**上层只需调用统一接口即可操作任意设备，无需关心底层具体实现；驱动程序只需按规范实现这些接口函数，即可被系统识别和调用
- **缓冲管理**：在内存中设置缓冲区，临时存放设备与进程之间交换的数据，以**缓和 CPU 与 I/O 设备速度不匹配**的矛盾，**减少中断频率**，**提高 CPU 与 I/O 设备之间的并行性**；具体策略包括单缓冲、双缓冲、循环缓冲和缓冲池
- **差错控制**：I/O 操作过程中可能出现的各类错误统一由该层截获和处理；产生的错误分为两类：**暂时性错误**（如网络丢包），通常通过重试纠正，仅当连续失败后才视为持久性错误，**持久性错误**（如磁盘划痕），操作系统可将其记录至坏块表，后续 I/O 自动绕过，避免更换整盘；常见的处理策略包括**重试**（重新执行失败的操作）、**忽略**（部分非关键错误）、**报告**（向调用者返回错误码，由上层决定如何处理）
- **独占设备的分配与回收**：独占设备（如打印机、刻录机）同一时刻只能由一个进程使用，该层负责**检查设备是否空闲**、**按算法分配设备**、**使用完毕后回收**；分配前需同时检查设备 → 控制器 → 通道三级是否均可用；支持静态分配（防死锁但利用率低）和动态分配（利用率高但可能死锁）
- **提供统一的逻辑数据块**：向上层屏蔽不同设备物理块大小的差异，呈现统一的**逻辑块**视图；例如磁盘的物理扇区大小可能不同（512B / 4KB），但上层看到的逻辑块大小是统一的，该层负责完成**逻辑块与物理扇区之间的映射和转换**
## 实现设备独立性的好处
- **设备分配灵活性**：系统可灵活地将逻辑设备映射到任一空闲的同类物理设备
- **易于实现 I/O 重定向**：无需修改程序即可将 I/O 数据流重定向到不同设备
- **程序可移植性**：应用程序不依赖特定物理设备，可在不同硬件配置的系统上运行
## 逻辑设备表(LUT)
操作系统通过**逻辑设备表**(Logical Unit Table, LUT)实现逻辑设备名到物理设备名的映射，LUT 每个表项包含：
- **逻辑设备名**
- **物理设备名**
- **设备驱动程序的入口地址**

# 磁盘高速缓存（Disk Cache）

OS 采用磁盘高速缓存技术以提高磁盘 I/O 性能，访问高速缓存中的数据比直接读/写磁盘更高效；磁盘高速缓存不同于通常意义下位于 CPU 和 主存之间的硬件高级缓存，而是指**利用内存中的存储空间暂存从磁盘读取的盘块数据**；这些数据在*逻辑上*是**磁盘内容的副本**，*物理上*则**驻留在内存中**；磁盘高速缓存在内存中有两种组织形式：一种是在内存中开辟固定大小的专用缓存区；另一种是将系统空闲内存作为动态缓冲池，共请求分页系统和磁盘 I/O 子系统共享使用

# 缓冲区（Buffer）
## 引入缓冲的目的
1. **缓和 CPU 与 I/O 设备速度不匹配的矛盾**：CPU 处理速度远快于 I/O 设备，缓冲可使两者更协调地工作
2. **减少对 CPU 的中断频率**：若没有缓冲，每传输一个字节就需中断一次；引入缓冲后可积累到一定数量再中断，放宽 CPU 对中断的响应时间要求
3. **提高 CPU 与 I/O 设备之间的并行性**：缓冲使数据在设备和内存之间批量传输，CPU 可在此期间处理其他任务
4. **解决数据粒度不匹配问题**：生产者与消费者之间数据单元大小不一致时，缓冲起到"数据转化"的作用
## 用户缓冲区和内核缓冲区
- 用户缓冲区是指用户进程在读取文件时，通常会预先申请一块内存区域作为缓冲区，用于暂存从文件中读取的数据；主要作用是**减少系统调用的次数，进而降低状态切换的开销**
- 内核缓冲区是操作系统在内核空间维护的缓冲区，当用户进程请求读取磁盘记录时，系统并不会直接访问磁盘，而是先检查内核缓冲区中是否已缓存所需数据；写数据时，也不是直接写入磁盘，而是先存入缓冲区，当满足一定条件才批量写入磁盘；主要目的是**提升磁盘 I/O 的整体效率，尤其对写操作具有突出优化效果**

## 缓冲区实现方式
- 采用硬件缓冲器，但由于成本较高，除一些关键部位外一般不采用
- 利用**内存**作为缓冲区
## 单缓冲
操作系统在**内存中**为 I/O 请求分配一个缓冲区，其大小通常为一个数据块
### 工作方式
- 设备先将数据输入缓冲区，装满后用户进程从缓冲区读取数据；对于块设备，必须等整块数据装入缓冲区后，才能将其传送到工作区
- 在处理当前数据块的同时可并行输入下一数据块
### 时间分析
设从磁盘读入一块数据到缓冲区的时间为 $T$，CPU 处理一块数据的时间为 $C$，将缓冲区数据传送到用户区的时间为 $M$：
- 单缓冲下，每块数据的平均处理时间约为 $\max(C, T) + M$
- 系统对每块数据的处理时间被制约，CPU 与 I/O 设备无法实现完全并行

>[!warning] 单缓冲局限
>单缓冲只能实现**部分并行**：CPU 处理当前数据块时，设备可以读入下一块到缓冲区，但但缓冲区是**共享资源**，设备与 CPU 对其访问必须**互斥**，不能同时访问同一缓冲区
## 双缓冲（缓冲对换）
在内存中设置**两个缓冲区**，采用交替使用策略
### 工作方式
- 设备向缓冲区 A 输入数据时，CPU 可从缓冲区 B 取出数据处理
- 当 A 装满、B 处理完毕时，两者**对换**：设备向 B 输入，CPU 从 A 取数据
### 时间分析
- 双缓冲下，每块数据的平均处理时间约为 $\max(C+M, T)$
- 若 $C+M \le T$（CPU 处理速度足够快），设备可连续不间断地工作，实现最大并行度
- 若 $C+M > T$，则 CPU 处理不会因等待数据而空闲

>[!important] 单缓冲 vs 双缓冲
>双缓冲消除了单缓冲中"设备和进程同时访问同一缓冲区"的限制，使设备与 CPU 的**并行程度更高**；在通信领域，双缓冲还用于实现**全双工通信**
## 循环缓冲
在内存中分配**多个大小相等的缓冲区**，并将它们链接成一个**循环链表**
### 组成
- **in 指针**：指向下一个待写入的空缓冲区，供生产者（设备）填入数据
- **out 指针**：指向下一个待读取的满缓冲区，供消费者（进程）取出数据

### 同步情形
- **in 追赶上 out**（系统受计算限制）：填充速度大于处理速度，所有缓冲区逐渐被写完但来不及取走，最终 in 追上 out（满缓冲），输入进程必须阻塞
- **out 追赶上 in**（系统受 I/O 限制）：处理速度大于填充速度，所有缓冲区逐渐被取空但来不及填入新数据，最终 out 追上 in（空缓冲），计算进程必须阻塞

## 缓冲池
缓冲池是一种包含管理数据结构和操作函数的**软件机制**，用于统一管理多个缓冲区，并支持多进程共享使用
### 缓冲池的缓冲区
缓冲区根据使用状态，这些缓冲区被组织为以下队列：
- **空缓冲队列**：由空缓冲区链接而成
- **输入队列**：由装满输入数据的缓冲区链接而成
- **输出队列**：由装满输出数据的缓冲区链接而成
缓冲区可以动态扮演以下四种角色：
- **收容输入缓冲区(hin)**：用于暂存输入数据
- **提取输入缓冲区(sin)**：用于读取输入数据
- **收容输出缓冲区(hout)**：用于暂存输出数据
- **提取输出缓冲区(sout)**：用于读取输出数据
### 四种工作方式
- **收容输入**：从空缓冲队列取空缓冲区，设备填入输入数据，完成后挂入输入队列
- **提取输入**：从输入队列取满缓冲区，进程取走数据，释放为空缓冲区挂回空缓冲队列
- **收容输出**：从空缓冲队列取空缓冲区，进程填入输出数据，完成后挂入输出队列
- **提取输出**：从输出队列取满缓冲区，设备取出数据输出，释放为空缓冲区挂回空缓冲队列

>[!important] 缓冲池与循环缓冲的区别
>循环缓冲是**专用**的（通常为某对生产者和消费者服务），而缓冲池是**公用**的——系统中的所有设备和进程都可以使用，利用率更高，管理也更复杂

# 高速缓存与缓冲区的对比

| 对比维度   | 高速缓存（Cache）                 | 缓冲区（Buffer）                       |
| ------ | --------------------------- | --------------------------------- |
| 目的     | 提高数据**重复访问**的速度，减少磁盘 I/O 次数 | 缓和 CPU 与 I/O 设备之间的**速度差异**，实现异步传输 |
| 存放内容   | 磁盘数据的**副本**（逻辑上属于磁盘，物理上在内存） | 设备与进程之间**正在传输**的临时数据              |
| 数据是否独占 | 一份数据可被多个进程**共享**读取          | 数据属于某次 I/O 传输，用完即释放               |
| 命中/未命中 | 有"命中"概念：命中则直接返回，未命中则读磁盘     | 无命中概念，数据流经缓冲区即被消费                 |
| 替换策略   | 需要淘汰算法（如 LRU）管理有限空间         | 按生产者-消费者模型自然流转，不涉及替换              |
| 生命周期   | 数据可多次访问，长期驻留                | 数据一次性流过，缓冲区可被复用                   |
| 逻辑位置   | 位于应用程序与磁盘之间，对上层透明           | 位于设备与进程之间，作为数据中转站                 |

>[!important] 核心区别
>**高速缓存用于"存"**（保留副本供反复读），**缓冲区用于"传"**（暂存数据以协调速度）。高速缓存解决的是**多次访问同一数据**的效率问题，缓冲区解决的是**一次传输中生产与消费速度不匹配**的问题

高速缓存和缓冲区**也存在相似之处**：二者都利用内存来提升 I/O 性能，物理上都驻留在内存中，都可能使用动态内存池管理，两者**均位于高速设备和低速设备之间，用于缓解两者之间的速度差异**
# 设备分配与回收

设备分配是指根据用户的 I/O 请求分配所需设备的过程；分配总体原则是：充分发挥设备使用效率的同时，避免因分配策略不当引发进程思索

## 设备分配中的数据结构
操作系统为设备管理维护以下核心数据结构：
### 系统设备表(SDT, System Device Table)
- 全系统**唯一**，每个表项记录一类设备的信息
- 表项包含：设备类型、设备标识符、指向该类型设备控制表(DCT)的指针、驱动程序入口等
### 设备控制表(DCT, Device Control Table)
- 每个设备**一个**，记录该设备的详细信息
- 表项包含：设备类型、设备标识符、设备状态（忙/闲）、指向控制器控制表(COCT)的指针、**等待此设备的进程队列（由 PCB 组成）**、可重复执行次数等
### 控制器控制表(COCT, Controller Control Table)
- 每个设备控制器**一个**，记录控制器的状态以及所辖的设备信息
- 表项包含：控制器标识符、控制器状态（忙/闲）、指向通道控制表(CHCT)的指针、等待此控制器的进程队列
### 通道控制表(CHCT, Channel Control Table)
- 每个通道**一个**，记录通道的状态以及其下属控制器
- 表项包含：通道标识符、通道状态（忙/闲）、指向控制器表(COCT)的指针、等待此通道的进程队列
### 数据结构之间的关联
SDT → DCT → COCT ↔ CHCT，形成**一条从设备类型到物理通路的完整映射链**；当进程请求 I/O 时，系统沿着这条链依次检查设备、控制器、通道是否可用

>[!important] 反复分配尝试
>当设备分配失败（设备忙）时，进程将被放入该设备的等待队列并阻塞；系统支持多次尝试分配，设备控制表中设置**"设备等待队列"**和**"重复执行次数"**字段以实现这一机制

## 设备分配的考虑因素

### 设备固有属性
- **独占设备**：同一时刻只能分配给一个进程（如打印机），便尤其独占使用，直至进程主动释放或终止
- **共享设备**：可同时分配给多个进程（如磁盘），需要调度机制协调访问顺序
- **虚拟设备**：通过 SPOOLing 技术将独占设备改造成逻辑上的共享设备，宏观上可同时使用

### 设备分配算法
在多个进程竞争同一设备时，按某种策略决定分配顺序：
- **先来先服务**：按请求先后分配，公平但忽略紧迫性
- **优先级高者优先**：紧迫进程优先，可能造成低优先级饥饿

### 设备分配中的安全性
动态分配可能导致**死锁**，系统需采用适当策略避免死锁发生
- **安全分配方式**：每当进程发出 I/O 请求后，便立即进入阻塞状态，直至 I/O 操作完成才被唤醒，在此期间，进程不再请求任何其他资源，从而有效**避免死锁**；其优点是安全性高，缺点是 CPU 和 I/O 设备串行工作，系统效率低
- **不安全分配方式**：进程发出 I/O 请求后可以继续运行并可随时申请其他资源，仅当所请求的设备已被其他进程占用时，才进入阻塞态；优点是 CPU 与 I/O 设备可**并行工作**，系统效率高；缺点是进程可能在持有部分资源的同时等待另一部分资源，存在**死锁风险**

## 设备分配步骤

1. **分配设备**：根据 I/O 请求中的物理/虚拟设备名，在 SDT 中查找对应设备类型的 DCT，检查设备状态
   - 空闲：分配设备，进入下一步
   - 忙：进程插入该设备的等待队列，阻塞等待
2. **分配控制器**：在 DCT 中找到关联的 COCT，检查控制器状态
   - 空闲：分配控制器，进入下一步
   - 忙：进程 PCB 插入该控制器的等待队列，阻塞等待
3. **分配通道**：在 COCT 中找到关联的 CHCT，检查通道状态
   - 空闲：分配通道，进入下一步
   - 忙：进程 PCB 插入该通道的等待队列，阻塞等待
4. **启动 I/O**：设备、控制器、通道三者均已分配，启动 I/O 操作

>[!warning] "自动连接" vs 显式连接
>某些系统允许设备与控制器**自动连接**，省去单独分配控制器的步骤，一定程度上简化了分配过程

## 逻辑设备名到物理设备名的映射

映射机制是实现设备独立性的关键，流程如下：

1. 进程以**逻辑设备名**发出 I/O 请求
2. 系统在 [[#逻辑设备表(LUT)|LUT]] 中查找该逻辑设备名对应的表项
3. 若找到空闲的同类物理设备，则返回**物理设备名**及**驱动程序入口地址**，并在 LUT 中建立相应表目；此后，该进程的所有 I/O 请求均通过该表项找到对应的物理设备及其驱动程序
4. 若所有同类设备均忙，进程进入等待队列（安全分配）或返回错误（不安全分配）

映射过程对用户透明，用户始终使用逻辑设备名编程，由操作系统在运行时动态绑定到具体物理设备，从而实现[[#实现设备独立性的好处|设备独立性]]
系统中可采用两种方式设计逻辑设备表：
- 整个系统中仅设置一张 LUT：所有进程的设备分配情况都记录在同一张 LUT 中，要求所有用户不能使用相同的逻辑设备名，适用于单用户系统
- 为每个用户设置一张LUT：系统仍维护全局的系统设备表（SDT）；不同用户可以使用相同的逻辑设备名，互不干扰，适用于多用户系统

# SPOOLing 技术（假脱机技术）

SPOOLing (Simultaneous Peripheral Operations OnLine，**假脱机技术**)是一种将**独占设备改造为共享设备**的技术，通过在磁盘上设置专门的 I/O 区域，将低速独占设备的输入/输出过程转化为对磁盘的高速读写过程
假脱机技术是一种**以空间换时间**的技术，使整体 I/O 效率得以提升，CPU 与 I/O 设备实现了并行工作，⭐**提高独占设备的利用率**
SPOOLing 技术的本质时以磁盘作为缓冲、用软件模拟脱机 I/O 的机制；”脱机“承袭自早期脱机 I/O 的思想，“假”则体现在以**多道程序和磁盘空间代替外围机**，进而**将一台物理独占设备虚拟为多台逻辑设备**，实现**设备共享**

>[!important] 脱机与假脱机
>- **脱机 I/O**：使用独立的外围计算机来控制 I/O 操作，主计算机仅与高速磁盘交互——需额外的硬件成本
>- **假脱机(SPOOLing)**：用**软件**模拟脱机 I/O 的功能，通过磁盘和多道程序技术实现，无需专用外围计算机，是脱机 I/O 的软件化替代方案

## SPOOLing 系统的组成
### 输入井和输出井
输入/输出井在**磁盘上**开辟的两个**专用**存储区域
- **输入井**：磁盘上用于暂存输入数据的区域，模拟脱机输入时的输入设备
- **输出井**：磁盘上用于暂存输出数据的区域，模拟脱机输出时的输出设备
每个进程的输入（或输出）数据以文件形式保存（称为**井文件**），所有文件按顺序链接成输入队列和输出队列

### 输入缓冲区和输出缓冲区

输入/输出缓冲区位于**内存**中，是设备与磁盘输入/输出井之间的数据中转站：
- **输入缓冲区**：暂存从输入设备读入的数据，积累到一定量后批量写入输入井，减少对磁盘的频繁 I/O
- **输出缓冲区**：暂存从输出井读出的数据，再逐次送往输出设备，平滑磁盘与低速设备之间的速度差异

>[!note] 井（磁盘）vs 缓冲区（内存）
>输入井/输出井是磁盘上的**持久暂存区**，存放完整的数据文件；输入/输出缓冲区是内存中的**临时中转区**，仅暂存当前传输的一小批数据。

### 输入进程和输出进程
- **输入进程（预输入程序/输入守护进程）**：模拟脱机输入时的外围控制机，将用户输入数据从输入设备读入输入井
- **输出进程（缓输出程序/输出守护进程）**：模拟脱机输出时的外围控制机，将输出井中的数据逐一送往输出设备
### 井管理程序
控制作业与磁盘输入/输出井之间的数据交换，当进程请求输入时从输入井读入，请求输出时写入输出井

## 共享打印机原理
以 SPOOLing 打印系统为例，完整的打印流程：
1. 用户进程请求打印输出时，SPOOLing 系统并不直接分配打印机，而是：
   - 在**输出井**中为其申请一块空闲盘块，将要打印的数据写入输出井
   - 申请一张用户请求打印表，记录用户的打印要求，并将其挂到假脱机打印队列上
1. 用户进程的打印请求被"快速完成"（实际仅是写入了磁盘），进程可继续执行
2. SPOOLing **输出守护进程**在打印机空闲时：
   - 从请求打印队列中取出队首表项
   - 将对应的输出井数据逐一送往打印机进行实际打印
   - 打印完毕后释放输出井空间

>[!important] SPOOLing 技术的特点
>- **提高了 I/O 速度**：将低速 I/O 操作转化为对磁盘的高速操作，缓解了 CPU 与低速设备的速度矛盾
>- **将独占设备改造为共享设备**：宏观上多个进程可"同时"使用同一台独占设备
>- **实现了虚拟设备功能**：每个进程认为自己独占了一台设备，实际是磁盘上的存储空间

# 设备驱动程序接口

设备驱动程序是 I/O 系统上层与设备控制器之间的通信桥梁，其主要任务是接受上层软件发来的**抽象 I/O 请求**，将这些请求转换为设备可识别的**具体命令**，并发送给设备控制器以启动设备执行相应的操作；同时，也将设备控制器返回的状态和中断信息反馈给上层

## 设备驱动程序的功能
- **接收命令与参数**：接收上层设备独立性软件发来的抽象 I/O 请求（如`read`、`write`），解析其中的命令和参数
- **将抽象请求转化为具体命令**：将通用的 I/O 请求转化为设备控制器能识别的**设备指令序列**（控制器命令）
- **检验用户 I/O 请求的合法性**，查询设备当前状态，**传递必要参数并设置其工作方式**
- **发出 I/O 命令**：若设备空闲，则立即启动设备执行操作；若设备正忙，则将请求进程的 PCB 挂到该设备的等待队列上以待调度
- **及时响应设备控制器发出的中断请求**，并根据中断类型调用相应的中断处理程序进行处理

## 设备驱动程序的特点
- 在抽象 I/O 请求与具体设备操作之间实现双向转换，并将设备状态及操作完成情况及时反馈给请求进程
- 其实现与设备所采用的 I/O 控制方式紧密相关，如中断驱动方式、DMA 方式
- 由于直接操作硬件，不同类型的设备必须配备相应的专用驱动程序
- 部分底层驱动功能可能由设备固件或操作系统 ROM 提供
- 驱动程序应设计为可重入的，以支持多个进程对同一设备的并发调用
为使各类设备驱动程序具有统一的接口，操作系统要求：
- 所有驱动程序与内核之间遵循相同或相近的接口规范，便于新增和维护
- 通过设备管理模块将逻辑设备名映射为物理设备名，并定位到相应的驱动程序入口
- 对设备的访问实施权限管理，防止未授权用户非法使用设备

>[!important] 驱动程序的层次定位
>回顾[[1.输入输出管理概述#I/O软件层次结构| I/O 软件层次结构]]：设备驱动程序位于设备独立性软件与中断处理程序之间，向下控制硬件，向上提供统一接口

# I/O 操作举例

1. **用户程序**在用户态调用 `scanf("%c", &d)` → **C 库**解析格式串、封装为 `read(STDIN_FILENO, buf, 1)` 系统调用 → CPU 陷入内核态，用户进程进入**阻塞态**
2. **设备独立性软件**在内核态通过进程打开文件表查到逻辑设备名，在 [[#逻辑设备表(LUT)|LUT]] 中查出对应的**物理键盘设备**及**键盘驱动入口地址** → 调用键盘驱动 `read()` 入口
3. **键盘驱动程序**在内核态向控制器寄存器写入读命令、构造 DCB 记录请求上下文 → 将进程 PCB 挂入键盘等待队列 → 返回，CPU 转去调度其他进程
4. **键盘硬件**：用户按下 `'A'` → 键盘电路生成**扫描码** `0x1E`，经串行线发至**键盘控制器** → 控制器校验后置"数据就绪"位，通过中断控制器向 CPU 发出 **IRQ 中断**
5. **CPU 硬件**在当前指令末尾检测到 IRQ 且 IF=1 → 自动压栈 PSW/PC，根据中断向量号查 [[1.输入输出管理概述#中断处理程序|IDT]] 获取**键盘 ISR 入口地址** → 跳转至 ISR
6. **键盘 ISR**在内核态从控制器数据寄存器读取扫描码 `0x1E` → （由 I/O 库函数）转换为 **ASCII 码** `'A'` → 写入内核中的**键盘输入缓冲区** → 向中断控制器发送 **EOI** → 从键盘等待队列唤醒阻塞进程
7. **被唤醒进程**在内核态执行 `read` 系统调用剩余逻辑，从缓冲区取出 `'A'` → 返回用户态 → `scanf` 将 `'A'` 写入 `&d`，返回 `1`

**全路径**：按键 → 扫描码 → 控制器 → IRQ → IDT → ISR（读码 → 转 ASCII → 入缓冲区 → EOI → 唤醒）→ `read` 返回 → `scanf` 写入 `&d`。硬件完成信号采集，内核四层软件完成映射、缓冲、调度和转换，一个字符跨越软硬件全栈存入用户变量。

> [!summary] 本章重点
>
> **一、设备独立性软件概述**
>
> 1. **层次定位**：内核 I/O 子系统最高层，位于设备驱动程序之上，向上提供统一抽象接口
> 2. **核心使命**：应用程序独立于具体物理设备，逻辑设备名与实际物理设备分离，运行时动态映射
> 3. **六大公共功能**：
>    - 统一驱动程序接口：上层调用不变，下层驱动可替换
>    - 缓冲管理：单缓冲 → 双缓冲 → 循环缓冲 → 缓冲池
>    - 差错控制：暂时性错误重试 → 持久性错误绕过/坏块表 → 关键错误上报
>    - 独占设备分配与回收：设备 → 控制器 → 通道三级检查，静态分配（无死锁、低效）与动态分配（高效、可能死锁）
>    - 统一逻辑数据块：屏蔽物理扇区差异（512B / 4KB），上层看到统一逻辑块
>    - 逻辑设备映射：LUT 实现逻辑名到物理名的转换
>
> **二、磁盘高速缓存（Disk Cache）**
>
> 1. **定义**：利用内存暂存磁盘盘块数据，逻辑上是磁盘副本，物理上驻留内存
> 2. **两种组织形式**：内存中开辟固定大小专用缓存区 / 系统空闲内存作为动态缓冲池
> 3. **与硬件 Cache 不同**：不是 CPU 与主存之间的硬件缓存，而是 OS 用软件管理的磁盘数据缓存
>
> **三、缓冲区（Buffer）**
>
> 1. **引入缓冲的四个目的**：缓和 CPU 与 I/O 速度不匹配 → 减少中断频率 → 提高并行性 → 解决数据粒度不匹配
> 2. **单缓冲**：一块，处理时间 $\max(C, T) + M$，部分并行，设备和 CPU 互斥访问
> 3. **双缓冲**：两块交替，处理时间 $\max(C+M, T)$，$C+M \le T$ 时设备连续工作、接近完全并行
> 4. **循环缓冲**：多块环形链表，in/out 指针；in 追 out（满缓冲、计算受限），out 追 in（空缓冲、I/O 受限）
> 5. **缓冲池**：公用软件机制，三类队列（空缓冲/输入/输出）、四种角色（hin/sin/hout/sout）、四种工作方式（收容输入/提取输入/收容输出/提取输出）
>
> **四、高速缓存 vs 缓冲区**
>
> 1. **核心区别**：高速缓存用于"**存**"（副本复用，有命中概念、需 LRU 淘汰），缓冲区用于"**传**"（速度协调，无命中概念、流经即消费）
> 2. **相似之处**：均利用内存提升 I/O 性能，均位于高速和低速设备之间缓解速度差异，均可使用动态内存池管理
>
> **五、设备分配与回收**
>
> 1. **数据结构链**：SDT（全局唯一）→ DCT（每设备一个）→ COCT（每控制器一个）→ CHCT（每通道一个），形成完整映射链
> 2. **考虑因素三类**：
>    - 设备固有属性：独占 / 共享 / 虚拟（SPOOLing）
>    - 分配算法：FCFS（公平）/ 优先级（紧迫优先、可能饥饿）
>    - 安全性：安全分配（I/O 后阻塞、无死锁、效率低）vs 不安全分配（继续运行、有死锁风险、效率高）
> 3. **分配步骤**：逐级检查空闲/忙 → 分配设备 → 分配控制器 → 分配通道 → 三级全空闲启动 I/O；任一级忙则入对应等待队列阻塞
> 4. **LUT 映射**：逻辑设备名 → LUT 查找 → 物理设备名 + 驱动入口；单系统单表（单用户）或单系统每用户一表（多用户）
>
> **六、SPOOLing 技术**
>
> 1. **本质**：以空间（磁盘）换时间，用软件模拟脱机 I/O，将独占设备改造为共享设备
> 2. **四组件**：输入井 + 输出井（磁盘持久暂存）→ 输入/输出缓冲区（内存临时中转）→ 输入/输出进程（守护进程）→ 井管理程序
> 3. **共享打印机流程**：进程写入输出井（快速完成）→ 输出守护进程依次送打印机实际打印
> 4. **三大特点**：提高 I/O 速度（低速→高速磁盘）→ 独占变共享 → 虚拟设备透明化
>
> **七、设备驱动程序**
>
> 1. **层次定位**：内核态，位于设备独立性软件与中断处理程序之间，是直接操控硬件的底层软件
> 2. **五大功能**：接收命令与参数 → 抽象请求转为设备指令序列 → 检验合法性、查询状态 → 发出 I/O 命令（空闲启动、忙则挂等待队列）→ 响应中断
> 3. **五大特点**：双向转换（抽象↔具体）、与 I/O 控制方式紧密相关、每类设备专用驱动、部分功能可由固件/ROM 提供、应设计为可重入
> 4. **接口统一要求**：驱动与内核遵循相同接口规范，通过设备管理模块映射逻辑名到驱动入口，实施权限管理
